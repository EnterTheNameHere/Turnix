Turnix AI workflow bootstrap
============================

Purpose
-------
This file is the prototype routing and workflow guide for language model
assistants working with the Turnix repository after the document-family split
experiment begins.

This file is NOT authoritative for Turnix runtime semantics.

This file does NOT override:
- design/core/CORE-04_CANONICAL_INDEX_TURNIX_SPECIFICATION.txt
- registered CORE documents
- registered DOCTRINE documents
- registered DESIGN AREA documents
- registered OPTIONAL DESIGN AREA documents
- registered IMPLEMENTATION SPECIFICATION documents

Authority and priority
----------------------
1. CORE-04 is the prototype canonical index for DOC_ID resolution and
   canonical paths.
2. CORE documents define document-system and highest-order design-system
   rules.
3. DOCTRINE documents define cross-cutting doctrine and cooperation modes.
4. DESIGN AREA, OPTIONAL DESIGN AREA, and IMPLEMENTATION SPECIFICATION
   documents define project semantics in their own scope.
5. AI WORKFLOW files define reusable operational guidance only unless a CORE
   document explicitly assigns additional authority.
6. MIGRATION NOTE files explain migration and rehydration; they do not replace
   CORE-04.
7. If this file conflicts with indexed documents, indexed documents win.

Internal design authority
-------------------------
Internal Turnix design authority lives under:
- design/

The docs/ directory is no longer the preferred home for internal design
authority in this prototype.

The docs/ directory may contain user-facing, creator-facing, developer-facing,
or explanatory documentation derived from internal design authority.

Internal design documents are development-facing source-of-truth documents,
not ordinary user manuals or tutorials.

Default loading rules
---------------------
For any Turnix design, specification, routing, or code task in this prototype
branch, first read:
- design/core/CORE-04_CANONICAL_INDEX_TURNIX_SPECIFICATION.txt

For work on the document-family split prototype, also read:
- design/migration/DOCUMENT_FAMILY_SPLIT_AND_ROUTING_PROTOTYPE.txt

For tracing design work, read:
- TRACE-00
- the specific TRACE-* authority slices relevant to the task

Do not load the whole tracing family by default when the task only concerns one
slice.

Family-aware routing
--------------------
A design area may be represented by a directory containing multiple registered
documents.

The family index routes the family.

Authority slices define connector-safe parts of the family semantics.

For tracing:
- TRACE-00 is the family index
- TRACE-01 defines substrate and authority
- TRACE-02 defines event model and DAG
- TRACE-03 defines spans and roots
- TRACE-04 defines reasons and attributes
- TRACE-05 defines retention, loss, and access
- TRACE-06 defines diagnostics, errors, and development mode

Legacy identity handling
------------------------
Legacy IDs remain usable only when CORE-04 lists their mapping.

Known prototype mappings:
- CMR-04 maps to CORE-04
- AI_WORKFLOW_BOOTSTRAP maps to AI-WORKFLOW-BOOTSTRAP
- DA-03 maps to TRACE-00

DA-03 section mappings are defined by CORE-04 and by the migration note.

Do not preserve old DA-* numbering merely because it exists.

Do preserve old meaning when it is still valid and intentionally migrated.

Reference-following rules
-------------------------
Do NOT automatically load every Upstream Documents or Downstream Documents
entry. Those blocks improve retrieval and review hygiene, but they are not
automatic read-everything instructions.

Load additional documents when one of the following is true:
- the user explicitly asks for it
- the current task clearly depends on that document
- the loaded document contains a concrete reference that should be checked
- the loaded document contains an explicit dependency marker

Concrete references include examples such as:
- TRACE-05 01.12
- DA-08 13.5
- CMR-03 16.19-16.24

When a concrete reference points to a legacy DOC_ID, resolve the legacy DOC_ID
through CORE-04 before loading the target document.

Explicit dependency markers
---------------------------
Use machine-friendly markers when possible.

Preferred forms:
- [ref: TRACE-05 01.12]
- [defined-in: CORE-04 05]
- [constrained-by: TRACE-01 03]
- [satisfied-by: DA-08 11.9]
- [relies-on: TRACE-02]
- [requires-read: CORE-04]
- [see-also: TRACE-06]

Interpretation:
- [ref: ...] is navigation only unless context makes dependency explicit
- [defined-in: ...] means the referenced location is the definition site
- [constrained-by: ...] means the current text is subordinate to that rule
- [satisfied-by: ...] means the referenced location fulfills the requirement
- [relies-on: ...] means load the referenced document for the current task
- [requires-read: ...] means read the referenced document before continuing
- [see-also: ...] is advisory and load-on-demand

Missing references
------------------
If a referenced DOC_ID or authority source cannot be resolved through CORE-04:
- do not guess
- do not substitute a similar document
- report that the reference is missing
- ask for clarification when the task cannot continue safely

Repository connector retrieval failures
---------------------------------------
The GitHub connector fetch_file operation can return file content that is
truncated.

The GitHub connector fetch_file operation can also return a truncated tool
response whose JSON object is incomplete or malformed.

When a fetched file content is truncated, the assistant treats the file as not
loaded.

When a fetch_file tool response is malformed, incomplete, or missing its
closing JSON object boundary, the assistant treats the file as not loaded.

The assistant does not treat truncated content, malformed JSON, incomplete
JSON, partial snippets, search results, web copies, local memory, or inferred
text as authoritative file text.

The assistant does not use workaround retrieval for the affected file.
Forbidden workaround retrieval includes:
- repository search results
- web retrieval
- file excerpts
- snippets
- cached memory
- inferred reconstruction
- any sha obtained outside the same successful fetch_file response

The assistant may use only a complete fetch_file response for read-only use
of that file during the task.

The assistant may use only the sha returned by the same complete fetch_file
response when updating that file through the GitHub connector.

The assistant does not replace the connector update sha with a public GitHub
file sha or with a sha retrieved by another route.

If complete file content or the connector update sha is unavailable, the
assistant stops the repository write path for that file.

When the assistant stops for this failure, the assistant informs the user that
the connector did not provide complete authoritative file state.

When the assistant has already prepared changes that cannot be committed
because the connector update sha is unavailable, the assistant prints the
updated file content or patch for manual user application.

Working behavior
----------------
Maintain a distinction between:
- known documents
- loaded documents
- cited documents
- assumed but not checked context

Do not claim that a document was checked unless it was actually loaded for the
current task.

Do not treat vague relevance as proof that a document is currently in working
context.

Do not create workaround routing when CORE-04 or this bootstrap should be
updated instead.

Suggested cooperation mode defaults
-----------------------------------
When working with Turnix, prefer explicit mode naming from DD-05 while DD-05
remains the active cooperation-mode source.

- Design Mode
  For architecture, invariants, semantics, and boundaries

- Refactor Planning Mode
  For planning structural change before code output

- Patch Map Mode
  For precise change locations and touched regions

- Materialization Mode
  For concrete code or artifact output after scope is accepted

- Review Mode
  For evaluating concrete output against contracts and scope

- Audit Mode
  For deeper verification of boundaries, tracing, continuity, and integrity

The active mode should be stated when useful, especially if the task could
drift between design, planning, and materialization.

Practical shorthand accepted from user
--------------------------------------
The assistant may resolve shorthand references such as:
- main, CORE-04
- branch, CORE-04
- design/area/tracing, TRACE-02
- old DA-03 tracing document

The assistant may resolve a DOC_ID through CORE-04 when the user names only
the DOC_ID.
