===============================================================================
CORE-04 - CANONICAL INDEX - TURNIX SPECIFICATION
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Single source of truth for Turnix internal design authority document identity,
canonical file locations, authority tiers, migration status, and family-aware
routing during the document-family split prototype.

Purpose:
--------
This document is the prototype successor to CMR-04.

It defines:
- canonical document identity for the prototype document system
- canonical repository locations for migrated documents
- authority tiers for conflict resolution
- design area family registration
- legacy identity mapping during migration
- the distinction between migrated authority and legacy compatibility

It does NOT define:
- domain semantics of indexed documents
- implementation behavior
- user-facing documentation
- creator-facing documentation
- explanatory documentation generated from internal design authority

All authoritative design content lives in the referenced documents.

This Canonical Index is authoritative only for document identity, location,
authority tier, authority status, and migration routing. It is not
authoritative for domain semantics.


===============================================================================
01. PROTOTYPE STATUS
===============================================================================

01.1. CORE-04 is a prototype successor to CMR-04.

01.2. CMR-04 is the former canonical index identity.

01.3. CORE-04 tests the future naming and routing model.

01.4. The prototype uses meaningful document-family identifiers instead of
      preserving old historical numbering as architecture.

01.5. Old DOC_ID values remain migration aliases when explicitly listed.

01.6. Old DOC_ID values do not force future document structure.

01.7. During the prototype, migrated documents use the new canonical paths
      listed in this document.

01.8. Unmigrated documents remain at their existing paths until they are
      migrated.

01.9. Legacy paths and legacy identifiers are compatibility handles, not the
      preferred future model.


===============================================================================
02. INTERNAL DESIGN AUTHORITY LOCATION
===============================================================================

02.1. Internal Turnix design authority documents live under:

      design/

02.2. Ordinary explanatory, user-facing, creator-facing, or tutorial
      documentation may live under:

      docs/

02.3. Internal design authority documents are development-facing source of
      truth documents.

02.4. Internal design authority documents are not ordinary user
      documentation.

02.5. User-facing and creator-facing documentation may be derived from
      internal design authority documents.

02.6. Derived documentation does not replace internal design authority.


===============================================================================
03. CANONICAL DIRECTORY MODEL
===============================================================================

03.1. The prototype canonical directory model is:

      design/core/
      design/doctrine/
      design/area/
      design/optional/
      design/implementation/
      design/ai/
      design/migration/
      design/legacy/

03.2. design/core/ contains document-system authority and highest-order
      design-system rules.

03.3. design/doctrine/ contains cross-cutting design doctrine documents.

03.4. design/area/ contains core design area families and authority slices.

03.5. design/optional/ contains optional design area documents and families.

03.6. design/implementation/ contains concrete protocol, schema, and
      implementation specification documents.

03.7. design/ai/ contains assistant workflow, routing, and helper documents.

03.8. design/migration/ contains migration and rehydration notes.

03.9. design/legacy/ contains legacy maps, redirect notes, and obsolete
      document compatibility records.


===============================================================================
04. AUTHORITY TIERS
===============================================================================

04.1. The prototype authority tiers are:
      1) CORE
      2) DOCTRINE
      3) DESIGN AREA
      4) OPTIONAL DESIGN AREA
      5) IMPLEMENTATION SPECIFICATION
      6) AI WORKFLOW
      7) MIGRATION NOTE
      8) LEGACY RECORD

04.2. CORE documents define document-system authority and highest-order
      design-system rules.

04.3. DOCTRINE documents define cross-cutting design, engineering,
      interface, and cooperation principles.

04.4. DESIGN AREA documents define Turnix correctness and core semantics.

04.5. OPTIONAL DESIGN AREA documents extend or contextualize DESIGN AREA
      semantics and do not contradict DESIGN AREA documents.

04.6. IMPLEMENTATION SPECIFICATION documents define concrete protocols,
      data formats, and execution flows constrained by higher tiers.

04.7. AI WORKFLOW documents define assistant loading and cooperation
      behavior. They do not define Turnix runtime semantics unless another
      authoritative document explicitly assigns that authority.

04.8. MIGRATION NOTE documents explain migration, rehydration, and
      continuation. They do not replace canonical index authority.

04.9. LEGACY RECORD documents preserve old identity and path interpretation.
      They do not define future architecture.


===============================================================================
05. DOCUMENT FAMILIES
===============================================================================

05.1. A design area may be represented by a directory containing multiple
      registered documents.

05.2. A family index document routes the design area family.

05.3. Authority slices define connector-safe parts of the family semantics.

05.4. A family index preserves conceptual unity.

05.5. A family index does not replace authority slices unless it explicitly
      defines the relevant rule itself.

05.6. Each registered family document has its own DOC_ID and DOC_FILE.

05.7. Each registered family document is a complete specification document.

05.8. Connector-safe authority slices are preferred over oversized
      monolithic authority documents.


===============================================================================
06. CONNECTOR-SAFE WORKING UNIT RULE
===============================================================================

06.1. A specification document is a safe working unit, not merely a
      conceptual unit.

06.2. A specification document used for connector-based assistant work is
      small enough to be retrieved as complete file content with its update
      SHA in the same connector response.

06.3. If a concept grows beyond the connector-safe working size, the concept
      is represented by a family index and multiple authority slices.

06.4. Workaround retrieval does not replace complete connector file state.

06.5. Workaround routing does not replace this Canonical Index.
