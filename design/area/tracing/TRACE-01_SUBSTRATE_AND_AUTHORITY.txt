===============================================================================
DESIGN AREA TRACE-01 - TRACE SUBSTRATE AND AUTHORITY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines trace evidence meaning, tracing authority, trace substrate ownership,
and the boundary between trace mechanics and domain-owned trace obligations.

Purpose:
--------
This document defines the highest-order tracing substrate semantics formerly
contained in the monolithic DA-03 tracing document, plus the anti-suppression
and migration boundaries needed to preserve one authority home per concept.

It defines:
- what trace evidence means in Turnix
- what trace evidence is authoritative for
- what trace evidence is not authoritative for
- trace substrate ownership
- the boundary between trace mechanics and domain-owned trace obligations
- no-heuristic trace suppression rules
- migration ownership for domain-specific trace obligations

It does NOT define:
- concrete trace event serialization
- trace DAG structure details
- span mechanics
- retention storage formats
- object-specific trace obligations
- domain-specific lifecycle transitions
- detailed section-by-section DA-03 compatibility routing

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE documents define the general rule language and interpretation
      model used by this document.

01.4. This document defines tracing substrate authority within that shared
      interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. Tracing answers:

          "What happened, what caused what, and why?"

02.2. Tracing is structured evidence with explicit reasons and causal
      links.

02.3. Tracing is not logging.

02.4. Plain logs are secondary and optional.

02.5. Logs do not replace trace evidence.

02.6. Tracing provides:
      - causal explanation
      - decision evidence
      - observed temporal sequence
      - diagnostic context

02.7. Tracing is authoritative evidence for:
      - causality represented inside retained trace evidence
      - decisions represented inside retained trace evidence
      - audit reconstruction within retained trace evidence
      - debugger and tooling correlation within retained trace evidence
      - policy-enforcement provenance within retained trace evidence

02.8. Tracing is not authoritative for:
      - committed state
      - committed-state restoration
      - replay semantics
      - rollback eligibility by itself
      - object lifecycle truth outside retained trace evidence

02.9. "If it is not traced, it did not happen" means:
      - trace evidence is the proof that a trace-using subsystem recorded
        the action
      - if an action cannot be found in retained trace evidence, the action
        is not provable from trace
      - trace retention is policy-driven
      - trace loss is acceptable only when loss is explicitly represented
        as loss evidence

02.10. This mantra does not make this document the owner of every domain
       object's tracing obligations.

02.11. Trace evidence can represent successful, failed, refused, cancelled,
       superseded, and other domain-defined outcomes.

02.12. Trace evidence supports explanation of an outcome only when the
       owning domain records evidence for that outcome.

02.13. A trace consumer does not infer an unrecorded outcome merely because
       failure evidence is absent.

02.14. Reason fields, causal links, and attributes allow Turnix to explain
       recorded events without treating unrecorded events as proven.

02.15. The design area that owns an object, lifecycle, workflow, or
       subsystem defines whether and how that subject uses the tracing
       substrate.

02.16. If a domain chooses not to emit trace evidence for a meaningful
       action, that absence is a weakness of that domain's design.

02.17. That absence does not change the tracing substrate defined by this
       document.


===============================================================================
03. TRACE SUBSTRATE OWNERSHIP
===============================================================================

03.1. TRACE owns the tracing substrate.

03.2. TRACE defines how trace evidence is represented, related, retained,
      accessed, and interpreted.

03.3. TRACE does not own domain lifecycles.

03.4. TRACE does not prescribe object-specific trace coverage.

03.5. TRACE does not define which transitions exist for another design
      area's object.

03.6. TRACE does not define which domain events are important enough to
      record.

03.7. The owner of each lifecycle, subsystem, workflow, or object defines
      its own tracing obligations when trace evidence is required for that
      subject.

03.8. Subject-specific trace obligations remain subordinate to the trace
      mechanics defined by the TRACE family.

03.9. Subject-specific documents do not redefine trace event structure,
      trace DAG semantics, span mechanics, retention-loss meaning, or trace
      access semantics.

03.10. Moving a subject-specific trace obligation out of the former DA-03
       document does not weaken the obligation when the subject-owning
       document preserves it.

03.11. The migration of subject-specific trace obligations preserves one
       authority home per concept.


===============================================================================
04. NO HEURISTIC TRACE SUPPRESSION
===============================================================================

04.1. Turnix does not infer reasons without recording them.

04.2. Turnix does not fabricate causal links from temporal ordering.

04.3. Turnix does not hide known trace loss.

04.4. Turnix does not present incomplete trace evidence as complete.

04.5. A subject-owning document may decide that a subject does not emit
      trace evidence for a given action.

04.6. If a subject-owning document requires trace evidence for an action,
      the trace substrate does not silently suppress that evidence.


===============================================================================
05. MIGRATED DOMAIN TRACE OBLIGATIONS
===============================================================================

05.1. Earlier revisions of the former DA-03 document contained
      domain-specific trace obligations.

05.2. Those obligations are moving to the documents that own the affected
      domains.

05.3. Discovery-specific trace obligations belong in the discovery-owning
      design area.

05.4. Resolution-specific trace obligations belong in the resolution-owning
      design area.

05.5. Activation-specific trace obligations belong in the activation-owning
      design area.

05.6. Capability-specific trace obligations belong in the capability-owning
      design area.

05.7. State and transaction trace obligations belong in the state-owning
      design area.

05.8. Configuration trace obligations belong in the configuration-owning
      design area.

05.9. Window trace obligations belong in the Window-owning design area.

05.10. Debugger trace obligations belong in the debugger-owning design area.

05.11. RPC trace obligations belong in the RPC implementation specification
       or a higher RPC semantic document when one exists.

05.12. This migration changes ownership location.

05.13. This migration does not intentionally weaken valid domain trace
       obligations that are preserved in their owning documents.


===============================================================================
06. LOCKED SUMMARY
===============================================================================

06.1. LOCKED:
      - TRACE owns the tracing substrate
      - TRACE does not own object-specific lifecycle truth
      - tracing is structured evidence rather than logging
      - trace evidence explains what happened, what caused what, and why
      - retained trace evidence is authoritative for retained causal proof
      - trace evidence is not authoritative committed state
      - trace evidence does not define rollback eligibility by itself
      - Turnix does not infer reasons without recording them
      - Turnix does not fabricate causal links from temporal ordering
      - Turnix does not hide known trace loss
      - Turnix does not present incomplete trace evidence as complete
      - domain-specific trace obligations live in the documents that own
        those domains
