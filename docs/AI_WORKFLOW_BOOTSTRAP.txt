Turnix AI workflow bootstrap
============================

Purpose
-------
This file is a short routing and workflow guide for language model assistants
working with the Turnix repository.

This file is NOT authoritative for Turnix semantics.

This file does NOT override:
- docs/CMR-04_CANONICAL_INDEX_TURNIX_SPECIFICATION.txt
- docs/CMR-01_DOCUMENT_FORMAT_SPECIFICATION_ASCII_79.txt
- docs/CMR-02_DOCUMENT_TITLE_AND_PREAMBLE_FORMAT.txt
- docs/CMR-03_DOCUMENT_CREATION_AND_CHANGE_RULES.txt
- docs/CMR-05_CROSS_THREAD_REFERENCE_AND_RETRIEVAL_RULES.txt
- any indexed DD, DA, ODA, or IS document

Authority and priority
----------------------
1. CMR-04 is the canonical index for DOC_ID resolution and canonical paths.
2. CMR-01 to CMR-05 define document-system rules.
3. DD documents define doctrine and cooperation modes.
4. DA, ODA, and IS documents define project semantics in their own scope.
5. AI helper files define reusable operational guidance only.
6. If this file conflicts with indexed documents, indexed documents win.

Default loading rules
---------------------
For any Turnix documentation or code task, first read:
- docs/CMR-04_CANONICAL_INDEX_TURNIX_SPECIFICATION.txt

For document creation, editing, rewriting, review, or reference hygiene,
read:
- CMR-01
- CMR-02
- CMR-03
- CMR-04
- CMR-05

For code creation, editing, review, refactor planning, or implementation
discussion, also read the relevant DD documents.

Recommended default DD set:
- DD-01
- DD-02
- DD-03
- DD-04
- DD-05

Task routing
------------
If the user asks to work on specification documents:
- load CMR-01 to CMR-05 first
- load the target document(s)
- load cited documents only when the task requires them

If the user asks to work on source code:
- load CMR-04
- load the relevant DD documents
- load the target code files
- load relevant DA, ODA, or IS documents when the code task depends on them

If the user asks to review design semantics:
- load CMR-04
- load the target DA, ODA, DD, or IS document(s)
- load precise cited references used by the target document when needed

Mode-sensitive helper loading
-----------------------------
If the task is an existing-document edit or integration pass:
- load existing_document_integration.txt
- load document_semantic_review.txt
- load turnix_semantic_anchors.txt when Turnix doctrine matters

If the task is a new-document draft:
- load new_document_drafting.txt
- load question_hardening_workflow.txt when semantics are still being
  narrowed
- load turnix_semantic_anchors.txt when Turnix doctrine matters

If the task is primarily DA-style document work:
- load design_area_document_mode.txt
- load document_semantic_review.txt
- load turnix_semantic_anchors.txt when Turnix doctrine matters

If the task is a multi-document semantic alignment or post-batch review:
- load cross_document_consistency_sweep.txt
- load document_semantic_review.txt
- load turnix_semantic_anchors.txt when Turnix doctrine matters

If the task is semantic document review or commit-message preparation for
Turnix docs:
- load document_semantic_review.txt
- load create_commit_message.txt when asked to print a git commit message

If the task is design discussion before rewriting:
- load question_hardening_workflow.txt
- do not begin rewriting until enough meaning is locked

Reference-following rules
-------------------------
Do NOT automatically load every Upstream Documents or Downstream Documents
entry. Those blocks improve retrieval and review hygiene, but they are not
automatic read-everything instructions.

Load additional documents when one of the following is true:
- the user explicitly asks for it
- the current task clearly depends on that document
- the loaded document contains a concrete reference that should be checked
- the loaded document contains an explicit dependency marker

Concrete references include examples such as:
- DA-08 13.5
- CMR-03 16.19-16.24

If a concrete reference is followed, load:
- the referenced file resolved through CMR-04
- the cited section or cited range

Explicit dependency markers
---------------------------
Use machine-friendly markers when possible.

Preferred forms:
- [ref: DA-08 13.5]
- [defined-in: CMR-03 16.1-16.24]
- [constrained-by: DA-04 07.3]
- [satisfied-by: DA-04 15.5]
- [relies-on: DA-08]
- [requires-read: CMR-03]
- [see-also: DA-16]

Interpretation:
- [ref: ...] is navigation only unless context makes dependency explicit
- [defined-in: ...] means the referenced location is the definition site
- [constrained-by: ...] means the current text is subordinate to that rule
- [satisfied-by: ...] means the referenced location fulfills the requirement
- [relies-on: ...] means load the referenced document for the current task
- [requires-read: ...] means read the referenced document before continuing
- [see-also: ...] is advisory and load-on-demand

Missing references
------------------
If a referenced DOC_ID or authority source cannot be resolved through CMR-04:
- do not guess
- do not substitute a similar document
- ask for clarification or report that the reference is missing

Working behavior
----------------
Maintain a distinction between:
- known documents
- loaded documents
- cited documents
- assumed but not checked context

Do not claim that a document was checked unless it was actually loaded for the
current task.

Do not treat vague relevance as proof that a document is currently in working
context.

Bootstrap metadata handling
---------------------------
This file defines loading workflow and cooperation guidance only.

Bootstrap metadata is for retrieval and version hygiene only.
It does not define Turnix semantics.

Assistants may use DOC_REV, DOC_GIT, DOC_ID, and DOC_FILE to detect whether
a newer revision of this bootstrap file should be reloaded.

If the user explicitly says "reload AI_WORKFLOW_BOOTSTRAP" or indicates that
this file changed, reload this file even if it was already read earlier in the
task.

Suggested cooperation mode defaults
-----------------------------------
When working with Turnix, prefer explicit mode naming from DD-05:

- Design Mode
  For architecture, invariants, semantics, and boundaries

- Refactor Planning Mode
  For planning structural change before code output

- Patch Map Mode
  For precise change locations and touched regions

- Materialization Mode
  For concrete code or artifact output after scope is accepted

- Review Mode
  For evaluating concrete output against contracts and scope

- Audit Mode
  For deeper verification of boundaries, tracing, continuity, and integrity

The active mode should be stated when useful, especially if the task could
drift between design, planning, and materialization.

Practical shorthand accepted from user
--------------------------------------
The assistant may resolve shorthand references such as:
- main, docs, CMR-04
- main, docs/design, DA-04
- main, docs/rpc, IS-01

The assistant may also resolve a DOC_ID through CMR-04 when the user names
only the DOC_ID.
