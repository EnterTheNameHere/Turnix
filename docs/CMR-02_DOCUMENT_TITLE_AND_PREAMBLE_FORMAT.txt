===============================================================================
CORE META-RULES CMR-02 - DOCUMENT TITLE AND PREAMBLE FORMAT
===============================================================================

Status:
-------
FROZEN

Description:
------------
Defines the required banner and preamble structure for authoritative
documents.

Purpose:
--------
This document defines the required structure and semantics of the document
title and preamble used by authoritative Turnix documents.

It defines:
- document banner requirements
- required Status block
- required Description block
- required Purpose and Scope framing
- required negative scope statements
- optional Upstream Documents block
- optional Downstream Documents block
- preamble boundary rules
- end-of-document marker rules
- Status term conventions (EVOLVING / STABLE / FROZEN / OBSOLETE)

It does NOT define:
- physical ASCII formatting primitives (see CMR-01)
- document change discipline (see CMR-03)
- authority tier semantics beyond naming and placement rules
- indexing rules and conflict resolution (see CMR-04)

Upstream Documents:
------------------
- CMR-01
- CMR-03
- CMR-04
- CMR-05


===============================================================================
01. NORMATIVE LANGUAGE
===============================================================================

01.1. Turnix specifications express normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning derives from sentence structure and document
      context.

01.3. RFC-style requirement keywords do not carry normative meaning in
      Turnix specification documents.

01.4. The tokens "SHOULD" and "SHOULD NOT" are forbidden in authoritative
      specification documents because they introduce ambiguous advisory
      semantics.

01.5. When such tokens appear in draft text, the drafting author removes
      or rewrites them before the document becomes authoritative.


===============================================================================
02. DOCUMENT BANNER (TITLE BLOCK)
===============================================================================

02.1. A specification document begins with a document banner.

02.1.1. Terminology:

02.1.1.1. The Canonical Index (CMR-04 02.1.1) defines the terms
          "Specification document" and "authoritative document".

02.1.2. These banner and preamble rules apply to all specification
        documents registered in the Canonical Index.

02.1.3. These rules also apply to documents whose
        DOC_AUTHORITATIVE_STATUS is DRAFT or OBSOLETE unless a CORE
        META-RULES document explicitly exempts them.

02.2. The document banner uses the following structure:

      Line 1: 79 "=" characters
      Line 2: DOCUMENT_TITLE_LINE
      Line 3: 79 "=" characters

02.3. The banner appears exactly once in the document.

02.4. The banner appears at the very top of the document.

02.5. The banner is not numbered.

02.6. The banner is not treated as a section.

02.7. The DOCUMENT_TITLE_LINE uses the following template:

      DOC_TIER DOC_ID - DOC_TITLE

02.7.1. The template contains exactly one space between DOC_TIER and DOC_ID.

02.7.2. The template contains exactly one space before and after "-".

02.8. DOC_TIER is the authority tier name registered in the Canonical
      Index (CMR-04).

02.9. DOC_ID is the canonical document identifier registered in the
      Canonical Index (CMR-04).

      Examples:
      - CMR-02
      - DA-01
      - ODA-A
      - IS-01

02.10. DOC_TITLE is the human-readable document title registered in the
       Canonical Index (CMR-04).

02.11. When a doc-meta block is present (CMR-01), the DOC_ID field matches
       the Canonical Index entry (CMR-04).

02.12. When a doc-meta block is present (CMR-01), the DOC_FILE field matches
       the Canonical Index entry (CMR-04).

02.13. In committed authoritative documents, the doc-meta block appears in
       the document.

02.14. In committed authoritative documents, the DOC_ID and DOC_FILE fields
       match the Canonical Index entry (CMR-04).


===============================================================================
03. STATUS AND DESCRIPTION BLOCKS
===============================================================================

03.1. A specification document contains a Status block immediately after
      the document banner.

03.2. The Status block uses the following structure:

      Status:
      -------
      <status term>

03.2.1. Preamble blocks (Status, Description, Purpose and Scope, Upstream
        Documents, Downstream Documents) are label blocks as defined by
        CMR-01 05.9 through 05.11.

03.3. The status term is exactly one of the following:

      - EVOLVING
      - STABLE
      - FROZEN
      - OBSOLETE

03.4. Status term meanings:

      EVOLVING:
      The document is under active refinement and is not final.
      Draft copies may exist in working threads.

      STABLE:
      The document is mature.
      Changes are expected to be rare and explicit.

      FROZEN:
      The document is not expected to change.
      Change is exceptional and has broad impact.

      OBSOLETE:
      The document is not in force.
      It exists for history or reference.

03.5. A specification document contains a Description block immediately
      after the Status block.

03.6. The Description block uses the following structure:

      Description:
      ------------
      <free text>

03.7. The Description block summarizes stability expectations.

03.8. The Description block summarizes document scope.

03.9. The Description block summarizes change impact in plain language.

03.10. The Description block remains non-normative.

03.10.1. The Status block and the Description block do not contain
         normative rules.

03.11. The Status value in the document matches the Canonical Index
       mapping rules defined in CMR-04.

03.12. The Canonical Index DOC_AUTHORITATIVE_STATUS field determines
       document authority.

03.13. When a mismatch exists between Status and the Canonical Index,
       the mismatch is treated as an error to fix.

03.14. Authority decisions follow the Canonical Index when a mismatch
       occurs.

03.15. A drafting author may produce Status and Description text.

03.16. Status and Description text becomes authoritative only after
       explicit acceptance by the responsible human authority
       (see CMR-03).


===============================================================================
04. PURPOSE AND SCOPE BLOCK
===============================================================================

04.1. A specification document contains a Purpose and Scope block.

04.2. The block may use one of the following titles:

      Purpose:
      Purpose and Scope:
      Core Problem Statement:

04.3. The Purpose and Scope block explains why the document exists.

04.4. The Purpose and Scope block defines the responsibility of the
      document.

04.5. The Purpose and Scope block explicitly states what the document
      does not define.

04.6. Negative scope statements remain explicit and unambiguous.


===============================================================================
05. PREAMBLE BOUNDARY
===============================================================================

05.1. The specification document preamble contains the following
      elements:

      - document banner
      - Status block
      - Description block
      - Purpose and Scope block
      - optional Upstream Documents block
      - optional Downstream Documents block

05.2. Numbered sections do not appear inside the preamble.

05.3. The first numbered section appears after the preamble.

05.4. The preamble does not define detailed behavior, mechanisms, or
      implementation rules.


===============================================================================
06. LANGUAGE AND INTERPRETATION RULES
===============================================================================

06.1. Preamble text remains self-contained, explicit, and
      interpretation-free.

06.2. Preamble text does not rely on reader intuition.

06.3. Preamble text does not rely on assumed common sense.

06.4. Preamble text does not rely on prior document versions.

06.5. Preamble text does not rely on conversational context.

06.6. The drafting author may repeat critical constraints to reinforce
      scope and authority boundaries.

06.7. A drafting author may produce preamble text.

06.8. Drafted text becomes authoritative only after explicit acceptance
      by the responsible human authority (see CMR-03).


===============================================================================
07. END-OF-DOCUMENT MARKER
===============================================================================

07.1. Each specification document ends with an explicit end marker.

07.2. The end marker uses the following structure:

      Line 1: 79 "=" characters
      Line 2: END OF DOC_ID - DOC_TITLE
      Line 3: 79 "=" characters

07.3. DOC_ID and DOC_TITLE in the end marker match the DOC_ID and
      DOC_TITLE used in the document banner.

07.4. No content appears after the end marker except the doc-meta block
      defined by CMR-01.

07.5. In committed authoritative documents, the doc-meta block appears
      after the end marker.


===============================================================================
08. UPSTREAM DOCUMENTS BLOCK (OPTIONAL)
===============================================================================

08.1. Upstream documents constrain this document.

08.1.1. "Upstream Documents" acts as a constraint set reference.

08.1.2. The Upstream Documents block does not define strict dependency
        ordering.

08.1.3. The Upstream Documents block does not represent publication
        chronology.

08.1.4. The Upstream Documents block does not require upstream documents
        to be read first.

08.1.5. Documents may mutually list each other as upstream without
        implying precedence or ordering.

08.2. A document may include an Upstream Documents block in the preamble.

08.3. When the Upstream Documents block or Downstream Documents block
      appears, the following order is used:

      1) Purpose and Scope block
      2) Upstream Documents block
      3) Downstream Documents block
      4) First numbered section

08.4. The Upstream Documents block uses the following structure:

      Upstream Documents:
      ------------------
      <free text, optional>
      - <DOC_ID>
      - <DOC_ID>

08.5. Retrieval convention:

08.5.1. Non-CMR documents include the following identifiers:

        - CMR-01
        - CMR-02
        - CMR-03
        - CMR-04
        - CMR-05

08.5.2. CMR documents include all CMR documents except the current
        document.

08.5.3. The inclusions in 08.5.1 and 08.5.2 exist for retrieval and
        navigation hygiene.

08.5.4. These inclusions do not imply semantic constraint.

08.6. The Upstream Documents block improves document retrieval.

08.7. The Upstream Documents block clarifies cross-document constraints
      for humans and language models.

08.8. The Upstream Documents block does not introduce new behavior,
      guarantees, or conflict resolution rules.


===============================================================================
09. DOWNSTREAM DOCUMENTS BLOCK (OPTIONAL)
===============================================================================

09.1. Downstream documents may be impacted by changes to this document.

09.2. A document may include a Downstream Documents block in the
      preamble.

09.3. The Downstream Documents block uses the following structure:

      Downstream Documents:
      --------------------
      <free text, optional>
      - <DOC_ID>
      - <DOC_ID>

09.4. Listed DOC_ID values exist in the Canonical Index (CMR-04).

09.5. The Downstream Documents block remains advisory.

09.6. The Downstream Documents block supports change-impact review.

09.7. The Downstream Documents block does not introduce new behavior,
      guarantees, or conflict resolution rules.

09.8. The Downstream Documents block is not required to be complete.
