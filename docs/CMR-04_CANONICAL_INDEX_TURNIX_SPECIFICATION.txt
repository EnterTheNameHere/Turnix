===============================================================================
CORE META-RULES CMR-04 - CANONICAL INDEX - TURNIX SPECIFICATION
===============================================================================

Status:
-------
FROZEN

Description:
------------
Single source of truth for which Turnix specification documents
are authoritative, where they live, and how conflicts are resolved.

Changes here have cascading impact and must be explicit.

Purpose:
--------
This document defines the canonical index for Turnix documentation.

It defines:
- which documents are authoritative
- canonical DOC_ID values, tiers, and file locations
- conflict resolution rules when documents disagree

It does NOT define:
- the content or semantics of indexed documents
- implementation behavior
- dependency graphs or explanatory prose

All authoritative content lives exclusively in the referenced documents.

- CMR-03

===============================================================================
01. NORMATIVE KEYWORDS
===============================================================================

01.1. The following keywords are normative and are used across ALL Turnix
      documents:
      - MUST
      - MUST NOT
      - MAY

01.2. No other normative keywords are recognized.


===============================================================================
02. IDENTIFIERS AND INDEX ENTRY FIELDS
===============================================================================

02.1. Each indexed document entry MUST define these fields:
      - DOC_ID
      - DOC_TITLE
      - DOC_TIER
      - DOC_FILE
      - DOC_AUTHORITATIVE_STATUS

02.2. DOC_ID is the canonical document identifier used for cross-document
      references and cross-thread retrieval.

      Examples:
      - CMR-01
      - DA-09
      - ODA-E
      - IS-01

02.3. DOC_TITLE is a human-readable name. It is not a stable identifier and
      MUST NOT be used as the sole identifier for retrieval.

02.4. DOC_TIER defines authority precedence for conflict resolution.

02.5. DOC_FILE is the canonical repository path of the current revision of
      the document.

02.6. If a doc-meta block is present (CMR-01):
      - Its DOC_ID MUST match the DOC_ID in the Canonical Index entry.
      - Its DOC_FILE MUST match the Canonical Index DOC_FILE for that DOC_ID.

02.7. DOC_AUTHORITATIVE_STATUS MUST be one of:
      - AUTHORITATIVE
      - DRAFT
      - OBSOLETE

02.8. DRAFT documents MAY be indexed, but MUST NOT be treated as
      authoritative for conflict resolution.

02.9. Index entry structure rules:

02.9.1. Each indexed document entry MUST begin with a bullet line:
        - DOC_ID: <value>

        The remaining required fields MUST follow as indented field lines.

02.9.2. Within an entry, each required field from 02.1:
        - MUST appear exactly once
        - MUST use the exact field name spelling from 02.1
        - MUST use the syntax "<FIELD>: <value>" on a single line

02.9.3. All non-first field lines in an entry MUST be indented by exactly two
        spaces.

02.9.4. No additional fields are permitted in an Index entry.

02.9.5. A blank line MAY appear between Index entries for readability.

02.9.6. Blank lines MUST NOT appear within an entry.

02.9.7. Field order within an entry MUST be:
        - DOC_ID
        - DOC_TITLE
        - DOC_TIER
        - DOC_FILE
        - DOC_AUTHORITATIVE_STATUS


===============================================================================
03. CONFLICT RESOLUTION
===============================================================================

03.1. This Canonical Index is the single source of truth for identifying the
      current authoritative document set.

03.2. If two documents conflict and this Canonical Index
      DOC_AUTHORITATIVE_STATUS marks one as AUTHORITATIVE and the other as
      OBSOLETE or DRAFT, the AUTHORITATIVE document wins.

03.3. If two documents conflict and both are AUTHORITATIVE, resolution
      follows authority tier precedence:

      1) CORE META-RULES
      2) DESIGN AREA
      3) OPTIONAL DESIGN AREA
      4) IMPLEMENTATION SPECIFICATION

03.3.1. Higher tiers override lower tiers when a conflict exists.

03.3.2. The tier names listed above are the only valid DOC_TIER values.

03.4. If a conflict exists within the same authority tier, the author MUST
      update documents and Index entries to restore consistency.


===============================================================================
04. STATUS TERM MAPPING RULES
===============================================================================

04.1. The Status term in a document's Status block (CMR-02) MUST obey the
      mapping rules in this section for the document's
      DOC_AUTHORITATIVE_STATUS value.

04.1.1. DOC_AUTHORITATIVE_STATUS is the sole source of truth for document
        authority. If Status conflicts with DOC_AUTHORITATIVE_STATUS,
        authority decisions MUST follow DOC_AUTHORITATIVE_STATUS.

04.2. DOC_AUTHORITATIVE_STATUS: DRAFT
      - Status MUST be EVOLVING.

04.3. DOC_AUTHORITATIVE_STATUS: OBSOLETE
      - Status MUST be OBSOLETE.

04.4. DOC_AUTHORITATIVE_STATUS: AUTHORITATIVE
      - Status MUST be either STABLE or FROZEN.

04.5. FROZEN is reserved for documents treated as not expected to change
      (see CMR-02).

===============================================================================
08. IMPLEMENTATION SPECIFICATION DOCUMENTS
===============================================================================

08.1. IMPLEMENTATION SPECIFICATION documents define concrete protocols,
      data formats, and execution flows constrained by DESIGN AREA and
      OPTIONAL DESIGN AREA documents.

08.2. IMPLEMENTATION SPECIFICATION documents MUST NOT introduce new system
      guarantees, authority semantics, or policy rules.

08.3. IMPLEMENTATION SPECIFICATION documents MUST be traceable to the DESIGN
      AREA and OPTIONAL DESIGN AREA documents that constrain them.

08.4. IMPLEMENTATION SPECIFICATION documents MUST be registered here when
      they become authoritative.
