===============================================================================
CORE META-RULES CMR-04 - CANONICAL INDEX - TURNIX SPECIFICATION
===============================================================================

Status:
-------
FROZEN

Description:
------------
Single source of truth for which Turnix specification documents
are authoritative, where they live, and how conflicts are resolved.

Changes here have cascading impact and must be explicit.

Purpose:
--------
This document defines the Canonical Index for Turnix documentation.

It defines:
- which documents are authoritative
- canonical DOC_ID values, tiers, and file locations
- conflict resolution rules when documents disagree
- status mapping rules for indexed documents

It does NOT define:
- the content or semantics of indexed documents
- implementation behavior
- dependency graphs or explanatory prose
- document creation and change discipline (see CMR-03)

All authoritative content lives exclusively in the referenced documents.

This Canonical Index is authoritative only for document identity, location,
and authority status. It is not authoritative for domain semantics.

===============================================================================
01. NORMATIVE LANGUAGE
===============================================================================

01.1. Turnix specifications express normative meaning through explicit,
      deterministic prose rather than through RFC-style keywords.

01.2. Normative meaning is expressed through clear actor-action statements
      that describe behavior, guarantees, constraints, or required
      handling.

01.3. Normative rules remain understandable without relying on special
      vocabulary markers.

01.4. Statements describing permitted alternatives express implementation
      freedom only within the constraints explicitly defined by the
      specification.

01.5. RFC-style requirement keywords carry no special meaning in Turnix
      specification documents.

01.6. The tokens "SHOULD" and "SHOULD NOT" are forbidden in authoritative
      documents because they introduce ambiguous advisory semantics.

01.7. When such tokens appear in draft text, the drafting author removes
      or rewrites them before the document is treated as authoritative.

01.8. Normative meaning derives from sentence structure and document
      context, not from reserved keywords.


===============================================================================
02. IDENTIFIERS AND INDEX ENTRY FIELDS
===============================================================================

02.1. Each indexed document entry defines all of the following fields:
      - DOC_ID
      - DOC_TITLE
      - DOC_TIER
      - DOC_FILE
      - DOC_AUTHORITATIVE_STATUS

02.1.1. Terminology:

02.1.1.1. Specification document: any document registered in this
          Canonical Index, regardless of DOC_AUTHORITATIVE_STATUS.

02.1.1.2. Authoritative document: a specification document whose
          DOC_AUTHORITATIVE_STATUS is AUTHORITATIVE.

02.2. DOC_ID is the canonical document identifier used for cross-document
      references and cross-thread retrieval.

      Examples:
      - CMR-01
      - DD-01
      - DA-09
      - ODA-E
      - IS-01

02.3. DOC_TITLE is a human-readable name. It is not a stable identifier
      and must not be used as the sole identifier for retrieval.

02.4. DOC_TIER defines authority precedence for conflict resolution.

02.5. DOC_FILE is the canonical repository path of the current revision of
      the document.

02.6. If a doc-meta block is present (CMR-01):
      - its DOC_ID is the same as the DOC_ID in the Canonical Index entry
      - its DOC_FILE is the same as the Canonical Index DOC_FILE for that
        DOC_ID

02.7. DOC_AUTHORITATIVE_STATUS is one of:
      - AUTHORITATIVE
      - DRAFT
      - OBSOLETE

02.8. DRAFT documents may be indexed, but conflict resolution does not
      treat them as authoritative.

02.9. Index entry structure rules:

02.9.1. Each indexed document entry begins with a bullet line:
        - DOC_ID: <value>

02.9.2. The remaining required fields follow as indented field lines.

02.9.3. Within an entry, each required field from 02.1 appears exactly once.

02.9.4. Within an entry, each required field uses the exact field name
        spelling defined in 02.1.

02.9.5. Each field line uses the syntax "<FIELD>: <value>" on a single line.

02.9.6. All non-first field lines in an entry are indented by exactly two
        spaces.

02.9.7. No additional fields appear in an Index entry.

02.9.8. A blank line may appear between Index entries for readability.

02.9.9. Blank lines do not appear within an entry.

02.9.10. Field order within an entry is:
         - DOC_ID
         - DOC_TITLE
         - DOC_TIER
         - DOC_FILE
         - DOC_AUTHORITATIVE_STATUS


===============================================================================
03. CONFLICT RESOLUTION
===============================================================================

03.1. This Canonical Index is the single source of truth for identifying the
      current authoritative document set.

03.2. If two documents conflict and this Canonical Index marks one entry as
      DOC_AUTHORITATIVE_STATUS: AUTHORITATIVE and the other as
      DOC_AUTHORITATIVE_STATUS: OBSOLETE or DRAFT, the AUTHORITATIVE
      document wins.

03.3. If two documents conflict and both are AUTHORITATIVE, resolution
      follows authority tier precedence in this order:
      1) CORE META-RULES
      2) DESIGN DOCTRINE
      3) DESIGN AREA
      4) OPTIONAL DESIGN AREA
      5) IMPLEMENTATION SPECIFICATION

03.3.1. Higher tiers override lower tiers when a conflict exists.

03.3.2. The tier names listed above are the only valid DOC_TIER values.

03.4. If a conflict exists within the same authority tier, the responsible
      human authority updates the affected documents and Index entries to
      restore consistency.


===============================================================================
04. STATUS TERM MAPPING RULES
===============================================================================

04.1. The Status term in a document's Status block (CMR-02) follows the
      mapping rules in this section for the document's
      DOC_AUTHORITATIVE_STATUS value.

04.1.1. DOC_AUTHORITATIVE_STATUS is the sole source of truth for document
        authority.

04.1.2. If Status conflicts with DOC_AUTHORITATIVE_STATUS, authority
        decisions follow DOC_AUTHORITATIVE_STATUS.

04.2. When DOC_AUTHORITATIVE_STATUS is DRAFT, Status is EVOLVING.

04.3. When DOC_AUTHORITATIVE_STATUS is OBSOLETE, Status is OBSOLETE.

04.4. When DOC_AUTHORITATIVE_STATUS is AUTHORITATIVE, Status is either
      STABLE or FROZEN.

04.5. FROZEN is reserved for documents treated as not expected to change
      (see CMR-02).

06. DESIGN DOCTRINE DOCUMENTS
===============================================================================

06.1. DESIGN DOCTRINE documents define cross-cutting design,
      engineering, interface, and cooperation principles for Turnix.

06.2. DESIGN DOCTRINE documents constrain or guide later DESIGN AREA,
      OPTIONAL DESIGN AREA, and IMPLEMENTATION SPECIFICATION documents
      where shared cross-cutting principles apply.

06.3. DESIGN DOCTRINE documents do not replace CORE META-RULES authority
      over document structure, indexing, change discipline, or
      cross-thread authority handling.

06.4. DESIGN DOCTRINE documents remain doctrinal. They do not silently
      drift into exact runtime semantics, concrete protocols, or
      document-system governance that belong more properly to DESIGN AREA,
      IMPLEMENTATION SPECIFICATION, or CORE META-RULES documents.

09. IMPLEMENTATION SPECIFICATION DOCUMENTS
===============================================================================

09.1. IMPLEMENTATION SPECIFICATION documents define concrete protocols,
      data formats, and execution flows constrained by DESIGN AREA and
      OPTIONAL DESIGN AREA documents.

09.2. IMPLEMENTATION SPECIFICATION documents must not introduce new system
      guarantees, authority semantics, or policy rules.

09.3. IMPLEMENTATION SPECIFICATION documents must remain traceable to the
      DESIGN AREA and OPTIONAL DESIGN AREA documents that constrain them.

09.4. IMPLEMENTATION SPECIFICATION documents must be registered here when
      they become authoritative.
