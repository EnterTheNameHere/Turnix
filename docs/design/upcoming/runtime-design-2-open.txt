01. Runtime host ownership
~~~~~~~~~~~~~~~~~~~~~~~~~~
Promoted owner:
- DA-28 Runtime Host, Workspace Ownership and Command Surface

Resolved direction:
- RuntimeHost is the running Turnix host boundary.
- RuntimeHost identity is distinct from Kernel, workspace, AppInstance,
  SavePack, View, adapter, RPC, Job, Orchestration Unit, transaction, and
  decision identity.
- RuntimeHost does not replace Kernel authority.
- RuntimeHost binds workspace or runtime scope and exposes controlled entry
  surfaces.
- RuntimeHost lifecycle and RuntimeHost-domain trace reason ownership belong
  to DA-28.

Remaining follow-up:
- exact runtimeHostId representation
- exact workspace identity representation
- exact process identity representation
- exact KernelContext identity representation
- exact host shutdown policy for active AppInstances
- exact retained work behavior after host stop, disconnect, or crash
- exact service, daemon, container, or virtual-machine wrapper behavior

02. Runtime host command surface
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Promoted owners:
- DA-28 Runtime Host, Workspace Ownership and Command Surface
- IS-04 Runtime Host Command Surface

Resolved direction:
- Command surfaces are entry surfaces and are not Kernel authority by
  themselves.
- Command receipt, classification, acceptance, dispatch, completion, failure,
  and cancellation are distinct.
- Command categories are semantic categories, not final CLI syntax.
- Command handling uses shared Kernel-mediated runtime operations.

Remaining follow-up:
- exact CLI syntax
- exact parser shape
- exact commandId representation
- exact runtimeOperationId representation
- exact command result JSON schema
- exact terminal output format
- exact script-stable output mode, including possible JSON output
- exact command extension mechanism
- exact command-to-Job handle surface
- exact command-to-subscription surface
- exact retained command result behavior

03. Workspace ownership and locking
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Promoted owners:
- DA-28 Runtime Host, Workspace Ownership and Command Surface
- IS-03 Workspace Ownership and Host Locking

Resolved direction:
- Writable workspace ownership is RuntimeHost-level ownership.
- A writable workspace is owned by at most one RuntimeHost at a time unless a
  later authoritative multi-host design defines another mode.
- Initial ownership record path is:
  .turnix/runtime-host/owner.json
- Existing ownership evidence is not silently overwritten.
- Stale-owner recovery is explicit, policy-governed, and trace-visible.
- Read-only inspection does not create writable workspace ownership.

Remaining follow-up:
- exact workspaceId representation
- exact runtimeHostId representation
- exact hostMachineId representation
- exact userId representation
- exact heartbeat interval
- exact heartbeat stale threshold
- exact process liveness behavior per operating system
- exact stale-owner recovery prompt text
- exact ownership record migration behavior
- exact optional read-only inspection output location
- exact ownership JSON schema publication format

04. Runtime host startup modes
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Promoted owners:
- DA-28 Runtime Host, Workspace Ownership and Command Surface
- IS-05 Runtime Host Startup Modes

Resolved direction:
- Startup modes are entry workflows over shared Kernel-mediated runtime
  operations.
- Startup mode selection does not define independent runtime semantics.
- Defined startup mode categories are interactiveHost, oneShotCommand, serve,
  connect, test, and launcher.
- Startup modes that require writable workspace ownership use IS-03.
- Startup modes that receive or dispatch commands use IS-04.

Remaining follow-up:
- exact startupRequestId representation
- exact startup result JSON schema
- exact mapping from CLI syntax to startupMode
- exact desktop launcher startup request representation
- exact service or daemon wrapper startup request representation
- exact process exit code mapping for startup results
- exact detached work behavior for oneShotCommand mode
- exact serve mode local bind defaults
- exact connect mode host discovery behavior
- exact startup-mode extension mechanism

05. Runtime host LAN and client connections
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Promoted owners:
- DA-28 Runtime Host, Workspace Ownership and Command Surface
- IS-06 Runtime Host LAN and Client Connections

Resolved direction:
- LAN exposure is explicit and policy-governed.
- localOnly is the default bind scope unless a later implementation
  specification defines a different default without weakening DA-28.
- lanExplicit is never selected silently.
- remoteExplicit is not enabled by IS-06.
- Client connection is not command acceptance, capability permission, user
  identity, or state authority.
- Pairing and authentication are distinct.

Remaining follow-up:
- exact clientConnectionId representation
- exact entry surface identity representation
- exact local bind behavior
- exact LAN bind behavior
- exact remote endpoint summary format
- exact pairing flow
- exact authentication flow
- exact token or certificate format
- exact client reconnect flow
- exact retained result retrieval after reconnect
- exact browser bootstrap interaction
- exact firewall guidance
- exact transport-specific connection schema
