===============================================================================
DESIGN AREA DA-09 - GENERIC RESOLUTION SEMANTICS
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the generic, dependency-agnostic resolution machinery used by Turnix
to interpret PackRequests, enumerate candidates, apply constraints, and
produce deterministic outcomes.

This document defines resolution mechanics only.

This area strictly separates:
- request parsing
- candidate enumeration
- constraint application
- deterministic selection
- ambiguity and failure signaling

Nothing in this area implies loading, activation, execution order, or runtime
access.

Purpose:
--------
This document defines the generic, side-effect-free resolution engine that
higher-level consumers invoke with their own policy and ordering criteria.

It defines:
- PackRequest grammar
- PackRequest parsing and internal representation
- candidate enumeration phases
- hard and soft constraint handling
- deterministic selection contract
- ambiguity and failure signaling
- successful-resolution observability requirements
- separation between resolution mechanics and consumer policy

It does NOT define:
- discovery boundaries (see DA-07)
- pack identity and visibility semantics (see DA-08)
- dependency meaning or strength
- save-specific override policy
- activation, execution, or lifecycle semantics
- capability authority or permission mechanics

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines generic resolution mechanics within that shared
      interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. Resolution answers:

          "Which concrete pack identity satisfies this PackRequest
           inside the current resolution context?"

02.2. Resolution is side-effect free.

02.3. Resolution consumes:
      - a PackRequest
      - a snapshot of discovered and indexed pack descriptors
      - a resolution context supplied by a higher consumer

02.4. Resolution does not:
      - rescan the filesystem
      - read manifests lazily
      - activate packs
      - load runtimes
      - guess policy

02.5. Resolution remains strictly separate from discovery.

02.6. Discovery builds the candidate universe. Resolution selects from that
      already-built universe.

02.7. Resolution remains strictly separate from activation and runtime use.

02.8. This document defines generic mechanics, not consumer-specific
      strategy.


===============================================================================
03. PACKREQUEST REFERENCE GRAMMAR (LOCKED)
===============================================================================

03.1. Canonical grammar:
      [<author>@]<packTreeId>[@<semverRequirement>]

03.2. Grammar rules:
      - case-sensitive
      - exactly zero, one, or two "@" characters
      - no filesystem paths
      - no slashes
      - dots are allowed only for pack hierarchy
      - no implicit defaults inside the raw reference string

03.3. Valid examples:
      - "ui"
      - "ui.controls"
      - "Turnix@ui.controls"
      - "ui.controls@^2.0"
      - "Turnix@ui.controls@~1.4"

03.4. Invalid examples:
      - "@ui"
      - "ui/controls"
      - "ui.controls:1.0"


===============================================================================
04. SINGLE-AT DISAMBIGUATION RULE (LOCKED)
===============================================================================

04.1. If a PackRequest contains exactly one "@", the resolver first attempts
      to parse the suffix as a semantic version requirement.

04.2. If semantic-version parsing succeeds, the suffix is treated as
      semverRequirement.

04.3. If semantic-version parsing fails, the suffix is treated as author.

04.4. Examples:
      - "foo@1.2" -> packTreeId = "foo", semverRequirement = "1.2"
      - "foo@bar" -> author = "foo", packTreeId = "bar"

04.5. Failure to parse semantic version in this case is not itself an error.
      It is part of the intentional disambiguation rule.


===============================================================================
05. PACKREQUEST INTERNAL REPRESENTATION (LOCKED)
===============================================================================

05.1. PackRequest is the parsed internal representation used by the generic
      resolver.

05.2. PackRequest structure:
      - author: str | None
      - packTreeId: str
      - semverRequirement: SemVerPackRequirement | None
      - kind: PackKind | None

05.3. PackRequest is created before generic resolution begins.

05.4. PackRequest remains immutable after creation.

05.5. PackRequest carries only normalized request identity and request
      constraints.

05.6. PackRequest does not carry higher-level resolution policy such as
      optionality, source preference, prerelease allowance, or persisted
      ambiguity decisions.

05.7. kind may be injected by the surrounding resolution context when that
      context requires kind restriction.


===============================================================================
06. RESOLUTION OVERVIEW (LOCKED)
===============================================================================

06.1. Resolution is a deterministic, side-effect-free process.

06.2. Inputs:
      - one PackRequest
      - one registry snapshot
      - one externally supplied resolution context

06.3. Outputs:
      - exactly one resolved concrete pack identity
      - or one classified resolution failure
      - or one classified ambiguity requiring external decision

06.4. Given the same inputs and the same resolution context, the outcome is
      identical.

06.5. The resolver does not mutate the registry snapshot.

06.6. The resolver does not trigger new discovery.


===============================================================================
07. CANDIDATE ENUMERATION - PHASE A (LOCKED)
===============================================================================

07.1. Candidate enumeration identifies packs that are structurally compatible
      with a PackRequest.

07.2. Enumeration steps:
      1) match packTreeId
      2) apply author constraint if present
      3) apply kind constraint if present
      4) collect remaining candidates

07.3. Candidate enumeration does not:
      - order candidates
      - select a winner
      - apply consumer preference
      - perform fallback
      - interpret optionality

07.4. This phase answers only:
      "Which candidates could structurally satisfy this request?"


===============================================================================
08. CONSTRAINT APPLICATION - PHASE B (LOCKED)
===============================================================================

08.1. Constraint application removes or restricts candidates after structural
      enumeration.

08.2. Constraint classes are:
      - hard constraints
      - soft constraints


08.1 HARD CONSTRAINTS (LOCKED)
------------------------------

08.1.1. Hard constraints permanently exclude candidates.

08.1.2. Examples include:
        - packTreeId mismatch
        - required author mismatch
        - kind mismatch
        - visibility violation
        - forbidden root or source within the current resolution context
        - concrete version mismatch

08.1.3. A candidate that fails a hard constraint is not selectable.

08.1.4. A candidate that fails a hard constraint is not offered as a valid
        outcome.

08.1.5. Hard-constraint failure remains observable through the higher
        diagnostic and tracing system.


08.2 SOFT CONSTRAINTS (LOCKED)
------------------------------

08.2.1. Soft constraints discourage selection without erasing candidate
        identity.

08.2.2. Examples include:
        - prerelease discouraged by policy
        - deprecated pack
        - policy-based discouragement

08.2.3. Soft-constrained candidates remain known to the resolver.

08.2.4. Soft-constrained candidates are not selected unless the surrounding
        consumer context explicitly allows them.

08.2.5. Soft constraints apply to candidates rather than to request
        mandatory or optional semantics.

08.2.6. Request strength or optionality is defined outside this document.

08.2.7. If soft-constrained candidates become allowed, they remain subject
        to deterministic ordering.


===============================================================================
09. RESOLUTION CONTEXT AND SOURCE BOUNDARY (LOCKED)
===============================================================================

09.1. Resolution always occurs inside one explicit resolution context.

09.2. The resolution context determines:
      - which candidate population is visible to the resolver
      - which source boundaries are legal
      - which policy constraints are in force
      - which ordering criteria will later be applied

09.3. The generic resolver does not invent or switch resolution context.

09.4. The generic resolver does not mix contexts silently.

09.5. If the surrounding consumer violates context assumptions, resolution
      fails explicitly.

09.6. This document does not define the final save-specific or consumer-
      specific source model. It only requires that one explicit context
      applies during one resolution attempt.


===============================================================================
10. DETERMINISTIC SELECTION (LOCKED)
===============================================================================

10.1. After constraints are applied, the resolver imposes a total,
      deterministic ordering on the remaining selectable candidates.

10.2. The resolver selects the first candidate in that ordering.

10.3. Ordering is:
      - stable
      - reproducible
      - independent of incidental iteration order
      - free of randomness

10.4. The exact ordering criteria are supplied by the surrounding consumer.

10.5. Silent substitution is forbidden.


===============================================================================
11. SUCCESSFUL RESOLUTION OBSERVABILITY (LOCKED)
===============================================================================

11.1. Successful resolution outcomes remain observable.

11.2. Observability covers successful resolution paths as well as failures.

11.3. A successful resolution outcome remains explainable after the fact.

11.4. Resolution observability supports explanation of:
      - why the selected candidate won
      - why another candidate did not win
      - why a candidate was excluded
      - why the active resolution context permitted this outcome

11.5. This document requires successful-resolution observability. DA-03
      defines the tracing and diagnostic mechanisms that realize it.

11.6. If a successful resolution path cannot be explained after the fact,
      that is an observability failure.


===============================================================================
12. AMBIGUITY HANDLING (LOCKED)
===============================================================================

12.1. If deterministic ordering cannot reduce the selectable candidates to
      one unambiguous result, resolution pauses.

12.2. Ambiguity is classified explicitly.

12.3. External policy or user decision is then required.

12.4. Once such a decision is made by the surrounding consumer, that
      decision is persisted by the consumer rather than by the generic
      resolver.

12.5. The generic resolver does not guess.

12.6. The generic resolver does not auto-persist decisions.


===============================================================================
13. FAILURE CONDITIONS (LOCKED)
===============================================================================

13.1. Resolution fails explicitly if:
      - no candidates exist after hard constraints
      - no candidates are selectable under the supplied context
      - context rules are violated
      - ambiguity requires decision and no decision path is available

13.2. Resolution failure is:
      - explicit
      - classified
      - observable

13.3. Resolution failure never falls back silently.


===============================================================================
14. RESOLUTION IS NOT ACTIVATION
===============================================================================

14.1. Resolution determines which concrete pack identity satisfies a
      request.

14.2. Resolution does not:
      - load code
      - activate packs
      - guarantee execution order
      - grant runtime access

14.3. Activation and lifecycle behavior are defined elsewhere.


===============================================================================
15. ERROR MODEL (LOCKED)
===============================================================================

15.1. Resolution errors include examples such as:
      - NotFoundError
      - VersionMismatchError
      - AmbiguousResolutionError
      - PermissionDeniedError

15.2. Resolution errors use the standard error envelope.

15.3. Resolution errors include:
      - PackRequest context
      - resolution context
      - reason classification

15.4. This document does not lock the final full error taxonomy beyond the
      requirement that classification remains explicit.


===============================================================================
16. LOCKED SUMMARY
===============================================================================

16.1. LOCKED:
      - PackRequest grammar and disambiguation rules
      - immutable PackRequest internal representation
      - side-effect-free resolution
      - no lazy discovery during resolution
      - phase-A candidate enumeration
      - phase-B hard and soft constraint handling
      - deterministic ordering and winner selection contract
      - concrete version mismatch is a hard exclusion
      - successful resolution outcomes remain observable
      - explicit ambiguity signaling
      - no silent substitution
      - resolution remains separate from activation

16.2. DEFERRED:
      - dependency semantics
      - save-specific override policy
      - pinning and copying rules
      - activation and lifecycle behavior
      - consumer-specific ordering criteria
