===============================================================================
DESIGN AREA DA-04 - MEMORY, STATE & AUTHORITATIVE COMMIT MODEL
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how Turnix distinguishes runtime memory from authoritative state, how
state is layered and mutated, and how committed state changes become
deterministic, restorable, and traceable without depending on observational
history.

Purpose:
--------
This document defines the authoritative memory and state model used by Turnix.

It defines:
- what "memory" means in Turnix
- what "state" means in Turnix
- authority boundaries between committed and non-committed material
- layer-based runtime memory structure
- transactional staging and commit or abort rules
- commit conflict detection for authoritative mutation
- orchestration boundaries for state mutation
- Orchestration Unit authority and outcome boundaries
- relationship between Orchestration Units, Jobs, and transactions
- state and transaction trace obligations
- persistence as protection against loss rather than the definition of truth
- static-data mutation intent
- revision and state-retention rules
- migration commit rules for authoritative state mutation
- state-boundary constraints for tooling-initiated mutation
- restoration boundary rules for committed state

It does NOT define:
- tracing structure, trace event schema, or trace storage semantics
  (see DA-03)
- payload identity, ChunkId, or ValueRef representation contracts
  (see DA-05)
- concrete persistent I/O authority, Readers, Writers, or filesystem rules
  (see DA-07)
- configuration-specific scope, schema, and migration semantics
  (see DA-06)
- windows, manifests, exports, replay workflows, or audit bundles
  (see DA-08)
- pack activation semantics or PackContext authority (see DA-14)
- service, capability, or permission semantics (see DA-16)
- Job lifecycle, Job result, Job timeout, or Job cancellation semantics
  (see DA-19)
- AppInstance lifecycle or runtime identity semantics (see DA-20)
- debugger pause, breakpoint, or execution-control behavior (see DA-18)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines the authoritative memory and state model within
      that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. This design area answers:

          "Which runtime values are authoritative, when do they become
           authoritative, and how does Turnix preserve deterministic
           committed-state evolution?"

02.2. Turnix distinguishes strictly between:
      - runtime memory
      - committed state
      - non-authoritative observational material

02.3. Runtime memory is not authoritative merely because it is resident,
      visible, active, or currently in use.

02.4. Committed state is authoritative only because it has crossed the
      explicit commit boundary defined by this document.

02.5. Turnix preserves deterministic committed-state evolution by making
      authoritative mutation explicit, staged, and commit-bounded.

02.6. Turnix preserves rollback and restoration by ensuring that committed
      state remains restorable without requiring non-authoritative
      historical material.

02.7. Observational material may explain what happened, but it does not
      define the committed truth of the system.

02.8. Turnix does not treat observation as authority.

02.9. Turnix does not require observational completeness for committed-state
      correctness.

02.10. Turnix does not imply determinism outside the committed-state
       boundary defined by this document.

02.11. Later sections of this document refine and apply these rules.
       They do not weaken them.


===============================================================================
03. MEMORY, STATE, AND AUTHORITY CLASSES
===============================================================================

03.1. Memory is the runtime representation of values.

03.2. Memory may:
      - exist only in-process
      - include speculative values
      - include staged transactional values
      - include transient runtime-only values

03.3. Memory is mutable.

03.4. Memory is not inherently authoritative.

03.5. State is the authoritative outcome of committed memory.

03.6. State exists independently of runtime residency.

03.7. State is the source of truth for restoration and deterministic
      comparison.

03.8. Committed state is immutable once committed except through a later
      committed revision.

03.9. Memory may exist without persistence.

03.10. Persisted state may exist without current runtime residency.

03.11. Only committed state is authoritative.

03.12. Runtime memory is not treated as authoritative unless it has become
       committed state through the commit rules defined by this document.


03.1 AUTHORITY CLASSES
----------------------

03.1.1. Turnix distinguishes exactly two authority classes:

        A) Authoritative
        B) Non-authoritative

03.1.2. Authoritative material includes:
        - committed memory layers
        - committed configuration
        - committed state captured by a SafePointWindow
        - other committed state governed by the Turnix commit model

03.1.3. Authoritative material defines current system truth.

03.1.4. Authoritative material is sufficient for deterministic restoration
        within the boundaries defined by this document.

03.1.5. Authoritative material is required for correctness.

03.1.6. Non-authoritative material includes:
        - transactional staging layers
        - uncommitted runtime memory
        - debugger staging artifacts
        - traces
        - journals
        - other observational or explanatory material

03.1.7. Non-authoritative material may support:
        - explanation
        - audit
        - debugging
        - diagnosis

03.1.8. Non-authoritative material is not required for state correctness.

03.1.9. Non-authoritative material does not affect authoritative-state
        equality.

03.1.10. Non-authoritative material may be incomplete, truncated, or
         missing according to explicit policy without changing committed
         state meaning.


03.2 AUTHORITY SEPARATION RULES
-------------------------------

03.2.1. Authoritative state is derivable without requiring access to:
        - traces
        - journals
        - transactional layers
        - debugger-only artifacts

03.2.2. Non-authoritative material is not required to:
        - restore committed state
        - evaluate committed-state equality
        - validate correctness of committed values

03.2.3. Loss of non-authoritative material does not:
        - corrupt committed state
        - change committed meaning
        - invalidate determinism guarantees for committed restoration

03.2.4. This authority separation preserves correctness without turning
        observation into a hidden dependency.

03.2.5. Turnix does not treat traces, journals, or debugger-only artifacts
        as authoritative state.

03.2.6. Turnix does not allow committed-state correctness to depend on
        whether such material is complete, retained, or present.


===============================================================================
04. IDENTITY AND REVISION TERMS
===============================================================================

04.1. A revisionId identifies one committed revision record for one key
      within one committed layer.

04.2. A baseRevisionId identifies the revisionId that a transaction
      observed when it first touched the key by read or write.

04.3. revisionId is a JSON-safe integer.

04.4. revisionId uniqueness and monotonicity are required only per
      (layer, key).

04.5. Clients may treat revisionId as opaque read-only metadata.

04.6. baseRevisionId supports an absent sentinel meaning:
      "the key was not present in the committed layer at capture time."

04.7. The absent sentinel is represented as baseRevisionId = 0.

04.8. The absent sentinel is not represented as null.

04.9. Valid committed revisionId values start at 1.

04.10. Revision metadata supports conflict detection and committed-state
       history. It does not by itself create authority outside committed
       state.


===============================================================================
05. MEMORY LAYERS
===============================================================================

05.1. Turnix manages runtime memory as a set of layered stores.

05.2. Layers represent:
      - authority
      - scope
      - lifetime
      - rollback boundaries

05.3. There is no single global memory object.

05.4. Each layer contains real values for its own scope.

05.5. Reads resolve according to the active layer stack for the requesting
      scope.

05.6. Example layer kinds include:
      - kernel-global
      - appInstance
      - session
      - view
      - transactional

05.7. Example layer kinds do not lock the complete future layer catalog.

05.8. A committed layer is representable at the commit boundary through the
      committed value representation contract defined elsewhere.

05.9. Layer serialization preserves committed value meaning rather than live
      runtime object identity.

05.10. Runtime residency is an implementation detail and does not change the
       meaning of a committed layer.

05.11. Layering defines visibility, rollback, and mutation boundaries. It
       does not blur the distinction between committed and non-committed
       material.


===============================================================================
06. TRANSACTIONS
===============================================================================

06.1. A transaction is a temporary staging overlay used to record
      speculative mutations before they become authoritative.

06.2. A transaction overlays one or more parent layers.

06.3. A transaction captures all writes during its lifetime.

06.4. A transaction may read from parent layers.

06.5. A transaction is resolved only by commit or abort.

06.6. Primary transaction use includes:
      - pipeline execution
      - capability calls with mutation
      - tooling-initiated staged mutation
      - other orchestrated speculative work

06.7. A transaction is not authoritative merely because it contains staged
      values.

06.8. Staged mutation does not become committed truth until commit into an
      authoritative boundary occurs under the rules of this document.


06.1 COMMIT AND ABORT
---------------------

06.1.1. A transaction ends in exactly one outcome:
        - commit
        - abort

06.1.2. Partial propagation does not occur.

06.1.3. Commit is atomic.

06.1.4. Abort is atomic.

06.1.5. Commit and abort outcomes are trace-visible.

06.1.6. Commit and abort preserve rollback guarantees.

06.1.7. On commit, staged values propagate to the target authoritative
        layer or to a parent transaction according to nesting rules.

06.1.8. On abort, all staged mutations are discarded deterministically.

06.1.9. A transaction does not remain in an unresolved half-committed
        state.

06.1.10. Commit does not silently drop part of staged mutation.

06.1.11. Abort does not silently preserve part of staged mutation.

06.1.12. A transaction commit does not directly finalize a
         SafePointWindow.

06.1.13. A transaction abort does not directly finalize an AbortedWindow.

06.1.14. Window finalization remains owned by DA-08.


06.2 NESTED TRANSACTION STACKING
--------------------------------

06.2.1. If a pipeline run creates or enters a child session while an
        enclosing pipeline run is active, the child session inherits the
        enclosing transactional view.

06.2.2. Reads resolve first-match from the child transaction downward
        through the inherited stack.

06.2.3. Writes target the child transaction.

06.2.4. Committing the child transaction propagates its writes into the
        parent transaction.

06.2.5. No child transaction commits directly into committed layers while
        an enclosing transaction exists.

06.2.6. Only the outermost transaction commit mutates committed layers.

06.2.7. This stacking rule prevents lost updates and desynchronization
        across nested pipeline activity.

06.2.8. Nested transactional propagation is not itself authoritative
        mutation into committed state.


06.3 COMMIT INTO COMMITTED LAYERS AND CONFLICT DETECTION
--------------------------------------------------------

06.3.1. Commit conflict detection applies only when a transaction attempts
        to commit into committed layers.

06.3.2. A transaction records, per touched key, the committed base revision
        state visible when the key was first read or first written.

06.3.3. The recorded base state includes:
        - whether the key existed in the committed layer at capture time
        - which committed revisionId the key had if it existed

06.3.4. If the key did not exist at capture time, baseRevisionId is the
        absent sentinel.

06.3.5. If the key existed at capture time, baseRevisionId equals the
        committed key's current revisionId at that time.

06.3.6. When committing into committed layers, the commit is in conflict if
        the recorded existence state differs from the committed layer's
        current existence state.

06.3.7. When committing into committed layers, the commit is in conflict if
        the recorded committed revisionId differs from the committed layer's
        current revisionId.

06.3.8. A conflicting commit fails deterministically.

06.3.9. Silent merge does not occur.

06.3.10. Last-write-wins behavior does not occur.

06.3.11. Heuristic reconciliation does not occur.

06.3.12. Any override requires explicit user decision or explicit policy
         defined elsewhere.

06.3.13. Conflict detection is not applied to child-transaction propagation
         into a parent transaction, because that propagation is not yet an
         authoritative mutation.

06.3.14. Turnix does not silently reconcile conflicting authoritative
         mutation.

06.3.15. Turnix does not silently change conflict meaning through fallback,
         guesswork, or convenience behavior.


06.4 STATE AND TRANSACTION TRACE OBLIGATIONS
--------------------------------------------

06.4.1. Authoritative state mutation remains trace-visible.

06.4.2. Transaction commit and abort outcomes remain trace-visible.

06.4.3. State trace evidence distinguishes:
        - staged mutation
        - committed mutation
        - aborted staged mutation
        - computed or runtime-only effect

06.4.4. Computed or runtime-only effects do not appear as committed
        mutations.

06.4.5. For any committed mutation of a memory key, Turnix emits or can
        reconstruct trace evidence with at least:
        - mutationKind
        - appInstanceId
        - memoryLayerId
        - memory key identity
        - baseRevisionId
        - newRevisionId
        - debuggerInitiated
        - transactionId when mutation was committed through a transaction
        - orchestrationUnitId when applicable
        - jobId when applicable

06.4.6. baseRevisionId follows the absent-sentinel and revision rules
        defined in section 04.

06.4.7. Mutation trace evidence is causally linked to:
        - the transaction commit event when a transaction commits the
          mutation
        - the owning Orchestration Unit when such ownership exists
        - the owning Job when such ownership exists
        - the owning trace span when such a span exists

06.4.8. Abort trace evidence does not imply committed mutation.

06.4.9. Commit trace evidence does not by itself define Window finalization.

06.4.10. Transaction trace evidence records enough context, when available,
         to explain:
         - transactionId
         - appInstanceId
         - orchestrationUnitId when applicable
         - jobId when applicable
         - affected committed layers when allowed
         - affected keys or scopes when allowed
         - base revision information when relevant
         - new revision information when relevant
         - abort reason when the transaction aborts

06.4.11. DA-03 defines the tracing substrate used to represent state and
         transaction trace evidence.


===============================================================================
07. ORCHESTRATION UNITS AND MUTATION BOUNDARIES
===============================================================================

07.1. An Orchestration Unit is a kernel-tracked execution unit that is
      permitted to mutate Turnix-managed state within an appInstance.

07.2. An Orchestration Unit is an execution-authority unit.

07.3. A Job is an observable lifecycle handle.

07.4. A transaction is a mutation staging and commit boundary.

07.5. Orchestration Units, Jobs, and transactions are related but distinct.

07.6. Example Orchestration Units include:
      - a pipeline run, including its stage callbacks
      - a capability handler invocation
      - a tooling-initiated mutation transaction
      - configuration mutation request processing

07.7. Work started outside Turnix orchestration is not an Orchestration Unit
      until it re-enters through orchestration boundaries.

07.8. Turnix guarantees apply only to work that occurs within Turnix
      orchestration boundaries or re-enters through them.

07.9. Mutation outside orchestration does not acquire Turnix guarantees by
      implication.

07.10. An Orchestration Unit may be associated with a Job.

07.11. An Orchestration Unit may execute without being associated with a
       Job.

07.12. A Job may coordinate more than one Orchestration Unit only when the
       Job-owning or workflow-owning design explicitly defines that
       coordination.

07.13. An Orchestration Unit may open, commit, or abort transactions under
       the rules of this document.

07.14. An Orchestration Unit outcome does not by itself define Job terminal
       outcome.

07.15. A Job terminal outcome does not by itself define Orchestration Unit
       outcome.

07.16. An Orchestration Unit outcome does not by itself define transaction
       commit or abort.

07.17. A transaction commit or abort does not by itself define the
       Orchestration Unit outcome.


07.1 ORCHESTRATION UNIT OUTCOMES
---------------------------------

07.1.1. An Orchestration Unit reaches exactly one terminal outcome.

07.1.2. Orchestration Unit terminal outcomes are:
        - Completed
        - Failed
        - Cancelled
        - Superseded

07.1.3. Completed means the Orchestration Unit finished its execution path
        without ending as Failed, Cancelled, or Superseded.

07.1.4. Failed means the Orchestration Unit ended because an unexpected or
        error outcome prevented successful completion.

07.1.5. Cancelled means the Orchestration Unit ended because intentional
        cancellation stopped the unit before completion.

07.1.6. Superseded means the Orchestration Unit ended because newer work,
        newer accepted input, or higher-priority work replaced it.

07.1.7. Superseded is not failure.

07.1.8. Superseded is not timeout.

07.1.9. Superseded does not roll back authoritative state that was already
        committed.

07.1.10. Completed may include:
         - committed authoritative mutation
         - no authoritative mutation
         - aborted or discarded transactional staging

07.1.11. The document or workflow that owns a concrete Orchestration Unit
         class may define narrower outcome meanings that remain compatible
         with this section.

07.1.12. An Orchestration Unit terminal outcome is trace-visible.


07.2 ORCHESTRATION UNITS AND TRANSACTIONS
-----------------------------------------

07.2.1. A mutation-bearing Orchestration Unit may open a transaction.

07.2.2. A non-mutating Orchestration Unit does not require a transaction.

07.2.3. A Completed Orchestration Unit may commit a transaction when the
        unit produced authoritative mutation.

07.2.4. A Completed Orchestration Unit may produce no authoritative
        mutation.

07.2.5. A Completed Orchestration Unit may abort or discard transactional
        staging when successful execution required no authoritative
        mutation.

07.2.6. A Failed Orchestration Unit aborts or discards open transactional
        staging according to transaction rules.

07.2.7. A Cancelled Orchestration Unit aborts or discards open
        transactional staging according to transaction rules.

07.2.8. A Superseded Orchestration Unit aborts or discards open
        transactional staging according to transaction rules.

07.2.9. An Orchestration Unit outcome does not convert already committed
        authoritative mutation into uncommitted mutation.

07.2.10. If committed state must be changed after an Orchestration Unit is
         cancelled, failed, or superseded, that change is a later
         authoritative mutation.


07.3 ORCHESTRATION UNITS AND WINDOWS
------------------------------------

07.3.1. Orchestration Unit outcomes occur inside the current active runtime
        region.

07.3.2. Transaction commit or abort evidence may contribute to later Window
        evidence.

07.3.3. Orchestration Unit outcome evidence may contribute to later Window
        evidence.

07.3.4. Orchestration Unit Completed does not mean a SafePointWindow was
        created.

07.3.5. Orchestration Unit Failed, Cancelled, or Superseded does not mean an
        AbortedWindow was created.

07.3.6. SafePointWindow and AbortedWindow finalization remain owned by
        DA-08.


07.4 ORCHESTRATION UNIT TRACE REASON IDENTIFIERS
------------------------------------------------

07.4.1. DA-04 owns the Orchestration Unit trace reason identifiers used for
        Orchestration Unit lifecycle, transaction association, and decision
        waiting inside state-mutating orchestration.

07.4.2. The following Orchestration Unit trace reasons are defined:

        - OrchestrationUnitCreated
        - OrchestrationUnitStarted
        - OrchestrationUnitCompleted
        - OrchestrationUnitFailed
        - OrchestrationUnitCancelled
        - OrchestrationUnitSuperseded
        - OrchestrationUnitTransactionOpened
        - OrchestrationUnitTransactionCommitted
        - OrchestrationUnitTransactionAborted
        - OrchestrationUnitDecisionPending
        - OrchestrationUnitDecisionResolved

07.4.3. OrchestrationUnitCreated is emitted when an Orchestration Unit is
        created.

07.4.4. OrchestrationUnitStarted is emitted when Orchestration Unit
        execution begins.

07.4.5. OrchestrationUnitCompleted is emitted when an Orchestration Unit
        reaches the Completed terminal outcome.

07.4.6. OrchestrationUnitFailed is emitted when an Orchestration Unit
        reaches the Failed terminal outcome.

07.4.7. OrchestrationUnitCancelled is emitted when an Orchestration Unit
        reaches the Cancelled terminal outcome.

07.4.8. OrchestrationUnitSuperseded is emitted when an Orchestration Unit
        reaches the Superseded terminal outcome.

07.4.9. OrchestrationUnitTransactionOpened is emitted when an Orchestration
        Unit opens or enters a transaction.

07.4.10. OrchestrationUnitTransactionCommitted is emitted when a transaction
         associated with an Orchestration Unit commits.

07.4.11. OrchestrationUnitTransactionAborted is emitted when a transaction
         associated with an Orchestration Unit aborts or is discarded.

07.4.12. OrchestrationUnitDecisionPending is emitted when an Orchestration
         Unit is waiting on a decision point that can affect its execution or
         mutation outcome.

07.4.13. OrchestrationUnitDecisionResolved is emitted when such a decision
         point resolves.

07.4.14. Orchestration Unit trace evidence records enough context to
         explain:
         - orchestrationUnitId
         - appInstanceId
         - owner or caller identity when available
         - work kind
         - parent jobId when applicable
         - parent capability identifier when applicable
         - parent RPC message identity when applicable
         - transactionId when applicable
         - terminal outcome when the unit reaches one
         - supersession source when applicable


07.5 SINGLE-LANE APPINSTANCE ORCHESTRATION
------------------------------------------

07.5.1. Turnix uses a deterministic single-lane orchestration model within
        one appInstance unless an explicit concurrency mode is defined
        elsewhere.

07.5.2. Within one appInstance, Turnix does not execute multiple
        Orchestration Units concurrently when those units may mutate the
        same appInstance state.

07.5.3. Awaited input/output or model calls may suspend the current unit.

07.5.4. Suspension does not authorize another state-mutating unit to run
        concurrently in the same appInstance unless an explicit concurrency
        mode permits that behavior.

07.5.5. Work started outside Turnix orchestration remains outside Turnix
        guarantees until it re-enters through orchestration boundaries.

07.5.6. Any mutation that seeks Turnix guarantees re-enters through
        orchestration boundaries before becoming authoritative.

07.5.7. Turnix does not imply concurrency safety merely because execution
        is asynchronous or suspended.

07.5.8. Turnix does not imply determinism for mutation paths that occur
        outside orchestration boundaries.


===============================================================================
08. PERSISTENCE AS PROTECTION, NOT TRUTH
===============================================================================

08.1. Persistence protects committed state against loss.

08.2. Persistence does not define what state truth is.

08.3. Committed state truth is determined by the commit model defined by
      this document.

08.4. Any committed value may be persisted.

08.5. Persistence of committed state occurs under Turnix authority.

08.6. Persistence ordering is controlled by Turnix.

08.7. Persistence failure is trace-visible.

08.8. Persistence does not imply:
      - immediate disk residency
      - continuous in-memory residency
      - unlimited retention

08.9. This section defines the state-model meaning of persistence only. It
      does not redefine the separate persistent I/O authority defined
      elsewhere.

08.10. Persistence failure does not silently redefine committed-state truth.

08.11. Persistence exists to protect committed state, not to decide whether
       committed state exists.


===============================================================================
09. STATIC DATA AND MUTATION INTENT
===============================================================================

09.1. Static data is data provided by:
      - packs
      - configuration defaults
      - bundled resources

09.2. Static data is read-only by default.

09.3. When a mutation is attempted on static data, the intended mutation
      mode is explicit.

09.4. Turnix recognizes exactly two mutation-intent modes for static data:

      A) Patch
      B) Copy

09.5. In Patch mode:
      - the original key remains authoritative
      - a patch is applied on each read
      - the patch is persisted as overlay data

09.6. In Copy mode:
      - a new key is created
      - data is copied and modified
      - the original remains unchanged

09.7. Static-data mutation intent may require explicit user decision or
      policy handling defined elsewhere.

09.8. Static-data mutation intent remains visible and explainable after the
      fact.

09.9. Static data does not become freely mutable merely because a runtime
      actor wants to modify it.

09.10. Turnix does not silently substitute Patch for Copy or Copy for Patch.


===============================================================================
10. REVISIONS AND STATE GARBAGE COLLECTION
===============================================================================

10.1. Committed state values may have revisions.

10.2. Revisions support:
      - rollback
      - audit
      - debugging
      - committed-state history

10.3. Revision retention is policy-driven.

10.4. State garbage collection may remove:
      - older revisions
      - unreachable historical committed states

10.5. State garbage collection does not remove data required to reconstruct
      the latest committed state for each committed key.

10.6. State garbage collection does not occur silently.

10.7. State garbage collection does not violate rollback guarantees that the
      retained policy still claims to preserve.

10.8. Removal of older revisions may make older committed value
      representations unreachable.

10.9. Historical removal does not invalidate the ability to restore the
      latest retained committed state.

10.10. Turnix does not silently discard material that retained-policy
       guarantees still depend on.

10.11. Retention policy may bound history. It does not silently rewrite
       current committed truth.


===============================================================================
11. TOOLING-INITIATED STATE MUTATION CONSTRAINTS
===============================================================================

11.1. Tooling may inspect Turnix-managed memory and committed state through
      appropriate controlled surfaces.

11.2. Tooling-initiated edits occur inside a transaction.

11.3. Tooling-created transactions follow the same commit or abort rules as
      runtime transactions.

11.4. Tooling commits propagate according to normal transaction-layer rules.

11.5. Tooling actions do not finalize committed state in the middle of
      active pipeline work merely because the tooling surface exists.

11.6. Tooling-originated state mutation is labeled distinctly in trace.

11.7. This document defines only the state-contract constraints for
      tooling-initiated mutation.

11.8. Breakpoints, pausing, step control, and debugger execution behavior
      are defined elsewhere.

11.9. Tooling does not bypass normal authority boundaries merely because it
      is privileged tooling.

11.10. Tooling-originated mutation does not become committed truth except
       through the same commit model that governs runtime mutation.


===============================================================================
12. COMMITTED-STATE RESTORATION BOUNDARY
===============================================================================

12.1. Turnix supports deterministic restoration of committed state.

12.2. Deterministic restoration means that committed state can be restored
      from retained authoritative material without requiring
      non-authoritative historical material.

12.3. Deterministic restoration does not imply deterministic reconstruction
      of in-flight runtime activity that never became committed.

12.4. Deterministic restoration does not imply deterministic replay of
      external side effects.

12.5. Restoration of committed state is authoritative.

12.6. Any later resumed or retried work after restoration is new work and is
      not retroactively part of the earlier committed mutation.

12.7. Later workflow concepts such as Replay, Resume, and Retry remain
      constrained by the restoration boundary defined in this section.

12.8. This document defines the restoration boundary only. It does not
      define the full workflow semantics of Replay, Resume, or Retry.

12.9. Turnix does not imply that uncommitted activity becomes reproducible
      merely because committed state is restorable.

12.10. Turnix does not imply that external effects are reversible,
       replayable, or reconstructible merely because committed state can be
       restored.


===============================================================================
13. MIGRATION COMMIT RULE
===============================================================================

13.1. Migration that changes authoritative committed state occurs as normal
      orchestrated work inside a transaction.

13.2. Migration that changes authoritative committed state results in a new
      committed state boundary on success.

13.3. Migration is abortable.

13.4. If migration aborts, partial authoritative mutation does not occur.

13.5. If one migration operation attempts multiple authoritative changes,
      the migration transaction is atomic with respect to the affected
      committed layers.

13.6. If migration cannot complete safely, it aborts and authoritative
      state remains identical to the pre-migration committed state.

13.7. Migration does not bypass the normal commit boundary.

13.8. Migration does not acquire permission to perform partial
      authoritative mutation merely because it is migration work.


===============================================================================
14. WHAT IS EXPLICITLY NOT REQUIRED
===============================================================================

14.1. Turnix does not require:
      - infinite committed history retention
      - journal-backed determinism
      - journal-backed recovery
      - trace-backed correctness
      - concurrency-first execution
      - deterministic reconstruction of uncommitted runtime activity

14.2. Turnix does not imply determinism where determinism does not exist.

14.3. Turnix does not treat observational completeness as a requirement for
      committed-state correctness.

14.4. Turnix does not require non-authoritative history to remain complete
      in order for committed state to remain authoritative.

14.5. Turnix does not require hidden fallback, reconciliation, or
      observational reconstruction in order to preserve committed-state
      correctness.

15. LOCKED SUMMARY
===============================================================================

15.1. LOCKED:
      - runtime memory and committed state are distinct
      - only committed state is authoritative
      - authoritative and non-authoritative material remain separate
      - memory is layered by authority, scope, lifetime, and rollback
        boundary
      - transactions stage speculative mutation and end only in atomic
        commit or atomic abort
      - transaction commit does not directly finalize a SafePointWindow
      - transaction abort does not directly finalize an AbortedWindow
      - nested transaction stacking prevents child authoritative commits
        while an enclosing transaction exists
      - conflict detection applies when committing into committed layers
      - state mutation and transaction outcomes remain trace-visible under
        this document's ownership
      - Orchestration Unit is an execution-authority unit
      - Job is an observable lifecycle handle
      - transaction is a mutation staging and commit boundary
      - Orchestration Units, Jobs, and transactions remain distinct
      - Orchestration Unit terminal outcomes are Completed, Failed,
        Cancelled, and Superseded
      - Superseded Orchestration Unit outcome is not failure
      - Orchestration Unit outcome does not by itself define Job terminal
        outcome, transaction outcome, or Window finalization
      - appInstance mutation follows deterministic single-lane orchestration
        unless another mode is explicitly defined elsewhere
      - persistence protects state against loss but does not define truth
      - static-data mutation intent is explicit as Patch or Copy
      - revision retention is policy-driven and does not invalidate latest
        committed-state restoration
      - tooling-initiated mutation obeys the normal transaction model
      - committed-state restoration is deterministic only within the
        committed-state boundary
      - migration that changes authoritative state is transactional and
        abortable
