Turnix document semantic review helper
=====================================

Purpose
-------
This helper is a reusable semantic-review checklist for specification,
doctrine, workflow, and related document work.

Use this helper when:
- reviewing HEAD vs STAGED for document changes
- checking whether a rewrite preserved meaning
- checking whether a wording change is really non-semantic
- checking for drift after integrating new meaning into an existing document
- preparing a semantic commit message for documentation changes

This helper is operational guidance.
It is not authoritative for Turnix runtime semantics.
Indexed CMR, DD, DA, ODA, and IS documents remain authoritative in their own
scope.

Core principle
--------------
Review meaning first, wording second.
A document change is not non-semantic merely because the wording is clearer,
shorter, more explicit, or more elegant.

Review levels
-------------
Evaluate document changes at three levels:

1. Local rule level
- added, removed, strengthened, weakened, or clarified rules
- changed guarantees
- changed failure handling
- changed allowed or forbidden behavior

2. Document architecture level
- changed scope of the document
- changed authority basis or source of truth
- changed interpretation basis
- changed what the document does or does not define
- changed relationship from standalone meaning to constrained meaning

3. Cross-document dependency level
- added or removed dependency on another indexed document
- stronger or weaker [constrained-by], [defined-in], [ref], or similar links
- changed ownership of a concept across documents
- changed where persistence, authority, lifecycle, validation, or decision
  meaning is defined

What to ignore
--------------
Ignore the following unless they also change meaning:
- DOC_REV changes
- DOC_GIT changes
- section renumbering
- formatting-only edits
- wording changes that improve clarity without changing meaning,
  authority, or interpretation

Re-anchoring is semantic
------------------------
Treat the following as semantic changes:
- a document becomes constrained by another document where it was previously
  self-contained
- a local definition is replaced by a reference to another document
- source-of-truth ownership moves to another document
- persistence, lifecycle, validation, mutation, or decision meaning is moved
  to another document
- document scope is reframed around another doctrine or design area

Unchanged meaning can still be weakened
---------------------------------------
Do not check only whether new meaning was added.
Also check whether valid prior meaning became:
- less direct
- less sharp
- easier to misread
- more abstract than designed
- more likely to be forgotten by a future reader

Narrowest correct term check
----------------------------
Prefer the narrowest correct term.
Before accepting a broader term, ask:
- Is the new term broader than the old one?
- Did the design actually cover that broader class?
- Could a future reader reasonably apply this wording to cases we did not
  design?
- Would that broadening change the contract even if the current examples
  still work?

If the answer is yes or unclear, keep the narrower term.
Do not generalize by vocabulary alone.
A wording cleanup must not silently widen the design domain.

Existing-document preservation check
------------------------------------
For existing documents, identify:
- preserved meaning
- changed meaning
- new meaning
- extended meaning
- valid old meaning that must remain

A rewrite is not good merely because it sounds stronger if it erased correct
prior meaning.

Drift checklist
---------------
After rewriting or integrating, check for:
- softened guarantees
- broadened terms
- changed source of truth
- hidden scope expansion
- accidental new dependency on another document
- lost explicitness
- new ambiguity where the older text was sharper
- hidden merge of several concepts that used to be separate

Suggested output behavior
-------------------------
When reporting review findings, distinguish:
- clearly correct and preserved
- drift or weakening issues
- new semantic changes
- architectural re-anchoring changes
- cross-document consistency risks

Do not collapse all findings into one undifferentiated summary.

Working mantra
--------------
Cleaner wording is not better if it weakens guarantees, broadens scope, or
moves authority silently.

End
---
This helper provides reusable semantic-review guidance only.
It does not define Turnix runtime semantics.
