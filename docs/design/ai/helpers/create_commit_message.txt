- create a conventional commit message, with type, scope, and description
- use "DD-<int>", "DA-<int>", "CMR-<int>", "DD-MN-<int>", "IS-<int>" or "ODA-<int>" found in the file header as the scope
- include a list of changes in the body when applicable

Input interpretation:
- the input may arrive in either of these forms:
  - inline file content, where the HEAD file content follows "HEAD:" and the STAGED file content follows "STAGED:"
  - attached text artifacts, where the first attachment is HEAD and the second attachment is STAGED
- if both inline labels and attachments are present, interpret the actual provided HEAD payload and STAGED payload by order and pairing, not by whether the UI rendered them as plain text or files
- treat attached text artifacts as equivalent to pasted file content

Comparison rules:
- compare HEAD against STAGED
- ignore revision changes
- ignore renumbering of sections
- ignore formatting-only edits
- ignore wording changes that improve clarity without changing meaning, dependency, authority, or interpretation

Semantic-change detection:
- treat as semantic changes not only direct rule additions, removals, or behavior changes, but also architectural re-anchoring changes

Architectural re-anchoring includes cases where a document:
- changes from self-contained semantics to semantics explicitly constrained by another document
- moves its source of truth, authority basis, or interpretation basis to another document
- replaces local definitions with dependency on another design area, doctrine, or meta-rule
- reframes its guarantees, mutation rules, persistence rules, lifecycle rules, validation rules, or interpretation rules as subordinate to another document
- introduces or removes cross-document constraints, references, or dependency wording that materially changes how the document should be interpreted

Treat such re-anchoring as semantic even if most local rules remain textually similar.

Do not classify a change as non-semantic merely because:
- wording became more normative
- structure became cleaner
- most individual rule outcomes appear preserved

Check for semantic changes at three levels:

1. Local rule level
- added, removed, strengthened, weakened, or clarified rules

2. Document-architecture level
- changes in scope, authority basis, interpretation basis, or source-of-truth wording
- changes from standalone semantics to semantics constrained by CMR, DD, DA, ODA, or IS documents
- changes in how this document relates to memory/state layers, transaction rules, migration rules, tracing rules, or other shared architecture

3. Cross-document dependency level
- added or removed [constrained-by], [defined-in], [satisfied-by], [ref] relationships that materially affect interpretation
- stronger or weaker dependency on another document family

If level 2 or level 3 changes occur, do not report "no semantic changes" even when local rule wording appears mostly preserved.

Practical commit-message guidance:
- if wording or structure was substantially rewritten but meaning stayed the same, prefer a commit message such as:
  - rewrite ... in normative prose
  - normalize ... into normative prose
  - restructure ... into explicit normative form
  - clarify ... without changing semantics
- if the conceptual foundation, authority relationship, or dependency on other documents changed, reflect that in the commit message
- if there are truly no semantic changes, you may say so explicitly

Output format:
- output as simple text in a text code block
