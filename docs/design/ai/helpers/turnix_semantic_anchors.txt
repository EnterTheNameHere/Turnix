Turnix semantic anchors helper
==============================

Purpose
-------
This helper records reusable Turnix-specific semantic anchors that should be
preserved when writing, rewriting, or reviewing Turnix documents.

Use this helper when:
- drafting or integrating Turnix design documents
- checking whether a rewrite still sounds like Turnix
- reviewing whether helpful wording accidentally weakened core doctrine
- performing cross-document consistency sweeps

Core anchors
------------
1. Deterministic core vs helpful tooling
- Helpful tooling such as Pack Manager or UI may assist.
- The deterministic loader/kernel/runtime core must not silently guess,
  merge, substitute, or fall back.

2. No silent magic
- no silent fallback
- no silent substitution
- no silent merge
- no hidden best-effort downgrade
- no hidden authority escalation
- no hidden policy creation

3. Whole-unit selection
- When several whole candidates, files, or artifacts exist:
  - one wins deterministically
  - others are shadowed
  - losing candidates do not contribute partial meaning
- Composition must be explicitly designed, not assumed.

4. Trace should show what happened
- Duplication, divergence, selection, and shadowing should be diagnosable.
- Trace or diagnostics should show which candidate won and which lost when
  the surrounding design area requires it.

5. Origin is context, not proof
- Origin, source, root, or layer may be shown as context.
- Origin alone is not authenticity proof unless explicitly designed as such.

6. File-first until redesigned
- If the current design is file-based, document it as file-based.
- Do not silently broaden to generic resources, database objects, in-memory
  objects, or arbitrary inputs unless that broader substrate was designed.

7. Advisory metadata stays advisory
- Advisory metadata for Pack Manager or UI does not silently become loader,
  activation, or runtime semantics.
- If advisory metadata is promoted into enforced semantics, that is an
  explicit semantic change.

8. One authority home per concept
- A concept should have one primary authoritative document.
- Other documents may depend on it or reference it.
- They should not silently take it over or duplicate it inconsistently.

Working use
-----------
When reviewing a Turnix document change, ask:
- Did the rewrite make the core more helpful than deterministic?
- Did any silent fallback or substitution creep in?
- Did wording imply loser-merging where Turnix expects one winner and others
  shadowed?
- Did source or origin begin to sound like authenticity proof?
- Did the rewrite broaden a file-based design into a generic-input design?
- Did advisory metadata start sounding enforceable by loader/runtime?
- Did one document silently take ownership from another?

End
---
This helper provides reusable Turnix-specific semantic anchors only.
It does not define Turnix runtime semantics by itself.
