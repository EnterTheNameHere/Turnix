===============================================================================
OPTIONAL DESIGN AREA ODA-C - TOOLING, AUTHORING & ECOSYSTEM SUPPORT
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines optional tooling, authoring support, and ecosystem-facing support
surfaces that improve author productivity and ecosystem health without
changing Turnix runtime semantics.

This area is optional in tier, but strategically important for long-term
ecosystem health, author productivity, and external adoption.

Purpose:
--------
This document defines optional tooling, authoring, and ecosystem-support
expectations for Turnix.

It defines:
- authoring-experience principles
- optional manifest and config authoring tools
- optional graph-visualization and catalog tooling
- optional adapter declaration and executable-artifact tooling
- optional scaffolding and static-analysis support
- optional PackManager user-experience extensions
- optional documentation-generation and ecosystem-policy support

It does NOT define:
- runtime rules
- conflict-resolution authority
- automatic pack activation or pack enabling
- execution adapter semantics (see DA-23)
- RPC, channel, or transport semantics (see DA-21)
- capability permission semantics (see DA-16)
- silent conflict resolution
- any new correctness guarantees or authority boundaries

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines optional tooling, authoring, and ecosystem-
      support expectations within that shared interpretation model.


===============================================================================
02. CORE PROBLEM
===============================================================================

02.1. Turnix is powerful, but it is also:
      - non-trivial to author for
      - policy-heavy
      - strict about correctness
      - adapter-mediated
      - unfamiliar to many mod authors

02.2. Without tooling support:
      - authors may make avoidable mistakes
      - specifications may be misunderstood
      - good ideas may fail because of friction
      - ecosystem cohesion may suffer

02.3. This area exists to reduce avoidable friction without weakening
      runtime truth or runtime correctness.

02.4. Tooling is allowed to explain and preview runtime rules, but tooling
      does not become a runtime authority source.


===============================================================================
03. AUTHORING-EXPERIENCE PRINCIPLE
===============================================================================

03.1. Turnix should be:
      - strict at runtime
      - more forgiving during development
      - explicit rather than magical
      - explainable rather than silent

03.2. Tooling exists to:
      - surface rules early
      - guide authors
      - prevent invalid states before runtime

03.3. Tooling support does not replace runtime enforcement.

03.4. Tooling previews do not grant execution authorization, capability
      permission, activation success, or persistence authority.


===============================================================================
04. MANIFEST AUTHORING TOOLS (OPTIONAL)
===============================================================================

04.1. Optional manifest-authoring tools may include:
      - manifest schema validation
      - manifest linting
      - dependency sanity checks
      - visibility and export previews
      - executable artifact declaration checks
      - adapter binding previews

04.2. Example diagnostics may include:
      - "this pack cannot see that pack"
      - "this dependency will never resolve"
      - "this export is unreachable"
      - "this executable artifact has no adapter binding"
      - "this adapter kind requires explicit trust policy"

04.3. These tools remain advisory and do not replace runtime checks.

04.4. Manifest tooling does not execute code merely because it recognizes a
      code artifact.


===============================================================================
05. CONFIG AUTHORING AND MIGRATION TOOLS (OPTIONAL)
===============================================================================

05.1. Optional tools may include:
      - config schema editors
      - default-value previews
      - migration simulation from old to new forms
      - diff viewers

05.2. The purpose of such tools is to:
      - reduce config breakage across versions
      - make migrations more predictable

05.3. Migration simulation does not commit authoritative state.


===============================================================================
06. PACK GRAPH VISUALIZATION (OPTIONAL)
===============================================================================

06.1. Optional graph tooling may show:
      - pack hierarchy and tree identity
      - dependency edges
      - visibility boundaries
      - activation order
      - AppPack to SavePack relationships when available
      - AppInstance relationships when viewing runtime evidence
      - adapter participation when trace evidence exists

06.2. Such tooling may be useful for:
      - debugging complex setups
      - education
      - PackManager user-experience enhancements

06.3. Visualized relationships do not create authority or dependency truth
      beyond the underlying evidence.


===============================================================================
07. CAPABILITY CATALOG (OPTIONAL)
===============================================================================

07.1. A structured capability catalog may include:
      - known capabilities
      - versions
      - providers
      - permissions required
      - adapter-facing or RPC-facing usage hints when applicable

07.2. Potential benefits include:
      - discoverability
      - reduced name collisions
      - stronger ecosystem conventions

07.3. Catalog entries do not grant capability permission.


===============================================================================
08. TEMPLATES AND SCAFFOLDING (OPTIONAL)
===============================================================================

08.1. Optional generators may create:
      - a new ModPack
      - a new AppPack
      - a new ViewPack
      - a capability provider
      - a service provider
      - an execution adapter binding example
      - a frontend View example

08.2. Such scaffolds may enforce or encourage:
      - correct manifest shape
      - declared executable artifacts
      - adapter binding declarations
      - canonical entrypoints
      - recommended patterns

08.3. Scaffolds remain starting points, not runtime authority.


===============================================================================
09. STATIC ANALYSIS AND LINTING (OPTIONAL)
===============================================================================

09.1. Optional static-analysis or linting tools may:
      - detect undeclared capability usage
      - detect unreachable exports
      - detect dead code entries
      - detect recognized code artifacts that are not declared executable
      - detect missing adapter binding metadata
      - warn about unsafe in-process adapter trust requirements
      - warn about background execution risks

09.2. Such tooling may run:
      - in development mode
      - as CLI tooling
      - as a pre-publish check

09.3. Static analysis may be incomplete and does not replace runtime
      enforcement.


===============================================================================
10. PACKMANAGER USER-EXPERIENCE EXTENSIONS (OPTIONAL)
===============================================================================

10.1. Optional PackManager user-experience features may include:
      - explaining why a pack is disabled
      - previewing the effect of enabling a pack
      - dry-run activation
      - conflict-resolution walkthroughs
      - showing execution adapter requirements
      - showing trust or policy prompts that may be required later

10.2. These remain user-experience layers over already-defined logic.

10.3. These features do not replace the underlying authority documents.

10.4. Dry-run activation does not become real activation.


===============================================================================
11. DOCUMENTATION GENERATION (OPTIONAL)
===============================================================================

11.1. Optional documentation-generation tooling may derive:
      - pack summaries
      - capability lists
      - dependency trees
      - config options
      - declared executable artifact summaries
      - adapter requirement summaries

11.2. Such output may be derived from:
      - manifests
      - schemas
      - trace metadata where appropriate

11.3. Generated documentation does not replace the manifest, schema, trace,
      or runtime authority source from which it was derived.


===============================================================================
12. ECOSYSTEM POLICY SUPPORT (OPTIONAL)
===============================================================================

12.1. Optional ecosystem-policy support may include:
      - recommended pack sets
      - compatibility badges
      - soft deprecation warnings
      - advisory version constraints
      - advisory adapter trust notes

12.2. Such support remains policy-driven and is never enforced silently by
      virtue of this document alone.

12.3. Ecosystem recommendations do not grant execution authorization or
      capability permission.


===============================================================================
13. WHAT THIS AREA MUST NOT DO
===============================================================================

13.1. This area does not justify:
      - changing runtime rules
      - auto-enabling packs
      - auto-resolving conflicts
      - auto-authorizing execution adapters
      - auto-granting capability permission
      - executing code because a file exists
      - hiding errors for convenience

13.2. Tooling convenience does not override runtime truth.


===============================================================================
14. LOCKED AND OPEN
===============================================================================

14.1. LOCKED:
      - tooling does not change runtime semantics
      - tooling is advisory rather than authoritative
      - runtime correctness wins over tooling convenience
      - code artifact recognition by tooling does not imply execution
      - adapter tooling previews do not grant adapter authority

14.2. OPEN:
      - which tools exist
      - where they live, such as CLI, UI, or external ecosystems
      - how much is bundled versus external
      - exact adapter declaration tooling behavior


===============================================================================
15. RELATION TO OTHER AREAS
===============================================================================

15.1. This area depends on higher documents such as:
      - DA-11 for generic resolution semantics
      - DA-13 for dependency semantics
      - DA-14 for activation
      - DA-16 for capabilities
      - DA-20 for AppInstance runtime identity
      - DA-21 for RPC, channels, and transports
      - DA-23 for execution adapter boundaries
      - ODA-B for diagnostics surfaces

15.2. This area does not replace those documents.
