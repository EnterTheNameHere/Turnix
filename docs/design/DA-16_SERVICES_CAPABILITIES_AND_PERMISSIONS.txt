===============================================================================
DESIGN AREA DA-16 - SERVICES, CAPABILITIES & PERMISSIONS
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how packs interact at runtime without importing each other's code,
and how Turnix mediates authority, power, safety, and Job-backed capability
execution.

This is the primary and only supported cross-pack interaction mechanism.

Purpose:
--------
This document defines the strict, auditable runtime interaction model used
by Turnix for cross-pack wiring and cross-pack execution.

It defines:
- Services as runtime wiring objects
- Capabilities as permissioned, Turnix-mediated operations
- permission classes and decision triggers
- capability identity and version rules
- conflict rules for capability identities
- capability invocation flow and execution modes
- capability relationship to Jobs when execution is Job-backed
- trace and audit requirements for capability calls
- cross-runtime symmetry

It does NOT define:
- importing and module-loading rules (see DA-12)
- activation semantics and execution guarantees (see DA-14)
- persistence rules and file I/O (see DA-07)
- configuration schemas and policy storage formats (see DA-06)
- remembered-decision precedence and user-decision persistence semantics
  (see DA-17)
- tracing schema or storage details (see DA-03)
- Job lifecycle, Job result, Job timeout, or Job cancellation semantics
  (see DA-19)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines services, capabilities, permission boundaries,
      and capability-to-Job correlation within that shared interpretation
      model.


===============================================================================
02. CORE DISTINCTION
===============================================================================

02.1. Turnix defines two orthogonal runtime interaction mechanisms:
      - Services
      - Capabilities

02.2. Services provide shared runtime wiring.

02.3. Capabilities provide named, permissioned operations.

02.4. Services and Capabilities serve different purposes and are not
      conflated.

02.5. Neither Services nor Capabilities implies persistence.

02.6. Persistence remains governed by:
      - DA-07 for file I/O and persistence authority
      - DA-06 for configuration and schema-driven persisted behavior
      - DA-17 for remembered-decision policy handling when user decisions
        are involved

02.7. Capabilities may use Jobs for observable execution lifecycle when the
      capability operation requires Job semantics.

02.8. A Job handle is not a Service.

02.9. A Job handle is not a raw implementation object.


===============================================================================
03. SERVICES - RUNTIME WIRING (LOCKED)
===============================================================================

03.1. A Service is:
      - a concrete runtime object or instance
      - registered by a pack during activation
      - stored in a scoped service registry
      - retrieved explicitly by other packs

03.2. Example:
      - ctx.registerService("llm", client)
      - client = ctx.services.get("llm")

03.3. Locked properties:
      - services are opaque runtime objects
      - service lifetime is bound to the providing pack's activation
      - service name collisions are forbidden
      - services must not perform direct filesystem I/O that bypasses Turnix
        persistence authority
      - services have no permission layer by themselves


===============================================================================
04. SERVICE ACCESS RULES (LOCKED)
===============================================================================

04.1. Service access is explicit.

04.2. Services are not inherited.

04.3. Visibility does not grant service access.

04.4. Service availability is never implicit.

04.5. If a service does not exist:
      - ctx.services.get(name) must fail
      - or must return None only when the API contract explicitly allows
        that behavior

04.6. Services are for trusted runtime wiring only.

04.7. Services are not used to bypass:
      - configuration schemas
      - persistence rules
      - permission checks
      - capability mediation
      - Job lifecycle boundaries


===============================================================================
05. CAPABILITIES - EXECUTION BOUNDARY (LOCKED)
===============================================================================

05.1. A Capability is:
      - a named operation
      - exported by a pack
      - invoked by other packs
      - mediated by Turnix
      - permission-checked

05.2. Capabilities are the only supported cross-pack execution boundary.

05.3. Example export:
      ctx.registerCapability("llm.pipeline@1", handler)

05.4. Example call:
      ctx.callCapability("llm.pipeline@1", args)

05.5. A capability invocation may complete without creating a Job.

05.6. A capability invocation may create or attach to a Job when the
      operation requires observable Job lifecycle semantics.

05.7. A Job-backed capability remains a capability invocation.

05.8. Job-backed execution does not bypass capability mediation.


===============================================================================
06. CAPABILITY IDENTITIES AND VERSIONING (LOCKED)
===============================================================================

06.1. Capability identifiers use the form:
      <name>@<major>

06.2. Examples:
      - "llm.pipeline@1"
      - "http.request@1"
      - "compile.typescript@1"

06.3. Locked rules:
      - capability identity uses major version only
      - incompatible change creates a new major version
      - minor or patch versions are not part of capability identity
      - capability identity is stable and global

06.4. Permission matching for remembered capability decisions uses the exact
      versioned capability identifier.

06.5. A permission remembered for storage.export@1 applies to
      storage.export@1.

06.6. This document does not introduce wildcard, prefix, family, or
      versionless fallback matching for remembered capability decisions.


===============================================================================
07. CAPABILITY EXPORT VS USAGE (LOCKED DISTINCTION)
===============================================================================

07.1. Capability export happens at runtime through ctx.registerCapability.

07.2. Capability export defines authoritative runtime behavior.

07.3. Capability usage happens through ctx.callCapability.

07.4. Capability usage invokes Turnix mediation.

07.5. Manifest capability declarations are optional.

07.6. Manifest capability declarations are advisory only.

07.7. Manifest capability declarations may support:
      - tooling
      - diagnostics
      - user experience

07.8. Manifest intent does not override runtime reality.

07.9. Runtime behavior remains authoritative over manifest intent.


===============================================================================
08. PERMISSIONS MODEL (LOCKED CORE)
===============================================================================

08.1. Each capability declares one permission class.

08.2. Permission class is immutable per major version.

08.3. Permission classes are:
      - none
      - soft
      - hard

08.4. none means the capability is always allowed.

08.5. soft means:
      - user decision may be required
      - allow, deny, for-this-run, and remember-style outcomes may exist

08.6. hard means:
      - explicit user allowance is required
      - the capability is not auto-granted

08.7. Permission classification is capability-defined and is not negotiated
      by callers.

08.8. Decision persistence scope and precedence are defined by DA-17.


===============================================================================
09. CAPABILITY INVOCATION FLOW (LOCKED)
===============================================================================

09.1. When ctx.callCapability(...) is invoked:
      1) Turnix evaluates permission
      2) if allowed, the handler executes or accepted Job-backed execution
         begins
      3) if undecided, execution pauses for decision handling

09.2. During undecided permission handling:
      - a decision surface may be shown
      - user or policy input is obtained
      - the resulting decision is recorded according to DA-17 policy rules
      - execution resumes only after that decision path completes

09.3. Locked rules:
      - Turnix remains responsive while execution is paused for decision
        handling
      - no background mutation may occur while paused
      - execution resumes deterministically from the pause point

09.4. Permission evaluation occurs before capability execution begins.

09.5. Job-backed execution does not skip permission evaluation.


09.1 CAPABILITY EXECUTION MODES AND JOBS
----------------------------------------

09.1.1. Capability invocation may use direct execution or Job-backed
        execution.

09.1.2. Direct execution means the capability handler completes the
        capability operation in the immediate invocation path.

09.1.3. Job-backed execution means the capability operation is represented
        by or attached to a Job lifecycle owned by DA-19.

09.1.4. Not every capability invocation creates a Job.

09.1.5. Short or immediate capability operations may complete directly.

09.1.6. Long-running capability execution may create or attach to a Job.

09.1.7. Cancellable capability execution may create or attach to a Job.

09.1.8. Progress-reporting capability execution may create or attach to a
        Job.

09.1.9. Timeout-managed capability execution may create or attach to a Job.

09.1.10. A capability invocation that creates or attaches to a Job remains
         subject to the same permission model as a direct capability
         invocation.

09.1.11. A Job-backed capability returns or exposes only the lifecycle
         surface defined for that operation.

09.1.12. A Job-backed capability does not expose service internals.

09.1.13. A Job-backed capability does not expose raw implementation objects.

09.1.14. A Job-backed capability does not grant callers direct access to the
         underlying execution mechanism.

09.1.15. Job lifecycle semantics, terminal outcomes, cancellation, timeout,
         supersession, progress, and late completion handling are defined by
         DA-19.

09.1.16. Capability semantics continue to own permission, mediation, and
         cross-pack execution boundary behavior.


===============================================================================
10. USER DECISIONS AND POLICY (LOCKED)
===============================================================================

10.1. User decisions may be persisted.

10.2. Remembered capability decisions may be scoped as:
      - run
      - save
      - global

10.3. User decisions remain explainable.

10.4. User decisions remain reversible.

10.5. Policy may define:
      - auto-allow behavior for soft permissions
      - never-auto-allow behavior for hard permissions

10.6. All decisions remain:
      - traceable
      - attributable
      - replayable within the meaning of the governing tracing and policy
        documents

10.7. DA-17 defines:
      - remembered-decision precedence
      - run/save/global evaluation order
      - exact remembered-decision key semantics
      - policy persistence behavior


===============================================================================
11. SERVICES VS CAPABILITIES - STRICT SEPARATION (LOCKED)
===============================================================================

11.1. Services provide direct object access for runtime wiring.

11.2. Services do not perform permission checks by themselves.

11.3. Capabilities provide an explicit execution boundary.

11.4. Capabilities represent user-visible or policy-visible power.

11.5. Capabilities remain permissioned and audited.

11.6. Locked rules:
      - capabilities must not expose service internals directly as a bypass
        path
      - one pack must not call another pack's service-provided behavior
        directly across pack boundaries
      - cross-pack execution must go through capability mediation
      - Job-backed capability execution does not expose service internals
      - Job handles do not provide direct service access


===============================================================================
12. COMPUTED AND REACTIVE BEHAVIOR (LOCKED CLARIFICATION)
===============================================================================

12.1. Packs may react to runtime state or configuration changes by:
      - observing values
      - computing derived behavior
      - adjusting runtime output

12.2. Locked rules:
      - such behavior remains runtime-only
      - such behavior does not mutate stored configuration or memory by
        itself
      - such behavior does not intercept persistence
      - such behavior does not replace the authoritative source of truth

12.3. Allowed mechanisms include:
      - capability invocation
      - service cooperation
      - config or watch-style notifications defined elsewhere

12.4. This preserves:
      - determinism
      - reversibility
      - non-invasive override behavior such as accessibility scaling


===============================================================================
13. CONFLICTS AND OVERRIDES (LOCKED)
===============================================================================

13.1. Multiple packs exporting the same capability identity are not allowed
      by default.

13.2. Such conflict is detected explicitly.

13.3. Resolution requires:
      - explicit user choice
      - or hard failure if no permitted resolution path exists

13.4. Silent override is forbidden.


===============================================================================
14. TRACE AND AUDIT (LOCKED)
===============================================================================

14.1. Capability calls remain trace-visible.

14.2. Capability-call trace evidence includes:
      - caller pack identity
      - target capability identifier
      - exact versioned capability identity used for permission matching
      - permission classification
      - permission decision when one occurs
      - invocation mode
      - execution outcome
      - jobId when a Job is created or attached
      - duration or elapsed execution evidence when available

14.3. Capability-call trace evidence distinguishes:
      - denied execution
      - deferred execution while waiting on decision handling
      - direct successful execution
      - direct failed execution
      - Job-backed execution accepted by the capability boundary

14.4. In this section, deferred execution means waiting on decision handling
      before the capability handler is allowed to run.

14.5. Job-backed execution acceptance is not a Job terminal outcome.

14.6. Job-backed execution acceptance is not capability success by itself
      unless the operation's contract defines successful acceptance as the
      direct capability result.

14.7. When a capability invocation creates or attaches to a Job, the Job
      lifecycle trace remains causally linked to the capability invocation
      trace evidence through DA-03 trace DAG parentage.

14.8. When a capability invocation creates or attaches to a Job, the Job
      lifecycle itself is traced through DA-19 Job-domain trace reasons.

14.9. DA-16 owns the capability-domain trace reason identifiers used for
      capability calls, permission evaluation, deferred execution, denied
      execution, direct successful execution, and direct failed execution.

14.10. The following capability-domain trace reasons are defined:

       - CapabilityCallRequested
       - CapabilityPermissionEvaluated
       - CapabilityExecutionDenied
       - CapabilityExecutionDeferred
       - CapabilityExecutionStarted
       - CapabilityExecutionSucceeded
       - CapabilityExecutionFailed

14.11. CapabilityCallRequested is emitted when a pack invokes a capability
       through Turnix mediation.

14.12. CapabilityPermissionEvaluated is emitted when Turnix evaluates the
       permission classification and applicable decision state for a
       capability call.

14.13. CapabilityExecutionDenied is emitted when a capability call is
       refused before handler execution.

14.14. CapabilityExecutionDeferred is emitted when a capability call waits
       on decision handling before handler execution.

14.15. CapabilityExecutionStarted is emitted when the capability handler is
       allowed to begin execution or accepted Job-backed execution begins.

14.16. CapabilityExecutionSucceeded is emitted when direct capability
       execution completes successfully, or when the capability contract
       defines successful Job acceptance as the direct capability result.

14.17. CapabilityExecutionFailed is emitted when direct capability execution
       fails after execution begins, or when Job-backed acceptance fails
       before a Job lifecycle exists.

14.18. Capability-call trace evidence does not expose service internals as a
       bypass path.

14.19. Capability-call trace evidence is required for:
       - debugging
       - trust
       - reproducibility
       - audit

14.20. DA-03 defines the tracing substrate used to represent capability-call
       evidence.

14.21. DA-17 defines decision trace obligations when capability permission
       decisions are required.

14.22. DA-19 defines Job lifecycle trace obligations when capability
       execution creates or attaches to a Job.


===============================================================================
15. DEVELOPMENT-TIME DIAGNOSTICS (LOCKED)
===============================================================================

15.1. In development mode, Turnix reports:
      - declared-but-unused capabilities
      - used-but-undeclared capabilities
      - exported-but-never-called capabilities

15.2. These reports are:
      - informational
      - non-fatal
      - tooling-oriented


===============================================================================
16. CROSS-RUNTIME SYMMETRY (LOCKED)
===============================================================================

16.1. The same interaction model applies across supported runtimes such as
      Python and JavaScript.

16.2. Python packs may expose capabilities used by JavaScript packs.

16.3. JavaScript packs may invoke Python logic through capabilities.

16.4. Direct cross-language object access is not part of this model.

16.5. Capabilities remain the universal mediated execution boundary.

16.6. Job-backed capability execution does not change cross-runtime
      boundary rules.


===============================================================================
17. EXPLICITLY OPEN QUESTIONS
===============================================================================

17.1. OPEN questions include:
      - fine-grained permission scopes beyond the current run/save/global
        decision model
      - capability discovery and browsing user interfaces
      - dynamic permission revocation
      - subscription-based or streaming capabilities
      - exact capability API form for Job-backed execution
      - exact returned Job handle representation
      - exact distinction between successful Job acceptance and final Job
        success in concrete capability APIs


===============================================================================
18. LOCKED SUMMARY
===============================================================================

18.1. LOCKED:
      - services are runtime wiring only
      - capabilities are the sole supported cross-pack execution boundary
      - capability invocation may complete directly or become Job-backed
      - not every capability invocation creates a Job
      - Job-backed capability execution remains capability-mediated
      - Job-backed capability execution does not expose service internals
      - Job handle is not a Service or raw implementation object
      - Job lifecycle semantics are owned by DA-19
      - the permission model is explicit and immutable per major version
      - remembered capability permission matching uses exact versioned
        capability identity
      - services and capabilities do not imply persistence
      - silent overrides are forbidden
      - capability calls remain fully trace-visible and auditable
      - capability trace evidence includes jobId when a Job is created or
        attached
      - Job lifecycle trace is causally linked to capability invocation trace
        when a capability creates or attaches to a Job
      - capability-domain trace reason identifiers are owned by DA-16

18.2. OPEN:
      - user experience, policy, streaming, and Job-backed API extensions
