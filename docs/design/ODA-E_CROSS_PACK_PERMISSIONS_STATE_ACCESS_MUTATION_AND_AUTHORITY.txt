===============================================================================
OPTIONAL DESIGN AREA ODA-E - CROSS-PACK PERMISSIONS, STATE ACCESS,
MUTATION & AUTHORITY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines an optional cross-pack state-access and mutation-authority model for
state owned by the kernel or by packs.

This area is cross-cutting in topic, but optional in tier. It does not
override higher-authority DA documents.

Purpose:
--------
This document defines an optional ownership-and-access model for cross-pack
state access, mutation, observation, and derived-value handling.

It defines:
- a state ownership model
- scope categories for kernel, pack-global, and save-bound state
- access-level concepts
- separation between read authority and write authority
- computed and derived-value handling
- cross-pack intentional modification requirements
- trace and audit expectations for state access

It does NOT define:
- filesystem access (see DA-07)
- service or capability invocation mechanics (see DA-16)
- dependency resolution or visibility semantics
- final authoritative memory, state-layer, or journaling semantics
- final authoritative config-schema representation for access metadata

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines an optional cross-pack state-access and
      mutation-authority model within that shared interpretation model.


===============================================================================
02. CORE INTENT
===============================================================================

02.1. This model exists to:
      - prevent accidental cross-pack corruption
      - preserve determinism and reproducibility
      - make authority explicit and traceable
      - avoid pretending that hostile local behavior can be fully prevented

02.2. This model exists to:
      - prevent mistakes
      - surface intent
      - make responsibility explicit

02.3. This model does not attempt:
      - sandboxing adversarial code
      - secrecy against a local user
      - moral or trust enforcement

02.4. This document remains subordinate to higher-authority documents that
      define the final state, memory, config, permission, and tracing model.


===============================================================================
03. WHAT THIS DOCUMENT GOVERNS
===============================================================================

03.1. This document governs an optional model for cross-pack access to:
      - kernel-owned state
      - pack-global state
      - pack-local save-bound state
      - computed or derived effective values

03.2. This document considers the following operations:
      - read
      - write or mutation
      - delete
      - observation or watching


===============================================================================
04. STATE OWNERSHIP MODEL (LOCKED)
===============================================================================

04.1. In this model, every state key has exactly one authority owner.

04.2. The owner is either:
      - the kernel
      - a specific pack

04.3. Ownership is:
      - defined by domain and key path
      - not inferred dynamically
      - immutable unless changed by migration or a higher-authority
        state-transition rule

04.4. This model does not use a shared undifferentiated global key bag.


===============================================================================
05. STATE SCOPES (LOCKED)
===============================================================================

05.1. This model recognizes the following scopes:
      - Kernel scope
      - Pack-global scope
      - Pack-local save-bound scope
      - Computed or effective layer


05.1 KERNEL SCOPE
-----------------

05.1.1. Kernel scope:
        - is owned by the kernel
        - is stored in userdata in this model
        - exists independently of packs and saves
        - is accessible without ctx where the higher API permits that

05.1.2. Examples include:
        - UI language
        - global UI scale
        - developer-mode toggles


05.2 PACK-GLOBAL SCOPE
----------------------

05.2.1. Pack-global scope:
        - is owned by a specific pack
        - is stored in userdata in this model
        - is shared across all SavePacks
        - available after pack indexing

05.2.2. Examples include:
        - UI theme defaults
        - code-formatting preferences
        - accessibility base parameters


05.3 PACK-LOCAL SAVE-BOUND SCOPE
--------------------------------

05.3.1. Pack-local save-bound scope:
        - is owned by a specific pack
        - is stored in a SavePack in this model
        - exists only when a save is loaded
        - is accessed through pack-aware runtime context

05.3.2. Examples include:
        - game difficulty
        - per-save gameplay modifiers
        - save-specific UI overrides


05.4 COMPUTED OR EFFECTIVE LAYER
--------------------------------

05.4.1. The computed or effective layer:
        - is not an ownership scope
        - is derived at read time or runtime-evaluation time
        - is not persisted directly

05.4.2. Examples include:
        - accessibility scaling applied to UI font size
        - late runtime transforms
        - temporary overrides


===============================================================================
06. ACCESS LEVELS (LOCKED)
===============================================================================

06.1. In this model, each state key declares an access level.

06.2. Access level is metadata that is:
      - declared in schema
      - immutable unless changed through migration or another higher-
        authority schema-evolution rule
      - enforced uniformly across all APIs that honor this model

06.3. Levels in this model are:
      - PUBLIC
      - PROTECTED
      - PRIVATE
      - INTERNAL

06.4. PUBLIC means:
      - readable by any pack that can address the scope
      - writable only by the owner unless other authority is granted

06.5. PROTECTED means:
      - readable by any pack
      - writable only with explicit permission or ownership

06.6. PRIVATE means:
      - readable and writable only by the owner
      - hidden or redacted by default in user-facing surfaces and traces

06.7. INTERNAL means:
      - readable and writable only by the owner and the kernel
      - not discoverable by default through enumeration
      - redacted by default in exports

06.8. These access levels are intended to prevent mistakes.
      They do not claim secrecy against the local user.


===============================================================================
07. READ AUTHORITY VS WRITE AUTHORITY (LOCKED)
===============================================================================

07.1. This model separates read authority from write or delete authority.


07.1 READING
------------

07.1.1. Reading is permissive by default in this model.

07.1.2. Restrictions exist only where needed to prevent misuse, probing, or
        violation of explicit access-level rules.

07.1.3. If access is denied for any reason:
        - the API returns NotFoundOrPermissionDenied
        - the distinction is not exposed programmatically
        - trace records the real reason

07.1.4. No silent fallback occurs.


07.2 WRITING AND DELETING
-------------------------

07.2.1. All mutation requires authority.

07.2.2. Delete is explicit and is treated as a write-class operation.

07.2.3. Subtree deletion may be supported.

07.2.4. Mutation authority may come from:
        - ownership
        - explicit permission
        - user-mediated decision where the governing model allows it

07.2.5. Unauthorized mutation does not occur.

07.2.6. Unauthorized mutation emits a trace event.

07.2.7. Unauthorized mutation may trigger a user decision only where a
        higher-authority decision-handling model permits that recovery path.


===============================================================================
08. MUTATION RULES (LOCKED)
===============================================================================

08.1. Defaults are not mutated.

08.2. Schemas are not mutated.

08.3. Only override layers are written in this model.

08.4. All writes:
      - go through IoManager or the higher-authority persistence path
      - are atomic where the underlying authority model requires atomicity
      - produce trace events
      - may produce backups
      - may trigger recovery handling when higher documents permit that


===============================================================================
09. COMPUTED AND DERIVED VALUES (LOCKED)
===============================================================================

09.1. Computed layers:
      - react to base-value changes
      - do not overwrite persisted values
      - are applied late
      - are not stored directly

09.2. Removal of a computed layer:
      - reverts behavior automatically
      - does not require migration by itself

09.3. Computed layers:
      - declare dependencies
      - remain traceable
      - avoid cycles


===============================================================================
10. REACTION AND OBSERVATION (LOCKED)
===============================================================================

10.1. Packs may:
      - watch values
      - react to changes
      - recompute derived values

10.2. Packs do not:
      - intercept reads
      - replace owners
      - suppress other observers

10.3. Observation remains additive rather than exclusive.


===============================================================================
11. CROSS-PACK INTENTIONAL MODIFICATION (LOCKED)
===============================================================================

11.1. If Pack B modifies state owned by Pack A:
      - Pack B depends on Pack A where the higher design requires that
        relationship
      - the target owner domain is explicit
      - the permission model applies
      - trace records the intent and outcome

11.2. This is explicit cooperation rather than an implicit override.


===============================================================================
12. ENUMERATION AND DISCOVERABILITY (LOCKED)
===============================================================================

12.1. Enumeration of keys is explicit rather than implicit.

12.2. INTERNAL keys do not appear by default.

12.3. PRIVATE keys appear only to the owner and to kernel tooling where the
      higher diagnostic model permits that access.

12.4. Enumeration exists for tooling rather than for probing.


===============================================================================
13. USER VISIBILITY AND DEFAULT REDACTION (LOCKED)
===============================================================================

13.1. Users may inspect all state through kernel tools.

13.2. However:
      - PRIVATE and INTERNAL values are redacted by default
      - traces redact raw values for PRIVATE or INTERNAL keys by default
      - exports redact sensitive values unless explicitly included through
        an allowed path

13.3. This model does not claim secrecy from the user.
      It defines leak-minimizing defaults for normal surfaces.


===============================================================================
14. TRACE AND AUDIT REQUIREMENTS (LOCKED)
===============================================================================

14.1. Cross-pack state access emits trace events including:
      - caller identity
      - owner domain
      - key path
      - access level
      - operation
      - outcome
      - reason

14.2. Raw values are excluded or redacted according to access level and the
      governing tracing model.


===============================================================================
15. FAILURE SEMANTICS (LOCKED)
===============================================================================

15.1. Denied access results in:
      - deterministic behavior
      - no partial mutation
      - no silent override

15.2. Errors remain:
      - traceable
      - explainable
      - reproducible

===============================================================================
17. OPEN QUESTIONS
===============================================================================

17.1. OPEN questions include:
      - how access-level metadata is represented in the schema dialect
      - policy granularity for write permissions
      - enumeration capability shape and scope

17.2. Watchers and observers are accepted conceptually, but the concrete
      watcher model remains OPEN.

17.3. OPEN watcher and observer questions include:
      - exact public API shape
      - registration and unregistration model
      - whether watching is by exact key, subtree, scope, owner domain, or
        predicate
      - whether notification defaults to committed mutation only
      - whether staged mutation may be observed explicitly
      - how transaction boundaries affect notification visibility
      - ordering guarantees between multiple watchers
      - whether ordering is deterministic and what defines it
      - sync versus async delivery semantics
      - batching, queueing, scheduling, or inline execution behavior
      - whether watcher logic may mutate state directly or only through
        normal orchestrated paths
      - cycle prevention, cycle classification, and cycle diagnostics
      - whether recomputation watchers differ from ordinary observers
      - failure isolation and whether one failing watcher affects others
      - trace expectations for registration, delivery, skipped delivery,
        suppression, and failure
      - visibility and permission constraints on observation
      - whether PRIVATE or INTERNAL values may be watched by non-owners
      - debugger and tooling interaction with watcher notifications
      - activation, deactivation, and reload lifecycle rules

17.4. Computed and effective behavior is accepted conceptually, but the
      concrete computed-handler model remains OPEN.

17.5. OPEN computed-handler questions include:
      - exact registration model
      - whether handlers attach by exact key, subtree, scope, owner domain,
        schema rule, or another declared dependency model
      - whether handlers execute on read, on dependency change, or both
      - caching semantics
      - invalidation semantics
      - ordering when multiple transforms or derivations exist
      - cycle handling and cycle diagnostics
      - whether handlers may expose derived values under the same key or
        only through explicit effective-layer resolution
      - permission and visibility rules for computed inputs and outputs
      - trace shape for computation, cache hit, invalidation, skipped
        computation, and failure
      - failure behavior when a handler errors
      - whether computed handlers may perform cross-pack reads freely or
        only within declared dependencies
      - interaction with staged versus committed mutation
      - debugger and tooling interaction with computed values

17.6. Transform ordering remains OPEN.

17.7. OPEN transform-ordering questions include:
      - whether ordering is declaration-based, priority-based, dependency-
        based, or another deterministic model
      - whether equal-priority transforms are allowed
      - tie-break behavior
      - whether transform ordering is global, per scope, per owner, or per
        key lineage
      - whether ordering is static after registration or may change during
        runtime
      - trace expectations for chosen transform order

17.8. Restriction metadata beyond configuration keys remains OPEN.

17.9. OPEN restriction-metadata questions include:
      - whether future non-config memory or state domains use schema-
        declared access metadata
      - whether a separate metadata model is required for non-config state
      - how such metadata would attach to non-config keys, values, or
        records
      - whether the same PUBLIC, PROTECTED, PRIVATE, and INTERNAL model
        applies unchanged outside configuration
      - migration and compatibility handling for such metadata

17.10. Cross-runtime transport details for these rules remain OPEN.

17.11. OPEN cross-runtime transport questions include:
       - how read, write, delete, watch, and computed-state operations are
         represented across runtime boundaries
       - how authority, owner domain, and access-level metadata are carried
         in transport envelopes where needed
       - how staged versus committed state distinctions are surfaced across
         runtimes
       - how denied access, missing state, and trace correlation are carried
         across runtime boundaries
       - how watcher delivery, backpressure, cancellation, and ordering are
         preserved when transport is involved
       - how effective or computed values are represented without implying
         authoritative stored state
       - which details belong in implementation specifications versus higher
         semantic documents

17.12. This document does not claim that no other design questions exist
       globally. It states only that these are the major open questions
       inside this optional model.


===============================================================================
18. LOCKED SUMMARY
===============================================================================

18.1. LOCKED:
      - explicit ownership
      - a scoped state model
      - schema-declared access levels
      - permissive reads and guarded writes
      - non-persistent computed layers
      - IoManager-mediated or equivalent authoritative mutation
      - trace-first enforcement
      - leak-minimizing diagnostics

18.2. OPEN:
      - exact schema representation details
      - permission-granularity details
      - enumeration-shape details
      - watcher and observer mechanics
      - computed-handler mechanics
      - transform ordering
      - restriction metadata beyond configuration keys
      - cross-runtime transport details for these rules
