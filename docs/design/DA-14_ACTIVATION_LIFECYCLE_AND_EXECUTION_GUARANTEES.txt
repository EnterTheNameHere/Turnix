===============================================================================
DESIGN AREA DA-14 - ACTIVATION, LIFECYCLE & EXECUTION GUARANTEES
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines when pack code is permitted to execute under Turnix-recognized
activation, under whose authority execution occurs, and which execution
behaviors Turnix explicitly guarantees or does not guarantee.

Activation is the sole mechanism that transitions a pack from inert metadata
to an executable participant under Turnix control.

Purpose:
--------
This document defines the strict boundary between inert pack metadata and
Turnix-recognized execution.

It defines:
- lifecycle phase ordering and the activation boundary
- the meaning of Activated state and PackContext authority
- pack-level activation semantics and the canonical entrypoint
- relationship between activation and execution adapters
- the separation between activation and later runtime execution
- rules for immediate execution and import-time side effects
- JavaScript runtime entry semantics relevant to activation
- orchestrated execution pathways and guarantees
- activation trace obligations
- failure behavior, including NotActivatedError requirements

It does NOT define:
- generic resolution mechanics (see DA-11)
- dependency semantics and source selection (see DA-13)
- importing and module-loading boundaries (see DA-12)
- execution adapter boundary semantics (see DA-23)
- tracing schema and storage (see DA-03)
- capability schemas and permission prompting details (see DA-16)
- user decision and remembered policy handling (see DA-17)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines activation, lifecycle, and execution-guarantee
      rules within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. Activation answers exactly one question:

          "Has Turnix intentionally brought this pack into recognized
           executable participation under Turnix orchestration?"

02.2. Activation is a controlled, explicit, one-time state transition.

02.3. Before activation:
      - a pack exists only as metadata
      - its code has no execution authority under Turnix
      - adapter loading or native module evaluation does not make it active

02.4. After activation:
      - the pack may participate in runtime behavior
      - execution is allowed only through Turnix-controlled orchestration
        pathways
      - PackContext defines the pack-facing authority context

02.5. Activation is not:
      - discovery
      - indexing
      - dependency resolution
      - importing
      - adapter selection
      - execution authorization
      - adapter loading
      - visibility granting
      - capability permission

02.6. Those earlier phases may occur before activation, but none of them
      implies activation.

02.7. DA-23 owns the execution adapter boundary in which adapter-mediated
      activation occurs.


===============================================================================
03. LIFECYCLE PHASES (LOCKED)
===============================================================================

03.1. Turnix operates in logically ordered, externally observable phases:

      1) Discovery
      2) Indexing
      3) Resolution
      4) Execution adapter preparation where applicable
      5) Activation
      6) Running or Finalized

03.2. Activation is the first moment Turnix may intentionally execute pack
      code with PackContext authority.

03.3. Execution adapter preparation may include adapter selection,
      execution authorization, adapter creation, adapter readiness, and
      artifact loading.

03.4. Execution adapter preparation is not activation.

03.5. After activation, pack code may execute during Running only through
      Turnix-orchestrated execution pathways defined by this document and
      related higher documents.

03.6. Code that executes outside Turnix orchestration remains outside Turnix
      guarantees.


===============================================================================
04. WHAT ACTIVATED MEANS (LOCKED)
===============================================================================

04.1. When a pack is activated:
      - Turnix creates a PackContext for that pack instance
      - Turnix dispatches the pack's activation entrypoint
      - the pack may participate in Turnix-managed runtime behavior

04.2. When the activated pack's executable artifact belongs to an adapter
      runtime, activation dispatch occurs through the appropriate execution
      adapter under DA-23.

04.3. Activated packs may, for example:
      - register services
      - register capabilities
      - subscribe to Turnix events
      - allocate runtime resources
      - participate in Turnix-managed orchestration

04.4. Before activation:
      - pack metadata may be resolved
      - the pack may appear in graphs
      - adapter selection may have occurred
      - artifact loading may have occurred
      - pack code must not execute under Turnix authority


===============================================================================
05. PACK-LEVEL ACTIVATION (LOCKED)
===============================================================================

05.1. Activation applies to the pack as a whole.

05.2. Activation does not apply to:
      - individual files
      - individual modules
      - individual imports
      - individual capabilities
      - individual adapter-loaded artifacts

05.3. A pack is either activated or not activated.

05.4. Partial, staged, deferred, or per-file activation is not defined by
      this document.

05.5. Adapter loading does not create partial activation.


===============================================================================
06. PACKCONTEXT (ctx) - EXECUTION AUTHORITY (LOCKED)
===============================================================================

06.1. Turnix provides exactly one pack-facing execution-authority context to
      pack code:
      PackContext (ctx).

06.2. PackContext is created only by Turnix.

06.3. PackContext is bound to exactly one pack instance.

06.4. PackContext represents execution authority on behalf of that pack.

06.5. PackContext is not a raw backend service object exposed to pack code.

06.6. PackContext is not unrestricted access to Kernel internals.

06.7. PackContext operations route through Kernel-owned boundaries.

06.8. Code executing with a PackContext:
      - executes under the authority of that pack
      - is subject to that pack's visibility, permissions, and policy
      - remains constrained by capability, transaction, persistence, trace,
        adapter, and lifecycle rules

06.9. PackContext may be represented differently in different adapters.

06.10. Representation difference does not change authority meaning.

06.11. In-process PackContext calls do not bypass capability mediation,
       policy, transaction, trace, persistence, or lifecycle rules.

06.12. Passing ctx to other code is an explicit delegation of authority.

06.13. The delegating pack remains fully responsible for actions performed
       using its ctx.


===============================================================================
07. ACTIVATION ENTRYPOINT (LOCKED)
===============================================================================

07.1. The sole activation entrypoint is:
      onLoad(ctx)

07.2. Locked rules:
      - onLoad(ctx) is called exactly once per pack instance
      - onLoad(ctx) is called only by Turnix
      - ctx is always provided by Turnix
      - failure aborts activation of that pack

07.3. No other function, construct, or mechanism activates a pack.

07.4. Adapter implementations do not introduce alternate activation
      entrypoints by themselves.

07.5. Future activation entrypoints, if introduced, follow the same rules
      and are explicitly defined by Turnix.


===============================================================================
08. ACTIVATION VS EXECUTION (CRITICAL DISTINCTION)
===============================================================================

08.1. Activation and execution are distinct concepts.

08.2. Activation:
      - is a one-time state transition
      - establishes PackContext authority
      - occurs exclusively through onLoad(ctx)
      - may dispatch through an execution adapter when applicable

08.3. Execution:
      - occurs only after activation when Turnix-recognized
      - may happen multiple times
      - occurs only when invoked by Turnix orchestration

08.4. No execution mechanism activates a pack.

08.5. No execution mechanism substitutes for onLoad(ctx).

08.6. Adapter execution without PackContext authority is not
      Turnix-recognized activated execution.


===============================================================================
09. IMMEDIATE EXECUTION AND LANGUAGE CONSTRUCTS (LOCKED)
===============================================================================

09.1. Some languages permit immediate execution constructs, including:
      - Immediately Invoked Function Expressions (IIFE)
      - module-level executable statements
      - top-level side effects

09.2. Turnix does not rely on, endorse, or coordinate immediate execution.

09.3. Immediate execution is discouraged in pack code.

09.4. Immediate execution does not activate a pack.

09.5. Immediate execution runs outside Turnix orchestration.

09.6. Code that executes immediately on import:
      - runs without PackContext authority
      - is not tracked by Turnix
      - is outside lifecycle guarantees
      - is not recoverable or abortable by Turnix guarantees

09.7. Such execution may technically occur, but its behavior is unsupported.

09.8. All consequences of such execution are the responsibility of the pack
      author.

09.9. These rules apply uniformly to:
      - Python
      - JavaScript
      - future supported languages

09.10. Same-language execution does not make immediate execution supported.

09.11. Adapter loading does not make immediate execution supported.


===============================================================================
10. RUNTIME ENTRIES - JAVASCRIPT SEMANTICS (LOCKED)
===============================================================================

10.1. Relevant manifest fields include:
      - runtimes.javascript.entry
      - runtimes.javascript.entries

10.2. runtimes.javascript.entry:
      - is imported by Turnix during activation
      - provides the module from which onLoad(ctx) is invoked
      - may be loaded through the browser JavaScript adapter
      - successful completion marks JavaScript activation success

10.3. runtimes.javascript.entries:
      - are never imported automatically by Turnix as activation entries
      - never receive ctx automatically
      - never have onLoad(ctx) invoked
      - are treated strictly as library modules

10.4. DA-23 owns the general adapter boundary that applies to JavaScript
      runtime entries.


===============================================================================
11. ENTRY VS ENTRIES - EXECUTION RULES (LOCKED)
===============================================================================

11.1. If runtimes.javascript.entry exists:
      - it must execute before pack code relies on activation state
      - entries must not assume activation unless caller logic enforces that

11.2. If runtimes.javascript.entry does not exist:
      - no JavaScript code is executed during activation
      - the pack may still activate based on non-JavaScript activation rules

11.3. If only runtimes.javascript.entries exist:
      - the pack is treated as a library-only pack
      - all such modules remain activation-independent

11.4. Library-only module loading remains distinct from activation.


===============================================================================
12. ACTIVATION-INDEPENDENT CODE (LOCKED)
===============================================================================

12.1. Activation-independent code:
      - must not assume its own pack is activated
      - must not implicitly depend on ctx
      - must not perform Turnix side effects at module top level

12.2. Such code may:
      - accept a PackContext explicitly as a parameter
      - execute under the authority of the caller pack that provides that ctx

12.3. In that case:
      - the caller pack owns responsibility
      - the authority is that of the caller, not the provider

12.4. If code requires activation-specific state of its own pack, that code
      is initialized from that pack's activation entry.

12.5. Passing PackContext into activation-independent code does not change
      the provider pack's activation state.


===============================================================================
13. ACTIVATION ORDER (LOCKED RESPONSIBILITY)
===============================================================================

13.1. Activation order is determined exclusively by:
      - PackManager
      - resolved dependencies
      - visibility constraints
      - user policy
      - future category-based ordering if later accepted

13.2. Packs must not assume:
      - another pack is activated unless explicitly required
      - services exist unless explicitly requested
      - capabilities are available unless explicitly granted

13.3. Adapter readiness does not define activation order.

13.4. Adapter loading does not define activation order.


===============================================================================
14. RESOLVEPACK VS ENSUREACTIVATED (LOCKED)
===============================================================================

14.1. resolvePack(selector):
      - resolves metadata
      - performs visibility checks
      - does not select an adapter for execution
      - does not authorize execution
      - does not load code
      - does not activate packs

14.2. ensureActivated(selector):
      - ensures the target pack is activated
      - may trigger activation
      - returns activation metadata only
      - activation metadata is immutable and diagnostic
      - does not return code or symbols
      - may raise NotActivatedError

14.3. ensureActivated may cause adapter-mediated activation when the target
      pack's executable artifact belongs to an adapter runtime.

14.4. ensureActivated does not expose adapter internals.


===============================================================================
15. ORCHESTRATED EXECUTION GUARANTEE (LOCKED)
===============================================================================

15.1. Turnix guarantees behavior only for execution that is invoked and
      tracked through Turnix orchestration.

15.2. Guaranteed execution pathways include:
      - activation entrypoints such as onLoad(ctx)
      - capability calls dispatched by Turnix
      - Turnix event dispatch for handlers registered through Turnix
      - Turnix-managed schedulers if such schedulers are later accepted
      - IoManager-mediated operations, including streaming reads and writes

15.3. Execution started outside Turnix orchestration is unsupported.

15.4. Unsupported execution may still run, but Turnix provides:
      - no guarantees
      - no tracing guarantees
      - no lifecycle control
      - no cleanup guarantees

15.5. Adapter-mediated execution has Turnix guarantees only when it is
      invoked and tracked through Turnix orchestration.

15.6. In-process execution has no extra guarantees merely because it runs in
      the same process as the backend implementation.


15.1 ACTIVATION TRACE OBLIGATIONS
---------------------------------

15.1.1. Activation lifecycle outcomes remain trace-visible.

15.1.2. Activation trace evidence distinguishes:
        - activation start
        - dependency activation ordering
        - adapter-mediated activation dispatch when applicable
        - activation entrypoint invocation
        - service registration during activation
        - capability registration during activation
        - activation success
        - activation failure

15.1.3. Activation trace evidence records enough context to explain:
        - which pack instance was activated
        - which activation entrypoint was invoked
        - which adapter kind or adapter instance was involved when
          applicable
        - which dependency or ordering condition caused activation to wait,
          proceed, or fail
        - which service or capability registration occurred during activation
        - which failure caused activation to abort when activation fails

15.1.4. Activation trace evidence defines runtime causality and activation
        scope boundaries.

15.1.5. Activation trace evidence does not redefine import, dependency
        resolution, adapter lifecycle, capability permission, or Job
        lifecycle semantics.

15.1.6. DA-03 defines the tracing substrate used to represent activation
        trace evidence.

15.1.7. DA-14 owns the activation-domain trace reason identifiers used for
        activation start, dependency activation ordering, activation
        entrypoint invocation, activation-time service registration,
        activation-time capability registration, activation success,
        activation failure, and not-activated access failure.

15.1.8. The following activation-domain trace reasons are defined:

        - ActivationStarted
        - ActivationOrderingEvaluated
        - ActivationEntrypointInvoked
        - ActivationServiceRegistered
        - ActivationCapabilityRegistered
        - ActivationSucceeded
        - ActivationFailed
        - NotActivated

15.1.9. ActivationStarted is emitted when activation begins for a pack
        instance.

15.1.10. ActivationOrderingEvaluated is emitted when a dependency or
         ordering condition causes activation to wait, proceed, or fail.

15.1.11. ActivationEntrypointInvoked is emitted when Turnix invokes the
         activation entrypoint for a pack instance.

15.1.12. ActivationServiceRegistered is emitted when a service registration
         occurs during activation.

15.1.13. ActivationCapabilityRegistered is emitted when a capability
         registration occurs during activation.

15.1.14. ActivationSucceeded is emitted when activation succeeds for a pack
         instance.

15.1.15. ActivationFailed is emitted when activation fails for a pack
         instance.

15.1.16. NotActivated is emitted when a Turnix-mediated action is refused
         because the target pack is not activated.

15.1.17. DA-23 owns adapter-domain trace evidence that may causally relate
         to activation dispatch.


===============================================================================
16. ACTIVATION FAILURE AND NOTACTIVATEDERROR (LOCKED)
===============================================================================

16.1. Activation failure indicates a system-level error such as:
      - unresolved dependency
      - policy violation
      - visibility violation
      - execution authorization failure
      - adapter loading failure
      - activation entry failure

16.2. Locked consequences:
      - activation failure leaves the pack inactive
      - dependent activations do not proceed
      - failure is surfaced to the user
      - silent fallback is forbidden

16.3. NotActivatedError must not be swallowed by pack code.

16.4. Swallowing NotActivatedError hides orchestration failure.

16.5. Activation failure is not a Job terminal outcome merely because the
      activation path used adapter-executed code.


===============================================================================
17. ACTIVATION RESULT (LOCKED)
===============================================================================

17.1. ensureActivated may return:
      - activation status
      - warnings
      - diagnostics

17.2. ensureActivated does not return:
      - executable code
      - module namespaces
      - runtime symbols
      - adapter internals


===============================================================================
18. EXPLICITLY OPEN QUESTIONS
===============================================================================

18.1. OPEN questions include:
      - category-based activation ordering
      - staged or phased activation
      - activation retry semantics
      - exact adapter-mediated activation diagnostics


===============================================================================
19. LOCKED SUMMARY
===============================================================================

19.1. LOCKED:
      - activation is explicit and pack-level
      - onLoad(ctx) is the only activation entrypoint
      - activation establishes PackContext authority
      - PackContext is a pack-facing authority context, not raw backend
        internals
      - execution is allowed only after activation when Turnix-recognized
      - adapter selection, execution authorization, adapter readiness, and
        adapter loading do not imply activation
      - immediate execution, including IIFE-style patterns and top-level
        code, is unsupported
      - execution requires Turnix orchestration
      - adapter-mediated execution has guarantees only when invoked and
        tracked through Turnix orchestration
      - activation is distinct from import, resolution, and adapter lifecycle
      - activation lifecycle outcomes remain trace-visible
      - activation-domain trace reason identifiers are owned by DA-14
      - adapter-domain trace evidence remains owned by DA-23
      - failures are visible and abort activation

19.2. OPEN:
      - ordering refinements
      - adapter-mediated activation diagnostics
