===============================================================================
DESIGN AREA DA-01 - PROJECT INVARIANTS & PHILOSOPHY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Foundational principles that constrain all other Turnix design areas.

Changes here have cascading impact and should remain rare and explicit.

Purpose:
--------
This document defines the non-negotiable principles that guide Turnix
design decisions.

It defines:
- Turnix core identity
- the Core Truth Doctrine
- locked project invariants
- highest-level runtime authority ordering
- adjustable design attitudes
- explicitly open philosophical questions

It does NOT define:
- pack formats, resolution rules, or activation mechanics
- RPC schemas or transport behavior
- persistence formats or storage layout
- UI behavior or tooling implementation details
- concrete adapter, runtime host, or transport implementation details

Upstream Documents:
------------------
- CMR-01
- CMR-02
- CMR-03
- CMR-04
- CMR-05


===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines project-level invariants and design attitudes
      within that shared interpretation model.


===============================================================================
02. CORE IDENTITY AND PROJECT GOAL
===============================================================================

02.1. Turnix exists to enable a persistent, local, identity-stable system
      built on modular packs without relying on server infrastructure or
      stateless execution assumptions.

02.2. Turnix is not:
      - a plugin loader
      - a dynamic script runner
      - a web app framework
      - a convenience-first modding system

02.3. Turnix is:
      - a continuity engine
      - a deterministic resolver
      - a policy-driven orchestrator
      - an explainable system

02.4. Turnix is defined by its contracts, guarantees, orchestration
      semantics, and continuity behavior rather than by any specific
      implementation language, framework, or current runtime stack.

02.5. Current implementation choices are pragmatic and may change.
      Platform truth remains valid across such changes.

02.6. Modularity applies both to creator-provided extensions and to
      first-party platform-realized functionality where modular treatment
      preserves platform truth and controllability.

02.7. Implementation convenience does not override platform identity.

02.8. Turnix runtime authority follows this highest-level ordering:
      - Kernel owns authority
      - adapters execute declared pack code
      - transports carry messages
      - RPC defines the logical communication boundary
      - capabilities define what code may do
      - SavePack stores durable executed state

02.9. The ordering in 02.8 is architectural, not implementation-specific.

02.10. FastAPI, WebSocket, browser JavaScript, Python, CLI, local sockets,
       subprocesses, containers, or other implementation vehicles do not
       redefine that ordering.

02.11. Same host language does not imply same authority.

02.12. Pack-provided code is not backend implementation code merely because
       it runs in the same language or process as backend implementation
       code.

02.13. Concrete adapter, RPC, transport, capability, AppInstance, and
       SavePack semantics are defined by later design areas.

02.14. This document defines the project-level anchor that those later
       documents preserve.


===============================================================================
03. CORE TRUTH DOCTRINE (CT-D)
===============================================================================

03.1. Turnix does not guarantee correctness.

03.2. Turnix guarantees:
      - reproducibility
      - reversibility
      - explicit uncertainty

03.3. Reproducibility means that given the same filesystem state,
      configuration, and user decisions, Turnix produces the same
      outcomes.

03.4. Reproducibility excludes silent substitution and implicit best-guess
      resolution.

03.5. Reversibility means that decisions remain traceable and that state
      changes do not silently destroy prior valid states.

03.6. Reversibility keeps recovery possible without reconstructing hidden
      context.

03.7. Explicit uncertainty means that ambiguity is classified, surfaced,
      and explainable after the fact.

03.8. Silent assumptions, guesses, and hidden defaults violate explicit
      uncertainty.

03.9. Any mechanism that hides uncertainty, makes silent decisions, or
      prevents tracing and recovery violates CT-D.


===============================================================================
04. LOCKED INVARIANTS
===============================================================================

04.1. The principles in this section are LOCKED and apply across Turnix
      unless an explicit accepted revision changes them.

04.2. Determinism is preferred over convenience.

04.3. Given the same filesystem state, configuration, and user decisions,
      Turnix makes the same choices every time.

04.4. Turnix does not perform silent substitution or implicit best-guess
      resolution.

04.5. If ambiguity exists, Turnix classifies it, explains it, resolves it
      through policy or user decision, and persists the outcome.

04.6. Continuity is preferred over performance.

04.7. Persistence failures are existential rather than cosmetic.

04.8. Partial writes to required state do not occur.

04.9. Quarantine is preferred over overwrite when state integrity is at
      risk.

04.10. Best-effort behavior is acceptable only for explicitly optional
       data such as:
       - logs
       - traces
       - analytics

04.11. Explicitness is preferred over magic.

04.12. Hidden imports, invisible dependency resolution, and behavior that
       cannot be explained after the fact do not belong in Turnix.

04.13. If something works, Turnix can explain why.

04.14. If something fails, Turnix can explain why.

04.15. User-visible decisions are surfaced.

04.16. Any decision that could reasonably surprise a user is surfaced.

04.17. If a user is asked once, the decision is persisted.

04.18. Silent retries or silent fallbacks do not occur unless explicit
       policy governs them.

04.19. Turnix is local-first.

04.20. Turnix assumes:
       - local filesystem
       - limited hardware
       - limited context window

04.21. Server-scale patterns are rejected by default, including:
       - lazy global registries
       - infinite retries
       - implicit availability

04.22. Platform truth is implementation-independent.

04.23. Backend language, frontend technology, adapter strategy, or other
       current implementation choices do not define Turnix platform truth.

04.24. Platform contracts and guarantees remain valid across
       implementation replacement.

04.25. No guarantees exist outside orchestration.

04.26. Turnix does not guarantee behavior for background work started
       outside Turnix orchestration. Only work triggered by Turnix
       entrypoints or capability calls is supported.

04.27. This limitation applies to:
       - timers
       - background threads
       - ad-hoc async tasks
       - side-effect-only module execution

04.28. If a pack starts such work anyway, the resulting consequences are on
       the pack author unless Turnix explicitly guaranteed otherwise.

04.29. Runtime authority ordering remains explicit.

04.30. Adapters, transports, frontends, tooling surfaces, and host-language
       runtimes do not acquire Kernel authority by implementation proximity,
       deployment topology, or host-language convenience.


===============================================================================
05. ADJUSTABLE DESIGN ATTITUDE
===============================================================================

05.1. The principles in this section are strong design preferences rather
      than locked invariants.

05.2. These preferences may evolve with experience provided that locked
      invariants remain preserved.

05.3. Explainability is preferred over elegance when the more elegant
      design weakens clarity, traceability, or diagnosis.

05.4. A slightly verbose system that can explain itself is preferred over
      a clever system that only appears to work.

05.5. Policy is preferred over blanket prohibition when policy preserves
      safety and makes the governing rule visible.

05.6. When practical, Turnix classifies behavior, gates it by policy, and
      keeps defaults conservative.

05.7. Tooling friendliness matters.

05.8. Design should support:
      - trace inspection
      - pack manager UI
      - conflict explanation
      - replay of decisions

05.9. These tooling-oriented expectations remain relevant even when the
      tooling itself is not implemented yet.


===============================================================================
06. EXPLICITLY OPEN QUESTIONS
===============================================================================

06.1. The questions in this section are intentionally OPEN.

06.2. Pack autonomy remains OPEN.

06.3. Current OPEN questions include:
      - whether packs may ever declare always-allow permissions
      - whether packs may auto-activate toolchains

06.4. AI-assisted recovery remains OPEN.

06.5. Current OPEN questions include:
      - whether LLM-assisted error recovery may exist
      - at which lifecycle phase it may exist
      - whether it would be optional, policy-gated, or first-class

06.6. The long-term strictness of local-first assumptions remains OPEN.

06.7. Current OPEN questions include:
      - remote assets
      - remote compilers
      - federated pack indexes

06.8. These questions are currently out of scope, but they are not
      philosophically banned.


===============================================================================
07. WHY THIS AREA EXISTS
===============================================================================

07.1. This area exists to prevent later specifications from drifting into:
      - convenience-driven shortcuts
      - accidental server assumptions
      - silent failure tolerance
      - deferred "fix it later" logic

07.2. Without this area, later specifications are more likely to weaken
      determinism, continuity, explainability, or orchestration truth in
      exchange for local convenience.

07.3. This document is the anchor that prevents that drift.
