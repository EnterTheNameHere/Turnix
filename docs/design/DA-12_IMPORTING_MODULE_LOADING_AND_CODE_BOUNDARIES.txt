===============================================================================
DESIGN AREA DA-12 - IMPORTING, MODULE LOADING & CODE BOUNDARIES
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how code is referenced, loaded, and made available across supported
languages, and where strict boundaries exist between resolution, importing,
adapter selection, execution authorization, activation, and execution.

This area is intentionally strict.

Importing is not resolution.
Importing is not activation.
Importing is not dependency declaration.
Importing is not adapter execution authority.
Importing does not imply permission to execute.

Purpose:
--------
This document defines how Turnix prevents native language import systems from
bypassing Turnix semantics such as resolution, visibility, source selection,
adapter selection, execution authorization, activation order, and execution
authority.

It defines:
- language-level import boundaries
- Turnix-mediated cross-pack access mechanisms
- turnixImport selector rules and loading behavior
- dependency consistency requirements for imports
- the separation between module evaluation and Turnix-recognized execution
- relationship between importing and execution adapter boundaries
- Python pack-code import limits under the adapter model
- canonical activation entrypoint constraints relevant to code boundaries

It does NOT define:
- generic resolution semantics (see DA-11)
- dependency semantics and solver behavior (see DA-13)
- execution adapter semantics (see DA-23)
- activation lifecycle beyond the boundary rules stated here (see DA-14)
- capability schemas and permission prompting details (see DA-16)
- user decision and remembered policy handling (see DA-17)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines importing, module-loading, and code-boundary
      rules within that shared interpretation model.


===============================================================================
02. CORE PROBLEM STATEMENT
===============================================================================

02.1. Native language import systems do not understand Turnix pack
      resolution, visibility, save scoping, adapter selection, execution
      authorization, activation boundaries, or execution authority.

02.2. Turnix therefore does not allow arbitrary language-native importing
      across packs.

02.3. Cross-pack code access routes only through Turnix-defined mechanisms.

02.4. Native imports remain allowed only for:
      - language internals
      - same-file or same-directory code under normal local language rules
      - explicitly allowed Turnix platform surfaces where such access is
        part of the supported execution model

02.5. Any mechanism that bypasses Turnix resolution, visibility, adapter
      selection, execution authorization, activation, or execution boundaries
      is forbidden.

02.6. Same host language does not imply same authority.

02.7. Python pack code remains adapter-executed pack code and does not
      become backend implementation code through Python import behavior.


===============================================================================
03. SEPARATION OF CONCERNS
===============================================================================

03.1. Importing is not dependency declaration.

03.2. Importing is not resolution.

03.3. Importing is not adapter selection.

03.4. Importing is not execution authorization.

03.5. Importing is not activation.

03.6. Importing is not execution authority.

03.7. Importing is not capability permission.

03.8. Resolution answers which pack identity satisfies a request.

03.9. Adapter selection answers which adapter kind or concrete adapter
      provider is eligible for a declared executable artifact.

03.10. Execution authorization answers whether a declared executable
       artifact may be loaded and executed in the current pack,
       AppInstance, SavePack, adapter, policy, and trust context.

03.11. Importing answers how a code artifact is obtained once Turnix rules
       permit that step.

03.12. Activation answers when Turnix intentionally starts recognized pack
       behavior.

03.13. Execution authority answers whether code may perform meaningful
       Turnix-recognized work.

03.14. Capability permission answers whether a capability invocation is
       allowed under the capability, policy, and decision model.

03.15. This document keeps these boundaries explicit because native language
       import systems would otherwise collapse them into one accidental path.

03.16. DA-23 owns the execution adapter boundary in which importing and
       module loading operate.


===============================================================================
04. PYTHON IMPORT RULES (LOCKED)
===============================================================================

04.1. Python pack code may use native Python imports only for local code
      under the same-file or same-directory rule and for explicitly allowed
      Turnix platform surfaces.

04.2. Local Python imports include examples such as:
      - from .llamacpp_client import LlamaCppClient
      - from .config import loadConfig

04.3. Explicitly allowed Turnix platform-surface imports include only the
      whitelisted supported surface.

04.4. Explicitly allowed Turnix platform-surface imports do not include raw
      backend internals merely because the pack code is Python.

04.5. Cross-pack native Python imports are not allowed.

04.6. Disallowed examples include:
      - from ui.controls import Button
      - from Turnix.ui.controls import Button
      - from first_party.mods.foo import bar
      - from backend.core.kernel import Kernel
      - from backend.persistence.io import IoManager

04.7. The rationale is:
      - Python import machinery has no awareness of pack resolution
      - Python import machinery has no awareness of visibility
      - Python import machinery has no awareness of save scoping
      - Python import machinery has no awareness of adapter selection
      - Python import machinery has no awareness of execution authorization
      - Python import machinery has no awareness of activation boundaries
      - Python import machinery has no awareness of capability permission

04.8. Allowing native cross-pack Python imports would bypass Turnix
      guarantees.

04.9. Allowing Python pack code to import backend internals as authority
      would bypass the execution boundary defined by DA-23.


===============================================================================
05. PYTHON CROSS-PACK ACCESS (LOCKED MECHANISMS)
===============================================================================

05.1. Python pack code interacts with other packs only through Turnix-
      mediated APIs.

05.2. Supported mediated categories include:
      - services where allowed by DA-16
      - capabilities
      - pack metadata and activation control

05.3. Example mediated patterns include:
      - ctx.registerService(name, instance)
      - ctx.services.get(name)
      - ctx.callCapability("llm.pipeline@1", args)
      - ctx.resolvePack(selector)
      - ctx.ensureActivated(selector)

05.4. There is no Python equivalent of turnixImport as a cross-pack native
      symbol or executable-code import mechanism.

05.5. Python code does not receive executable code objects or symbols from
      other packs through native import bypass.

05.6. Python pack code may still receive Turnix-created context,
      capability, service, metadata, and activation-control surfaces when
      those surfaces are granted by the owning documents.

05.7. In-process Python access remains mediated by DA-23, DA-14, and DA-16.

05.8. In-process Python access does not become backend authority merely
      because it uses function calls instead of network transport.


===============================================================================
06. JAVASCRIPT IMPORT MODEL (LOCKED)
===============================================================================

06.1. JavaScript pack code runs in a browser environment.

06.2. Current constraints include:
      - no Node.js runtime assumptions
      - no filesystem access
      - no per-pack import maps
      - no native cross-pack imports

06.3. Dynamic cross-pack JavaScript loading routes through:
      turnixImport(selector)

06.4. turnixImport is a Turnix-defined mediated loader.
      It is not equivalent to native import().

06.5. Reserved names for future refinement do not change current semantics.

06.6. Browser JavaScript code remains adapter-executed frontend code under
      DA-23 and DA-22.


===============================================================================
07. TURNIXIMPORT - WHAT IT IS
===============================================================================

07.1. turnixImport(selector: string) is:
      - a Turnix-aware resolver
      - a controlled loader
      - a mediated ESM importer

07.2. turnixImport behavior includes:
      - parsing a Turnix selector
      - asking the backend to resolve pack identity and load target
      - performing visibility and policy checks
      - fetching the permitted JavaScript artifact
      - performing dynamic ESM import of that permitted artifact

07.3. turnixImport does not create a second independent resolution system.
      It depends on Turnix resolution and boundary checks already defined
      elsewhere.

07.4. turnixImport participates in adapter loading semantics without
      becoming activation or execution authorization by itself.


===============================================================================
08. TURNIXIMPORT - WHAT IT IS NOT
===============================================================================

08.1. turnixImport is not:
      - a dependency declaration
      - an adapter selection request
      - an execution authorization request
      - an activation request
      - a capability permission request
      - a visibility override
      - a permissive or best-effort loader
      - a version selector
      - an execution authorization

08.2. If importing fails, the failure is explicit.

08.3. turnixImport does not silently guess or broaden the requested target.

08.4. turnixImport does not grant Kernel authority to imported module code.


===============================================================================
09. TURNIXIMPORT SELECTOR FORMS (LOCKED)
===============================================================================

09.1. A turnixImport selector is not:
      - a filesystem path
      - a URI
      - a semver-qualified reference

09.2. A turnixImport selector is an identity-based Turnix selector within
      the importing rules of this document.

09.3. Supported conceptual forms are:

      A) Core or internal runtime files
         "core/rpc.js"
         "core/layout.js"

      B) Pack default entry
         "ui.controls"

      C) Pack sub-entry or export
         "ui.controls/Button"

09.4. Pack default-entry selectors require the corresponding declared
      dependency relationship already to exist where dependency rules require
      it.

09.5. Pack sub-entry selectors resolve against declared JavaScript entry
      mappings rather than filesystem guessing.

09.6. Disallowed selector forms include:
      - "../"
      - absolute paths
      - filesystem guessing
      - semver in selector
      - author mismatch against the already-resolved dependency identity


===============================================================================
10. DEPENDENCY CONSISTENCY RULE (LOCKED)
===============================================================================

10.1. Importing respects the concrete identity already chosen during
      dependency resolution.

10.2. If a pack declares a dependency that resolves to one concrete pack
      identity, importing must remain consistent with that chosen identity.

10.3. Allowed imports therefore remain those that refer to that already-
      resolved identity.

10.4. Disallowed imports include attempts to:
      - switch author identity
      - restate semver in the selector
      - bypass the already-resolved dependency choice

10.5. Examples:

      If a pack declares dependency:
          "packs": ["Turnix@ui.controls@^1.2"]

      Allowed imports:
          turnixImport("ui.controls")
          turnixImport("ui.controls/Button")

      Disallowed imports:
          turnixImport("Other@ui.controls")
          turnixImport("ui.controls@2.0")

10.6. Violations are hard errors.

10.7. Importing does not reopen dependency resolution.


===============================================================================
11. SEMVER IN IMPORTS (LOCKED)
===============================================================================

11.1. Semver is not allowed in turnixImport selectors.

11.2. Semver belongs to:
      - manifests
      - dependency resolution
      - policy decisions

11.3. Importing is identity-based rather than version-selection-based.


===============================================================================
12. FILE MATCHING RULES (LOCKED)
===============================================================================

12.1. For sub-entry imports such as:
      turnixImport("ui.controls/Button")

      Turnix resolves against runtimes.javascript.entries.

12.2. Matching logic:
      - exact "Button"
      - or "Button.<extension>"

12.3. Allowed current extension forms include:
      - .js
      - .mjs
      - .ts compiled to the served artifact form

12.4. Ambiguous matches are errors.

12.5. Missing matches are errors.

12.6. There is no fallback from sub-entry import to default entry.


===============================================================================
13. DEFAULT ENTRY BEHAVIOR (LOCKED)
===============================================================================

13.1. If the selector has no sub-path, Turnix loads the declared default
      JavaScript entry for that pack.

13.1.1. Example:
            turnixImport("ui.controls")

13.2. runtimes.javascript.entry must exist.

13.3. That entry is loaded.

13.4. Module exports are then returned.

13.5. There is no implicit index.js, filename guessing, or directory
      guessing.


===============================================================================
14. IMPORTING VS ACTIVATION (CRITICAL DISTINCTION)
===============================================================================

14.1. turnixImport loads a JavaScript module and may trigger normal ESM
      evaluation by the JavaScript runtime.

14.2. That ESM module evaluation is not Turnix-recognized activation.

14.3. turnixImport does not:
      - activate the pack
      - grant execution authority
      - grant capability permission
      - imply lifecycle participation

14.4. Activation remains a separate Turnix-controlled action handled through
      the appropriate activation mechanisms.

14.5. Module evaluation that occurs merely because the JavaScript runtime
      evaluated an imported module is not treated as Turnix-recognized pack
      execution by itself.

14.6. Adapter loading or module evaluation is not activation under DA-23 and
      DA-14.


===============================================================================
15. RESOLVEPACK VS ENSUREACTIVATED (LOCKED SEMANTICS)
===============================================================================

15.1. resolvePack(selector):
      - resolves metadata only
      - performs visibility checks
      - does not load code
      - does not select an adapter for execution
      - does not authorize execution
      - does not activate

15.2. ensureActivated(selector):
      - ensures pack activation
      - may trigger activation
      - does not return exports or symbols
      - may return activation diagnostics
      - may raise NotActivatedError

15.3. NotActivatedError indicates orchestration failure rather than native
      import failure.

15.4. NotActivatedError must surface to the user.


===============================================================================
16. IMMEDIATE EXECUTION AND SIDE EFFECTS (LOCKED POLICY)
===============================================================================

16.1. Some languages permit immediate execution constructs such as:
      - module-level statements
      - IIFE-style patterns
      - top-level side effects

16.2. Turnix does not coordinate, track, or rely on immediate execution.

16.3. Immediate execution is discouraged in pack code.

16.4. Import-time execution is not an activation mechanism.

16.5. Importing a module does not imply permission to execute meaningful
      Turnix-recognized logic.

16.6. Code that executes immediately:
      - runs outside Turnix orchestration
      - runs without PackContext authority
      - is outside Turnix guarantees
      - is not considered activated even if it appears to work

16.7. Meaningful Turnix-recognized execution waits for the supported
      activation entrypoint.

16.8. Adapter execution that occurs outside Turnix-recognized activation or
      orchestration remains unsupported by Turnix guarantees.


===============================================================================
17. CANONICAL ENTRYPOINTS (LOCKED)
===============================================================================

17.1. The currently supported activation entrypoint is:
      onLoad(ctx)

17.2. onLoad(ctx):
      - is called exactly once by Turnix
      - is the only supported activation entrypoint
      - is the first moment Turnix intentionally executes pack code
      - receives a Turnix-created PackContext
      - prepares the pack for later runtime behavior

17.3. Turnix recognizes no alternative activation path.

17.4. If a pack has executable code, Turnix activates that pack through
      onLoad(ctx) using the appropriate execution adapter when the artifact
      belongs to an adapter runtime.


===============================================================================
18. BACKGROUND WORK GUARANTEE (LOCKED)
===============================================================================

18.1. Turnix does not guarantee behavior for background work started outside
      Turnix orchestration.

18.2. Only work triggered by:
      - supported activation entrypoints
      - capability calls
      - other Turnix-defined orchestrated execution pathways

      is supported by Turnix guarantees.

18.3. Background work started otherwise:
      - may execute
      - remains unsupported
      - remains untracked
      - is not abortable by Turnix guarantees
      - is not cleaned up by Turnix guarantees

18.4. Adapter implementation details do not convert unsupported background
      work into supported Turnix execution.


===============================================================================
19. ERROR PHILOSOPHY (LOCKED)
===============================================================================

19.1. Import errors are hard errors.

19.2. The rationale is:
      - importing implies certainty
      - permissive loading creates nondeterminism
      - failures must remain visible and explainable

19.3. Import-related errors:
      - use the standard error envelope
      - include requesting pack identity
      - include target identity
      - include reason classification

19.4. Import-related errors do not become capability permission failures or
      Job terminal outcomes merely because later execution might have used
      capabilities or Jobs.


===============================================================================
20. EXPLICITLY OPEN QUESTIONS
===============================================================================

20.1. OPEN questions include:
      - possible rename turnixImport
      - whether turnixImport remains strictly internal to Turnix, with mods
        using ctx.resolvePack or ctx.ensureActivated instead
      - tooling for authoring entry mappings
      - documentation wording that avoids Python-import confusion


===============================================================================
21. LOCKED SUMMARY
===============================================================================

21.1. LOCKED:
      - no Python cross-pack native imports
      - Python pack code is not backend implementation code
      - JavaScript cross-pack loading through turnixImport only
      - importing does not imply adapter selection
      - importing does not imply execution authorization
      - importing does not imply activation
      - importing does not imply execution authority
      - importing does not imply capability permission
      - no semver in imports
      - dependency consistency is enforced
      - immediate execution, including IIFE-style patterns and top-level code,
        is discouraged and unsupported
      - onLoad(ctx) is the only supported activation entrypoint
      - import errors are strict and visible
      - activation remains separate from import
      - native imports do not bypass DA-23 execution boundaries

21.2. OPEN:
      - naming polish
      - authoring ergonomics
