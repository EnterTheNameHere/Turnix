===============================================================================
DESIGN AREA TRACE-06 - TRACE DIAGNOSTICS, ERRORS AND DEVELOPMENT MODE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines user-facing trace diagnostics, error-envelope correlation, and
explicit development-mode trace enhancement.

Purpose:
--------
This document defines diagnostic, error-correlation, and development-mode trace
semantics formerly contained in DA-03 sections 13 through 15 and section 18.

It defines:
- user-facing diagnostic constraints
- trace correlation for surfaced errors
- development-mode trace enrichment
- scoped open questions for diagnostic tooling

It does NOT define:
- concrete error-envelope schemas
- UI layout for trace viewers or debugger tooling
- debugger pause or mutation safety semantics
- object-specific trace obligations

-------------------

===============================================================================
01. USER-FACING DIAGNOSTICS
===============================================================================

01.1. Trace events may be surfaced to users as:
      - warnings
      - explanations
      - decision prompts
      - error dialogs
      - debugger inspection views

01.2. User-facing views may simplify presentation.

01.3. User-facing views preserve causality.

01.4. User-facing views do not fabricate reasons.

01.5. Raw trace remains developer-facing unless a surface explicitly exposes
      it.


===============================================================================
02. ERROR ENVELOPE CORRELATION
===============================================================================

02.1. Surfaced Turnix errors include trace correlation evidence when
      available.

02.2. A surfaced error includes or remains linkable to:
      - error code
      - human-readable message
      - reason
      - trace correlation identifiers when available

02.3. Error envelopes do not replace trace evidence.

02.4. Surfaced errors and trace evidence remain linked when trace evidence
      exists for the failing operation.

02.5. A surfaced error maps to one or more trace events when the error
      occurs inside Turnix-traced execution.

02.6. If trace evidence for a surfaced error is unavailable, incomplete,
      pruned, dropped, corrupt, or not yet initialized, that limitation
      remains explicit.

02.7. Concrete error-envelope schemas are defined by the owning protocol or
      implementation specification.

02.8. The owning protocol or implementation specification remains
      constrained by the trace-correlation semantics defined here.

02.9. DD-02 defines the interface doctrine for surfaced failure structure.

02.10. TRACE defines only the trace-correlation obligation for surfaced
       errors.


===============================================================================
03. DEVELOPMENT-MODE TRACE ENHANCEMENTS
===============================================================================

03.1. Development mode may add trace and diagnostic enrichment.

03.2. Development-mode trace enrichment may include:
      - verbose traces
      - speculative branches
      - why-not-chosen explanations
      - layer-resolution breakdowns
      - debugger-specific annotations

03.3. Development-mode trace enrichment is explicit.

03.4. Development-mode trace enrichment is never enabled silently.

03.5. Development-mode trace enrichment does not change runtime behavior.

03.6. Development-mode trace enrichment does not create authoritative state.

03.7. Development-mode trace enrichment does not create additional
      authority.

03.8. Development-mode trace enrichment does not replace domain-owned trace
      obligations.


===============================================================================
04. EXPLICITLY OPEN QUESTIONS
===============================================================================

04.1. OPEN questions include:
      - concrete trace visualization schemas
      - debugger-specific trace presentation
      - diagnostic export presentation


===============================================================================
05. LOCKED SUMMARY
===============================================================================

05.1. LOCKED:
      - user-facing diagnostics preserve causality
      - user-facing diagnostics do not fabricate reasons
      - surfaced errors correlate with trace evidence where available
      - unavailable trace evidence for surfaced errors remains explicit
      - development-mode trace enrichment is explicit and non-authoritative
      - development-mode enhancements do not change runtime behavior
      - development-mode enhancements do not replace domain-owned trace
        obligations
