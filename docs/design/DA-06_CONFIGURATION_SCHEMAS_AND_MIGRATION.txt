===============================================================================
DESIGN AREA DA-06 - CONFIGURATION, SCHEMAS & MIGRATION
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines configuration as schema-governed authoritative state, explains how
configuration is layered and materialized, defines how configuration is
stored and located, and defines how configuration evolves without silent
breakage.

Configuration is data with identity.
Breaking it silently is forbidden.

Purpose:
--------
This document defines how configuration exists, how configuration is
validated, how configuration evolves over time, how configuration is
materialized into persistent files, and how user state is protected.

This area defines:
- configuration as a specialized use of the Turnix state model
- configuration scopes and precedence rules
- schema role for configuration
- defaults and override-input rules
- configuration ownership identity
- the configuration materialization pipeline
- effectiveConfig and computed transform constraints
- config access API semantics
- persistent config storage layout and file placement
- policies.config.json and paths.config.json placement
- duplicate schema/default/config file selection rules
- migration rules and state-safety boundaries
- IoManager mediation requirements for persistence
- error classification and recovery boundaries

This area does NOT define:
- the general memory/state model (see DA-04)
- direct filesystem I/O rules beyond the IoManager bridge (see DA-07)
- tracing schema and storage (see DA-03)
- debugger pause and mutation safety semantics (see DA-18)
- the full root/bootstrap candidate model (see DA-09)
- a fully generalized schema system for all future structured data domains

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines configuration, schema use for configuration,
      storage materialization for configuration, and migration constraints
      within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. Configuration answers:

          "Which parameters influence Turnix and packs at runtime,
           how are they made safe, stable, and evolvable,
           and where are they persisted?"

02.2. Configuration is authoritative state.

02.3. Configuration is not:
      - arbitrary JSON blobs
      - free-form user data
      - implicit defaults hidden in code

02.4. Configuration follows the general memory/state model defined in
      [constrained-by: DA-04 02], [constrained-by: DA-04 06],
      [constrained-by: DA-04 07], and [constrained-by: DA-04 09].

02.5. This document refines that general model only for:
      - configuration scopes
      - schema-governed validation and interpretation
      - defaults and override inputs
      - effectiveConfig materialization
      - persistent configuration file placement
      - configuration migration and recovery

02.6. Configuration remains explainable, reversible, and protected from
      silent breakage.

02.7. Schema is introduced in this document for configuration first.

02.8. The schema model described here may later serve other structured data
      domains, but this document does not yet define the full generalized
      schema system for those domains.


===============================================================================
03. CONFIGURATION AS STATE
===============================================================================

03.1. Configuration uses the same authority boundary as other Turnix state.

03.2. Committed configuration is authoritative.

03.3. Runtime-visible configuration views are not authoritative merely
      because they are resident or currently effective.

03.4. Configuration persistence is protection against loss rather than the
      definition of truth, [constrained-by: DA-04 09].

03.5. Storage layout and file placement defined later in this document do
      not redefine authority. They define where authoritative
      configuration is persisted, not what makes it authoritative.

03.6. Configuration mutation that changes authoritative committed state is
      normal orchestrated work and follows the general transaction and
      commit rules from [constrained-by: DA-04 07] and
      [constrained-by: DA-04 19].

03.7. This document does not create a separate state architecture for
      configuration.


===============================================================================
04. CONFIGURATION SCOPES, OWNERSHIP, AND IDENTITY (LOCKED)
===============================================================================

04.1. Turnix distinguishes configuration by scope:

      A) Engine
      B) Pack-global
      C) Save-bound

04.2. Engine scope is kernel-global configuration.

04.3. Pack-global scope is configuration owned by one specific pack and
      shared across saves.

04.4. Save-bound scope is configuration owned by one specific pack and
      bound to one save.

04.5. Scopes are strictly ordered by precedence:

      Save-bound > Pack-global > Engine > Defaults

04.6. Each scope has:
      - schema-governed value shape
      - defaults
      - explicit override inputs
      - migration path

04.7. Ownership rules:

04.7.1. Engine scope is owned by the kernel.

04.7.2. Pack-global scope is owned by one specific canonicalPackId.

04.7.3. Save-bound scope is owned by one specific canonicalPackId inside
        one save context.

04.8. Ownership is immutable without migration.

04.9. There is no shared global bag of configuration keys.

04.10. canonicalPackId is the persistent configuration-ownership identity
       for pack-global and save-bound configuration.

04.11. canonicalPackId is exactly:

       resolvedAuthor + "@" + packTreeId

04.12. canonicalPackId does not include version or semver.

04.13. Version metadata may be stored inside configuration files,
       migration metadata, or both, but version is not part of the
       persistent ownership identity.

04.14. resolvedAuthor must exist through:
       - declared author
       - or inherited author

04.15. If resolvedAuthor cannot be determined, the pack is invalid.

04.16. Configuration ownership and cross-pack mutation authority remain
       compatible with [ref: ODA-E 4], [ref: ODA-E 5], and
       [ref: ODA-E 6].


===============================================================================
05. SCHEMA ROLE FOR CONFIGURATION (LOCKED)
===============================================================================

05.1. Schemas define:
      - structure
      - types
      - constraints
      - allowed value shapes
      - compatibility expectations
      - access-level metadata when configuration authority rules require it

05.2. Schemas are not a source of runtime defaults.

05.3. Schemas do not by themselves create authoritative values.

05.4. In this document, schema governs how configuration values are:
      - interpreted
      - validated
      - migrated safely
      - exposed with correct ownership and access metadata

05.5. The schema model described here is storage-agnostic at the semantic
      level.

05.6. Storage placement and file realization defined later in this document
      do not redefine schema meaning.

05.7. Current schema realization may use JSON-like structured forms, but
      this document does not lock the full implementation dialect.

05.8. Schema evolution rules:

05.8.1. Released schema meaning does not rewrite old meaning silently.

05.8.2. Breaking schema change requires an explicit new compatibility
        boundary and matching migration treatment.

05.8.3. Access-level metadata does not change silently.

05.9. This document defines schema use for configuration first. It does not
      yet define the full generalized schema system for manifests and other
      structured data domains.


===============================================================================
06. DEFAULTS AND OVERRIDE INPUTS (LOCKED)
===============================================================================

06.1. Defaults define initial values only.

06.2. Defaults are immutable input material.

06.3. Defaults are not user configuration.

06.4. Defaults are not mutable state.

06.5. Override inputs record explicit user-authored deviations from
      defaults.

06.6. Override inputs never modify defaults.

06.7. Override inputs are loaded after defaults.

06.8. Override inputs are migrated explicitly.

06.9. Override inputs are not modified silently.

06.10. Override inputs are not silently rewritten by storage
       materialization, migration, normalization, or persistence side
       effects.

06.11. Override inputs are the only user-authored persisted configuration
       input layer.

06.12. This does not create an exception to the general DA-04 state model.
       Authoritative committed configuration still follows normal state
       mutation and commit rules.

06.13. Defaults may change between pack or platform versions at author
       responsibility.

06.14. Changed defaults remain authored input change, not user-authored
       override change.

06.15. If changed defaults alter effective behavior, that change remains
       distinguishable from user-authored overrides.

06.16. Duplicate schema, default, or configuration file is selected, not
       stacked.

06.17. Duplicate schema, default, or configuration file is shadowed, not
       merged.

06.18. Deterministic selection rule:
       - first candidate in deterministic order wins
       - later duplicates remain shadowed
       - no additive combination occurs

06.19. Diagnostics and tracing should surface duplicate schema, default, or
       configuration files as strongly as they can, including when
       duplicates are equivalent, which file was selected, and which files
       were shadowed.


===============================================================================
07. CONFIGURATION MATERIALIZATION PIPELINE (LOCKED)
===============================================================================

07.1. Configuration is materialized through the following semantic stages:

      1) schema interpretation
      2) defaults input
      3) explicit override inputs
      4) migration when required
      5) validation of stored values
      6) computed transforms when allowed
      7) effectiveConfig materialization

07.2. Stages 1 through 5 are gating stages.

07.3. Stages 6 and 7 are runtime-view stages.

07.4. Runtime-view stages do not by themselves create authoritative
      committed state.

07.5. Materialization produces a runtime-visible configuration view from
      authoritative committed configuration plus permitted derived behavior.

07.6. Materialization remains explainable and traceable.

07.7. Computed layers and effectiveConfig remain compatible with
      [constrained-by: DA-04 02.2] and [constrained-by: DA-04 02.3].


===============================================================================
08. EFFECTIVECONFIG AND COMPUTED TRANSFORMS (LOCKED)
===============================================================================

08.1. effectiveConfig is the final runtime-visible configuration view.

08.2. effectiveConfig is not a separate authoritative state class.

08.3. Some configuration values may be:
      - computed
      - derived
      - transformed
      - adjusted based on other values

08.4. Computed transforms operate only on effectiveConfig.

08.5. Computed transforms do not mutate stored values.

08.6. Computed transforms do not write configuration.

08.7. Removal of a computed transform restores prior behavior automatically.

08.8. Computed layers remain traceable.

08.9. Computed layers remain compatible with [ref: ODA-E 9].

08.10. Diagnostic validation of effectiveConfig may exist, but stored-value
       validation remains authoritative.


===============================================================================
09. CONFIG ACCESS API (LOCKED)
===============================================================================

09.1. Configuration access is scope-explicit and ownership-aware.

09.2. Engine scope API:

      config.engine.get(key)
      config.engine.set(key, value)
      config.engine.delete(key)
      config.engine.deleteTree(prefixKey)
      config.engine.tryGet(key)

09.3. Pack-global scope API:

      config.user.get(packSelector, key)
      config.user.set(packSelector, key, value)
      config.user.delete(packSelector, key)
      config.user.deleteTree(packSelector, prefixKey)
      config.user.tryGet(packSelector, key)

09.4. Save-bound scope API for the current owner pack context:

      config.save.get(key)
      config.save.set(key, value)
      config.save.delete(key)
      config.save.deleteTree(prefixKey)
      config.save.tryGet(key)

09.5. Save-bound scope API for explicit target pack ownership:

      config.save.getFrom(packSelector, key)
      config.save.setFrom(packSelector, key, value)
      config.save.deleteFrom(packSelector, key)
      config.save.deleteTreeFrom(packSelector, prefixKey)
      config.save.tryGetFrom(packSelector, key)

09.6. Read semantics:

09.6.1. get() returns value or raises NotFoundOrPermissionDenied.

09.6.2. tryGet() returns value when present or None when the addressed key
        does not exist.

09.6.3. The API does not distinguish not-found from permission-denied
        programmatically.

09.6.4. The real denial reason remains trace-visible internally.

09.7. Write semantics:

09.7.1. Writes target an explicit owner scope.

09.7.2. Writes may trigger permission decisions or user decisions.

09.7.3. delete() removes exactly the addressed override key. It never
        mutates defaults.

09.7.4. deleteTree() removes the addressed override node and its subtree.
        It never mutates defaults.

09.7.5. Delete is a write-class operation.

09.8. Cross-pack configuration access remains compatible with
      [ref: ODA-E 6], [ref: ODA-E 11], and [ref: ODA-E 13].


===============================================================================
10. CONFIG STORAGE LAYOUT (LOCKED)
===============================================================================

10.1. Turnix-written live and save-bound configuration files use strict JSON.

10.2. Human-facing defaults and other authored source material may use
      comment-capable JSON dialects where other authoritative documents
      permit it.

10.3. Turnix-written live/save configuration files do not use comments as a
      semantic mechanism.

10.4. Comments never carry structure, authority, migration meaning, or
      loader instructions.

10.5. Engine-global live configuration is stored under:

      userdata/config/engine/

10.6. Save-bound engine configuration is stored under:

      saves/<appPackId>/<instanceId>/config/engine/

10.7. Pack-global live configuration is stored under:

      userdata/config/packs/

10.8. Save-bound pack configuration is stored under:

      saves/<appPackId>/<instanceId>/config/packs/

10.9. Engine-global config file form is:

      userdata/config/engine/<name>.config.json

10.10. Save-bound engine config file form is:

       saves/<appPackId>/<instanceId>/config/engine/<name>.config.json

10.11. Pack-global config file form is:

       userdata/config/packs/<canonicalPackId>.config.json

10.12. Save-bound pack config file form is:

       saves/<appPackId>/<instanceId>/config/packs/<canonicalPackId>.config.json

10.13. paths.config.json is engine-owned configuration and is stored at:

       userdata/config/engine/paths.config.json

10.14. Global policies.config.json is stored at:

       userdata/config/engine/policies.config.json

10.15. Save-bound policies.config.json is stored at:

       saves/<appPackId>/<instanceId>/config/engine/policies.config.json

10.16. One clearly owned file per owner/name is the current rule.

10.17. For engine config, one engine config domain maps to one file.

10.18. For pack-owned config, one owning pack maps to one file.

10.19. Later expansion to more complex grouping is possible, but this
       document currently locks one clearly owned file per owner/name.

10.20. Defaults and authored configuration material remain load-only at
       runtime. Turnix runtime does not rewrite them as part of ordinary
       persistence.

10.21. The root/bootstrap behavior of paths.config.json fields is defined
       in DA-09. This document is authoritative for field naming,
       ownership classification, storage placement, and persistence
       materialization.

10.22. paths.config.json is engine-owned persisted configuration. Its file
       placement does not by itself redefine the higher bootstrap semantics
       that DA-09 governs.

10.23. Locked field names for paths.config.json are:
       - redirectUserdataDirectoryTo
       - userdataDirectoryRoots
       - savesDirectoryRoots
       - customDirectoryRoots
       - firstPartyDirectoryRoots
       - thirdPartyDirectoryRoots
       - selectedUserdata
       - selectedSaves
       - preferredUserdata
       - preferredSaves
       - preferredCustom
       - preferredFirstParty
       - preferredThirdParty

10.24. selected* fields express explicit user-selected roots for active
       state categories.

10.25. preferred* fields express the last used usable directory for their
       category.

10.26. preferred* is sticky.

10.27. Stored preferred* values are always rechecked for continued
       usability and, where writing is required, writability before reuse.

10.28. preferred* is not replaced merely because a different candidate
       becomes writable later.

10.29. selected* for userdata and saves overrides preferred* when usable.

10.30. selected* and preferred* for userdata or saves may differ when a
       previously selected path becomes unusable and controlled recovery
       causes Turnix to use another path.

10.31. DA-09 defines how selected* and *DirectoryRoots participate in
       bootstrap and candidate-root behavior. This document does not
       duplicate that bootstrap logic.


===============================================================================
11. IOMANAGER MEDIATION (LOCKED)
===============================================================================

11.1. Configuration services and providers do not perform filesystem I/O
      directly.

11.2. Configuration persistence routes through IoManager.

11.3. IoManager enforces:
      - root boundaries
      - permissions
      - atomic writes
      - backup and snapshot policy
      - error classification
      - recovery surfaces

11.4. IoManager mediation remains [constrained-by: DA-04 03.5] and
      [constrained-by: DA-07].

11.5. This document defines storage layout, file placement, and ownership
      mapping for configuration.

11.6. This document does not redefine low-level filesystem behavior,
      direct filesystem I/O rules, or DA-07 persistence mechanics.

11.7. If selected/preferred directories or canonical persisted config
      locations cannot be written:
      - Turnix does not silently fall through to some other writable path
      - this is surfaced through IoManager or delegated manager flow
      - a failed persistence action fails explicitly rather than faking
        success elsewhere

11.8. Path/config placement failure follows the same explicit recovery
      model as other configuration persistence failure. It does not create
      a separate silent fallback path.

11.9. Directory/path queries may yield no usable target when no valid
      directory is currently available.

11.10. Consumers that can tolerate such absence may handle it in controlled
       form. Consumers that cannot tolerate it propagate failure normally
       into the surrounding error-handling model.

11.11. User prompting and recovery remain centralized through IoManager or
       delegated manager flow rather than ad hoc consumer logic.

11.12. This remains aligned with the broader decision-handling and failure
       propagation model defined in DA-17.


===============================================================================
12. CONFIGURATION MIGRATION (LOCKED)
===============================================================================

12.1. Configuration migration evolves authoritative configuration state.

12.2. Configuration migration is not a side path outside normal state rules.

12.3. Configuration migration that changes authoritative committed state
      follows [constrained-by: DA-04 19].

12.4. Configuration migration runs before stored-value validation of the
      migrated result.

12.5. Migrators are versioned and ordered.

12.6. Multiple migrators may run sequentially.

12.7. A migrator is a deterministic transformation from one configuration
      representation to another.

12.8. Migration may inspect previous values.

12.9. Migration does not discard user intent silently.

12.10. Migration preserves ownership and access metadata unless an explicit
       accepted migration rule changes them.

12.11. If migration cannot complete safely, migration aborts and
       authoritative configuration remains identical to the pre-migration
       committed state.


===============================================================================
13. MIGRATION SAFETY, BACKUP, AND RECOVERY (LOCKED)
===============================================================================

13.1. Before writing migrated configuration, IoManager may create a
      recoverable snapshot of the last known good state.

13.2. Migrated authoritative configuration writes are atomic.

13.3. Rollback capability remains available when policy requires it.

13.4. No silent fallback is allowed.

13.5. User-mediated recovery options may include:
      - restore last known good
      - reset to defaults
      - quarantine broken configuration
      - abort affected startup or affected operation

13.6. Recovery remains explicit and trace-visible.


===============================================================================
14. MANIFEST MIGRATION AND CONFIGURATION MIGRATION (LOCKED DISTINCTION)
===============================================================================

14.1. Manifest migration and configuration migration are separate concerns.

14.2. Manifest migration evolves pack or artifact metadata.

14.3. Configuration migration evolves user or engine configuration state.

14.4. These migration classes remain separate even when they use related
      schema ideas.

14.5. This document defines configuration migration only.


===============================================================================
15. SAVE-BOUND CONFIGURATION (LOCKED)
===============================================================================

15.1. Save-bound configuration is isolated per save.

15.2. Save-bound configuration overrides pack-global and engine values
      within that save context.

15.3. Save-bound configuration migrates independently.

15.4. Save-bound configuration is included in save-bound authoritative
      state snapshots and restoration boundaries.

15.5. Corruption or deletion of save-bound configuration does not by itself
      alter global configuration state.

15.6. Engine save-bound configuration is allowed.

15.7. Pack-owned save-bound configuration follows the same owner/file model
      as pack-global configuration, but under the save-bound location.

15.8. policies.config.json may therefore exist in save-bound engine config
      storage without changing the general ownership model.


===============================================================================
16. POLICY STORAGE (LOCKED)
===============================================================================

16.1. Global persisted policies are stored in:

      userdata/config/engine/policies.config.json

16.2. Save-scoped persisted policies are stored in:

      saves/<appPackId>/<instanceId>/config/engine/policies.config.json

16.3. Run-scoped policy is memory-only and is never persisted.

16.4. policies.config.json contains top-level sections:
      - permissionPolicies
      - fingerprintPolicies

16.5. permissionPolicies and fingerprintPolicies are semantically distinct
      remembered-decision classes.

16.6. permissionPolicies persist remembered permission decisions.

16.7. fingerprintPolicies persist remembered fingerprint-mismatch
      acceptance decisions.

16.8. Exact internal JSON record shape inside those sections remains
      intentionally flexible for now.

16.9. Identity keys and higher-level decision semantics for these policy
      classes are defined by the permission and decision-handling areas
      that use them.

16.10. This document is authoritative for storage location, file placement,
       and top-level persistent config structure.


===============================================================================
17. ERROR HANDLING AND RECOVERY BOUNDARIES (LOCKED)
===============================================================================

17.1. Configuration errors are classified.

17.2. Fatal examples include:
      - missing required schema material
      - missing required defaults material where defaults are mandatory
      - failure before minimal recovery UI is ready

17.3. Recoverable examples include:
      - invalid user value
      - parse errors
      - migration-required state
      - restore-from-backup decisions

17.4. Recovery remains UI-driven or policy-driven where the surrounding
      design permits it.

17.5. Configuration handling does not invent defaults, ignore invalid
      stored values, or auto-reset silently.


===============================================================================
18. DEV-MODE DIAGNOSTICS (LOCKED)
===============================================================================

18.1. Development diagnostics may report:
      - unused keys
      - deprecated keys
      - schema drift
      - effective-config diagnostics
      - access denials as informational diagnostics
      - duplicate schema, default, or configuration files
      - shadowed equivalent or divergent config files

18.2. Development diagnostics are non-fatal unless some other authoritative
      document explicitly states otherwise.

18.3. Development diagnostics do not rewrite stored configuration.


===============================================================================
19. NO IMPLICIT FALLBACKS (LOCKED)
===============================================================================

19.1. Turnix does not:
      - invent defaults
      - ignore invalid stored values
      - auto-reset without consent or explicit policy
      - treat parse errors as empty configuration
      - silently write to some other config file because the chosen one
        failed
      - silently move selected/preferred config state elsewhere

19.2. Explicitness takes precedence over convenience.


===============================================================================
20. EXPLICITLY OPEN QUESTIONS
===============================================================================

20.1. OPEN questions include:
      - hot-reload semantics
      - config-diff UI
      - cross-scope promotion UX
      - when the broader generalized schema system should split from
        configuration-first design
      - whether later implementation requires finer-grained policy file
        partitioning

20.2. Storage meaning, file placement, and owner/path identity defined in
      this document are otherwise treated as semantically locked.


===============================================================================
21. LOCKED SUMMARY
===============================================================================

21.1. LOCKED:
      - configuration is authoritative state
      - configuration follows the DA-04 memory/state model
      - explicit configuration scopes and ownership
      - canonicalPackId for persistent config ownership is
        resolvedAuthor + "@" + packTreeId
      - canonicalPackId excludes version
      - a pack without resolvable author is invalid
      - schema governs configuration structure, constraints,
        compatibility, and access metadata
      - schemas are not the runtime source of defaults
      - defaults are immutable input material
      - overrides are explicit user-authored persisted deviations
      - effectiveConfig is a derived runtime-visible view, not a separate
        authoritative class
      - configuration migration follows DA-04 migration commit rules
      - IoManager mediates persistence
      - recovery is explicit
      - the public config API is scope-explicit and ownership-aware
      - delete() removes exactly the addressed override key
      - deleteTree() removes the addressed override node and subtree
      - read semantics use NotFoundOrPermissionDenied
      - tryGet() returns None for absent keys and does not expose
        not-found versus permission-denied programmatically
      - engine live config path is
        userdata/config/engine/<name>.config.json
      - save-bound engine config path is
        saves/<appPackId>/<instanceId>/config/engine/<name>.config.json
      - pack-global config path is
        userdata/config/packs/<canonicalPackId>.config.json
      - save-bound pack config path is
        saves/<appPackId>/<instanceId>/config/packs/<canonicalPackId>.config.json
      - paths.config.json is stored at
        userdata/config/engine/paths.config.json
      - policies.config.json is stored at
        userdata/config/engine/policies.config.json
      - save-bound policies.config.json is stored at
        saves/<appPackId>/<instanceId>/config/engine/policies.config.json
      - one clearly owned file per owner/name is the current rule
      - Turnix-written live/save config files use strict JSON
      - duplicate schema, default, or configuration file is
        selected, not stacked

21.2. OPEN:
      - generalized schema-system expansion beyond configuration-first use
      - UX and tooling only
      - internal record JSON shape inside persisted policy sections
