===============================================================================
OPTIONAL DESIGN AREA ODA-A - MINIMAL BOOTSTRAP & ERROR UI SUBSYSTEM
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines a minimal bootstrap and error-presentation layer for failures that
occur before normal appPack activation and before the main user-facing
Turnix surfaces are available.

This area is optional in tier, but strategically important for robustness,
debuggability, and user trust.

Purpose:
--------
This document defines the minimal bootstrap and error-user-interface model
used when Turnix encounters blocking failures before normal runtime surfaces
are available.

It defines:
- bootstrap rings as a conceptual startup model
- the role and scope of the minimal error UI
- kernel-owned error communication and action handling
- frontend handling during early blocking failures
- fallback behavior when minimal UI is unavailable

It does NOT define:
- core Turnix correctness semantics
- normal appPack user-interface behavior
- PackManager behavior
- mod-defined or themeable early UI extension rules
- LLM-assisted recovery as required behavior

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines a minimal bootstrap and error-user-interface
      subsystem within that shared interpretation model.


===============================================================================
02. CORE PROBLEM
===============================================================================

02.1. Turnix may fail before:
      - any appPack is active
      - normal user interface exists
      - mods are loaded
      - services or capabilities are available for normal application use

02.2. Examples include:
      - corrupted global config
      - unreadable manifest
      - schema mismatch
      - missing runtime files
      - discovery failure

02.3. Without a minimal bootstrap-facing error surface, such failures may
      become:
      - silent
      - terminal-only
      - confusing
      - harder to recover for non-technical users

02.4. This document defines an early minimal subsystem intended to reduce
      those failure modes without depending on normal pack-driven runtime
      behavior.


===============================================================================
03. BOOTSTRAP RINGS (LOCKED CONCEPT)
===============================================================================

03.1. Turnix startup may be understood as a ring-based conceptual model.


03.1 RING 0 - KERNEL BOOTSTRAP (LOCKED CONCEPT)
-----------------------------------------------

03.1.1. Ring 0 includes early kernel bootstrap responsibilities such as:
        - process startup
        - logging
        - IoManager
        - error-envelope handling
        - config schema loading
        - default-config loading
        - user-config loading

03.1.2. Failures in Ring 0 are fatal.

03.1.3. No user interface is guaranteed in Ring 0.


03.2 RING 1 - MINIMAL UI BOOTSTRAP (LOCKED CONCEPT)
---------------------------------------------------

03.2.1. Ring 1 includes the minimal bootstrap-facing error subsystem.

03.2.2. Ring 1 responsibilities may include:
        - serving index.html
        - serving bootstrap.js
        - establishing RPC
        - registering early error-reporting paths
        - rendering a minimal error UI

03.2.3. Failures in Ring 1 are severe.

03.2.4. Ring-1 failures may still allow retry, fallback, or user action when
        the surrounding bootstrap environment supports that.


03.3 RING 2 - PACK DISCOVERY AND ACTIVATION (LOCKED CONCEPT)
------------------------------------------------------------

03.3.1. Ring 2 includes later startup work such as:
        - content scanning
        - indexing
        - activation
        - appInstance creation

03.3.2. Failures in Ring 2 are expected to be more user-resolvable than
        Ring-0 failures because the minimal interaction layer may already
        exist by then.


===============================================================================
04. MINIMAL ERROR UI PURPOSE (LOCKED)
===============================================================================

04.1. The minimal error UI exists only to:
      - display fatal or blocking errors
      - present recovery choices supplied by the kernel
      - return a decision to the kernel

04.2. The minimal error UI is not:
      - a full application UI
      - a PackManager
      - themeable by normal packs
      - customizable by mods


===============================================================================
05. ERROR UI CHARACTERISTICS (LOCKED)
===============================================================================

05.1. The minimal error UI is:
      - small
      - fast to load
      - independent of packs
      - independent of LLM availability
      - independent of user config
      - independent of saves
      - independent of asset compilation

05.2. Acceptable forms may include:
      - an Electron window
      - a browser page
      - a native dialog
      - a text overlay fallback

05.3. The exact presentation technology is an implementation choice so long
      as the minimal-bootstrap constraints in this document remain preserved.


===============================================================================
06. ERROR COMMUNICATION MODEL (LOCKED)
===============================================================================

06.1. Kernel-originated blocking errors are sent to the minimal UI as:
      - the standard error envelope
      - a human-readable message
      - structured payload
      - an allowed-actions list

06.2. The UI returns:
      - the selected action
      - optional parameters
      - acknowledgement when required

06.3. Action meaning originates in the kernel rather than in the UI.


===============================================================================
07. SUPPORTED USER ACTIONS (LOCKED)
===============================================================================

07.1. Typical kernel-provided actions may include:
      - Retry
      - Reset config to defaults
      - Ignore and continue when allowed
      - Abort startup
      - Open logs directory

07.2. Available actions are provided by the kernel.

07.3. The UI does not invent actions.


===============================================================================
08. UI AVAILABILITY GUARANTEE
===============================================================================

08.1. Once Ring 1 is complete:
      - the kernel may assume that a minimal decision surface exists
      - user decisions may be requested
      - execution may pause safely for decision handling

08.2. Before Ring 1 completes:
      - no user interaction is relied upon
      - fatal errors abort immediately unless an even earlier fallback path
        exists outside this document's guaranteed scope


===============================================================================
09. MULTI-FRONTEND HANDLING (LOCKED)
===============================================================================

09.1. If multiple frontends connect to the minimal error subsystem:
      - the blocking error is broadcast
      - the first valid response wins
      - other pending presentations are dismissed

09.2. The kernel remains the authority.


===============================================================================
10. FAILURE MODES
===============================================================================

10.1. If the minimal UI fails to load:
      - Turnix may fall back to a launcher surface
      - Turnix may fall back to console reporting
      - Turnix may abort with a clear error when no safe fallback exists

10.2. If the UI disconnects:
      - execution may pause
      - reconnect may be allowed
      - timeout handling may apply when configured


===============================================================================
11. OPTIONAL PACK-BASED ERROR UI (OPEN)
===============================================================================

11.1. A future design may define a special early-loaded errorUiPack model.

11.2. Such a model might:
      - load before normal discovery
      - require explicit whitelisting
      - replace the built-in minimal error UI

11.3. Potential advantages include:
      - themed presentation
      - extensibility
      - reuse

11.4. Potential costs include:
      - increased bootstrap complexity
      - increased dependency risk

11.5. This concept remains OPEN.


===============================================================================
12. LLM-ASSISTED RECOVERY (OPEN)
===============================================================================

12.1. A future design may allow LLM-assisted explanation or guidance when an
      LLM service is available.

12.2. Constraints would include:
      - LLM assistance does not block recovery
      - LLM assistance does not replace user choice
      - LLM assistance is never required for recovery

12.3. This concept remains OPEN and optional.


===============================================================================
13. WHAT THIS AREA DOES NOT DO
===============================================================================

13.1. This area does not include:
      - save inspection
      - pack browsing
      - mod installation
      - normal capability invocation as a user feature surface
      - scripting
      - theming


===============================================================================
14. LOCKED SUMMARY
===============================================================================

14.1. LOCKED:
      - bootstrap rings as a conceptual model
      - the minimal error UI role
      - kernel-driven actions
      - structured error-envelope communication
      - pause-and-resume handling once minimal UI exists

14.2. OPEN:
      - pack-based error UI
      - LLM-assisted guidance
      - user-experience refinements
