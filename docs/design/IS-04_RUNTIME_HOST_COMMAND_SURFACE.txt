===============================================================================
IMPLEMENTATION SPECIFICATION IS-04 - RUNTIME HOST COMMAND SURFACE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the concrete RuntimeHost command-surface flow, command category
classification, command identity, command acceptance and dispatch behavior,
command result envelopes, command-to-runtime-operation mapping, and trace mapping
for RuntimeHost command surfaces.

This document implements the command-surface semantics defined by DA-28. It
remains constrained by RPC, AppInstance, adapter, Job, Orchestration Unit,
workspace ownership, policy, persistence, and committed-state documents.

Purpose:
--------
This document defines how a RuntimeHost command surface receives, classifies,
accepts, rejects, dispatches, and reports commands without becoming Kernel
authority by itself.

It defines:
- command identity and command correlation
- command input normalization
- command classification categories
- command acceptance and rejection flow
- command dispatch flow
- command result envelope shape
- command-to-runtime-operation mapping
- command relationship to Jobs and Orchestration Units
- command relationship to workspace ownership
- command trace mapping to DA-28 trace reasons
- command error categories

It does NOT define:
- RuntimeHost semantics (see DA-28)
- Kernel authority ordering (see DA-01 and DA-02)
- committed-state or transaction semantics (see DA-04)
- persistent I/O safety semantics (see DA-07)
- AppInstance lifecycle or runtime identity semantics (see DA-20)
- RPC, channel, or transport semantics (see DA-21)
- UI, View, or frontend runtime semantics (see DA-22)
- execution adapter authority boundaries (see DA-23)
- Orchestration Unit mutation authority semantics (see DA-24)
- adapter lifecycle, trust, and trace semantics (see DA-27)
- workspace ownership and host-lock record mechanics (see IS-03)
- concrete shell syntax, argparse parser structure, or terminal rendering
- concrete LAN pairing or authentication

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines implementation-level RuntimeHost command-surface
      behavior within that shared interpretation model.

01.5. This implementation specification does not create new authority
      semantics. It implements DA-28 command-surface rules.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. RuntimeHost command-surface behavior answers:

          "How does an entry surface submit a command to RuntimeHost-owned
           command handling, and how does Turnix distinguish receipt,
           classification, acceptance, dispatch, completion, failure, and
           downstream work outcome?"

02.2. A command surface receives command input and submits it to RuntimeHost
      command handling.

02.3. A command surface is not Kernel authority by itself.

02.4. Command receipt is not command acceptance.

02.5. Command acceptance is not command completion.

02.6. Command completion is not Job success unless the owning command
      contract maps those outcomes explicitly.

02.7. Command failure is not Job failure by default.

02.8. Command failure is not Orchestration Unit failure by default.

02.9. Command syntax does not define authority.

02.10. Command handling remains constrained by DA-28 and by each owning
       domain document for the work requested by the command.


===============================================================================
03. COMMAND IDENTITY AND CORRELATION
===============================================================================

03.1. commandId identifies one received command.

03.2. commandId is distinct from:
      - runtimeHostId
      - runtimeOperationId
      - appInstanceId
      - jobId
      - orchestrationUnitId
      - RPC message identity
      - adapter identity
      - viewId
      - transactionId
      - decision identity

03.3. commandId is stable for the lifetime of the command handling record.

03.4. commandId is used for command trace correlation.

03.5. A command may have a parent or source relation to:
      - CLI invocation
      - frontend interaction
      - RPC message
      - remote client request
      - test harness request
      - launcher interaction

03.6. Source relation does not collapse command identity into the source
      identity.

03.7. A command may create or correlate with a runtimeOperationId.

03.8. A command may create or correlate with a jobId when command handling
      becomes Job-backed work.

03.9. A command may create or correlate with an orchestrationUnitId when
      command handling performs or may perform mutation-bearing orchestration.

03.10. A command may correlate with appInstanceId, viewId, adapter identity,
       or capability identity when the command reaches those domains.


===============================================================================
04. COMMAND INPUT NORMALIZATION
===============================================================================

04.1. Command input is normalized before classification.

04.2. Normalization may include:
      - command name parsing
      - argument parsing
      - option parsing
      - environment input capture
      - current workspace resolution
      - current entry-surface identity capture
      - current startup mode capture

04.3. Normalization does not grant authority.

04.4. Normalization does not perform mutation-bearing work.

04.5. Normalization failure rejects the command before runtime operation
      acceptance.

04.6. Normalization failure emits RuntimeHostCommandReceived when receipt
      occurred and RuntimeHostCommandRejected when rejection is produced.

04.7. Concrete parser syntax belongs to implementation code or a later CLI
      implementation specification.

04.8. Normalized command representation preserves enough original source
      information to explain what the user or caller requested when practical.


===============================================================================
05. COMMAND CATEGORY CLASSIFICATION
===============================================================================

05.1. Command classification assigns a command category before dispatch.

05.2. Command categories include:
      - StartHost
      - StopHost
      - ConnectHost
      - RunAppPack
      - CreateSavePack
      - LoadSavePack
      - ListPacks
      - ListAppPacks
      - ListSavePacks
      - ListAppInstances
      - InspectHost
      - InspectTraceEvidence
      - InspectPolicyDecisions
      - ServeFrontend
      - ExposeLanSurface
      - CloseEntrySurface
      - ReadOnlyInspectWorkspace
      - RecoverStaleWorkspaceOwner
      - UnknownCommand

05.3. Command category names define semantic categories, not final CLI
      command names.

05.4. UnknownCommand is rejected unless a later extension mechanism defines
      explicit handling.

05.5. Classification does not grant permission.

05.6. Classification does not acquire workspace ownership.

05.7. Classification does not create an AppInstance.

05.8. Classification does not create a Job by itself.

05.9. Classification does not create an Orchestration Unit by itself.

05.10. Classification emits RuntimeHostCommandClassified when classification
       succeeds.

05.11. Classification failure emits RuntimeHostCommandRejected.


===============================================================================
06. COMMAND ACCEPTANCE AND REJECTION
===============================================================================

06.1. Command acceptance means the command passed normalization,
      classification, preliminary validation, and was accepted for dispatch or
      operation handling.

06.2. Command acceptance does not mean the requested work completed.

06.3. Command acceptance does not mean workspace ownership was acquired.

06.4. Command acceptance does not mean AppInstance creation succeeded.

06.5. Command acceptance does not mean adapter execution succeeded.

06.6. Command acceptance does not mean Job success.

06.7. Command acceptance does not mean Orchestration Unit completion.

06.8. Command rejection means the command did not enter dispatch for the
      requested operation.

06.9. Command rejection may occur because of:
      - malformed command input
      - unknown command category
      - unsupported startup mode
      - invalid workspace selection
      - missing required argument
      - policy denial
      - permission denial
      - existing workspace owner
      - unavailable RuntimeHost
      - incompatible command and entry surface
      - unavailable required implementation feature

06.10. Command rejection before Job-backed work exists does not produce Job
       terminal outcome.

06.11. Command rejection before Orchestration Unit creation does not produce
       Orchestration Unit terminal outcome.

06.12. Command rejection before adapter-mediated work exists does not produce
       adapter failure.

06.13. Command acceptance emits RuntimeHostCommandAccepted.

06.14. Command rejection emits RuntimeHostCommandRejected.


===============================================================================
07. COMMAND DISPATCH FLOW
===============================================================================

07.1. Command dispatch begins after command acceptance.

07.2. Dispatch routes the command to one or more RuntimeHost runtime
      operations.

07.3. Dispatch emits RuntimeHostCommandDispatched.

07.4. Dispatch does not bypass domain-owned semantics.

07.5. A command that requests AppInstance creation follows DA-20.

07.6. A command that communicates through RPC follows DA-21.

07.7. A command that affects UI or View behavior follows DA-22.

07.8. A command that starts adapter execution follows DA-23 and DA-27.

07.9. A command that mutates Turnix-managed state follows DA-04 and DA-24.

07.10. A command that performs persistent I/O follows DA-07.

07.11. A command that invokes a capability follows DA-16.

07.12. A command that requires policy or user decision follows DA-17.

07.13. A command that creates, observes, cancels, or retrieves Job-backed
       work follows DA-19.

07.14. A command that acquires, releases, inspects, or recovers workspace
       ownership follows IS-03.

07.15. Dispatch may produce immediate completion, immediate failure, accepted
       Job-backed work, accepted RuntimeHost operation, accepted adapter-
       mediated work, or accepted subscription-like observation.

07.16. Dispatch result shape is represented by the command result envelope.


===============================================================================
08. RUNTIME OPERATION MAPPING
===============================================================================

08.1. A command may map to zero, one, or multiple runtime operations.

08.2. StartHost may map to:
      - ResolveWorkspace
      - BindWorkspace
      - AcquireWorkspaceOwnership
      - StartRuntimeHost
      - CreateEntrySurface

08.3. StopHost may map to:
      - StopRuntimeHost
      - ReleaseWorkspaceOwnership
      - CloseEntrySurface

08.4. ConnectHost may map to:
      - ResolveWorkspace
      - DiscoverExistingRuntimeHost
      - ConnectRuntimeHost

08.5. RunAppPack may map to:
      - ResolveWorkspace
      - AcquireWorkspaceOwnership when host ownership is required
      - ResolveAppPack
      - CreateSavePack or LoadSavePack
      - ValidateSavePackCompatibility
      - BindSaveContext
      - CreateAppInstance
      - StartAppInstance
      - CreateView when a view is requested

08.6. CreateSavePack may map to:
      - ResolveAppPack
      - CreateSavePack
      - ValidateSavePackCompatibility
      - BindSaveContext when immediate binding is requested

08.7. LoadSavePack may map to:
      - LoadSavePack
      - ValidateSavePackCompatibility
      - BindSaveContext

08.8. List and inspect commands map to read-only discovery or inspection
      operations unless an explicit policy grants writable repair behavior.

08.9. ExposeLanSurface maps to RuntimeHost LAN exposure operations and
      policy handling under DA-28 and DA-17.

08.10. RecoverStaleWorkspaceOwner maps to stale-owner detection and recovery
       behavior under IS-03.

08.11. Runtime operation mapping does not create independent semantics for a
       command category.

08.12. Runtime operation mapping is trace-visible when runtime operation trace
       evidence exists.


===============================================================================
09. COMMAND RESULT ENVELOPE
===============================================================================

09.1. A command result envelope represents command handling outcome.

09.2. A command result envelope contains:
      - schemaVersion
      - commandId
      - runtimeHostId when available
      - commandCategory
      - commandStatus
      - message
      - details
      - runtimeOperationId when applicable
      - jobId when applicable
      - orchestrationUnitId when applicable
      - appInstanceId when applicable
      - viewId when applicable
      - errorCategory when applicable
      - traceCorrelation when available

09.3. commandStatus is one of:
      - accepted
      - rejected
      - completed
      - failed
      - cancelled
      - detached
      - retained

09.4. accepted means command handling accepted the command for further work.

09.5. rejected means command handling rejected the command before dispatch or
      before accepted work exists.

09.6. completed means command handling reached the command-defined completed
      surface.

09.7. failed means command handling reached the command-defined failed
      surface.

09.8. cancelled means command handling was cancelled at the command level.

09.9. detached means command handling detached from the waiting entry surface
      while owning workflow rules define continued work or retained result
      behavior.

09.10. retained means command handling retained result or evidence for later
       retrieval under owning workflow rules.

09.11. commandStatus does not redefine Job terminal outcome.

09.12. commandStatus does not redefine Orchestration Unit terminal outcome.

09.13. commandStatus does not redefine AppInstance lifecycle outcome.

09.14. commandStatus does not redefine adapter lifecycle outcome.

09.15. details contains structured command-specific data when available.

09.16. message is a user-facing or caller-facing explanation summary.

09.17. Concrete JSON field names may be revised by a future schema
       implementation specification if semantic meaning is preserved.


===============================================================================
10. RELATION TO JOBS
===============================================================================

10.1. A command may create or expose Job-backed work.

10.2. Command acceptance is not Job creation unless the command contract
      defines accepted command handling as Job creation.

10.3. Job-backed command handling returns jobId when a Job exists and the
      caller is allowed to observe it.

10.4. Command completion is not Job success unless the command contract maps
      command completion to Job success explicitly.

10.5. Command failure is not Job failure by default.

10.6. Command cancellation is not Job cancellation by default.

10.7. A command that requests Job cancellation routes the request through
      DA-19 Job lifecycle semantics.

10.8. A command result envelope may report Job state only as Job-domain
      information governed by DA-19.


===============================================================================
11. RELATION TO ORCHESTRATION UNITS
===============================================================================

11.1. A command may cause creation of an Orchestration Unit when it performs
      or may perform orchestrated mutation.

11.2. Command acceptance is not Orchestration Unit creation unless the owning
      workflow defines that relationship.

11.3. Command completion is not Orchestration Unit completion by default.

11.4. Command failure is not Orchestration Unit failure by default.

11.5. Command cancellation is not Orchestration Unit cancellation by default.

11.6. A command rejected before Orchestration Unit creation has no
      Orchestration Unit terminal outcome.

11.7. Mutation-bearing command dispatch enters DA-24 Orchestration Unit
      boundaries before mutation becomes authoritative.

11.8. Command trace evidence may correlate commandId with orchestrationUnitId
      when such a unit exists.


===============================================================================
12. RELATION TO WORKSPACE OWNERSHIP
===============================================================================

12.1. Commands that require writable workspace ownership use IS-03 ownership
      acquisition behavior.

12.2. Commands that release workspace ownership use IS-03 ownership release
      behavior.

12.3. Commands that encounter an existing owner use IS-03 second-owner
      rejection behavior.

12.4. Commands that inspect an already-owned workspace use IS-03 read-only
      inspection behavior when read-only inspection is explicitly requested or
      permitted.

12.5. Commands that recover stale ownership use IS-03 stale-owner recovery
      behavior.

12.6. Command acceptance does not imply workspace ownership acquisition.

12.7. Workspace ownership acquisition does not imply command completion by
      itself.

12.8. Workspace ownership rejection may cause command rejection or command
      failure depending on when the rejection occurs in the command workflow.

12.9. Workspace ownership errors preserve the IS-03 error distinctions when
      reported through command error categories.


===============================================================================
13. TRACE MAPPING
===============================================================================

13.1. IS-04 maps implementation behavior to DA-28 RuntimeHost command and
      operation trace reasons.

13.2. Command receipt maps to RuntimeHostCommandReceived.

13.3. Command classification maps to RuntimeHostCommandClassified.

13.4. Command acceptance maps to RuntimeHostCommandAccepted.

13.5. Command rejection maps to RuntimeHostCommandRejected.

13.6. Command dispatch maps to RuntimeHostCommandDispatched.

13.7. Command completion maps to RuntimeHostCommandCompleted.

13.8. Command failure maps to RuntimeHostCommandFailed.

13.9. Command cancellation maps to RuntimeHostCommandCancelled.

13.10. Runtime operation handling maps to:
       - RuntimeHostOperationRequested
       - RuntimeHostOperationClassified
       - RuntimeHostOperationAccepted
       - RuntimeHostOperationRejected
       - RuntimeHostOperationDispatched
       - RuntimeHostOperationCompleted
       - RuntimeHostOperationFailed
       - RuntimeHostOperationCancelled
       - RuntimeHostOperationSuperseded
       - RuntimeHostOperationDetached
       - RuntimeHostOperationRetained

13.11. Trace evidence includes enough context, when available, to identify:
       - commandId
       - command category
       - runtimeHostId
       - runtimeOperationId
       - startup mode
       - entry surface identity
       - workspace identity
       - appInstanceId
       - jobId
       - orchestrationUnitId
       - RPC message identity
       - policy decision identity
       - error category

13.12. Command trace evidence does not replace RuntimeHost trace evidence.

13.13. Command trace evidence does not replace Job trace evidence.

13.14. Command trace evidence does not replace Orchestration Unit trace
       evidence.

13.15. Command trace evidence does not replace workspace ownership trace
       evidence under IS-03.


===============================================================================
14. ERROR CATEGORIES
===============================================================================

14.1. RuntimeHost command-surface error categories include:
      - CommandMalformed
      - CommandUnknown
      - CommandUnsupported
      - CommandInvalidForStartupMode
      - CommandInvalidForEntrySurface
      - CommandMissingRequiredArgument
      - CommandWorkspaceRequired
      - CommandWorkspaceAlreadyOwned
      - CommandWorkspaceOwnershipFailed
      - CommandPolicyDenied
      - CommandPermissionDenied
      - CommandAppPackNotFound
      - CommandSavePackNotFound
      - CommandSavePackIncompatible
      - CommandRuntimeHostUnavailable
      - CommandRuntimeOperationFailed
      - CommandJobUnavailable
      - CommandAdapterUnavailable
      - CommandImplementationFeatureUnavailable

14.2. Error category names in this section define semantic categories, not
      final exception class names.

14.3. Concrete exception class names belong to implementation code or a later
      error-schema specification.

14.4. Error reporting preserves distinctions between command parsing,
      command classification, policy denial, permission denial, workspace
      ownership failure, missing entities, incompatible save state, and
      downstream runtime operation failure.

14.5. Error reporting does not collapse all command failure into generic
      execution failure when the implementation can distinguish the cause.


===============================================================================
15. WHAT IS EXPLICITLY NOT SUPPORTED
===============================================================================

15.1. The following are not supported:
      - treating command syntax as authority
      - treating command receipt as command acceptance
      - treating command acceptance as command completion
      - treating command completion as Job success by default
      - treating command failure as Job failure by default
      - treating command failure as Orchestration Unit failure by default
      - treating command cancellation as Job cancellation by default
      - treating command cancellation as Orchestration Unit cancellation by
        default
      - silently selecting a command alternative after workspace ownership
        rejection
      - silently downgrading a writable command to read-only inspection
      - silently starting a second writable RuntimeHost for the same workspace
      - silently creating separate SavePack creation semantics for one command
        surface
      - silently creating separate AppInstance creation semantics for one
        command surface
      - command dispatch bypassing capability, policy, persistence,
        transaction, Orchestration Unit, AppInstance, adapter, RPC, or trace
        boundaries

17. LOCKED SUMMARY
===============================================================================

17.1. LOCKED:
      - IS-04 implements DA-28 RuntimeHost command-surface semantics.
      - A command surface is not Kernel authority by itself.
      - commandId is distinct from runtimeHostId, runtimeOperationId, jobId,
        orchestrationUnitId, appInstanceId, adapter identity, viewId, and RPC
        message identity.
      - Command receipt is not command acceptance.
      - Command acceptance is not command completion.
      - Command completion is not Job success by default.
      - Command failure is not Job failure by default.
      - Command failure is not Orchestration Unit failure by default.
      - Command syntax does not define authority.
      - Command categories define semantic categories, not final CLI syntax.
      - Runtime operation mapping does not create independent semantics for a
        command category.
      - Commands that require writable workspace ownership use IS-03.
      - Mutation-bearing command dispatch enters DA-24 Orchestration Unit
        boundaries before mutation becomes authoritative.
      - Command handling preserves DA-20, DA-21, DA-22, DA-23, DA-27, DA-28,
        DA-04, DA-07, DA-16, DA-17, DA-19, and DA-24 boundaries.
      - Command receipt, classification, acceptance, rejection, dispatch,
        completion, failure, and cancellation are trace-visible.

17.2. OPEN:
      - concrete command identifiers, parser shape, CLI syntax, result schema,
        retention behavior, subscription surface, error classes, and help text.
