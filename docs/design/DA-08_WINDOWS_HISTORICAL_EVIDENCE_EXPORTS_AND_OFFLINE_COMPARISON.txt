===============================================================================
DESIGN AREA DA-08 - WINDOWS AND FINALIZED RUNTIME HISTORY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how Turnix groups runtime history into bounded Windows, how
SafePointWindow, ActiveWindow, and AbortedWindow regions behave, and how
Window finalization preserves committed boundaries and non-authoritative
evidence without turning retained history into a hidden correctness
dependency.

This document defines Window and finalization semantics around committed
state. It does not redefine the committed-state model itself.

Purpose:
--------
This document defines the Turnix model for Windows and finalized runtime
history.

It defines:
- what Windows mean in Turnix
- SafePointWindow, ActiveWindow, and AbortedWindow semantics
- Window identity terms
- finalization of in-flight runtime regions
- relationship between Window finalization, Orchestration Units, and Jobs
- SafePointWindow creation request modes and lifecycle visibility
- journal as non-authoritative evidence
- trace correlation rules for finalized Windows

It does NOT define:
- the general memory and authoritative-state model (see DA-04)
- payload identity, ChunkId, or ValueRef representation contracts
  (see DA-05)
- concrete persistent I/O authority, Readers, Writers, or filesystem rules
  (see DA-07)
- tracing structure, trace event schema, span semantics, or trace storage
  semantics (see DA-03)
- configuration-specific state, schema, or migration semantics
  (see DA-06)
- debugger pause, breakpoint, or execution-control behavior (see DA-18)
- Job lifecycle, Job terminal states, or Job result semantics (see DA-19)
- Orchestration Unit lifecycle or mutation authority semantics (see DA-24)
- WindowManifest, WindowExport, Minimal Audit Bundle, digest, or offline
  inspection artifact semantics (see DA-25)
- equality modes, ComparisonResult, Replay, Resume, Retry, or
  cross-session audit workflow semantics (see DA-26)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines Window and finalized runtime history semantics
      within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. This design area answers:

          "How does Turnix preserve bounded committed-state history and
           bounded historical evidence without making retained history a
           hidden correctness dependency?"

02.2. Turnix groups runtime history into bounded Windows.

02.3. A Window is a bounded region of runtime history and retained evidence
      associated with one AppInstance.

02.4. Windows distinguish:
      - authoritative committed boundaries
      - non-authoritative in-flight activity
      - non-authoritative historical evidence

02.5. Finalized Windows preserve what was committed, what was aborted, and
      what evidence remains retained for explanation, audit, and offline
      inspection.

02.6. This document does not redefine authoritative state.

02.7. This document defines how finalized Windows relate to authoritative
      state and retained evidence.

02.8. Historical evidence may support explanation, audit, and debugging.

02.9. Historical evidence does not become a hidden correctness dependency.

02.10. Job lifecycle evidence and Orchestration Unit evidence may contribute
       to Window evidence.

02.11. Job lifecycle evidence and Orchestration Unit evidence do not by
       themselves finalize a Window.

02.12. Window finalization remains owned by this document.

02.13. Window-derived artifact, export, offline comparison, Replay, Resume,
       Retry, and audit workflow semantics are defined by DA-25 and DA-26.


===============================================================================
03. JOURNAL AS NON-AUTHORITATIVE EVIDENCE
===============================================================================

03.1. The journal records observational evidence of system activity.

03.2. Journal content is:
      - append-oriented
      - non-authoritative
      - non-blocking
      - evidence-oriented rather than correctness-defining

03.3. Journal content may include:
      - raw user input attempts
      - raw assistant output, including aborted streams
      - intermediate pipeline artifacts
      - debugging captures
      - other explanatory material

03.4. Journal data is not state.

03.5. Journal data is not required for correctness.

03.6. Journal loss does not change committed-state meaning.

03.7. Journal retention follows explicit best-effort policy.

03.8. If retention limits are exceeded, older journal material may be
      pruned.

03.9. If storage is unavailable or under severe pressure, journal writes may
      be dropped rather than block execution.

03.10. Journal pruning or dropped writes are trace-visible and classified.

03.11. Journal entries may declare a retention class.

03.12. Retention classes may include:
       - ephemeral
       - bounded
       - pinned

03.13. Retention class affects garbage collection only.

03.14. Journal retention policy does not alter authority boundaries.


03.1 JOURNAL RETENTION POLICIES
-------------------------------

03.1.1. Journal retention policy is explicit.

03.1.2. Journal retention policy is inspectable.

03.1.3. Journal retention policy is configurable.

03.1.4. Journal retention policy may use one or more policy axes such as:
        - maximum total size
        - maximum age
        - per-appInstance quota
        - per-save quota
        - per-category quota

03.1.5. Journal garbage collection may operate incrementally.

03.1.6. Incremental journal garbage collection does not block execution.

03.1.7. Journal garbage collection remains trace-visible.

03.1.8. Journal garbage collection does not weaken the non-authoritative
        status of journal evidence.


===============================================================================
04. WINDOW MODEL
===============================================================================

04.1. An AppInstance maintains one explicit ordered Window sequence.

04.2. Every Window in that sequence is exactly one of:
      - SafePointWindow
      - AbortedWindow
      - ActiveWindow

04.3. No other Window kind is part of this model unless a later
      authoritative revision defines it explicitly.

04.4. At most one ActiveWindow exists for one AppInstance at a time.

04.5. If an ActiveWindow exists, it is the unique mutable tail of the
      Window sequence.

04.6. SafePointWindow and AbortedWindow are finalized Window forms.
      ActiveWindow is not finalized.

04.7. Window ordering is explicit.

04.8. Window ordering does not rely on timestamps, inference, or
      presentation order.

04.9. The ordered Window sequence preserves:
      - finalized committed boundaries
      - finalized abort boundaries
      - the current in-flight region, if one exists

04.10. Window ordering is distinct from Job lifecycle ordering.

04.11. Window ordering is distinct from Orchestration Unit lifecycle
       ordering.

04.12. Job and Orchestration Unit lifecycle evidence remains correlated to
       Windows through trace and retained evidence rather than through
       timestamp inference.


04.1 WINDOW IDENTITY TERMS
--------------------------

04.1.1. safePointWindowId identifies one SafePointWindow.

04.1.2. abortedWindowId identifies one AbortedWindow.

04.1.3. safePointWindowId and abortedWindowId are UUID values.

04.1.4. safePointWindowId and abortedWindowId are representable in plain
        text.


===============================================================================
05. SAFEPOINTWINDOW
===============================================================================

05.1. A SafePointWindow is a finalized authoritative committed boundary.

05.2. A SafePointWindow contains committed layers and committed config as
      authoritative material.

05.3. The authoritative portion of a SafePointWindow contains committed
      state material only.

05.4. Transactional layer contents are not part of the authoritative
      SafePointWindow boundary.

05.5. A SafePointWindow remains deterministically restorable within the
      committed-state restoration boundary defined by
      [defined-in: DA-04 12].

05.6. A retained SafePointWindow includes retained trace identity and scope
      evidence sufficient to identify:
      - the SafePointWindow itself
      - the committed scope or scopes it covers
      - its relation to the committed work it finalizes

05.7. A SafePointWindow does not contain transactional layer contents as
      authoritative material.

05.8. A SafePointWindow may additionally retain, by policy:
      - revision history
      - journal material
      - additional trace evidence
      - Job lifecycle evidence compatible with retained scope
      - Orchestration Unit lifecycle evidence compatible with retained scope
      - other non-authoritative evidence compatible with this document

05.9. Optional retained evidence does not add authority.

05.10. A SafePointWindow has safePointWindowId as stable identity.

05.11. Trace, journal, revision, Job lifecycle, and Orchestration Unit
       lifecycle material retained within a SafePointWindow may be truncated
       or removed according to explicit policy.

05.12. Retention reduction within a SafePointWindow does not invalidate the
       retained authoritative committed-state boundary, provided enough
       authoritative material remains to restore that SafePointWindow
       deterministically.

05.13. A retained SafePointWindow is not treated as complete if retained
       trace identity and scope evidence are insufficient to identify the
       Window and its committed scope.

05.14. A Job reaching Succeeded does not by itself create a SafePointWindow.

05.15. An Orchestration Unit reaching Completed does not by itself create a
       SafePointWindow.

05.16. A transaction commit does not by itself create a SafePointWindow.

05.17. SafePointWindow artifact and export representation is defined by
       DA-25.


===============================================================================
06. ACTIVEWINDOW
===============================================================================

06.1. ActiveWindow is the current in-flight region after the last finalized
      committed boundary.

06.2. ActiveWindow contains activity that has not yet been finalized as a
      SafePointWindow or AbortedWindow.

06.3. ActiveWindow contains:
      - trace events emitted after the last finalized SafePointWindow
      - journal entries produced after the last finalized SafePointWindow
      - metadata about active Orchestration Units sufficient for diagnosis
      - metadata about active Jobs sufficient for diagnosis when Jobs exist
      - transactional staging layers and in-flight transactional state

06.4. ActiveWindow is non-authoritative.

06.5. ActiveWindow does not define committed truth.

06.6. ActiveWindow may be discarded during recovery.

06.7. ActiveWindow is not interpreted as a deterministic replay boundary.

06.8. ActiveWindow does not guarantee deterministic reconstruction of:
      - in-flight activity
      - active or pending Job execution
      - active Orchestration Unit execution
      - external side effects
      - pre-crash execution ordering beyond what later committed boundaries
        explicitly retain

06.9. ActiveWindow trace and journal material are not pruned out from the
      in-flight region merely because they are non-authoritative.

06.10. If storage pressure occurs:
       - journal writes may be dropped according to explicit policy
       - trace records still preserve the fact of omission

06.11. Omission handling remains trace-visible and classified.

06.12. ActiveWindow does not require its own stable Window identity while it
       remains the unique mutable tail of the Window sequence.

06.13. ActiveWindow may contain Jobs in Pending or Running state.

06.14. ActiveWindow may contain Orchestration Units that have not yet reached
       a terminal outcome.

06.15. ActiveWindow evidence does not convert Pending or Running Jobs into
       replayable work.


===============================================================================
07. ABORTEDWINDOW
===============================================================================

07.1. An AbortedWindow is a finalized ActiveWindow that did not produce a
      new authoritative committed boundary.

07.2. An AbortedWindow contains:
      - retained trace evidence describing what happened after the previous
        finalized SafePointWindow
      - a finalization outcome identifying the abort path and reason
        category

07.3. An AbortedWindow may retain, by policy:
      - journal evidence
      - Job lifecycle evidence compatible with retained scope
      - Orchestration Unit lifecycle evidence compatible with retained scope
      - other non-authoritative evidence compatible with this document

07.4. An AbortedWindow does not contain authoritative committed-state
      changes as part of its finalized outcome.

07.5. An AbortedWindow does not preserve transactional layer contents as
      authoritative material.

07.6. An AbortedWindow is finalized evidence of non-commit outcome, not a
      committed boundary.

07.7. An AbortedWindow does not acquire authority merely because it is
      finalized, retained, or exported.

07.8. An AbortedWindow has abortedWindowId as stable identity.

07.9. An AbortedWindow may be truncated or partially removed according to
      explicit policy.

07.10. If an AbortedWindow is retained at all, it retains enough trace
       meaning to explain:
       - that the Window was aborted rather than committed
       - why it was aborted
       - how it relates causally to the retained evidence still present

07.11. AbortedWindow finalization trace evidence records enough context to
       explain:
       - appInstanceId
       - abortedWindowId
       - abort reason category
       - causal relation to retained evidence still present

07.12. An AbortedWindow may be fully removed only if explicit policy permits
       the removal and either:
       - at least one later SafePointWindow exists
       - or the engine explicitly accepts loss of that audit region

07.13. When such removal occurs, the platform remains explicit and honest
       about the loss of that audit region.

07.14. A Job reaching Failed, Cancelled, TimedOut, or Superseded does not by
       itself create an AbortedWindow.

07.15. An Orchestration Unit reaching Failed, Cancelled, or Superseded does
       not by itself create an AbortedWindow.

07.16. A transaction abort does not by itself create an AbortedWindow.

07.17. AbortedWindow artifact and export representation is defined by
       DA-25.


===============================================================================
08. FINALIZATION RULES
===============================================================================

08.1. ActiveWindow finalizes as exactly one outcome:
      - SafePointWindow
      - AbortedWindow

08.2. Finalization never produces both outcomes for the same ActiveWindow.

08.3. Finalization never produces a partial finalized artifact.

08.4. Finalization never preserves an unresolved mixed state in which a
      Window is partly authoritative and partly aborted.

08.5. Finalization into SafePointWindow requires that all transactions
      relevant to the finalized committed boundary are resolved.

08.6. Finalization into SafePointWindow produces:
      - a new SafePointWindow
      - a new safePointWindowId
      - trace-visible success of finalized committed-boundary creation

08.7. Finalization into AbortedWindow occurs when in-flight work concludes
      without producing a new SafePointWindow and the authoritative committed
      state remains that of the last finalized SafePointWindow.

08.8. Finalization into AbortedWindow produces:
      - a new AbortedWindow
      - a new abortedWindowId
      - trace-visible abort finalization with reason category

08.9. Finalization preserves the distinction between:
      - committed authoritative outcome
      - non-authoritative retained evidence about failed or aborted work

08.10. Job terminal outcome does not by itself define Window finalization.

08.11. Orchestration Unit terminal outcome does not by itself define Window
       finalization.

08.12. Transaction commit or abort does not by itself define Window
       finalization.

08.13. Window finalization may include retained evidence about Job,
       Orchestration Unit, and transaction outcomes when that evidence falls
       within the finalized Window scope.

08.14. Including such evidence does not make the evidence authoritative
       committed state.


===============================================================================
09. SAFEPOINTWINDOW CREATION REQUEST MODES
===============================================================================

09.1. Turnix may support multiple SafePointWindow creation request modes.

09.2. Request-mode semantics are explicit.

09.3. Request-mode semantics are trace-visible.

09.4. Turnix does not silently substitute one request mode for another.


09.1 IMMEDIATECOMMITTED
-----------------------

09.1.1. ImmediateCommitted means that SafePointWindow creation is attempted
        without waiting for quiescence.

09.1.2. If an eligible SafePointWindow can be created immediately, it is
        created.

09.1.3. If an eligible SafePointWindow cannot be created immediately, the
        request fails deterministically with an explicit reason.

09.1.4. ImmediateCommitted does not silently degrade into waiting behavior.


09.2 ATQUIESCENCE
-----------------

09.2.1. AtQuiescence means that SafePointWindow creation is attempted only
        when no Orchestration Unit is active for the target AppInstance.

09.2.2. If AtQuiescence is requested while an Orchestration Unit is active,
        Turnix may queue the request until quiescence is reached.

09.2.3. Queued AtQuiescence requests are explicit.

09.2.4. Queued AtQuiescence requests are trace-visible.

09.2.5. If queued AtQuiescence requests are not persisted, that limitation
        remains explicit.

09.2.6. A queued AtQuiescence request may be lost on crash or restart only
        when that loss is compatible with explicit policy and explicit trace
        visibility before the crash point when possible.

09.2.7. Queued AtQuiescence does not mean queued Job execution.

09.2.8. Queued AtQuiescence does not alter Job lifecycle state by itself.


09.3 QUEUEING, TIMEOUT, AND SAFE-POINT PAUSE POLICY
---------------------------------------------------

09.3.1. AtQuiescence waiting is bounded by explicit policy.

09.3.2. Bounded waiting may use one or more policy mechanisms such as:
        - timeout
        - pause-at-safe-point attempt

09.3.3. If timeout policy applies and quiescence is not reached within the
        configured bound, the request fails deterministically.

09.3.4. If pause-at-safe-point policy is used, Turnix may attempt a
        deterministic pause at a defined safe point.

09.3.5. If safe-point pause cannot be achieved safely, the request fails
        with an explicit reason.

09.3.6. Turnix does not silently wait forever.

09.3.7. Turnix does not silently cancel in-flight work in order to satisfy
        SafePointWindow creation.

09.3.8. Turnix does not silently downgrade AtQuiescence into
        ImmediateCommitted.

09.3.9. Any user or policy decision that changes creation behavior remains
        explicit and trace-visible.

09.3.10. SafePointWindow creation timeout is not Job timeout.

09.3.11. SafePointWindow creation cancellation is not Job cancellation.

09.3.12. If Window creation policy affects a Job or Orchestration Unit, that
         effect remains governed by the Job, Orchestration Unit, and policy
         documents that own those lifecycle outcomes.


===============================================================================
10. SAFEPOINTWINDOW CREATION LIFECYCLE VISIBILITY
===============================================================================

10.1. SafePointWindow creation has an explicit lifecycle.

10.2. The lifecycle includes:
      - request
      - queued state when applicable
      - creation start
      - creation success
      - creation failure
      - creation cancellation when cancellation occurs

10.3. Turnix emits trace-visible lifecycle evidence for each applicable
      stage of that lifecycle.

10.4. SafePointWindow creation trace evidence distinguishes:
      - SafePointWindow creation requested
      - SafePointWindow creation queued
      - SafePointWindow creation started
      - SafePointWindow creation succeeded
      - SafePointWindow creation failed
      - SafePointWindow creation cancelled

10.5. DA-08 owns the Window-domain trace reason identifiers used for
      SafePointWindow creation and ActiveWindow abort finalization.

10.6. The following Window-domain trace reasons are defined:

      - SafePointWindowCreationRequested
      - SafePointWindowCreationQueued
      - SafePointWindowCreationStarted
      - SafePointWindowCreationSucceeded
      - SafePointWindowCreationFailed
      - SafePointWindowCreationCancelled
      - ActiveWindowFinalizedAsAborted

10.7. SafePointWindowCreationRequested trace evidence records enough context
      to explain:
      - appInstanceId
      - requestMode
      - requestId

10.8. SafePointWindowCreationQueued trace evidence records enough context to
      explain:
      - appInstanceId
      - requestId
      - queuePolicyId when applicable

10.9. SafePointWindowCreationStarted trace evidence records enough context to
      explain:
      - appInstanceId
      - requestId

10.10. SafePointWindowCreationSucceeded trace evidence records enough context
       to explain:
       - appInstanceId
       - requestId
       - safePointWindowId
       - committedStateRoot when available
       - includedEvidencePolicyId when applicable

10.11. SafePointWindowCreationFailed trace evidence records enough context
       to explain:
       - appInstanceId
       - requestId
       - failureReasonCategory

10.12. SafePointWindowCreationCancelled trace evidence records enough context
       to explain:
       - appInstanceId
       - requestId
       - cancelReasonCategory

10.13. ActiveWindowFinalizedAsAborted trace evidence records enough context
       to explain:
       - appInstanceId
       - abortedWindowId
       - abortReasonCategory

10.14. Queued state remains distinguishable from started creation.

10.15. Failure does not create a partial finalized artifact.

10.16. Cancellation is explicit.

10.17. Lifecycle visibility remains consistent with the creation request mode
       in effect.

10.18. SafePointWindow creation lifecycle trace evidence may include jobId or
       orchestrationUnitId when Window creation is causally linked to a Job
       or Orchestration Unit.

10.19. DA-03 defines the tracing substrate used to represent
       SafePointWindow creation and ActiveWindow finalization evidence.


===============================================================================
11. TRACE CORRELATION WITH WINDOWS AND FINALIZED EVIDENCE
===============================================================================

11.1. This document consumes the trace substrate defined by DA-03.

11.2. This document defines Window-specific trace obligations.

11.3. This document does not redefine trace event structure, trace DAG
      structure, span semantics, or trace storage semantics.

11.4. Finalized Windows are correlated to trace evidence through explicit
      retained trace identity and causal relationships rather than through
      timestamps alone.

11.5. Each SafePointWindow is identifiable through retained trace evidence
      that states:
      - safePointWindowId
      - the committed scope or scopes covered
      - the reason or creation path
      - whether optional evidence classes are included by policy

11.6. Each AbortedWindow is identifiable through retained trace evidence
      that states:
      - abortedWindowId
      - the abort outcome
      - the abort reason category
      - the causal relation to the retained evidence still present

11.7. Trace correlation distinguishes:
      - events that contributed to committed state included in a
        SafePointWindow
      - events that remained in later ActiveWindow or AbortedWindow regions

11.8. This distinction does not rely on timestamps alone.

11.9. If retained finalized evidence includes a trace span end event, the
      corresponding span begin event is also included.

11.10. Evidence carry-forward may reference older retained trace material
       only to preserve internal consistency of the retained evidence
       closure.

11.11. Trace incompleteness, pruning, corruption, or partial export remains
       explicit when it affects retained finalized evidence.

11.12. Job lifecycle evidence retained inside finalized Window evidence
       remains non-authoritative evidence unless it is also part of
       committed state under DA-04.

11.13. Orchestration Unit evidence retained inside finalized Window evidence
       remains non-authoritative evidence unless it is also part of
       committed state under DA-04.


===============================================================================
12. RELATION TO WINDOW ARTIFACTS AND WORKFLOWS
===============================================================================

12.1. DA-25 defines WindowManifest, WindowExport, Minimal Audit Bundle,
      optional digest representation, partial export, missing material, and
      offline inspection artifact semantics.

12.2. DA-26 defines equality modes, ComparisonResult, Replay, Resume, Retry,
      and cross-session audit workflow semantics.

12.3. A Window finalization outcome does not by itself create a WindowExport.

12.4. A Window finalization outcome does not by itself create a Minimal
      Audit Bundle.

12.5. A Window finalization outcome does not by itself perform comparison,
      Replay, Resume, Retry, or audit workflow work.

12.6. Artifact and workflow documents consume Window finalization semantics.
      They do not redefine Window finalization.

12.7. This document remains the authority for Window kind, Window ordering,
      Window finalization, and Window-domain trace reason ownership.

14. LOCKED SUMMARY
===============================================================================

14.1. LOCKED:
      - runtime history is grouped into ordered Windows
      - Windows are SafePointWindow, AbortedWindow, and ActiveWindow
      - safePointWindowId and abortedWindowId are shared Window identity
        terms
      - SafePointWindow is a finalized authoritative committed boundary
      - ActiveWindow is the current non-authoritative in-flight region
      - AbortedWindow is a finalized non-authoritative abort region
      - finalization produces exactly one finalized outcome from
        ActiveWindow
      - Job terminal outcome does not by itself define Window finalization
      - Orchestration Unit terminal outcome does not by itself define Window
        finalization
      - transaction commit or abort does not by itself define Window
        finalization
      - SafePointWindow creation modes and lifecycle visibility are explicit
      - SafePointWindow creation timeout is not Job timeout
      - SafePointWindow creation cancellation is not Job cancellation
      - SafePointWindow creation trace obligations are owned here
      - Window-domain trace reason identifiers are owned by DA-08
      - journal is non-authoritative evidence and retention loss does not
        affect committed-state meaning
      - journal retention policy is explicit, inspectable, and configurable
      - a retained SafePointWindow is not complete unless retained trace
        identity and scope evidence remain sufficient to identify the Window
        and its committed scope
      - finalized Windows are correlated to retained trace evidence through
        explicit causal identity rather than timestamps alone
      - retained Job and Orchestration Unit evidence inside finalized Window
        evidence remains non-authoritative unless committed under DA-04
      - DA-25 owns Window artifact, export, Minimal Audit Bundle, digest, and
        offline inspection artifact semantics
      - DA-26 owns equality, ComparisonResult, Replay, Resume, Retry, and
        cross-session audit workflow semantics
      - artifact and workflow documents consume Window finalization
        semantics; they do not redefine Window finalization

14.2. OPEN:
      - exact Window finalization request API
      - exact ActiveWindow and finalized Window retention policy shape
      - exact Window sequence storage representation
      - exact Window-domain trace payload taxonomy
