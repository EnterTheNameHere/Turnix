===============================================================================
DESIGN DOCTRINE DD-01 - TURNIX CREATOR INTERACTION DOCTRINE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the relationship between Turnix, creators, users, and Turnix
authors.

Purpose:
--------
This document defines how Turnix treats creators, where platform
responsibility begins and ends, and when the platform steps in.

It defines:
- role terminology
- creator trust model
- Turnix intervention boundaries
- diagnostic preference
- observability expectations

It does NOT define:
- pack loading semantics
- capability permission semantics
- tracing data structures
- memory or journaling storage format

Upstream Documents:
------------------
- CMR-01
- CMR-02
- CMR-03
- CMR-04
- CMR-05


===============================================================================
01. ROLE DEFINITIONS
===============================================================================

01.1. A user is the person running a Turnix application instance.

01.2. A user may run:
      - the main menu
      - an appPack
      - a headless evaluation
      - an AI buddy
      - a Turnix-based game or simulation

01.3. A creator is a person producing content that Turnix runs.

01.4. A creator may produce:
      - mods
      - asset packs
      - schemas
      - appPacks
      - save-related content
      - other Turnix-executed content

01.5. A creator uses Turnix as a platform.

01.6. A creator may also be a user when operating in a development role.

01.7. Turnix authors are the developers of the Turnix platform itself.

01.8. Turnix authors design and maintain:
      - internal architecture
      - runtime boundaries
      - continuity infrastructure
      - first-party packs
      - core invariants and guarantees


===============================================================================
02. CREATOR TRUST MODEL
===============================================================================

02.1. Turnix treats creators as competent adults.

02.2. Turnix does not assume that creator mistakes should be prevented by
      default.

02.3. Creators remain free to:
      - experiment
      - bypass conventions
      - write unusual code
      - ignore warnings
      - take deliberate risks in their own content

02.4. Creator freedom ends where Turnix integrity, platform-tracked
      authoritative state, or guaranteed platform behavior would be
      compromised.

02.5. Turnix protects platform guarantees and integrity. Turnix does not
      restrict creator behavior without platform-level justification.


===============================================================================
03. PLATFORM INTERVENTION BOUNDARY
===============================================================================

03.1. Turnix intervenes when platform integrity or guaranteed behavior is
      at risk.

03.2. Creator-local mistakes do not by themselves justify broader platform
      intervention.

03.3. Platform integrity risk includes cases such as:
      - corrupted tracing state
      - corrupted memory state
      - broken journaling continuity
      - invalid pack loader invariants
      - invalid runtime boundary behavior

03.4. Guaranteed behavior risk includes cases where explicit Turnix
      guarantees would be broken if execution continued silently.

03.5. If a creator mistake remains local to creator-owned behavior and does
      not threaten Turnix integrity or guarantees, Turnix prefers
      observation and diagnosis over platform takeover.

03.6. Turnix does not expand local creator mistakes into broader platform
      intervention without clear reason.


===============================================================================
04. DIAGNOSTICS BEFORE PROHIBITION
===============================================================================

04.1. When a misuse pattern is visible and reliably identifiable, Turnix
      may detect it and record diagnostics.

04.2. Turnix records what happened, where it happened, and the resulting
      consequence.

04.3. Turnix may surface creator-facing diagnostics such as:
      - warnings
      - post-run issue reports
      - development-time notices
      - structured trace records

04.4. Turnix does not add fragile pattern detection merely because a misuse
      is undesirable.

04.5. Turnix adds system-level handling only when the pattern is
      sufficiently specific, the signal is reliable, and the response is
      worth the added complexity.

04.6. Diagnostic visibility is preferred over blanket prohibition when
      creator freedom remains compatible with platform safety.


===============================================================================
05. CONTAINMENT AND ERROR OWNERSHIP
===============================================================================

05.1. Execution boundaries matter.

05.2. When creator code runs inside a defined boundary, boundary handling
      determines whether a failure remains creator-local or constitutes a
      platform defect.

05.3. A creator error that escapes a boundary which Turnix defined as a
      containment boundary indicates a Turnix boundary failure.

05.4. A creator error that remains inside its defined boundary remains a
      creator-side failure unless additional platform guarantees were
      violated.

05.5. Turnix distinguishes between:
      - creator mistakes inside creator-owned space
      - Turnix failures to contain, translate, or observe those mistakes


===============================================================================
06. OBSERVABILITY REQUIREMENT
===============================================================================

06.1. This doctrine requires observability.

06.2. Turnix records enough information to support:
      - diagnosis
      - audit
      - creator feedback
      - post-run review
      - continuity-safe recovery decisions

06.3. Without observability, creator freedom cannot be diagnosed,
      audited, or contained reliably.

06.4. Tracing is part of platform accountability rather than optional
      decoration.


===============================================================================
07. SUMMARY
===============================================================================

07.1. Turnix gives creators room to build, experiment, and fail.

07.2. Turnix protects platform integrity, authoritative state, and
      guaranteed behavior.

07.3. Turnix prefers boundaries, diagnostics, and observability over
      unnecessary restriction.

07.4. Creator freedom and platform safety coexist through containment,
      explicit guarantees, and truthful runtime observation.
