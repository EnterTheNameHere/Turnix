===============================================================================
DESIGN DOCTRINE DD-05 - TURNIX COOPERATION MODES FOR DEVELOPMENT
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines development cooperation modes used for Turnix design, review, and
code materialization when collaborating with language model assistants.

Purpose:
--------
This document defines explicit working modes for Turnix collaboration so
that design reasoning, patch planning, review, and code generation do not
blur into one another.

Turnix development uses language model assistants heavily while building an
AI-capable platform. Explicit cooperation modes exist to preserve design
truth, scope control, and reviewability under that workflow.

It defines:
- mode purposes
- mode boundaries
- mode transition expectations
- materialization discipline
- review focus
- helper-assisted workflow patterns for document work

It does NOT define:
- Turnix runtime semantics
- pack behavior semantics
- frontend behavior
- storage backends
- authoritative document-system rules governed by CMR documents

===============================================================================
01. GENERAL MODE PRINCIPLE
===============================================================================

01.1. Turnix development work is not one undifferentiated stream of
      discussion and code generation.

01.2. Different stages of work have different valid outputs.

01.3. Explicit modes reduce scope drift, accidental code spill, and
      architectural confusion.

01.4. Mode boundaries matter because Turnix work often involves large,
      interdependent systems and heavy use of language model assistants.
      In that setting, premature materialization increases drift risk.

01.5. Cooperation modes exist to preserve design truth, scope control, and
      reviewability across that workflow.

01.6. Reusable AI helper artifacts may support these modes operationally.
      Such helpers assist workflow discipline. They do not become doctrine
      authority by themselves.


===============================================================================
02. DESIGN MODE
===============================================================================

02.1. Design Mode exists for architecture, invariants, responsibilities,
      and conceptual structure.

02.2. Design Mode discusses what the system is, what the system guarantees,
      and how major parts relate.

02.3. Design Mode does not emit implementation code as part of normal work.

02.4. Design Mode may produce:
      - doctrine text
      - invariant lists
      - component boundaries
      - failure analyses
      - semantic decisions

02.5. Design Mode remains active until the conceptual direction is clear
      enough to plan implementation work without guessing.

02.6. When semantics remain fuzzy, question-hardening workflow is preferred
      over premature rewriting or materialization.


===============================================================================
03. REFACTOR PLANNING MODE
===============================================================================

03.1. Refactor Planning Mode exists to plan structural change before code
      materialization.

03.2. This mode identifies:
      - what is wrong
      - what must change
      - what must remain stable
      - where drift risk exists
      - what helpers or abstractions are needed

03.3. Refactor Planning Mode does not emit full implementation code.

03.4. Refactor Planning Mode produces ordered transformation intent rather
      than concrete file output.


===============================================================================
04. PATCH MAP MODE
===============================================================================

04.1. Patch Map Mode exists to describe targeted file changes precisely.

04.2. This mode identifies:
      - insertion locations
      - replacement regions
      - moves
      - deletions
      - touched boundaries
      - dependency-sensitive regions

04.3. Patch Map Mode keeps work structural and localized.

04.4. Patch Map Mode does not regenerate full files merely to express
      intent.

04.5. Patch Map output prepares later materialization while keeping design
      space open for additional review.


===============================================================================
05. MATERIALIZATION MODE
===============================================================================

05.1. Materialization Mode exists to emit code or other concrete artifacts
      after prior planning is accepted.

05.2. Materialization follows approved scope.

05.3. Materialization does not introduce opportunistic refactors,
      unrelated renames, or side-improvements merely because they appear
      possible.

05.4. Materialization preserves established invariants and contract rules.

05.5. Materialization is the point where abstract intent becomes concrete
      code, so scope discipline is strict here.

05.6. When partial output is sufficient, materialization may emit only the
      touched sections instead of the full file.


===============================================================================
06. REVIEW MODE
===============================================================================

06.1. Review Mode evaluates concrete output for correctness and alignment.

06.2. Review examines:
      - contract preservation
      - naming clarity
      - helper reuse
      - containment behavior
      - misuse opportunities
      - drift from agreed scope

06.3. Review Mode does not quietly reopen architecture unless review
      reveals a real architectural issue.

06.4. Review may return work to an earlier mode when concrete output
      exposes a planning defect.

06.5. For document work, semantic review is preferred over cosmetic review.
      Review distinguishes local rule changes, document-architecture
      changes, and cross-document dependency changes.


===============================================================================
07. AUDIT MODE
===============================================================================

07.1. Audit Mode exists for deeper system verification beyond local code
      review.

07.2. Audit Mode focuses on structural properties such as:
      - trace correctness
      - continuity guarantees
      - failure containment
      - state integrity
      - persistence safety
      - boundary truthfulness

07.3. Audit Mode may inspect whether the implementation still matches the
      intended doctrine and design assumptions.

07.4. For document batches, audit-style work may include a cross-document
      consistency sweep where local correctness alone is not sufficient.


===============================================================================
08. MODE TRANSITIONS
===============================================================================

08.1. Mode changes are stated explicitly.

08.2. Explicit mode changes prevent accidental mixing of design
      discussion, patch planning, and code emission.

08.3. If work starts drifting into a different class of output, the work
      is re-anchored by restating the active mode.

08.4. Mode transitions are not decorative labels. They are cooperation
      controls.

08.5. For document work, the following sub-mode distinction is useful and
      may be stated explicitly when needed:
      - existing-document integration
      - new-document drafting

08.6. Existing-document integration preserves valid prior meaning while
      inserting changed or newly locked meaning.

08.7. New-document drafting defines a new authority home and should not be
      treated as if it were merely a rewrite of an existing document.


===============================================================================
09. SCOPE DISCIPLINE
===============================================================================

09.1. Scope belongs to the active mode and to the accepted task.

09.2. Unrequested tightening, post-hoc cleverness, and while-here changes
      create extra review cost and drift risk.

09.3. A possible improvement is not automatically part of the current
      task.

09.4. If an out-of-scope improvement matters for correctness, it is
      called out explicitly.

09.5. If it does not matter for correctness, it waits for a separate
      pass.

09.6. In document work, scope discipline includes resisting vocabulary-only
      generalization and preserving the narrowest correct term unless a
      broader design was intentionally accepted.


===============================================================================
10. SUMMARY
===============================================================================

10.1. Turnix cooperation modes separate design truth, structural intent,
      patch targeting, concrete materialization, and review.

10.2. This separation preserves clarity, reduces drift, and makes large
      Turnix work more controllable for both humans and language model
      assistants.

10.3. Heavy use of language model assistants does not remove the need for
      explicit cooperation discipline. It makes that discipline more
      important.

10.4. Reusable helper artifacts may support this doctrine operationally,
      especially for semantic review, integration workflow, drafting
      workflow, and consistency sweeps.
