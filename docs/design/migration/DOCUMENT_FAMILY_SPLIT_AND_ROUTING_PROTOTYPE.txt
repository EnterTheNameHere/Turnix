===============================================================================
DOCUMENT FAMILY SPLIT AND ROUTING PROTOTYPE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Records the intent, constraints, and continuation rules for the prototype that
splits monolithic Turnix design documents into connector-safe design area
families.

Purpose:
--------
This note exists to rehydrate future work on the document-family migration
without requiring the assistant or human reader to reconstruct the experiment
from conversation history.

This note is a migration and rehydration note. It does not define Turnix
runtime semantics.

It defines:
- why the document split prototype exists
- what structural change is being tested
- which documents participate in the prototype
- how future work should continue the experiment
- which workarounds are intentionally rejected

It does NOT define:
- tracing semantics
- CMR interpretation rules
- final document numbering policy
- final repository-wide migration results
- user-facing or creator-facing documentation


===============================================================================
01. PROBLEM STATEMENT
===============================================================================

01.1. Turnix design documents were originally written as large, coherent
      monolithic documents.

01.2. Large monolithic documents are difficult for language model assistants
      to load, reason about, and edit safely.

01.3. The GitHub connector can truncate large file responses.

01.4. When the connector truncates a file response, the assistant cannot
      safely treat the returned content as complete authoritative file text.

01.5. When the connector does not provide the file SHA in the same complete
      file response, the assistant cannot safely update that file through the
      connector.

01.6. Therefore, large normative documents are not reliable working units
      for connector-based design work.

01.7. The correct structural response is to split large conceptual areas into
      smaller connector-safe authority slices.

01.8. Workaround retrieval is rejected because it creates a second authority
      path outside the canonical document system.


===============================================================================
02. DESIGN INTENT
===============================================================================

02.1. A design area is modeled as a directory.

02.2. A design area directory contains one family index and one or more
      connector-safe authority slices.

02.3. The family index preserves conceptual unity.

02.4. The authority slices preserve safe retrieval, review, and update.

02.5. The path carries human and assistant routing meaning.

02.6. The document identifier remains the stable reference handle.

02.7. Historical numbering does not define the future architecture.

02.8. Legacy identifiers remain useful for migration and old reference
      resolution, but they do not force the new structure to preserve old
      numbering.


===============================================================================
03. PROTOTYPE LAYOUT
===============================================================================

03.1. The prototype design area layout is:

      docs/design/area/<area-name>/<AREA>-NN_TOPIC.txt

03.2. The tracing prototype uses:

      docs/design/area/tracing/

03.3. The tracing family index is:

      docs/design/area/tracing/TRACE-00_INDEX.txt

03.4. The tracing authority slices are intended to be:

      TRACE-01_SUBSTRATE_AND_AUTHORITY.txt
      TRACE-02_EVENT_MODEL_AND_DAG.txt
      TRACE-03_SPANS_AND_ROOTS.txt
      TRACE-04_REASONS_AND_ATTRIBUTES.txt
      TRACE-05_RETENTION_LOSS_AND_ACCESS.txt
      TRACE-06_DIAGNOSTICS_ERRORS_AND_DEV_MODE.txt

03.5. The former source document is:

      docs/design/DA-03_TRACING_DIAGNOSTICS_AND_OBSERVABILITY.txt

03.6. The former source document carried DOC_ID DA-03.

03.7. The new tracing family uses TRACE-* document identifiers as the
      prototype replacement identity family.


===============================================================================
04. AUTHORITY MODEL UNDER TEST
===============================================================================

04.1. The prototype tests whether CMR-04 should register connector-safe
      authority slices instead of one monolithic design area document.

04.2. The prototype rejects a hidden helper-routing system that bypasses
      CMR-04.

04.3. CMR-04 should participate in the experiment.

04.4. Bootstrap routing documents should be rewritten when needed so that
      assistant loading behavior follows the canonical system rather than a
      workaround system.

04.5. Migration notes may explain legacy mapping and continuation rules.

04.6. Migration notes do not replace CMR-04 as the canonical index.

04.7. AI helper documents may explain how to work with the system.

04.8. AI helper documents do not define canonical document identity or
      authority by themselves.


===============================================================================
05. INTERNAL DESIGN DOCUMENT MEANING
===============================================================================

05.1. Turnix design documents are internal development-facing authority
      documents.

05.2. They are not ordinary user documentation.

05.3. They are not tutorials, manuals, public API explanations, or
      creator-facing guides by default.

05.4. User-facing and creator-facing documentation may be derived from them.

05.5. Derived documentation does not replace internal design authority.

05.6. Internal design documents may be strict, dense, and definitive because
      their purpose is to define project reality for implementation and
      future design work.


===============================================================================
06. REJECTED WORKAROUNDS
===============================================================================

06.1. The prototype rejects keeping oversized documents and relying on
      partial snippets.

06.2. The prototype rejects using repository search results as authoritative
      replacement for full file retrieval.

06.3. The prototype rejects using web copies as authoritative replacement for
      connector file retrieval.

06.4. The prototype rejects using cached assistant memory as authoritative
      replacement for loaded file text.

06.5. The prototype rejects obtaining an update SHA through a separate route
      when the connector did not return it with the complete file response.

06.6. The prototype rejects a permanent parallel routing index that acts like
      CMR-04 without being CMR-04.


===============================================================================
07. CONTINUATION INSTRUCTIONS FOR FUTURE ASSISTANTS
===============================================================================

07.1. Start by loading this migration note when the task concerns the
      document-family split prototype.

07.2. Load CMR-04 before deciding canonical document identity or canonical
      paths.

07.3. Load AI_WORKFLOW_BOOTSTRAP before changing assistant workflow or
      loading behavior.

07.4. Load the current tracing prototype files from
      docs/design/area/tracing/ before editing the tracing family.

07.5. Do not assume that old DA-* numbering remains the desired future
      structure.

07.6. Do not preserve old numbering merely because it exists.

07.7. Preserve old meaning when it is still valid.

07.8. Preserve legacy mappings enough for migration and old-reference
      interpretation.

07.9. Prefer direct CMR/bootstrap updates over workaround helper notes when
      the change affects canonical routing, loading, authority, or identity.

07.10. Use migration notes for explanation and rehydration, not for hidden
       authority.

07.11. Treat each new authority slice as a complete document with its own
       title, purpose, scope, locked summary, and doc-meta block.

07.12. Keep each authority slice small enough for complete connector
       retrieval and update.

07.13. Preserve the current Turnix plain-text specification style unless the
       document-system rules are intentionally revised.

07.14. Avoid deep directory hierarchies unless they express stable routing
       meaning.

07.15. Prefer shallow semantic paths such as:

       docs/design/area/tracing/

       over deeply nested paths that encode every subtopic.
