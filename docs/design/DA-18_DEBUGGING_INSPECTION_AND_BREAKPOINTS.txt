===============================================================================
DESIGN AREA DA-18 - DEBUGGING, INSPECTION & BREAKPOINTS
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how Turnix can be inspected while running, where execution may be
paused, and how state can be safely observed and modified without breaking
determinism.

This area is developer-facing.

Purpose:
--------
This document defines the Turnix debugging model for semantic inspection,
scoped pausing, and controlled mutation.

It defines:
- semantic breakpoint types and their trigger semantics
- pause behavior and safe mutation constraints
- inspection surfaces and debugger action boundaries
- manual pause and apply-at-safe-point semantics
- Job-aware breakpoint and pause behavior when Jobs exist
- debugger-originated Job lifecycle control boundaries
- undo and redo semantics for debugger tooling
- trace integration requirements for debugger actions and breakpoint events

It does NOT define:
- the tracing model, trace event schema, or retention rules (see DA-03)
- authoritative committed state, transactions, or windows (see DA-04)
- activation semantics beyond pause boundaries (see DA-14)
- decision and policy handling semantics (see DA-17)
- Job lifecycle states, terminal outcomes, or Job result semantics
  (see DA-19)
- concrete user-interface layout for debugger tooling

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines debugging, inspection, and breakpoint behavior
      within that shared interpretation model.


===============================================================================
02. CORE INTENT
===============================================================================

02.1. Debugging in Turnix is not:
      - print-statement debugging
      - ad-hoc logging
      - attaching debuggers to arbitrary threads as the primary model

02.2. Debugging in Turnix is:
      - intentional
      - observable
      - reversible where the surrounding state model allows reversal
      - scoped

02.3. Turnix debugging operates on Turnix concepts rather than on language
      internals as the primary debugging surface.

02.4. Primary debugging targets include:
      - packs
      - activation
      - pipelines
      - memory layers
      - capabilities
      - Jobs when Jobs exist

02.5. Python frames or JavaScript call stacks may appear as contextual
      information, but they are not the primary debugging model.


===============================================================================
03. BREAKPOINT TYPES
===============================================================================

03.1. Turnix supports semantic breakpoints rather than code-line
      breakpoints.

03.2. Supported semantic breakpoint classes include:
      - pipeline-stage breakpoints
      - memory breakpoints
      - capability breakpoints


03.1 PIPELINE-STAGE BREAKPOINTS
-------------------------------

03.1.1. A pipeline-stage breakpoint pauses execution when a pipeline reaches
        a specific stage.

03.1.2. Use cases include:
        - inspecting a prompt before sending it to an LLM
        - verifying memory composition
        - observing transformations

03.1.3. Pipeline-stage breakpoint triggers may include:
        - stageId
        - pipelineId
        - optional condition


03.2 MEMORY BREAKPOINTS
-----------------------

03.2.1. A memory breakpoint pauses execution when memory changes according
        to the configured trigger.

03.2.2. Memory-breakpoint triggers may include:
        - memory key
        - memory layer
        - change type such as set, delete, or overwrite
        - optional value predicate


03.2.1 MEMORY BREAKPOINTS AND TRANSACTIONAL STAGING (LOCKED)
------------------------------------------------------------

03.2.1.1. Memory breakpoints observe state transitions rather than storage
          mechanics.

03.2.1.2. Definitions:
          - a staged mutation is a write into a transactional layer
          - a committed mutation is a write that becomes visible in a
            committed authoritative memory layer after successful commit
          - an aborted mutation is a staged mutation discarded during abort

03.2.1.3. By default, memory breakpoints trigger on committed mutations
          only.

03.2.1.4. Writing into a transactional layer does not trigger a memory
          breakpoint by default.

03.2.1.5. Aborting a transaction does not trigger a memory breakpoint by
          default.

03.2.1.6. Only the moment a value becomes authoritative is breakpoint-
          visible by default.

03.2.1.7. Memory breakpoints may be configured to observe staged mutations
          explicitly.

03.2.1.8. If staged observation is enabled, a breakpoint may specify:
          - observeStaged = true
          - observeCommitted = true or false

03.2.1.9. When observeStaged is enabled, the breakpoint may trigger when a
          staged value is written.

03.2.1.10. Staged-mutation breakpoint context includes:
           - transactionId
           - staging-layer identity
           - parent-layer identity
           - commit-eligibility status

03.2.1.11. Every memory-breakpoint hit is classified as exactly one of:
           - StagedMutation
           - CommittedMutation

03.2.1.12. Breakpoints do not fire twice for the same mutation phase.

03.2.1.13. If a breakpoint observes both staged and committed mutation
           phases, the staged hit and committed hit are distinct events.

03.2.1.14. Those distinct events are trace-linked to the same mutation
           lineage.

03.2.1.15. Nested transactions inherit breakpoint semantics unless
           explicitly scoped otherwise.

03.2.1.16. A child transaction commit into a parent transaction is not
           treated as a committed mutation.

03.2.1.17. Only the outermost transaction commit produces authoritative
           committed-mutation events.

03.2.1.18. This model preserves determinism, prevents breakpoint storms
           during speculative execution, and aligns debugging semantics with
           the transactional authority model in DA-04.


03.3 CAPABILITY BREAKPOINTS
---------------------------

03.3.1. A capability breakpoint pauses execution before or after a
        capability call according to the configured trigger.

03.3.2. Capability-breakpoint triggers may include:
        - capability name
        - caller pack
        - permission decision point

03.3.3. Capability breakpoints may observe capability invocations that later
        create or attach to Jobs.

03.3.4. If a capability invocation creates or attaches to a Job, the Job
        lifecycle remains owned by DA-19 rather than by the capability
        breakpoint.


===============================================================================
04. BREAKPOINT BEHAVIOR (LOCKED)
===============================================================================

04.1. When a breakpoint triggers:
      - execution of the triggering unit pauses
      - mutation-capable execution for the owning AppInstance pauses
      - other AppInstances may continue if the runtime supports that
      - non-mutating kernel activity may continue unless configured
        otherwise
      - debugger tooling is notified
      - relevant state becomes inspectable

04.2. Breakpoints do not:
      - auto-resume silently
      - alter execution order
      - mutate state by default
      - silently cancel Jobs
      - silently supersede Jobs


===============================================================================
05. LIVE DEBUGGING WITHOUT PAUSE (LOCKED)
===============================================================================

05.1. The debugger may be used while Turnix is running normally.

05.2. Debugger inspection does not require a breakpoint hit.

05.3. Debugger inspection does not require Turnix to be paused.

05.4. A user may open inspectors, browse state, and request read-only views
      at any time.

05.5. Such inspection is traceable and marked as debugger or tooling
      activity.


===============================================================================
06. DEBUGGER ACTIONS VS ORCHESTRATION (LOCKED)
===============================================================================

06.1. Debugger actions execute through Turnix-controlled remote procedure
      calls, capabilities, or equivalent Turnix-mediated interfaces.

06.2. Debugger tooling does not bypass Turnix orchestration.

06.3. Debugger tooling does not access raw filesystem input or output
      directly as a bypass path.

06.4. Any persistence-affecting debugger action executes through the same
      authority path as normal runtime actions, for example IoManager or
      services that use it.

06.5. Debugger-originated Job control actions execute through Turnix
      lifecycle and policy surfaces.

06.6. Debugger tooling does not mutate Job terminal state directly.


===============================================================================
07. INSPECTION SURFACES
===============================================================================

07.1. Inspectable objects include:
      - PackDescriptors
      - activation state
      - memory layers
      - pipeline objects
      - capability registry
      - services registry
      - Jobs when Jobs exist

07.2. Inspection is read-only by default.

07.3. Debugger inspection may enumerate memory layers and their composition.

07.4. Debugger inspection may inspect values together with semantic context
      such as:
      - origin layer
      - revision identifiers
      - transactional visibility

07.5. Debugger inspection may view active transactional stacks when doing so
      is safe and meaningful in the current execution context.

07.6. Debugger inspection may distinguish between:
      - committed values currently authoritative
      - staged values present only in transactional layers
      - unresolved or unavailable state when the requested surface cannot be
        read safely or meaningfully in the current execution context

07.7. Inspection surfaces do not silently collapse committed and staged
      state into one undifferentiated view.

07.8. If a debugger surface presents both committed and staged state, the
      distinction remains explicit.

07.9. Inspection surfaces may show revision identifiers, origin-layer
      identity, and transactional visibility as surfaced inspection metadata.

07.10. Inspecting transactional state does not turn staged values into
       authoritative state.

07.11. Inspecting committed-state history does not bypass the committed-state
       authority model defined elsewhere.

07.12. Debugger inspection may inspect Job lifecycle state, owner, progress,
       terminal outcome, and causal trace links when Jobs exist.

07.13. Inspecting a Job does not grant direct access to the underlying
       execution mechanism.


===============================================================================
08. CONTROLLED MUTATION (LOCKED)
===============================================================================

08.1. Explicit mutation is allowed only through:
      - debugger actions
      - explicit user confirmation where required
      - trace logging

08.2. Examples include:
      - editing a memory value
      - disabling or modifying a breakpoint
      - forcing resume or abort of a debugger pause scope

08.3. Every debugger-initiated mutation emits a trace event.

08.4. Debugger tooling may create its own temporary transactional overlay
      for controlled staged edits.

08.5. A debugger-created temporary transactional overlay remains subject to
      the same authority, scope, ownership, and trace rules that govern
      other transactional mutation.

08.6. A debugger-created temporary transactional overlay does not become
      authoritative merely because the debugger created it.

08.7. If debugger tooling uses such an overlay, the overlay remains distinct
      from committed state until normal commit rules make the mutation
      authoritative.

08.8. Job lifecycle control is not controlled mutation of committed state.

08.9. Job lifecycle control remains constrained by DA-19 and by policy
      handling when policy participates.


===============================================================================
09. MUTATION WHILE RUNNING (LOCKED)
===============================================================================

09.1. Controlled mutation may be performed while Turnix is active and not
      paused.

09.2. Rules:
      - mutation is explicit user action in debugger tooling
      - mutation is traceable and labeled as debugger-initiated
      - mutation obeys the same permission, ownership, and scope rules as
        normal runtime actions

09.3. If a mutation affects data that active execution units may read
      concurrently, the debugger either:
      - requires a manual pause
      - or enforces deterministic apply-at-safe-point semantics


===============================================================================
10. MANUAL PAUSE (LOCKED)
===============================================================================

10.1. Turnix may support manual pausing even when no breakpoint is hit.

10.2. Manual pause exists to:
      - allow deterministic inspection and mutation
      - prevent important work from being voided by cancellation

10.3. Manual pause is deterministic.

10.4. Manual pause does not silently cancel in-flight work.

10.5. Manual pause does not silently supersede in-flight work.

10.6. If pausing cannot be achieved safely, Turnix refuses the pause
      request with an explicit, traceable reason.

10.7. Manual pause may operate by enforcing quiescence at defined safe
      points such as:
      - between pipeline stages
      - between capability calls
      - before commit points for persistence

10.8. Manual pause of a Job remains non-terminal unless a later explicit
      lifecycle or policy outcome changes the Job state under DA-19.


===============================================================================
11. APPLY-AT-SAFE-POINT (LOCKED)
===============================================================================

11.1. When Turnix is not paused, debugger-initiated mutations may be staged
      and applied only at a deterministic safe point.

11.2. Rules:
      - staging is traceable
      - the exact safe point where the mutation becomes visible is traceable
      - staged mutations do not partially apply
      - if the safe point is never reached, for example because execution
        aborts, the staged mutation is discarded deterministically and traced

11.3. Debugger tooling may use a temporary transactional overlay as the
      staging vehicle for apply-at-safe-point mutation.

11.4. If such an overlay is used, the overlay remains debugger-owned staging
      state until the mutation is applied or discarded according to the
      normal rules of this document and DA-04.

11.5. Discard of a debugger-staged overlay is explicit and trace-visible.

11.6. Application of a debugger-staged overlay does not bypass normal
      transaction, ownership, scope, or authority rules.


===============================================================================
12. UNDO AND REDO SEMANTICS
===============================================================================

12.1. Debugger tooling may support undo of debugger-initiated changes within
      the boundaries defined by the Turnix state model.

12.2. Undo that affects authoritative state operates on committed revisions.

12.3. Undo that affects authoritative state is performed as a new explicit
      debugger-initiated mutation.

12.4. Undo that affects authoritative state is trace-visible.

12.5. Undo does not bypass transaction, permission, ownership, scope, or
      safe-point rules defined elsewhere in this document and in DA-04.

12.6. Redo history is debugger-local, ephemeral, and non-authoritative.

12.7. Redo history does not become committed state merely because debugger
      tooling remembers it.

12.8. Redo history is discarded deterministically when a new committed
      revision makes the stored redo path no longer valid.

12.9. Debugger tooling does not present ephemeral redo state as if it were
      authoritative system history.


===============================================================================
13. TRACE RETENTION AND DEBUGGER ACTIONS (LOCKED)
===============================================================================

13.1. Debugger tooling does not create untraced gaps for debugger actions
      that inspect, pause, resume, abort, stage, apply, discard, or mutate
      Turnix-managed state.

13.2. Debugger actions that affect trace evidence, including export, prune,
      clear, or retention-policy changes, remain trace-visible.

13.3. Debugger-originated trace-affecting actions are marked as
      debugger-initiated through trace attributes.

13.4. Trace retention and loss-evidence rules apply equally to debugger-
      initiated trace-affecting actions.

13.5. DA-03 defines the tracing substrate used to represent debugger trace
      evidence.

13.6. This document defines debugger-specific trace obligations.


===============================================================================
14. DEBUGGER USER-INTERFACE INTEGRATION
===============================================================================

14.1. Debugger user interfaces may:
      - subscribe via RPC
      - receive breakpoint-hit events
      - show causal context
      - allow resume, step, abort of a debugger pause scope, undo, or redo
        where supported
      - request Job pause-at-safe-point, cancellation, or supersession where
        supported

14.2. Debugger user interfaces are optional but supported.

14.3. Debugger user interfaces do not present Superseded Job outcomes as
      failures.

14.4. Debugger user interfaces do not present paused-at-safe-point Jobs as
      terminal.


===============================================================================
15. STEP MODES
===============================================================================

15.1. Supported step modes include:
      - Resume
      - Step over, meaning the next stage or memory change
      - Abort, meaning fail the current debugger-controlled execution scope

15.2. Unsupported step modes include:
      - instruction-level stepping
      - time travel as a runtime debugger mode in this document

15.3. Step modes do not redefine Job lifecycle states.

15.4. If step or abort actions affect a Job, the resulting Job lifecycle
      outcome remains governed by DA-19.


===============================================================================
16. JOBS, LONG-RUNNING WORK, AND BREAKPOINTING
===============================================================================

16.1. Turnix may represent long-running work as explicit Jobs.

16.2. If Jobs exist:
      - Jobs are observable through status, owner, progress, terminal
        outcome, and causal trace links
      - Jobs are controlled only through Turnix lifecycle, orchestration, or
        policy surfaces
      - Job lifecycle semantics remain owned by DA-19

16.3. Debugger interaction with Jobs may include requests for:
      - pause-at-safe-point for a Job
      - breakpoint-at-safe-point for a Job
      - resume of a paused Job
      - cancellation request for a Job
      - supersession request for a Job where supported

16.4. Manual pause does not implicitly cancel Jobs.

16.5. Manual pause does not implicitly supersede Jobs.

16.6. Safe-point pausing or breakpointing of Jobs occurs at boundaries such
      as:
      - between pipeline stages
      - before or after capability boundaries
      - before commit points for persistence

16.7. If a Job cannot reach a safe point:
      - that fact is surfaced explicitly
      - no silent cancellation is permitted
      - no silent supersession is permitted

16.8. A paused-at-safe-point Job remains non-terminal.

16.9. Debugger-originated cancellation, supersession, continuation, or hold
      behavior may require work-lifecycle policy handling under DA-17.

16.10. The debugger does not directly write a Job terminal state.

16.11. Job terminal outcomes caused or requested through debugger action are
       recorded through Job lifecycle evidence owned by DA-19.


===============================================================================
17. SCOPING RULES
===============================================================================

17.1. Breakpoints may be scoped to:
      - global
      - AppInstance
      - view
      - pack
      - pipeline
      - Job when Jobs exist

17.2. Unscoped breakpoints are discouraged.

17.3. A Job-scoped breakpoint does not create a Job.

17.4. A Job-scoped breakpoint applies only to the Job lifecycle or work
      region selected by the debugger configuration.


===============================================================================
18. DEVELOPMENT VS PRODUCTION
===============================================================================

18.1. In production:
      - breakpoints are disabled by default
      - inspection is limited
      - mutation is disallowed unless explicitly enabled

18.2. In development:
      - all debugging features in this document are available

18.3. Development availability does not allow debugger tooling to bypass
      Job lifecycle, policy, transaction, or capability authority boundaries.


===============================================================================
19. FAILURE MODES (LOCKED)
===============================================================================

19.1. If the debugger disconnects:
      - execution resumes, cancels, supersedes, aborts the affected
        debugger-controlled scope, or refuses to hold according to policy
      - the chosen outcome is trace-visible
      - deadlock is not allowed

19.2. Debugger disconnect outcome trace evidence distinguishes:
      - execution resumed after debugger disconnect by policy
      - active work cancelled, superseded, or aborted after debugger
        disconnect by policy
      - hold refused because holding is unsupported, unsafe, or impossible

19.3. Debugger disconnect outcome trace evidence records enough context to
      explain:
      - debugger client identity
      - policy identity
      - disconnect kind
      - affected target when work is cancelled, superseded, or aborted
      - refusal reason category when hold is refused

19.4. If breakpoint logic errors:
      - the breakpoint is disabled
      - the error is trace-visible

19.5. If debugger disconnect policy affects a Job, the Job lifecycle outcome
      remains governed by DA-19.

19.6. If debugger disconnect policy is a work-lifecycle policy decision,
      the policy handling remains governed by DA-17.


===============================================================================
20. RELATION TO TRACING
===============================================================================

20.1. Debugger connection lifecycle trace evidence distinguishes:
      - debugger connected
      - debugger disconnected

20.2. Debugger connection trace evidence records enough context to explain:
      - debugger client identity
      - connectionKind when a debugger connects
      - disconnect kind when a debugger disconnects

20.3. Every breakpoint hit emits trace evidence.

20.4. Breakpoint-hit trace evidence links to the causal chain.

20.5. Breakpoint-hit trace evidence records inspection and mutation actions
      when applicable.

20.6. Breakpoint-hit trace evidence distinguishes breakpoint type.

20.7. Breakpoint-hit trace evidence includes:
      - breakpointId
      - breakpoint type
      - breakpoint scope
      - appInstanceId when applicable

20.8. Pipeline breakpoint-hit trace evidence additionally includes:
      - pipelineRunId
      - stageId

20.9. Memory breakpoint-hit trace evidence additionally includes:
      - memoryLayerId
      - memory-key identity
      - mutation phase
      - mutationKind
      - transactionId when applicable

20.10. Capability breakpoint-hit trace evidence additionally includes:
       - capability identifier
       - caller-pack identity
       - decisionPoint when applicable

20.11. Job-scoped breakpoint-hit trace evidence additionally includes:
       - jobId
       - Job state when available
       - Job terminal outcome when available

20.12. Breakpoint-hit trace evidence includes orchestrationUnitId or Job
       identity when applicable.

20.13. Manual pause trace evidence distinguishes:
       - pause requested
       - pause granted
       - pause refused
       - pause-at-safe-point requested
       - pause-at-safe-point applied

20.14. Manual pause trace evidence records enough context to explain:
       - request identity
       - pause scope
       - target identity
       - whether a safe point was reached
       - safe-point descriptor when applicable
       - refusal reason category when pause is refused
       - jobId when the target is a Job

20.15. Debugger inspection trace evidence records:
       - inspection kind
       - target kind
       - target identity when applicable

20.16. Debugger mutation trace evidence distinguishes:
       - mutation requested
       - mutation staged
       - mutation applied
       - mutation discarded

20.17. Debugger mutation trace evidence records enough context to explain:
       - mutation request identity
       - mutation lineage identity
       - target kind
       - target identity
       - mutation kind
       - intended safe point when staged
       - applied safe point when applied
       - discard reason category when discarded

20.18. Debugger-originated Job control is trace-visible and remains linked
       to Job lifecycle evidence when it affects a Job.

20.19. Debugger-originated Job control trace evidence records policy source
       when policy participates.

20.20. Debugger actions that stage and apply mutations at safe points are
       traced as distinct events that share a stable mutation-lineage
       identifier.

20.21. DA-18 owns the debugger-domain trace reason identifiers used for
       debugger inspection, breakpoints, manual pause, safe-point pause,
       debugger disconnect outcomes, and debugger-initiated mutation.

20.22. The following debugger-domain trace reasons are defined:

       - DebuggerConnected
       - DebuggerDisconnected
       - DebuggerDisconnectedResume
       - DebuggerDisconnectedAbort
       - DebuggerDisconnectedHoldRefused
       - BreakpointHit
       - BreakpointResumed
       - ManualPauseRequested
       - ManualPauseGranted
       - ManualPauseRefused
       - PauseAtSafePointRequested
       - PauseAtSafePointApplied
       - DebuggerInspection
       - DebuggerMutationRequested
       - DebuggerMutationStaged
       - DebuggerMutationApplied
       - DebuggerMutationDiscarded

20.23. Tracing and debugging are complementary.

20.24. DA-03 defines trace event structure, trace DAG semantics, span
       mechanics, and trace retention semantics.

20.25. DA-17 defines work-lifecycle policy trace obligations when policy
       participates in debugger-originated Job control.

20.26. DA-19 defines Job lifecycle trace obligations when debugger-originated
       control affects a Job.

20.27. This document defines debugger-specific trace obligations.


===============================================================================
21. WHAT IS EXPLICITLY NOT SUPPORTED
===============================================================================

21.1. The following are not supported:
      - arbitrary Python debugger injection
      - JavaScript eval debugging
      - modifying pack code at runtime
      - hot-patching logic without reload
      - directly writing Job terminal state from debugger tooling
      - silently cancelling or superseding Jobs from debugger tooling


===============================================================================
22. EXPLICITLY OPEN QUESTIONS
===============================================================================

22.1. OPEN questions include:
      - conditional breakpoint language
      - snapshot or restore state
      - time-travel debugging
      - replay-from-trace tooling
      - exact debugger UI affordances for Job lifecycle control


===============================================================================
23. LOCKED SUMMARY
===============================================================================

23.1. LOCKED:
      - semantic breakpoints
      - pipeline, memory, capability, and Job-scoped breakpoints
      - transaction-aware breakpoint semantics
      - read-only inspection by default
      - controlled mutation with tracing
      - debugger usability while Turnix is running
      - deterministic manual pause
      - manual pause does not silently cancel or supersede Jobs
      - apply-at-safe-point mutation semantics
      - undo of authoritative debugger changes operates on committed
        revisions as new explicit debugger-initiated work
      - redo history is debugger-local, ephemeral, and non-authoritative
      - debugger-specific trace obligations are owned here
      - debugger-domain trace reason identifiers are owned by DA-18
      - Job-safe breakpointing and pausing align with DA-19 Job lifecycle
      - debugger-originated Job control is explicit, trace-visible, and
        constrained by DA-17 and DA-19
      - paused-at-safe-point Jobs remain non-terminal
      - debugger tooling does not directly write Job terminal state
      - non-blocking debugger behavior

23.2. OPEN:
      - advanced conditional logic
      - replay and time travel
      - exact debugger UI affordances for Job lifecycle control
