===============================================================================
IMPLEMENTATION SPECIFICATION IS-01 - RPC MESSAGE SCHEMA
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the concrete wire-envelope schema used for RPC messages exchanged
between Turnix runtime participants.

This document is an implementation specification. It does not introduce new
system guarantees, authority semantics, or policy rules.

Purpose:
--------
This document defines the concrete RPC message envelope, its fields, helper
construction expectations, serialization responsibilities, and envelope
completion-marker behavior.

It defines:
- core RPC wire models
- envelope fields and their meanings
- per-type field expectations
- helper-constructor responsibilities
- serialization and routing expectations
- adapter and AppInstance correlation metadata carried by the envelope
- envelope completion-marker behavior

It does NOT define:
- new runtime authority semantics
- new permission or policy rules
- abstract RPC, channel, and transport semantics (see DA-21)
- execution adapter boundary semantics (see DA-23)
- frontend View semantics (see DA-22)
- AppInstance lifecycle semantics (see DA-20)
- tracing semantics beyond carrying correlation metadata
- Job lifecycle semantics beyond carrying Job metadata (see DA-19)
- WebSocket lifecycle behavior beyond what the message schema requires
- higher-level routing, visibility, activation, or capability semantics

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines the RPC wire schema within that shared
      interpretation model.


===============================================================================
02. CANONICAL MODEL LOCATION
===============================================================================

02.1. The canonical backend RPC message model lives in:
      backend/rpc/models.py

02.2. The frontend mirrors this schema in:
      - frontend/assets/types.d.ts
      - RpcClient helper utilities

02.3. This document describes the concrete message shape expected on the
      wire.

02.4. If code and this document drift, the project updates them together so
      that the implementation remains traceable to this specification.

02.5. The schema may be used by WebSocket, in-process, testing, or future
      transport implementations when those implementations carry the same
      concrete RPCMessage envelope.


===============================================================================
03. CORE WIRE MODELS
===============================================================================

03.1. The core RPC wire models are:
      - Gen
      - Route
      - RPCMessage


03.1 GEN
--------

03.1.1. Gen wraps the connection generation assigned by the server.

03.1.2. Gen contains the generation components used to detect stale traffic
        and reject frames from previous sessions.

03.1.3. Gen is embedded in RPC envelopes where session continuity validation
        is required.

03.1.4. Gen does not define AppInstance lifecycle, Job lifecycle, or adapter
        lifecycle.


03.2 ROUTE
----------

03.2.1. Route points a message to either:
        - a capability
        - an object

03.2.2. If the explicit lane is missing, backend logic may infer the default
        delivery lane from Route values.

03.2.3. Route does not grant permission, activate packs, or authorize
        adapter execution by itself.


03.3 RPCMESSAGE
---------------

03.3.1. RPCMessage is the wire envelope for all RPC communication.

03.3.2. RPCMessage responsibilities include:
        - enforcing camelCase aliases
        - forbidding extra fields
        - validating message shape
        - inferring delivery lane through model validation when the lane
          is omitted and inference is permitted
        - carrying isComplete as the final serialized top-level field for
          complete envelopes

03.3.3. RPCMessage carries correlation metadata for higher-level systems.

03.3.4. RPCMessage does not create authority by carrying that metadata.


===============================================================================
04. ENVELOPE FIELDS
===============================================================================

04.1. Unless stated otherwise, fields are optional and default to null,
      None, or the implementation default for that side.

04.2. The envelope fields are defined below.

04.3. JSON object field order is not treated as general semantic meaning by
      Turnix.

04.4. This schema nevertheless requires canonical serializer field order for
      the top-level RPC envelope so that isComplete can act as an envelope
      tail sentinel.

04.5. Complete envelopes serialize isComplete as the final top-level field.

04.6. No top-level envelope field appears after isComplete in canonical
      serialized form.


04.1 v
------

04.1.1. v is the schema version.

04.1.2. The current version is:
        "0.1"


04.2 id
-------

04.2.1. id uniquely identifies the message.

04.2.2. id uses UUIDv7.

04.2.3. id identifies the RPC message only.

04.2.4. id is distinct from appInstanceId, jobId, viewId, and adapter
        identity.


04.3 type
---------

04.3.1. type identifies the message category.

04.3.2. Valid control-message values are:
        - "hello"
        - "welcome"
        - "clientReady"
        - "heartbeat"
        - "ack"

04.3.3. Valid workload-message values are:
        - "request"
        - "emit"
        - "reply"
        - "subscribe"
        - "stateUpdate"
        - "unsubscribe"
        - "cancel"
        - "error"

04.3.4. type does not define capability permission, Job outcome, adapter
        outcome, or AppInstance lifecycle by itself.


04.4 correlatesTo
-----------------

04.4.1. correlatesTo links the message to a parent message.

04.4.2. correlatesTo is used for:
        - replies
        - acknowledgements
        - cancels
        - error propagation

04.4.3. correlatesTo is message correlation. It is not Job identity or
        adapter identity.


04.5 gen
--------

04.5.1. gen carries the server-assigned generation.

04.5.2. The client echoes gen to validate session continuity.

04.5.3. Generation mismatch is session or transport continuity evidence. It
        does not define Job cancellation or adapter failure by itself.


04.6 ts
-------

04.6.1. ts is the monotonic send timestamp populated on message creation.

04.6.2. ts may represent milliseconds since process start or another
        monotonic local representation used by the implementation.

04.6.3. ts is not used as the authority source for causal ordering when
        trace or explicit correlation evidence exists.


04.7 lane
---------

04.7.1. lane identifies the delivery lane.

04.7.2. The default system lane is:
        "sys"

04.7.3. For non-control messages, if lane is empty and inference is
        permitted, the lane may be derived from route:
        - "cap:<capability>" when targeting a capability
        - "obj:<id>" when targeting a specific object

04.7.4. Lanes maintain independent sequencing and flow control per logical
        stream.

04.7.5. Lane selection does not grant permission or execution authority.


04.8 budgetMs
-------------

04.8.1. budgetMs is the optional time budget for the operation in
        milliseconds.

04.8.2. Defaults may be derived from protocol settings and priority
        classes.

04.8.3. Callers may override budgetMs explicitly.

04.8.4. budgetMs does not define Job timeout unless a higher-level operation
        explicitly maps it to Job timeout semantics.


04.9 ackOf
----------

04.9.1. ackOf is reserved for richer acknowledgement semantics.

04.9.2. ackOf is currently unused.

04.9.3. Acknowledgement metadata confirms message receipt or transport-level
        handling only.


04.10 job
---------

04.10.1. job carries Job lifecycle snapshot or Job handle metadata when a
         message reports, accepts, updates, or completes Job-backed work.

04.10.2. The concrete job structure is job-type specific.

04.10.3. job does not define Job lifecycle semantics.

04.10.4. DA-19 defines Job lifecycle states, terminal outcomes, progress,
         cancellation, timeout, supersession, and late completion handling.

04.10.5. Presence of job metadata does not mean the current RPC message is
         final Job success.


04.11 idempotencyKey
--------------------

04.11.1. idempotencyKey enables backend deduplication and cached replies.

04.11.2. For request and emit messages, helpers may default
         idempotencyKey to the message id.

04.11.3. Retries using the same effective idempotencyKey are naturally
         deduplicated by the backend when that behavior is enabled.

04.11.4. idempotencyKey does not bypass permission checks, adapter
         authorization, or capability mediation.


04.12 route
-----------

04.12.1. route identifies the target capability or object.

04.12.2. route is mandatory for:
         - request
         - emit

04.12.3. route is generally present for workload messages that require
         routing.

04.12.4. route is also used to:
         - derive lanes
         - enforce routing and permission checks in the surrounding system

04.12.5. route may identify adapter-facing or adapter-originated endpoints
         only when the higher RPC and adapter documents define that route
         class.

04.12.6. route is carried by the envelope; authority remains defined by
         higher DA documents.


04.13 op
--------

04.13.1. op is an operation qualifier.

04.13.2. Examples include:
         - "call"
         - "event"

04.13.3. op allows handlers to multiplex multiple behaviors on the same
         route.

04.13.4. op does not bypass routing, permission, adapter, or lifecycle
         checks.


04.14 path
----------

04.14.1. path is an additional routing hint used by handlers to select an
         operation variant or sub-endpoint.

04.14.2. path does not provide filesystem authority.


04.15 args
----------

04.15.1. args carries positional arguments for capability calls initiated by
         the client.

04.15.2. For most RPC calls, payload carries the structured data and args is
         empty or small.


04.16 seq
---------

04.16.1. seq is the per-lane delivery sequence number maintained by the
         sender.

04.16.2. Each lane has its own counter.

04.16.3. seq is a lane-ordering field and not a committed-state revision.


04.17 origin
------------

04.17.1. origin carries opaque pass-through metadata.

04.17.2. origin does not affect permissions by itself.

04.17.3. origin may include:
         - trace identifiers
         - span identifiers
         - calling context
         - appInstanceId
         - viewId
         - frontend client identity
         - adapter kind
         - adapter instance identity
         - Job identity when available

04.17.4. origin metadata is correlation evidence only.

04.17.5. origin metadata does not grant authority, permission, activation,
         adapter execution, or state mutation.


04.17.1 TRACE AND RUNTIME CORRELATION METADATA
----------------------------------------------

04.17.1.1. RPC envelopes may carry trace and runtime correlation metadata in
           origin.

04.17.1.2. Trace and runtime correlation metadata may include:
          - trace identifiers
          - span identifiers
          - parent event identifiers
          - calling context
          - appInstanceId when available
          - viewId when available
          - adapter kind when available
          - adapter instance identity when available
          - jobId when available

04.17.1.3. Trace correlation metadata carried by an RPC envelope does not
           define trace semantics.

04.17.1.4. Runtime correlation metadata carried by an RPC envelope does not
           define AppInstance, View, adapter, or Job lifecycle semantics.

04.17.1.5. DA-03 defines trace event structure, trace DAG semantics, span
           mechanics, and trace-correlation meaning.

04.17.1.6. DA-20 defines AppInstance lifecycle and runtime identity.

04.17.1.7. DA-22 defines View and frontend runtime semantics.

04.17.1.8. DA-23 defines execution adapter semantics.

04.17.1.9. RPC helpers preserve trace and runtime correlation metadata when
           constructing replies, errors, acknowledgements, cancellation
           messages, and other messages that causally relate to an earlier
           message.

04.17.1.10. If trace or runtime correlation metadata is unavailable, the
            envelope remains valid unless the message type or higher-level
            owner explicitly requires that metadata.


04.18 chunkNo
-------------

04.18.1. chunkNo is the chunk index when payloads are streamed or sent as an
         ordered message series.

04.18.2. chunk numbering starts at 0 or 1 depending on helper convention.

04.18.3. chunkNo is monotonically increasing within the ordered series.

04.18.4. chunkNo does not define Job progress by itself.


04.19 final
-----------

04.19.1. final marks the final message in a streamed or ordered message
         series.

04.19.2. final accepts:
         - boolean
         - integer flag, where non-zero is treated as true

04.19.3. final is a series-level marker.

04.19.4. final is not a single-envelope completion marker.

04.19.5. A message may be complete as an individual envelope even when final
         is false or absent.

04.19.6. final does not define Job terminal outcome by itself.


04.20 payload
-------------

04.20.1. payload carries application data.

04.20.2. Helpers enforce payload shape by message type.

04.20.3. Examples include:
         - reply requires a result payload
         - error requires an error object
         - stateUpdate carries state deltas

04.20.4. Payload content does not gain authority merely because the envelope
         is valid.


04.21 isComplete
----------------

04.21.1. isComplete marks that the sender intentionally completed
         construction of the full top-level RPC envelope before serialized
         send.

04.21.2. isComplete is boolean.

04.21.3. A complete RPC envelope sets isComplete = true.

04.21.4. isComplete is serialized as the final top-level field of a
         complete RPC envelope.

04.21.5. No top-level envelope field is serialized after isComplete in
         canonical serialized form.

04.21.6. If isComplete is missing, the receiver treats the envelope as
         incomplete.

04.21.7. If isComplete is false, null, or not boolean true, the receiver
         treats the envelope as incomplete.

04.21.8. An incomplete envelope is not processed as a valid RPC message.

04.21.9. isComplete is not a semantic correctness marker.

04.21.10. isComplete does not prove that the message content is correct.

04.21.11. isComplete does not replace JSON parsing, schema validation,
          generation validation, routing validation, permission checks,
          trace correlation, transport framing, stream or ordered-series
          sequencing, or final.

04.21.12. isComplete exists to detect accidental incomplete envelope
          construction, unintended partial content emission, and truncation
          where the tail of the serialized top-level envelope did not make it
          through.

04.21.13. Because isComplete is the final top-level field, a serialized
          envelope that lacks isComplete has not reached the declared end of
          the complete envelope.

04.21.14. A receiver accepts isComplete only after the complete JSON frame
          has parsed successfully and the top-level envelope validates.

04.21.15. isComplete does not imply capability permission, Job success,
          adapter readiness, AppInstance readiness, or committed-state
          mutation.


===============================================================================
05. HELPER CONSTRUCTORS
===============================================================================

05.1. Backend helper functions construct validated envelopes while enforcing
      required fields, correlating identifiers, and copying idempotency keys
      from the triggering message when appropriate.

05.2. Helper constructors exist to avoid hand-building envelopes across the
      codebase.

05.3. Typical helper responsibilities include:
      - setting v to the current protocol version
      - generating id with UUIDv7 for new messages
      - copying correlatesTo for replies, acknowledgements, and cancels
      - populating gen from the active RPCConnection
      - computing lane when not provided and when inference is permitted
      - setting default budgetMs for the operation kind
      - copying idempotencyKey from the triggering message when applicable
      - preserving trace and runtime correlation metadata when applicable
      - enforcing required fields per message type
      - setting isComplete = true only after the envelope has been fully
        constructed
      - ensuring isComplete is the final serialized top-level field

05.4. The frontend mirrors these helper expectations in RpcClient so that
      both sides construct symmetric envelopes.

05.5. Shared helper-default behavior exists for:
      - schema version
      - system lane
      - acknowledgement budget where applicable
      - envelope completion marking

05.6. This shared helper behavior exists to keep message construction
      deterministic and free of ad-hoc exceptions.

05.7. Helper constructors do not set isComplete = true before required
      top-level fields and required payload structures for the message type
      have been supplied.

05.8. Before sending, serialization verifies that the canonical serialized
      top-level envelope ends with isComplete = true.

05.9. Helper constructors do not use origin metadata to skip permission,
      adapter, activation, routing, or lifecycle checks.


===============================================================================
06. PER-TYPE FIELD EXPECTATIONS
===============================================================================

06.1. "hello":
      - carries gen
      - carries client identity information in payload
      - carries isComplete = true

06.2. "request":
      - carries route
      - carries op and or path as required by the target
      - carries payload or args as required by the operation
      - carries isComplete = true

06.3. "reply":
      - carries correlatesTo
      - carries payload
      - carries isComplete = true

06.4. "error":
      - carries correlatesTo
      - carries an error payload
      - carries or preserves trace correlation metadata when available
      - carries isComplete = true

06.5. "stateUpdate":
      - carries state-update payload
      - carries isComplete = true

06.6. Other message types obey the field expectations required by their role
      and the helper constructors that create them.

06.7. All complete message types carry isComplete = true.

06.8. Compatibility mode may accept older envelopes without isComplete only
      when that compatibility behavior is explicit and scoped.

06.9. Adapter-facing message types do not gain separate authority merely by
      being adapter-facing.


===============================================================================
07. SERIALIZATION AND LOGGING
===============================================================================

07.1. Backend send flow is:
      1) build or mutate an RPCMessage through helper constructors
      2) complete the envelope by setting isComplete = true as the final
         top-level field
      3) serialize using a JSON helper such as safeJsonDumps
      4) verify the serialized top-level envelope ends with isComplete = true
         according to the canonical serializer field order
      5) feed the frame into the RPC logging-decision engine
      6) transmit the JSON string over the transport

07.2. Serialization steps may include:
      - dumping the model to a plain structure
      - applying deterministic separators or ordering where appropriate
      - ensuring isComplete remains the final serialized top-level field

07.3. Logging-decision flow may include:
      - deciding whether to log based on lane, type, and level
      - emitting structured log entries when enabled

07.4. Frontend receive flow is:
      1) parse incoming text frames as JSON
      2) validate the basic structure
      3) verify that isComplete is present and true unless explicit
         compatibility handling applies
      4) reject incomplete envelopes before routing
      5) apply optional logging hooks
      6) route messages by type, lane, and route

07.5. Frontend routing behavior includes:
      - control frames update connection and generation state
      - workload frames dispatch to handlers, promises, or subscriptions

07.6. This shared schema exists so that any message emitted on one side can
      be:
      - interpreted correctly
      - routed deterministically
      - logged consistently
      - correlated for tracing
      - rejected when the envelope completion marker is absent or false

07.7. This document does not require ad-hoc special cases outside the schema
      and helper model defined here.

07.8. The completion marker is checked before a parsed envelope is routed.

07.9. Failure of completion-marker validation is treated as malformed or
      incomplete message handling rather than as a routed workload error.

07.10. Valid serialization does not imply valid authority, successful work,
       or accepted lifecycle outcome.


===============================================================================
08. RELATION TO HIGHER DOCUMENTS
===============================================================================

08.1. This document is constrained by higher-authority documents.

08.2. This document does not introduce new system guarantees, authority
      semantics, or policy rules.

08.3. Routing, visibility, activation, capability, permission,
      decision-handling, AppInstance lifecycle, View lifecycle, execution
      adapter semantics, transport semantics, and Job lifecycle semantics
      remain governed by higher DA documents.

08.4. This document defines only the concrete wire structure used to carry
      those higher-level concepts.

08.5. isComplete defines a wire-envelope completion marker only.

08.6. isComplete does not add authority to the payload, result, error,
      Job snapshot, trace correlation metadata, adapter metadata, runtime
      correlation metadata, or routed operation.

08.7. DA-21 defines RPC, channel, and transport semantics.

08.8. DA-23 defines execution adapter semantics.

08.9. DA-20 defines AppInstance runtime identity.

08.10. DA-22 defines View and frontend runtime semantics.
