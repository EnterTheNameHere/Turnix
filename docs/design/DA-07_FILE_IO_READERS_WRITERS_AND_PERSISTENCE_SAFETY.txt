===============================================================================
DESIGN AREA DA-07 - FILE I/O, READERS / WRITERS & PERSISTENCE SAFETY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how Turnix performs persistent I/O, who is allowed to perform it,
and how failures are handled without corrupting authoritative state,
history, or user trust.

All persistence in Turnix is centralized and mediated.

Purpose:
--------
This document defines how Turnix centralizes persistent I/O behind a single
authority so that persistence remains deterministic, reversible, and
explicitly controlled.

It defines:
- centralized I/O authority and boundaries
- Reader / Writer provider model
- raw access containment rules
- root read/write permissions
- persistence safety requirements
- backup, snapshot, and rollback boundaries
- error classification and recovery boundaries
- streaming I/O rules and where streaming is forbidden

It does NOT define:
- the general state model (see DA-04)
- activation and lifecycle semantics (see DA-14)
- dependency resolution semantics (see DA-13)
- capability schemas and permission prompting (see DA-16)
- tracing schema or trace storage semantics (see DA-03)
- pack discovery rules beyond save scanning implications (see DA-09)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines persistent I/O authority, Reader / Writer
      boundaries, and persistence-safety rules within that shared
      interpretation model.


===============================================================================
02. CORE PRINCIPLE
===============================================================================

02.1. Persistent I/O in Turnix routes through a centralized I/O authority.

02.2. Packs, tools inside runtime orchestration, and runtime components do
      not read or write persistent data by bypassing that authority.

02.3. This applies to:
      - configuration
      - manifests
      - save data
      - logs and traces
      - caches
      - compiled assets
      - future databases or storage backends

02.4. Persistent I/O centralization exists to preserve:
      - determinism
      - reversibility
      - explicit uncertainty
      - root-boundary control
      - user-trustworthy recovery behavior

02.5. This document refines the general persistence and state-safety rules
      already constrained by [constrained-by: DA-04 03],
      [constrained-by: DA-04 09], and
      [constrained-by: DA-04 19].


===============================================================================
03. CENTRAL AUTHORITY: IOMANAGER (LOCKED)
===============================================================================

03.1. IoManager is the sole authority responsible for persistent I/O.

03.2. IoManager responsibilities include:
      - mediating reads and writes
      - enforcing root boundaries and permissions
      - normalizing and validating storage targets
      - selecting appropriate Readers and Writers
      - classifying errors
      - ensuring trace-visible persistence outcomes
      - providing atomic-write guarantees where required
      - managing backups and snapshots per policy
      - preserving rollback points for identity-bearing mutations

03.3. No other component bypasses IoManager for persistent I/O.

03.4. IoManager does not replace the DA-04 state model. IoManager enforces
      how persistence occurs for state and other persisted artifacts.


===============================================================================
04. READERS AND WRITERS (LOCKED CONCEPT)
===============================================================================

04.1. IoManager does not hardcode all persistence formats.

04.2. IoManager delegates concrete format handling to registered providers:
      - Readers
      - Writers

04.3. A Reader loads data from storage under IoManager control.

04.4. A Writer persists data to storage under IoManager control.

04.5. Each Reader or Writer:
      - supports one or more specific kinds
      - declares its supported formats explicitly
      - is registered with IoManager
      - operates strictly within IoManager control
      - executes in the Turnix-controlled backend environment

04.6. Reader / Writer providers refine persistence mechanics. They do not
      redefine platform persistence authority.


===============================================================================
05. RAW ACCESS BOUNDARY (LOCKED)
===============================================================================

05.1. Reader / Writer implementations may internally use:
      - raw filesystem handles
      - platform-native APIs
      - low-level I/O operations

05.2. Raw handles do not escape the Reader / Writer boundary.

05.3. Raw paths do not become part of the calling runtime surface.

05.4. Packs do not receive filesystem handles or unrestricted path APIs.

05.5. Readers return loaded data or streaming views when explicit streaming
      mode allows it.

05.6. Writers accept data or explicit streams to persist under IoManager
      control.

05.7. The raw filesystem remains an internal implementation detail of
      IoManager and its providers.


===============================================================================
06. BUILT-IN AND EXTENDED PROVIDERS (LOCKED POLICY)
===============================================================================

06.1. Turnix core provides built-in persistence kinds required before
      discovery begins.

06.2. Built-in kinds currently include:
      - utf8Text
      - binary
      - json
      - json5
      - jsonl

06.3. These built-in kinds remain available before discovery and indexing.

06.4. Additional kinds may be provided by packs or later platform
      extensions.

06.5. Extended providers:
      - register with IoManager
      - declare supported kinds explicitly
      - are not usable until the providing pack is activated

06.6. Turnix does not auto-discover or load providers in the middle of a
      read or write operation.

06.7. Early phases use only built-in providers.


===============================================================================
07. DISCOVERY- AND INDEXING-TIME I/O LIMITS (LOCKED)
===============================================================================

07.1. Discovery and indexing use built-in Readers only.

07.2. During discovery and indexing, Turnix does not use:
      - pack-provided Readers
      - pack-provided Writers
      - compilers
      - transformers
      - streaming I/O

07.3. Early phases remain deterministic, fast, and independent of pack
      activation.

07.4. These limits remain compatible with [constrained-by: DA-09] and
      [constrained-by: DA-14].


===============================================================================
08. ROOT PERMISSIONS AND WRITE AUTHORITY (LOCKED)
===============================================================================

08.1. Writable roots at runtime are:
      - userdata/
      - saves/

08.2. Read-only roots at runtime are:
      - custom/
      - first-party/
      - third-party/

08.3. Runtime orchestration enforces these root rules strictly.

08.4. Authoring tools outside runtime orchestration may define broader
      editing behavior, but those behaviors do not weaken runtime rules.

08.5. IoManager enforces root-boundary rules for runtime persistence.

08.6. Root writability does not by itself authorize state mutation.
      State-mutation authority remains constrained by the relevant higher
      design areas.


===============================================================================
09. ERROR CLASSIFICATION (LOCKED)
===============================================================================

09.1. Persistent I/O errors are classified into exactly one category:

      A) Unrecoverable
      B) User-decidable
      C) Optional / best-effort

09.2. Unrecoverable errors include failures such as:
      - failure to load required configuration schema material
      - failure to load required default configuration material
      - corruption of identity-bearing state required before minimal
        recovery UI is available

09.3. Unrecoverable errors terminate the affected operation or runtime
      path as fatal.

09.4. User-decidable errors include failures such as:
      - corrupted user configuration
      - missing optional files required for a chosen action
      - version mismatch requiring migration or reset
      - recoverable restore decisions

09.5. User-decidable errors pause normal continuation and require explicit
      recovery flow.

09.6. Optional / best-effort errors include data such as:
      - logs
      - traces
      - caches

09.7. Optional / best-effort failures do not corrupt authoritative state.

09.8. Identity-bearing data never uses silent fallback.


===============================================================================
10. ERROR PRESENTATION AND RECOVERY BOOTSTRAP
===============================================================================

10.1. User recovery is mediated through a minimal guaranteed recovery
      surface.

10.2. Before that recovery surface is available, recovery decisions do not
      occur.

10.3. Failures before minimal recovery availability are fatal for the
      affected boot path.

10.4. This document does not lock the permanent implementation stack of the
      recovery surface.

10.5. The platform guarantee is:
      - a minimal recovery surface becomes available before user-decidable
        recovery is required
      - if that recovery surface is not yet available, the failure is
        treated as unrecoverable in that moment

10.6. This document does not define frontend implementation details for the
      recovery surface.


===============================================================================
11. TRACE VISIBILITY OF I/O OUTCOMES (LOCKED)
===============================================================================

11.1. Refused, skipped, failed, retried, recovered, or completed I/O
      outcomes are trace-visible.

11.2. Trace naming, reason naming, and trace storage semantics remain
      governed by [constrained-by: DA-03].

11.3. This document defines the persistence-side obligation to surface I/O
      outcomes to trace. It does not define the full tracing taxonomy.


===============================================================================
12. SAVE DATA HANDLING (LOCKED)
===============================================================================

12.1. Save data is discovered and loaded under the discovery and indexing
      limits already defined by this document and by DA-09.

12.2. Save-local copied packs or save-internal artifacts do not become part
      of initial global indexing merely because they exist in save data.

12.3. Deeper save inspection occurs only when:
      - a save is explicitly loaded
      - a management tool explicitly inspects the save
      - explicit configuration or policy enables deeper inspection

12.4. Save-local visibility remains save-local unless some other
      authoritative document explicitly states otherwise.


===============================================================================
13. CONFIGURATION LOADING RELATIONSHIP
===============================================================================

13.1. Configuration loading and persistence use IoManager mediation.

13.2. Configuration-specific semantics are defined by [constrained-by: DA-06].

13.3. This document does not redefine configuration scopes, defaults,
      override semantics, or effectiveConfig semantics.

13.4. This document defines only the I/O authority and persistence-safety
      obligations that configuration obeys.


===============================================================================
14. MIGRATORS AND ROLLBACK GUARANTEES (LOCKED)
===============================================================================

14.1. Migrators may exist for:
      - configuration
      - manifests
      - save structures
      - future persisted artifact classes

14.2. Migrators are versioned and applied sequentially toward a target
      compatibility state.

14.3. Migrators are deterministic and side-effect controlled within their
      defined scope.

14.4. Migrators do not provide inverse operations by themselves.

14.5. Before any identity-bearing mutation is committed, IoManager creates
      a rollback point according to policy.

14.6. Identity-bearing mutations include, but are not limited to:
      - configuration migrations
      - save structure changes
      - manager-driven save edits
      - multi-file writes that must remain internally consistent

14.7. Rollback safety is provided by IoManager and by the DA-04 commit
      model, not by per-migrator inverse logic.

14.8. This section remains [constrained-by: DA-04 19].


===============================================================================
15. BACKUPS AND SNAPSHOTS (LOCKED CONCEPT)
===============================================================================

15.1. IoManager supports durability mechanisms such as:
      - file backups
      - snapshots

15.2. File backups are suited to small critical files and prior versions.

15.3. Snapshots are suited to logically related multi-file or multi-artifact
      units.

15.4. Policy controls:
      - retention
      - storage location
      - cleanup behavior
      - restoration availability

15.5. Automatic rollback does not occur without explicit policy or
      user-visible consent where such consent is required.


===============================================================================
16. STREAMING I/O (PARTIALLY LOCKED)
===============================================================================

16.1. Streaming is an explicit I/O mode for large or long-lived operations.

16.2. Streaming read returns a stream or async iterator.

16.3. Streaming write accepts a stream and stages output incrementally.

16.4. Core streaming rules:
      - streaming is never implicit
      - the caller requests streaming explicitly
      - staging is mandatory for streaming write
      - cancellation remains safe

16.5. Non-streaming kinds remain non-streaming unless explicitly extended.


16.1 STREAMING KINDS (LOCKED)
----------------------------

16.1.1. Examples of streaming kinds include:
        - utf8TextStream
        - binaryStream
        - jsonlStream
        - tarStream
        - saveSnapshotStream

16.1.2. Example names do not by themselves lock the full future streaming
        catalog.


16.2 CANCELLATION AND CLEANUP (LOCKED)
--------------------------------------

16.2.1. Streaming read cancellation closes underlying handles safely.

16.2.2. Streaming read cancellation does not create persistence side
        effects.

16.2.3. Streaming write cancellation does not produce partial valid final
        output.

16.2.4. Final destination update occurs only on commit.

16.2.5. Staging artifacts remain garbage-collectable and cleanup-compatible.


16.3 WHERE STREAMING IS FORBIDDEN (LOCKED)
------------------------------------------

16.3.1. Streaming is not used for:
        - discovery
        - indexing
        - manifest loading
        - configuration schema loading
        - default configuration loading
        - userdata configuration loading
        - dependency resolution
        - activation-critical reads

16.3.2. These phases require full deterministic materialization rather than
        partial progressive access.


===============================================================================
17. RECOVERY USING BACKUPS AND SNAPSHOTS (LOCKED)
===============================================================================

17.1. When user-decidable persisted data fails to load, IoManager may offer
      recovery actions such as:
      - restore last known good
      - reset to defaults where that concept exists
      - quarantine broken data
      - proceed without the failed data only if explicit policy allows it

17.2. Recovery actions remain UI-mediated or policy-mediated according to
      the governing higher documents.

17.3. Recovery actions remain trace-visible.


===============================================================================
18. NO BACKDOORS (LOCKED)
===============================================================================

18.1. Packs do not bypass IoManager.

18.2. Packs do not access arbitrary persistent paths through unrestricted
      runtime APIs.

18.3. Packs do not intercept core persistence flows.

18.4. Persistent I/O authority remains centralized.


===============================================================================
19. EXPLICITLY OPEN QUESTIONS
===============================================================================

19.1. OPEN questions include:
      - streaming backpressure API
      - resumable streaming
      - verification hooks such as hashing or limits
      - encrypted storage backends
      - the future boundary between logical addressing and physical storage
        mapping


===============================================================================
20. LOCKED SUMMARY
===============================================================================

20.1. LOCKED:
      - centralized IoManager authority
      - Reader / Writer provider abstraction
      - controlled raw-access boundary
      - strict runtime root permissions
      - trace-visible I/O outcomes
      - rollback points before identity-bearing mutations
      - UI-mediated recovery
      - deterministic early phases
      - no silent fallback for identity-bearing data
      - explicit streaming mode and explicit streaming-forbidden phases

20.2. OPEN:
      - performance refinements
      - future storage backends
      - future logical-addressing / physical-mapping refinement
