===============================================================================
DESIGN DOCTRINE DD-04 - TURNIX SOURCE FORMATTING AND CODE DOCUMENTATION RULES
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines general formatting and code-adjacent documentation rules for
Turnix source files and non-specification development artifacts.

Purpose:
--------
This document defines repository-wide formatting expectations for source
files and code-adjacent documentation while keeping those rules separate
from the ASCII-79 specification document format used by authoritative
Turnix documents.

It defines:
- separation from specification document formatting
- repository-wide indentation rules
- default line-length target for source files
- general documentation-in-code expectations
- language-specific refinement allowance

It does NOT define:
- authoritative specification document formatting
- runtime semantics
- specific language parser behavior
- complete language-specific style guides
- frontend protocol details

Upstream Documents:
------------------
- CMR-01
- CMR-02
- CMR-03
- CMR-04
- CMR-05


===============================================================================
01. GENERAL SEPARATION
===============================================================================

01.1. Turnix distinguishes between authoritative specification documents
      and source or code-adjacent development artifacts.

01.2. Authoritative specification documents follow the ASCII-79 format
      rules defined by CMR-01.

01.3. Source files, schemas, manifests, scripts, and other development
      artifacts do not follow the ASCII-79 document format unless an
      authoritative document explicitly requires that format.

01.4. Repository-wide source formatting rules exist to improve clarity,
      consistency, and cross-language reviewability.

01.5. Source formatting rules do not replace language-specific syntax
      rules. They operate within those syntax rules.


===============================================================================
02. INDENTATION RULES
===============================================================================

02.1. Turnix source formatting uses spaces rather than tabs.

02.2. Python, JavaScript, TypeScript, HTML, CSS, and similar source files
      use four-space indentation unless a language-specific Turnix
      document explicitly states otherwise.

02.3. JSON, JSONL, JSON5, and similar structured data files use
      two-space indentation unless a format-specific Turnix document
      explicitly states otherwise.

02.4. Tabs are not used for indentation in repository-maintained
      Turnix files.

02.5. Indentation rules exist to preserve readability, stable review, and
      predictable formatting across languages and tools.


===============================================================================
03. LINE LENGTH
===============================================================================

03.1. Source files and code-adjacent development artifacts use a default
      line-length target of 119 characters.

03.2. The 119-character target does not apply to authoritative
      specification documents governed by CMR-01.

03.3. When a language-specific Turnix document defines a narrower limit
      for a certain artifact class, that narrower limit applies to that
      artifact class.

03.4. Long literals, paths, generated identifiers, or language-specific
      constructs may exceed the default target when forced wrapping would
      weaken clarity more than it helps.

03.5. The line-length target exists to support readability and review. It
      does not justify semantic distortion or misleading rewrites.


===============================================================================
04. CODE-ADJACENT DOCUMENTATION
===============================================================================

04.1. Turnix distinguishes between authoritative specification documents
      and documentation written inside or alongside source files.

04.2. Code-adjacent documentation follows the formatting and structural
      rules appropriate to the host language or artifact type.

04.3. Code-adjacent documentation does not need to mimic ASCII-79
      specification document structure.

04.4. Documentation embedded in source files remains readable, local to
      the relevant code, and compatible with the conventions of the host
      language.

04.5. Documentation placed outside immediate code bodies remains
      acceptable when the host language or tooling convention supports it
      clearly and predictably.

04.6. Turnix does not force one documentation placement model across all
      languages when the languages themselves structure documentation
      differently.


===============================================================================
05. LANGUAGE-SPECIFIC REFINEMENT
===============================================================================

05.1. This document defines repository-wide defaults for source
      formatting and code-adjacent documentation.

05.2. Language-specific Turnix documents may refine these defaults for
      particular languages or artifact types.

05.3. A language-specific refinement preserves the general separation
      between:
      - authoritative specification documents
      - source formatting rules
      - code-adjacent documentation rules

05.4. Language-specific refinement does not weaken repository-wide
      readability, reviewability, or semantic clarity without clear
      justification.

05.5. When a language-specific document exists, it acts as the primary
      reference for that language within the boundaries defined by this
      document.


===============================================================================
06. REVIEW AND CONSISTENCY EXPECTATION
===============================================================================

06.1. Repository formatting should help a reader move between languages
      without unnecessary friction.

06.2. Cross-language consistency matters, but consistency does not require
      erasing language-specific structure where that structure is
      meaningful.

06.3. Formatting choices support review, collaboration with language model
      assistants, and long-term maintainability.

06.4. Mechanical consistency is useful, but semantic clarity remains more
      important than blind formatting uniformity.


===============================================================================
07. FUTURE LANGUAGE DOCUMENTS
===============================================================================

07.1. Turnix may define language-specific source formatting or
      documentation documents for languages such as:
      - Python
      - JavaScript
      - TypeScript
      - HTML and CSS
      - JSON-family artifacts
      - future supported artifact classes

07.2. Such documents may define:
      - comment style expectations
      - documentation placement expectations
      - language-specific exceptions
      - tool-facing formatting details
      - artifact-specific refinement rules

07.3. Language-specific documents should reference this document as a
      general upstream source when they refine repository-wide defaults.


===============================================================================
08. SUMMARY
===============================================================================

08.1. Turnix keeps authoritative specification document formatting
      separate from source-file formatting and code-adjacent
      documentation.

08.2. Repository-wide defaults use spaces rather than tabs, four-space
      indentation for source code, two-space indentation for JSON-family
      artifacts, and a 119-character line-length target for source files.

08.3. Language-specific documents may refine how documentation and
      formatting work inside particular languages while preserving the
      general separation defined here.
