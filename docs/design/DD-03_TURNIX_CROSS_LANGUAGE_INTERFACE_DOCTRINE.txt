===============================================================================
DESIGN DOCTRINE DD-03 - TURNIX CROSS-LANGUAGE INTERFACE DOCTRINE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines cross-language interface doctrine for Turnix creator-facing APIs,
host-language adaptation, same-language authority separation, and
platform-controlled integration points.

Purpose:
--------
This document defines how Turnix preserves creator-facing semantic
consistency across supported languages while allowing host-language
adaptation where adaptation does not weaken platform guarantees.

Turnix platform truth is defined by contracts, guarantees, orchestration
semantics, and continuity behavior rather than by any specific host
language or current implementation stack. Cross-language interface
doctrine exists to preserve that platform truth across supported
languages.

It defines:
- cross-language semantic consistency
- same-language authority separation
- host-language adaptation boundaries
- platform-controlled integration points
- creator-facing API coherence
- language-native freedom inside Turnix-safe boundaries

It does NOT define:
- specific pack APIs
- specific capability schemas
- concrete frontend protocols
- concrete module loader internals
- concrete execution adapter semantics
- language-specific formatting minutiae

===============================================================================
01. GENERAL DOCTRINE
===============================================================================

01.1. Turnix presents one creator-facing platform across supported host
      languages.

01.2. A creator moving between supported languages encounters the same
      platform concepts, the same major guarantees, and the same
      creator-facing mental model.

01.3. Host-language adaptation is allowed, but adaptation does not change
      Turnix platform meaning.

01.4. Cross-language consistency matters because Turnix is one platform
      expressed through multiple host languages rather than separate
      language-specific platforms.

01.5. Host-language convenience does not override platform integrity,
      platform guarantees, creator-facing semantic coherence, or
      implementation-independent platform truth.

01.6. Same host language does not imply same authority.

01.7. Pack-provided code does not become platform implementation code merely
      because it uses the same host language as platform implementation
      code.


===============================================================================
02. CROSS-LANGUAGE SEMANTIC CONSISTENCY
===============================================================================

02.1. If two supported languages expose the same Turnix concept, they use
      the same creator-facing meaning.

02.2. Creator-facing semantic consistency is derived from shared Turnix
      contracts rather than from host-language convention.

02.3. If two supported languages expose the same Turnix concept and the
      host language permits it without distortion, they use the same
      creator-facing name.

02.4. A language-specific surface difference is acceptable only when the
      host language requires or strongly justifies that difference.

02.5. A language-specific adaptation does not create a different platform
      concept unless Turnix explicitly defines a distinct concept.

02.6. Creator-facing semantics remain primary. Host-language presentation
      remains secondary.


===============================================================================
03. SAME-LANGUAGE AUTHORITY SEPARATION
===============================================================================

03.1. Turnix distinguishes language identity from authority identity.

03.2. Code written in the same language may still occupy different
      authority roles.

03.3. Backend implementation code may realize platform authority.

03.4. Pack-provided code remains pack-provided code even when written in the
      backend implementation language.

03.5. Python pack code is not backend implementation code merely because
      the current backend implementation is Python.

03.6. In-process execution does not erase Turnix authority boundaries.

03.7. Same-language direct calls may be used as implementation technique
      only when they preserve Turnix contract meaning and boundary meaning.

03.8. Same-language direct calls do not bypass lifecycle, tracing,
      capability, persistence, policy, or execution adapter boundaries.

03.9. This doctrine constrains language-facing design only. Concrete
      execution adapter semantics remain owned by DESIGN AREA documents.


===============================================================================
04. HOST-LANGUAGE ADAPTATION BOUNDARY
===============================================================================

04.1. Turnix may adapt syntax, calling style, or surface mechanics to fit
      the host language.

04.2. Such adaptation preserves:
      - contract meaning
      - boundary behavior
      - platform guarantees
      - creator-facing expectations

04.3. Host-language adaptation does not weaken lifecycle control,
      observability, permission boundaries, tracing, continuity
      guarantees, or other platform truth defined independently of the
      host language.

04.4. A host-language convention is not adopted merely because it is the
      native convention of that language.

04.5. Turnix accepts host-language differences where those differences
      improve usability without weakening platform truth.

04.6. Host-language adaptation does not convert pack code into backend
      implementation authority.


===============================================================================
05. PLATFORM-CONTROLLED INTEGRATION POINTS
===============================================================================

05.1. Turnix may require platform-specific integration points when native
      host-language mechanisms would bypass Turnix guarantees, Turnix
      contracts, or boundary control.

05.2. Such integration points may include:
      - Turnix-specific import mechanisms
      - Turnix-controlled module loading
      - Turnix-defined lifecycle entrypoints
      - Turnix-defined capability access paths
      - Turnix-defined tracing or continuity hooks
      - Turnix-defined execution adapter boundaries

05.3. A native host-language mechanism is not used as the default path if
      that mechanism bypasses Turnix control over loading, permissions,
      tracing, lifecycle, continuity, or other guaranteed behavior.

05.4. Platform-controlled integration points exist to preserve guaranteed
      behavior and contract truth rather than to imitate host-language
      conventions at the expense of platform control.

05.5. When Turnix requires a platform-specific mechanism, the mechanism
      remains explicit and creator-facing rather than hidden behind a
      misleading language-native surface.


===============================================================================
06. CREATOR-FACING API COHERENCE
===============================================================================

06.1. Creator-facing APIs are designed so that moving between supported
      languages preserves conceptual familiarity.

06.2. Equivalent concepts use equivalent names when host-language rules
      allow that equivalence.

06.3. Equivalent concepts produce equivalent categories of runtime
      outcome.

06.4. Equivalent concepts preserve the same major contract expectations
      across languages.

06.5. A creator should not need to relearn the platform merely because the
      host language changed.

06.6. Differences that remain unavoidable are documented as language
      adaptations rather than treated as separate platform truths.


===============================================================================
07. LANGUAGE-NATIVE FREEDOM INSIDE SAFE BOUNDARIES
===============================================================================

07.1. Turnix does not forbid host-language expression merely because it is
      language-native.

07.2. Creators may use host-language features freely inside boundaries that
      do not compromise Turnix integrity or guarantees.

07.3. Turnix restricts host-language freedom only where unrestricted use
      would bypass or weaken:
      - platform guarantees
      - containment boundaries
      - authority boundaries
      - continuity guarantees
      - observability requirements

07.4. Creator freedom remains broad inside safe boundaries.

07.5. Turnix does not expand platform restrictions without platform-level
      justification.

07.6. Language-native freedom inside safe boundaries does not include
      native mechanisms that bypass Turnix authority boundaries.


===============================================================================
08. NAMING AND SEMANTIC TRANSFER
===============================================================================

08.1. Turnix favors names that preserve semantic transfer across
      languages.

08.2. A creator familiar with a Turnix concept in one supported language
      should be able to recognize the corresponding concept in another.

08.3. Semantic naming is preferred over unnecessary language-specific
      drift.

08.4. Host-language idiom may influence local surface form, but semantic
      identity remains recognizable.

08.5. Language-specific renaming is justified only when the host language
      would otherwise make the API misleading, unusable, or structurally
      unnatural.


===============================================================================
09. LANGUAGE ADAPTER ACCOUNTABILITY
===============================================================================

09.1. A language adapter or host-language integration layer preserves the
      creator-facing semantics defined by Turnix.

09.2. A language adapter does not silently weaken, reinterpret, or replace
      platform contracts or guarantees.

09.3. If a host-language integration cannot preserve a required guarantee,
      contract expectation, or boundary behavior, that limitation is made
      explicit.

09.4. Adapter convenience does not take precedence over platform truth.

09.5. A language adapter is judged by whether it preserves platform
      meaning, boundary discipline, contract truth, and creator-facing
      coherence.

09.6. A language adapter is not judged valid merely because it uses the
      host language's most convenient native mechanism.


===============================================================================
10. FORMAT AND STYLE CONVENTIONS
===============================================================================

10.1. Mechanical formatting conventions are subordinate to semantic
      coherence and platform guarantees.

10.2. Turnix may define repository-wide formatting conventions for source
      files and data files.

10.3. Such conventions may differ by artifact type when differing
      conventions improve usability or maintainability without weakening
      clarity.

10.4. Formatting conventions do not justify semantic drift between
      languages.

10.5. Language-specific style choices do not override creator-facing API
      coherence.


===============================================================================
11. SUMMARY
===============================================================================

11.1. Turnix presents one platform across supported languages rather than
      separate language-specific platforms.

11.2. Host-language adaptation is allowed only when semantic meaning,
      platform guarantees, and creator-facing coherence remain preserved.

11.3. Same host language does not imply same authority.

11.4. Pack-provided code remains pack-provided code even when written in the
      same host language as platform implementation code.

11.5. Platform-specific integration points are justified when native
      language mechanisms would bypass guaranteed behavior, contract
      truth, or boundary control.

11.6. Creator freedom remains broad inside safe boundaries.

11.7. Cross-language consistency reduces drift, reduces relearning cost,
      and preserves platform truth across the Turnix ecosystem.
