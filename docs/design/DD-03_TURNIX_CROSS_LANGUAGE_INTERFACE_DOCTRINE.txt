===============================================================================
DESIGN DOCTRINE DD-03 - TURNIX CROSS-LANGUAGE INTERFACE DOCTRINE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines cross-language interface doctrine for Turnix creator-facing APIs,
host-language adaptation, and platform-controlled integration points.

Purpose:
--------
This document defines how Turnix preserves creator-facing semantic
consistency across supported languages while allowing host-language
adaptation where adaptation does not weaken platform guarantees.

It defines:
- cross-language semantic consistency
- host-language adaptation boundaries
- platform-controlled integration points
- creator-facing API coherence
- language-native freedom inside Turnix-safe boundaries

It does NOT define:
- specific pack APIs
- specific capability schemas
- concrete frontend protocols
- concrete module loader internals
- language-specific formatting minutiae

------------------

===============================================================================
01. GENERAL DOCTRINE
===============================================================================

01.1. Turnix presents a coherent creator-facing platform across supported
      host languages.

01.2. A creator moving between supported languages should encounter the
      same platform concepts, the same major guarantees, and the same
      creator-facing mental model.

01.3. Host-language adaptation is allowed, but adaptation does not change
      the semantic meaning of Turnix concepts.

01.4. Cross-language consistency matters because Turnix is a platform,
      not a disconnected collection of language-specific ecosystems.

01.5. Language-specific convenience does not override platform integrity,
      platform guarantees, or creator-facing semantic coherence.


===============================================================================
02. CROSS-LANGUAGE SEMANTIC CONSISTENCY
===============================================================================

02.1. If two supported languages expose the same Turnix concept, they use
      the same creator-facing meaning.

02.2. If two supported languages expose the same Turnix concept and the
      host language allows it reasonably, they use the same creator-facing
      name.

02.3. A language-specific surface difference is acceptable only when the
      host language requires or strongly justifies that difference.

02.4. A language-specific adaptation does not create a different platform
      concept unless Turnix explicitly defines a distinct concept.

02.5. Creator-facing semantics remain primary. Host-language presentation
      remains secondary.


===============================================================================
03. HOST-LANGUAGE ADAPTATION BOUNDARY
===============================================================================

03.1. Turnix may adapt syntax, calling style, or surface mechanics to fit
      the host language.

03.2. Such adaptation preserves:
      - concept meaning
      - boundary behavior
      - platform guarantees
      - creator-facing expectations

03.3. Host-language adaptation does not weaken lifecycle control,
      observability, permission boundaries, tracing, or continuity
      guarantees.

03.4. A host-language convention is not adopted merely because it is the
      native convention of that language.

03.5. Turnix accepts host-language differences where those differences
      improve usability without weakening platform truth.


===============================================================================
04. PLATFORM-CONTROLLED INTEGRATION POINTS
===============================================================================

04.1. Turnix may require platform-specific integration points when native
      host-language mechanisms would bypass Turnix guarantees or boundary
      control.

04.2. Such integration points may include:
      - Turnix-specific import mechanisms
      - Turnix-controlled module loading
      - Turnix-defined lifecycle entrypoints
      - Turnix-defined capability access paths
      - Turnix-defined tracing or continuity hooks

04.3. A native host-language mechanism is not used as the default path if
      that mechanism bypasses Turnix control over loading, permissions,
      tracing, lifecycle, or other guaranteed behavior.

04.4. Platform-controlled integration points exist to preserve guaranteed
      behavior, not to mimic host-language conventions superficially.

04.5. When Turnix requires a platform-specific mechanism, the mechanism
      remains explicit and creator-facing rather than hidden behind
      misleading language-native appearance.


===============================================================================
05. CREATOR-FACING API COHERENCE
===============================================================================

05.1. Creator-facing APIs are designed so that moving between supported
      languages preserves conceptual familiarity.

05.2. Equivalent concepts use equivalent names when host-language rules
      allow that equivalence.

05.3. Equivalent concepts produce equivalent categories of runtime outcome.

05.4. Equivalent concepts preserve the same major contract expectations
      across languages.

05.5. A creator should not need to relearn the platform merely because the
      host language changed.

05.6. Differences that remain unavoidable are documented as language
      adaptations rather than treated as separate platform truths.


===============================================================================
06. LANGUAGE-NATIVE FREEDOM INSIDE SAFE BOUNDARIES
===============================================================================

06.1. Turnix does not forbid host-language expression merely because it is
      language-native.

06.2. Creators may use host-language features freely inside boundaries that
      do not compromise Turnix integrity or guarantees.

06.3. Turnix restricts host-language freedom only where unrestricted use
      would bypass or weaken:
      - platform guarantees
      - containment boundaries
      - authority boundaries
      - continuity guarantees
      - observability requirements

06.4. Creator freedom remains broad inside safe boundaries.

06.5. Turnix does not expand platform restrictions without platform-level
      justification.


===============================================================================
07. NAMING AND SEMANTIC TRANSFER
===============================================================================

07.1. Turnix favors names that preserve semantic transfer across languages.

07.2. A creator familiar with a Turnix concept in one supported language
      should be able to recognize the corresponding concept in another.

07.3. Semantic naming is preferred over unnecessary language-specific drift.

07.4. Host-language idiom may influence local surface form, but semantic
      identity remains recognizable.

07.5. Language-specific renaming is justified only when the host language
      would otherwise make the API misleading, unusable, or structurally
      unnatural.


===============================================================================
08. LANGUAGE ADAPTER ACCOUNTABILITY
===============================================================================

08.1. A language adapter or host-language integration layer preserves the
      creator-facing semantics defined by Turnix.

08.2. A language adapter does not silently weaken or reinterpret platform
      guarantees.

08.3. If a host-language integration cannot preserve a required guarantee,
      that limitation is made explicit.

08.4. Adapter convenience does not take precedence over platform truth.

08.5. A language adapter is judged by whether it preserves platform
      meaning, boundary discipline, and creator-facing coherence.


===============================================================================
09. FORMAT AND STYLE CONVENTIONS
===============================================================================

09.1. Mechanical formatting conventions are subordinate to semantic
      coherence and platform guarantees.

09.2. Turnix may define repository-wide formatting conventions for source
      files and data files.

09.3. Such conventions may differ by artifact type when differing
      conventions improve practicality without weakening clarity.

09.4. Formatting conventions do not justify semantic drift between
      languages.

09.5. Language-specific style choices do not override creator-facing API
      coherence.


===============================================================================
10. SUMMARY
===============================================================================

10.1. Turnix presents one platform across supported languages rather than
      separate language-specific platforms.

10.2. Host-language adaptation is allowed only when semantic meaning,
      platform guarantees, and creator-facing coherence remain preserved.

10.3. Platform-specific integration points are justified when native
      language mechanisms would bypass guaranteed behavior.

10.4. Creator freedom remains broad inside safe boundaries.

10.5. Cross-language consistency reduces drift, reduces relearning cost,
      and preserves platform truth across the Turnix ecosystem.
