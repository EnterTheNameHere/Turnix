===============================================================================
DESIGN AREA DA-03 - TRACING, DIAGNOSTICS & OBSERVABILITY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the Turnix tracing model, including trace events, causal structure,
reasons, span rules, and retention-loss evidence, enabling offline audit and
explanation without making trace retention a hidden correctness dependency.

Purpose:
--------
This design area defines the Turnix tracing, diagnostics, and observability
model.

It defines:
- what tracing means in Turnix and what it is authoritative for
- trace event fields and causal DAG semantics
- trace spans, ordering constraints, and trace roots
- trace reasons and required reason sets
- retention, pruning, and loss evidence requirements
- required tracing coverage for discovery, resolution, activation, and
  capabilities
- configuration and state trace integration rules
- trace access and user-facing diagnostics constraints
- error envelope integration requirements

It does NOT define:
- authoritative committed state, transactions, or windows (see DA-04)
- debugger pause and mutation safety semantics (see DA-18)
- concrete storage formats, codecs, or digest algorithms
  (see DA-05, DA-06, DA-08)
- UI layout for trace viewers or debugger tooling (see OPTIONAL DESIGN AREAS)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines tracing, diagnostics, and observability
      semantics within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. Tracing answers:

          "What happened, what caused what, and why?"

02.2. Tracing provides:
      - causal explanation (authoritative)
      - decision evidence (authoritative)
      - observed temporal sequence (non-authoritative)
      - diagnostic context (non-authoritative)

02.3. Tracing is not logging.

02.4. Tracing is structured evidence with explicit reasons and causal links.

02.5. Turnix uses tracing to support:
      - debugging
      - user explanation
      - post-mortem analysis
      - reproducibility and audit

02.6. Tracing is authoritative evidence for:
      - causality
      - decisions
      - audit reconstruction within retained evidence
      - debugger and tooling correlation within retained evidence
      - policy-enforcement provenance within retained evidence

02.7. Tracing is not authoritative for:
      - committed state
      - committed-state restoration
      - replay semantics
      - rollback eligibility by itself

02.8. "If it is not traced, it did not happen" means:
      - Turnix emits trace events for all decision-producing actions
      - if an action cannot be found in retained trace evidence, the action
        is not provable from trace
      - trace retention is policy-driven and loss is acceptable only when
        loss is explicitly represented as loss evidence

02.9. Success paths are traceable too, not only failures.

02.10. If only failures are observable, accidental success remains invisible
       until it stops being successful.

02.11. A successful path that cannot be explained after the fact is an
       observability failure.

02.12. Tracing therefore supports questions such as:
       - why a capability grant was accepted
       - why a resource resolved to a given pack
       - why a state value existed at a given moment
       - why one handler path was selected over another
       - why a UI result was delivered

02.13. Reason fields, causal links, and before/after state exist so that
       Turnix can explain not only that something happened, but why it
       happened.


===============================================================================
03. TRACE FIRST, LOG SECOND (LOCKED)
===============================================================================

03.1. Turnix prioritizes:
      - structured trace events
      - explicit reasons
      - causal relationships

03.2. Plain logs are secondary and optional.

03.3. Logs are not relied upon for correctness or explanation.

03.4. Logs do not replace trace evidence.


===============================================================================
04. TRACE EVENT MODEL (LOCKED)
===============================================================================

04.1. Each trace event has:
      - eventId
      - timestamp
      - level (info | warn | error | fatal)
      - domain (discovery, resolution, activation, io, config,
        capability, debugger, etc.)
      - message (human-readable)
      - reason (PascalCase enum)
      - attributes (structured key-value data)
      - optional parentEventIds

04.2. Events form a trace DAG rather than a flat stream.

04.3. Causal ordering is authoritative and is represented by explicit DAG
      links formed through parentEventIds and other causal span links
      defined in this document.

04.4. Temporal ordering is observational only.

04.5. Temporal ordering does not determine causality or correctness.

04.6. Turnix does not use temporal ordering to infer causality or
      correctness.

04.7. Causality is preserved across subsystems.

04.8. Correlation attributes should include stable identifiers that allow
      offline linking to Turnix objects such as:
      - kernelRunId
      - appInstanceId
      - pipelineRunId
      - orchestrationUnitId
      - transactionId
      - memoryLayerId
      - memory key identity

04.9. Identifier presence may vary by domain.

04.10. Boundary events and mutation events remain linkable to their owning
       objects.


04.1 PARENT LINKS AND DAG STRUCTURE (LOCKED)
--------------------------------------------

04.1.1. A trace event may reference zero, one, or multiple parent events.

04.1.2. parentEventIds, when present, is the set of explicit causal
        parents of that trace event.

04.1.3. Multiple parent links are permitted.

04.1.4. Absence of parent links means independent trace origin rather than
        simultaneity.

04.1.5. A trace event does not list the same parent event more than once.

04.1.6. Parent links express causal structure rather than temporal order.


04.2 TRACE DAG NAMING AND MEANING (LOCKED)
------------------------------------------

04.2.1. "Trace Directed Acyclic Graph" is the canonical name for the causal
        trace structure defined by this document.

04.2.2. "trace DAG" is the permitted abbreviation.

04.2.3. The trace DAG is the directed acyclic graph formed by trace events
        and their explicit parent and span-causality links.

04.2.4. The trace DAG is authoritative for causal structure within
        retained trace evidence.

04.2.5. The trace DAG is not a substitute for authoritative committed
        state.


04.3 TRACE SPANS AND ORDERING (LOCKED)
--------------------------------------

04.3.1. Trace spans represent bounded work with a start and an end.

04.3.2. A span has a begin event and an end event.

04.3.3. Span begin and span end events are trace events and participate in
        the trace DAG.

04.3.4. Span begin occurs before span end in the same trace DAG.

04.3.5. A span end does not exist without a reachable span begin.

04.3.6. Any export, snapshot, or offline trace view that includes span end
        also includes the corresponding span begin.

04.3.7. Span begin and span end share a stable span identity attribute.

04.3.8. The exact span identity encoding remains OPEN.

04.3.9. Span begin and span end are connected by explicit causality links.

04.3.10. The trace DAG allows reconstruction of:
         - which span is open at a SafePointWindow boundary
         - which spans were closed by that boundary

04.3.11. If a snapshot includes trace evidence, it satisfies the rule:
         "no included span end without included span begin."

04.3.12. This snapshot requirement is [satisfied-by: DA-08 11.8].


04.4 TRACE ROOTS AND SPAN STRATIFICATION (LOCKED)
-------------------------------------------------

04.4.1. Trace storage may contain multiple unconnected trace DAG roots.

04.4.2. Each root represents an independent trace origin, such as a kernel
        run or an appInstance activation region.

04.4.3. Unconnected roots are expected and remain representable without
        ambiguity.

04.4.4. Root span types are:

        A) KernelRunSpan
        - Represents one Turnix kernel run.
        - KernelRunSpan begin occurs when Turnix begins running.
        - KernelRunSpan end occurs when Turnix ends running.
        - KernelRunSpan begin and end share a stable span identity
          attribute.

        B) AppInstanceActiveSpan
        - Represents an appInstance being active while a pack is loaded.
        - Begins when an appInstance becomes active.
        - Ends when the appInstance is deactivated, or when the
          KernelRunSpan ends.
        - Remains linkable to:
          - appInstanceId
          - active pack identity
          - active save identity when applicable

        C) WorkSpans
        - Nested spans representing bounded work inside an
          AppInstanceActiveSpan.
        - Examples:
          - pipeline runs
          - capability calls
          - jobs or long-running orchestration units

04.4.5. Span begin and span end events participate in the trace DAG.

04.4.6. Span begin occurs before span end in the same trace DAG.

04.4.7. A span end does not exist without a reachable span begin.

04.4.8. If an exported trace view includes a span end, it also includes the
        corresponding span begin.

04.4.9. WorkSpans remain causally reachable from their owning
        AppInstanceActiveSpan begin event when such ownership exists.

04.4.10. Root spans define when Turnix ran and when an appInstance was
         active.

04.4.11. WorkSpans define what happened inside that active region.

05. TRACE REASONS (LOCKED)
===============================================================================

05.1. Reasons are:
      - stable identifiers
      - PascalCase
      - machine-parseable
      - semantically precise

05.2. Example reasons include:
      - ManifestInRootLayerDir
      - SymlinkForbidden
      - ScanDepthLimitReached
      - PermissionDenied
      - NotActivated
      - SemVerParseError
      - ConfigValueOverridden
      - ConfigValueComputed
      - ConfigWriteBlocked
      - RestoreFromBackupChosen

05.3. Debugger-related examples include:
      - DebuggerInspection
      - DebuggerMutationRequested
      - DebuggerMutationApplied
      - DebuggerMutationStaged
      - DebuggerMutationDiscarded
      - ManualPauseRequested
      - ManualPauseGranted
      - ManualPauseRefused
      - JobPausedAtSafePoint
      - JobPauseDeferred
      - JobAbortRequested

05.4. Turnix emits reasons whenever:
      - a decision is made
      - a value source changes
      - a write is accepted or refused
      - behavior differs from defaults
      - debugger tooling inspects or mutates state
      - execution is paused, staged, resumed, or aborted via debugging
        facilities


05.1 TRACE REASONS FOR RETENTION, CONTINUITY, AND LOSS (LOCKED)
---------------------------------------------------------------

05.1.1. This subsection defines required trace reasons for trace retention
        behavior, continuity evidence, and explicit loss reporting.

05.1.2. These reasons preserve the meaning of:
        "If it is not traced, it did not happen"
        under policy-driven retention.

05.1.3. Trace retention events use domain = trace or persistence.

05.1.4. Manual user actions use domain = debugger or policy.

05.1.5. Events in this subsection include correlation attributes when
        available:
        - kernelRunId
        - appInstanceId
        - pipelineRunId
        - orchestrationUnitId
        - transactionId
        - activeWindow identity (if applicable)
        - policyId (if applicable)


05.1.1 REQUIRED RETENTION AND LOSS REASONS (LOCKED)
---------------------------------------------------

05.1.1.1. The following reasons exist and are emitted when applicable.

05.1.1.2. Trace system lifecycle and storage behavior:

        - TraceStorageWriteSucceeded
          Emitted when a trace chunk, segment, or write unit is persisted
          successfully.

          Required attributes:
          - storageTarget (implementation detail label)
          - byteCount
          - eventCount
          - chunkId or storageUnitId

        - TraceStorageWriteFailed
          Emitted when a trace write attempt fails.

          Required attributes:
          - storageTarget
          - errorCode (or error identifier)
          - failureMode (example: PermissionDenied, DiskFull, IoUnavailable)
          - droppedEventCount (0 if none)
          - policyId (if policy influenced behavior)

        - TraceStorageWriteDropped
          Emitted when trace events are intentionally dropped due to
          pressure or non-blocking policy.

          Required attributes:
          - policyId
          - dropReasonCategory (StoragePressure, QuotaExceeded, Backpressure)
          - droppedEventCount
          - affectedDomainSet (optional)
          - scopeHint (kernelRun, appInstance, unknown)

05.1.1.3. Explicit pruning and removal:

        - TracePruneRequested
          Emitted when pruning is initiated.

          Required attributes:
          - policyId
          - pruneReasonCategory (PolicyRotation, QuotaExceeded, AgeExceeded,
            ManualRemoval, OtherExplicitReason)
          - intendedScope (kernelRun, appInstance, window, global)
          - prunePlanId (stable identifier for the pruning attempt)

        - TracePruneApplied
          Emitted when pruning is successfully applied.

          Required attributes:
          - policyId
          - prunePlanId
          - prunedTargetDescriptor
            (a structured description such as chunkId ranges, eventId
             ranges, or named retention buckets)
          - prunedUnitCount
          - prunedEventCount (if known)
          - scopeIdentity (kernelRunId, appInstanceId, window identity if
            applicable)

        - TracePruneFailed
          Emitted when pruning fails.

          Required attributes:
          - policyId
          - prunePlanId
          - errorCode (or error identifier)
          - failureMode
          - scopeIdentity (if known)

05.1.1.4. Continuity and completeness claims:

        - TraceContinuityAnchorCreated
          Emitted when a continuity anchor is created.

          Meaning:
          - Establishes a durable point in the trace evidence chain.

          Required attributes:
          - continuityAnchorId
          - anchorScope (kernelRun, appInstance, window, global)
          - scopeIdentity (kernelRunId, appInstanceId, window identity if
            applicable)
          - prevAnchorId (absent if first)
          - anchorDigest (if present, computed deterministically)
          - storageUnitId or chunkId (where the anchor is stored)

        - TraceContinuityBroken
          Emitted when continuity cannot be maintained.

          Examples:
          - missing previous anchor
          - corruption detected
          - storage reset

          Required attributes:
          - anchorScope
          - scopeIdentity (if known)
          - breakReasonCategory (Corrupt, Missing, Reset, Unknown)
          - lastKnownAnchorId (if any)

        - TraceEvidenceDeclaredIncomplete
          Emitted when Turnix explicitly declares that retained trace
          evidence is not complete for a scope.

          Required attributes:
          - scopeIdentity (kernelRunId, appInstanceId, window identity if
            applicable)
          - incompletenessCategory (Pruned, Dropped, Corrupt, PartialExport,
            Unknown)
          - policyId (if applicable)

05.1.1.5. Kernel run and appInstance span boundaries:

        - KernelRunSpanBegin
          Emitted at the start of a Turnix kernel run.

          Required attributes:
          - kernelRunId
          - spanIdentity

        - KernelRunSpanEnd
          Emitted at the end of a Turnix kernel run.

          Required attributes:
          - kernelRunId
          - spanIdentity
          - endReasonCategory (NormalExit, Crash, ForcedExit, Restart)

        - AppInstanceActiveSpanBegin
          Emitted when an appInstance becomes active.

          Required attributes:
          - appInstanceId
          - spanIdentity
          - activePackId (or equivalent pack identity)
          - activeSaveIdentity (optional)

        - AppInstanceActiveSpanEnd
          Emitted when an appInstance is no longer active.

          Required attributes:
          - appInstanceId
          - spanIdentity
          - endReasonCategory (NormalDeactivate, Crash, ForcedDeactivate,
            KernelExit)

05.1.1.6. Window boundary trace reasons aligned with DA-08:

        - SafePointWindowCreationRequested
          Emitted when SafePointWindow creation is requested.

          Required attributes:
          - appInstanceId
          - requestMode (ImmediateCommitted, AtQuiescence)
          - requestId (stable identifier for the request)

        - SafePointWindowCreationQueued
          Emitted when an AtQuiescence request is queued.

          Required attributes:
          - appInstanceId
          - requestId
          - queuePolicyId (if applicable)

        - SafePointWindowCreationStarted
          Emitted when SafePointWindow creation begins.

          Required attributes:
          - appInstanceId
          - requestId

        - SafePointWindowCreationSucceeded
          Emitted when SafePointWindow creation completes successfully.

          Required attributes:
          - appInstanceId
          - requestId
          - safePointWindowId
          - committedStateRoot (optional reference)
          - includedEvidencePolicyId (optional)

        - SafePointWindowCreationFailed
          Emitted when SafePointWindow creation fails.

          Required attributes:
          - appInstanceId
          - requestId
          - failureReasonCategory (NotQuiescent, Conflict, IoFailure,
            PolicyRefused, OtherExplicitReason)

        - SafePointWindowCreationCancelled
          Emitted when SafePointWindow creation is cancelled.

          Required attributes:
          - appInstanceId
          - requestId
          - cancelReasonCategory (UserCancelled, PolicyCancelled, Timeout,
            Other)

        - ActiveWindowFinalizedAsAborted
          Emitted when an ActiveWindow is finalized into an AbortedWindow.

          Required attributes:
          - appInstanceId
          - abortedWindowId
          - abortReasonCategory (ExplicitAbort, Failure, Timeout,
            PolicyRefused, CrashRecovered, OtherExplicitReason)


05.1.2 REQUIRED ATTRIBUTES FOR KEY MUTATION TRACE (LOCKED)
----------------------------------------------------------

05.1.2.1. Turnix remains able to emit audit-skeleton mutation evidence even
          under aggressive pruning.

05.1.2.2. For any committed mutation of a memory key, Turnix emits or can
          reconstruct a trace event with at least:
          - mutationKind (Set, Delete, Overwrite)
          - appInstanceId
          - memoryLayerId
          - memoryKey identity
          - baseRevisionId
          - newRevisionId
          - debuggerInitiated (true | false)
          - transactionId (if mutation was committed via a transaction)
          - orchestrationUnitId (if applicable)

05.1.2.3. baseRevisionId follows the absent-sentinel and revision rules
          defined in DA-04 04.6-04.9.

05.1.2.4. Mutation evidence is causally linked to:
          - the transaction commit event
          - the owning work span or orchestration unit span when such a
            span exists

05.1.4 DEBUGGER TRACE REASONS (REQUIRED) (LOCKED)
-------------------------------------------------

05.1.4.1. This subsection defines required trace reasons for debugger
          inspection, breakpoints, pausing, and debugger-initiated
          mutation.

05.1.4.2. These reasons are emitted when applicable.

05.1.4.3. These events are causally linked to the owning span or unit when
          such ownership exists.

05.1.4.4. These events include correlation attributes when available:
        - appInstanceId
        - pipelineRunId
        - orchestrationUnitId
        - transactionId
        - Job identity (if applicable)

05.1.4.5. Debugger connection lifecycle:

        - DebuggerConnected
          Emitted when a debugger client successfully connects.

          Required attributes:
          - debuggerClientId (stable per connection)
          - connectionKind (UI, headless, other)

        - DebuggerDisconnected
          Emitted when a debugger client disconnects unexpectedly or
          intentionally.

          Required attributes:
          - debuggerClientId
          - disconnectKind (Normal, Unexpected, Timeout, TransportError)

05.1.4.6. Debugger disconnect outcome policy effect:

        Exactly one of the following is emitted when a debugger disconnect
        causes or triggers a policy outcome for active debugging control.

        - DebuggerDisconnectedResume
          Emitted when execution continues after debugger disconnect by
          policy.

          Required attributes:
          - debuggerClientId
          - policyId

        - DebuggerDisconnectedAbort
          Emitted when active work is aborted after debugger disconnect by
          policy.

          Required attributes:
          - debuggerClientId
          - policyId
          - abortTarget (Job, orchestrationUnit, appInstance)

        - DebuggerDisconnectedHoldRefused
          Emitted if policy would require holding execution but holding is
          not allowed or cannot be achieved safely.

          Required attributes:
          - debuggerClientId
          - policyId
          - refusalReasonCategory (NotSupported, NotSafe, NotQuiescent,
            Other)

05.1.4.7. Breakpoint hit and handling:

        - BreakpointHit
          Emitted when any breakpoint triggers.

          Required attributes:
          - breakpointId
          - breakpointType (PipelineStage, Memory, Capability)
          - scope (global, appInstance, view, pack, pipeline)

          Required attributes by type:
          - PipelineStage:
            - pipelineRunId
            - stageId
          - Memory:
            - memoryLayerId
            - memoryKey identity
            - mutationPhase (StagedMutation, CommittedMutation)
            - mutationKind (Set, Delete, Overwrite)
            - transactionId (if applicable)
          - Capability:
            - capabilityId
            - callerPackId
            - decisionPoint (Before, After, PermissionDecision)

        - BreakpointResumed
          Emitted when execution resumes after a breakpoint hit.

          Required attributes:
          - breakpointId
          - resumeMode (Resume, StepOver, AbortRequested)

05.1.4.8. Manual pause and safe-point pause:

        - ManualPauseRequested
          Emitted when a manual pause is requested.

          Required attributes:
          - requestId
          - pauseScope (appInstance, Job, global)
          - targetIdentity (appInstanceId or Job identity)

        - ManualPauseGranted
          Emitted when manual pause is achieved.

          Required attributes:
          - requestId
          - achievedAtSafePoint (true | false)
          - safePointDescriptor (if achievedAtSafePoint is true)

        - ManualPauseRefused
          Emitted when manual pause is refused.

          Required attributes:
          - requestId
          - refusalReasonCategory (NotSafe, NotSupported, NotQuiescent,
            PolicyRefused, Other)

        - PauseAtSafePointRequested
          Emitted when pause-at-safe-point is requested.

          Required attributes:
          - requestId
          - targetIdentity (Job identity or orchestrationUnitId)

        - PauseAtSafePointApplied
          Emitted when pause-at-safe-point is reached and applied.

          Required attributes:
          - requestId
          - safePointDescriptor

05.1.4.9. Debugger inspection and mutation:

        - DebuggerInspection
          Emitted when the debugger performs an inspection action.

          Required attributes:
          - inspectionKind (ReadValue, Enumerate, Query, Other)
          - targetKind (Memory, Layer, Pipeline, CapabilityRegistry,
            ServicesRegistry)
          - targetIdentity (as applicable)

        - DebuggerMutationRequested
          Emitted when debugger requests a mutation.

          Required attributes:
          - mutationRequestId
          - targetKind (Memory, Breakpoint, Policy, Other)
          - targetIdentity
          - mutationKind (Set, Delete, Overwrite, Enable, Disable, Modify)

        - DebuggerMutationStaged
          Emitted when a debugger mutation is staged for apply-at-safe-point.

          Required attributes:
          - mutationRequestId
          - mutationLineageId (stable id linking staged and applied)
          - targetIdentity
          - intendedSafePointDescriptor

        - DebuggerMutationApplied
          Emitted when a debugger mutation becomes effective.

          Required attributes:
          - mutationRequestId
          - mutationLineageId
          - appliedAtSafePointDescriptor (if applied at safe point)

        - DebuggerMutationDiscarded
          Emitted when a staged debugger mutation is discarded.

          Required attributes:
          - mutationRequestId
          - mutationLineageId
          - discardReasonCategory (Abort, Timeout, PolicyCancelled,
            TargetInvalid, Other)

===============================================================================

07.1. Resolution emits:
      - candidate sets
      - filters applied
      - soft rejections
      - hard rejections
      - scoring results
      - ambiguity detection
      - user decision points

07.2. Resolution trace remains sufficient to:
      - reconstruct resolution logic offline
      - explain why one pack or version was chosen
      - explain why another was not


===============================================================================
08. TRACE DURING ACTIVATION (LOCKED)
===============================================================================

08.1. Activation emits:
      - activation start
      - dependency activation ordering
      - entrypoint invocation
      - service registration
      - capability registration
      - activation failure or success

08.2. Activation trace defines runtime causality and scope boundaries.


===============================================================================
09. CAPABILITY CALL TRACING (LOCKED)
===============================================================================

09.1. Every capability call emits:
      - caller pack identity
      - target capability identifier
      - permission classification
      - permission decision
      - execution outcome
      - duration

09.2. Capability trace distinguishes:
      - denied execution
      - deferred execution
      - successful execution

09.3. In this section, deferred execution means waiting on UI.

09.4. Capability call tracing is required for trust and audit.


===============================================================================
10. CONFIG AND STATE TRACE INTEGRATION (LOCKED)
===============================================================================

10.1. Configuration and state interactions are traced distinctly.

10.2. Trace differentiates between:

      A) Stored mutation
      - change written via IoManager
      - persisted to userdata or savePack
      - attributed to scope (engine / pack-global / save)

      B) Computed effect
      - derived value
      - reactive transform
      - accessibility scaling
      - runtime-only override

      C) Read resolution
      - which layer provided the value
      - defaults vs user override vs save override

10.3. Debugger-initiated state changes are labeled explicitly and are not
      conflated with runtime or pack-originated mutations.

10.4. Computed effects do not appear as stored mutations.

10.5. Trace attributes include:
      - config key or state key
      - scope (engine | pack-global | save | runtime)
      - source layer
      - pack identity (if applicable)
      - debuggerInitiated: true | false

10.6. State trace integration remains [constrained-by: DA-04 05],
      [constrained-by: DA-04 06], and
      [constrained-by: DA-18 07-12].


===============================================================================
11. TRACE STORAGE (LOCKED POLICY)
===============================================================================

11.1. Trace data is:
      - best-effort persistence
      - non-blocking
      - bounded or rotating
      - safe to drop under pressure

11.2. Failure to write trace remains trace-visible.

11.3. Failure to write trace does not block execution.

11.4. Trace loss is acceptable. Silent trace loss is not.


11.1 TRACE RETENTION, LOSS EVIDENCE, AND PRUNING LEDGER (LOCKED POLICY)
----------------------------------------------------------------------

11.1.1. Trace retention is policy-driven.

11.1.2. Turnix may:
        - retain traces indefinitely
        - rotate traces
        - prune traces
        - drop traces under severe pressure

11.1.3. Turnix does not:
        - silently lose trace evidence
        - silently prune trace evidence
        - silently claim completeness that does not exist

11.1.4. If trace events or trace chunks are pruned, Turnix emits a trace
        event describing the pruning action.

11.1.5. If trace writes are dropped under pressure, Turnix emits a trace
        event describing the omission.

11.1.6. Pruning and drop events include:
        - policy identity
        - scope identity
        - target identity
        - reason category
          (for example: PolicyPruned, StoragePressureDropped,
          CorruptedIsRejected, ManualRemoval, OtherExplicitReason)

11.1.7. Turnix maintains a minimal continuity record sufficient to
        establish:
        - trace existed for decision-producing actions
        - missing material is missing due to explicit pruning or dropping

11.1.8. The continuity record remains suitable for offline inspection.

11.1.9. Trace retention policy does not affect committed state correctness.

11.1.10. Loss evidence allows an auditor to distinguish:
         - not provable due to missing evidence
         from
         - not recorded by Turnix at the time of action

11.1.11. These rules remain [constrained-by: DA-04 03.2],
         [constrained-by: DA-08 17], and
         [constrained-by: DA-08 18].


===============================================================================
12. TRACE ACCESS (LOCKED)
===============================================================================

12.1. Trace access occurs through:
      - TraceMonitor UI
      - backend APIs
      - future export formats

12.2. Trace consumers do not rely on direct file scraping.

12.3. Trace consumers rely on structured access.


===============================================================================
13. USER-FACING DIAGNOSTICS (LOCKED)
===============================================================================

13.1. Trace events may be surfaced to users as:
      - warnings
      - explanations
      - decision prompts
      - error dialogs
      - debugger inspection views

13.2. User-facing views:
      - simplify presentation
      - preserve causality
      - do not fabricate reasons

13.3. Raw trace remains developer-facing.


===============================================================================
14. ERROR ENVELOPE INTEGRATION (LOCKED)
===============================================================================

14.1. All surfaced errors include:
      - error code
      - human message
      - reason
      - trace correlation identifiers

14.2. Errors and traces remain inseparable.

14.3. Every surfaced error maps to trace events.


===============================================================================
15. DEV-MODE ENHANCEMENTS (LOCKED)
===============================================================================

15.1. Development mode may add:
      - verbose traces
      - speculative branches
      - why-not-chosen explanations
      - layer-resolution breakdowns
      - debugger-specific annotations

15.2. Dev-mode enhancements are explicit.

15.3. Dev-mode enhancements are never enabled silently.

15.4. Dev-mode enhancements do not change runtime behavior.


===============================================================================
16. NO HEURISTIC SUPPRESSION (LOCKED)
===============================================================================

16.1. Turnix does not:
      - suppress obvious traces
      - collapse distinct decisions
      - infer reasons without recording them
      - merge computed effects with stored changes
      - hide debugger-originated actions

16.2. Explicit evidence takes precedence over clever omission.


===============================================================================
17. EXPLICITLY OPEN QUESTIONS
===============================================================================

17.1. OPEN questions include:
      - trace visualization schemas
      - time-travel replay tooling
      - cross-session trace stitching
      - diff-oriented trace views


===============================================================================
18. LOCKED SUMMARY
===============================================================================

18.1. LOCKED:
      - structured tracing everywhere
      - stable PascalCase reasons
      - canonical Trace Directed Acyclic Graph naming and meaning
      - causal DAG model
      - multi-parent causal event linkage
      - successful paths remain observable, not only failures
      - a successful path that cannot be explained is an observability
        failure
      - trace spans with explicit begin/end and carry-forward constraints
      - kernel-run, appInstance, and work-span stratification
      - required retention, loss, continuity, and window-boundary reasons
      - required per-event attributes for retention, continuity, window, and
        debugger reason families
      - full discovery, resolution, activation coverage
      - capability auditing
      - config vs computed distinction
      - debugger inspection and mutation traceability
      - scope-aware tracing
      - retention-loss evidence and continuity claims
      - DA-08-aligned SafePoint and ActiveWindow correlation rules

18.2. OPEN:
      - visualization and tooling only
