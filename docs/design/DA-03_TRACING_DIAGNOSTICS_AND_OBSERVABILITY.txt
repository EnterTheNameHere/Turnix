===============================================================================
DESIGN AREA DA-03 - TRACING, DIAGNOSTICS & OBSERVABILITY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the Turnix tracing substrate, including trace events, causal
structure, spans, trace evidence, retention-loss evidence, and diagnostic
access, without owning domain-specific lifecycle or tracing obligations.

Purpose:
--------
This design area defines the shared Turnix tracing, diagnostics, and
observability substrate.

It defines:
- what trace evidence means in Turnix
- what trace evidence is authoritative for
- trace event structure
- trace DAG semantics
- parent links and causal links
- trace span mechanics
- trace roots and span stratification
- trace reason field semantics
- trace attribute field semantics
- trace retention, pruning, and loss evidence
- trace access and user-facing diagnostic constraints
- error-envelope correlation with trace evidence

It does NOT define:
- which domain objects must be traced
- which domain lifecycle transitions exist
- which domain events are important enough to record
- which object-specific trace reasons must be emitted
- which object-specific trace attributes are required
- discovery, resolution, activation, capability, configuration, state,
  Window, debugger, RPC, or Job lifecycle semantics
- authoritative committed state, transactions, or windows (see DA-04)
- debugger pause and mutation safety semantics (see DA-18)
- concrete storage formats, codecs, or digest algorithms
  (see DA-05, DA-06, DA-08)
- UI layout for trace viewers or debugger tooling

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines tracing, diagnostics, and observability
      substrate semantics within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. Tracing answers:

          "What happened, what caused what, and why?"

02.2. Tracing is structured evidence with explicit reasons and causal
      links.

02.3. Tracing is not logging.

02.4. Plain logs are secondary and optional.

02.5. Logs do not replace trace evidence.

02.6. Tracing provides:
      - causal explanation
      - decision evidence
      - observed temporal sequence
      - diagnostic context

02.7. Tracing is authoritative evidence for:
      - causality represented inside retained trace evidence
      - decisions represented inside retained trace evidence
      - audit reconstruction within retained trace evidence
      - debugger and tooling correlation within retained trace evidence
      - policy-enforcement provenance within retained trace evidence

02.8. Tracing is not authoritative for:
      - committed state
      - committed-state restoration
      - replay semantics
      - rollback eligibility by itself
      - object lifecycle truth outside retained trace evidence

02.9. "If it is not traced, it did not happen" means:
      - trace evidence is the proof that a trace-using subsystem recorded
        the action
      - if an action cannot be found in retained trace evidence, the action
        is not provable from trace
      - trace retention is policy-driven
      - trace loss is acceptable only when loss is explicitly represented
        as loss evidence

02.10. This mantra does not make this document the owner of every domain
       object's tracing obligations.

02.11. Trace evidence can represent successful, failed, refused, cancelled,
      superseded, and other domain-defined outcomes.

02.12. Trace evidence supports explanation of an outcome only when the
       owning domain records evidence for that outcome.

02.13. A trace consumer does not infer an unrecorded outcome merely because
       failure evidence is absent.

02.14. Reason fields, causal links, and attributes allow Turnix to explain
       recorded events without treating unrecorded events as proven.

02.15. The design area that owns an object, lifecycle, workflow, or
       subsystem defines whether and how that subject uses the tracing
       substrate.

02.16. If a domain chooses not to emit trace evidence for a meaningful
       action, that absence is a weakness of that domain's design.

02.17. That absence does not change the tracing substrate defined by this
       document.


===============================================================================
03. TRACE SUBSTRATE OWNERSHIP
===============================================================================

03.1. DA-03 owns the tracing substrate.

03.2. DA-03 defines how trace evidence is represented, related, retained,
      accessed, and interpreted.

03.3. DA-03 does not own domain lifecycles.

03.4. DA-03 does not prescribe object-specific trace coverage.

03.5. DA-03 does not define which transitions exist for another design
      area's object.

03.6. DA-03 does not define which domain events are important enough to
      record.

03.7. The owner of each lifecycle, subsystem, workflow, or object defines
      its own tracing obligations when trace evidence is required for that
      subject.

03.8. Subject-specific trace obligations remain subordinate to the trace
      mechanics defined by this document.

03.9. Subject-specific documents do not redefine trace event structure,
      trace DAG semantics, span mechanics, retention-loss meaning, or trace
      access semantics.

03.10. Moving a subject-specific trace obligation out of DA-03 does not
       weaken the obligation when the subject-owning document preserves it.

03.11. The migration of subject-specific trace obligations preserves one
       authority home per concept.


===============================================================================
04. TRACE EVENT MODEL
===============================================================================

04.1. A trace event is one structured evidence record.

04.2. Each trace event has:
      - eventId
      - timestamp
      - level
      - domain
      - message
      - reason
      - attributes
      - optional parentEventIds

04.3. eventId identifies one trace event.

04.4. timestamp records observed emission time.

04.5. timestamp is observational.

04.6. timestamp does not define causality.

04.7. level classifies presentation severity.

04.8. domain classifies the subsystem or subject area that emitted the
      event.

04.9. message provides a human-readable explanation.

04.10. reason provides a machine-readable reason identifier.

04.11. attributes provide structured data required to interpret the event.

04.12. parentEventIds records explicit causal parents when such parents
       exist.

04.13. The exact concrete serialization of trace events is defined outside
       this document.

04.14. Concrete serialization does not change trace event semantics.


===============================================================================
05. TRACE DAG
===============================================================================

05.1. Events form a trace DAG rather than a flat stream.

05.2. "Trace Directed Acyclic Graph" is the canonical name for the causal
      trace structure defined by this document.

05.3. "trace DAG" is the permitted abbreviation.

05.4. The trace DAG is the directed acyclic graph formed by trace events
      and their explicit parent and span-causality links.

05.5. Causal ordering is authoritative within retained trace evidence.

05.6. Causal ordering is represented by explicit DAG links.

05.7. Temporal ordering is observational only.

05.8. Temporal ordering does not determine causality or correctness.

05.9. Turnix does not infer causality from timestamps.

05.10. Cycles do not occur in the trace DAG.

05.11. No trace event is causally reachable from itself through parent or
       span-causality links.

05.12. The trace DAG is not a substitute for authoritative committed state.


===============================================================================
06. PARENT LINKS
===============================================================================

06.1. A trace event may reference zero, one, or multiple parent events.

06.2. parentEventIds, when present, is the set of explicit causal parents
      of that trace event.

06.3. Multiple parent links are permitted.

06.4. Absence of parent links means independent trace origin.

06.5. Absence of parent links does not imply simultaneity.

06.6. A trace event does not list the same parent event more than once.

06.7. Parent links express causal structure rather than temporal order.


===============================================================================
07. TRACE SPANS
===============================================================================

07.1. Trace spans represent bounded work with a start and an end.

07.2. A span has a begin event and an end event.

07.3. Span begin and span end events are trace events.

07.4. Span begin and span end events participate in the trace DAG.

07.5. Span begin occurs before span end in the same trace DAG.

07.6. A span end does not exist without a reachable span begin.

07.7. Span begin and span end share a stable span identity attribute.

07.8. The exact span identity encoding remains OPEN.

07.9. Span begin and span end are connected by explicit causality links.

07.10. Any export, snapshot, or offline trace view that includes span end
       also includes the corresponding span begin.

07.11. If a snapshot includes trace evidence, it satisfies the rule:
       "no included span end without included span begin."

07.12. This snapshot requirement is [satisfied-by: DA-08 11.9].


===============================================================================
08. TRACE ROOTS AND SPAN STRATIFICATION
===============================================================================

08.1. Trace storage may contain multiple unconnected trace DAG roots.

08.2. Each root represents an independent trace origin.

08.3. Unconnected roots are expected and remain representable without
      ambiguity.

08.4. Root spans identify the outer retained causal region for trace
      evidence.

08.5. Nested spans identify bounded work inside a retained causal region.

08.6. Span stratification allows trace consumers to distinguish broad
      runtime regions from nested bounded work.

08.7. A subject-owning document may define subject-specific span kinds.

08.8. Subject-specific span kinds do not redefine the span mechanics in
      this document.

08.9. A subject-specific span remains a trace span when it follows the
      begin, end, identity, and causal-link rules in this document.

08.10. DA-03 owns the trace-substrate reason identifiers used for kernel-run
       trace root spans.

08.11. The following kernel-run trace span reasons are defined:

       - KernelRunSpanBegin
       - KernelRunSpanEnd

08.12. KernelRunSpanBegin is emitted when Turnix begins a kernel run.

08.13. KernelRunSpanBegin trace evidence records enough context to explain:
       - kernelRunId
       - spanIdentity

08.14. KernelRunSpanEnd is emitted when Turnix ends a kernel run.

08.15. KernelRunSpanEnd trace evidence records enough context to explain:
       - kernelRunId
       - spanIdentity
       - endReasonCategory

08.16. KernelRunSpanEnd endReasonCategory distinguishes:
       - NormalExit
       - Crash
       - ForcedExit
       - Restart


===============================================================================
09. TRACE REASON FIELD
===============================================================================

09.1. reason is the machine-readable reason field on a trace event.

09.2. Reason values identify why an event was emitted.

09.3. Reason values are interpreted within the domain that owns them.

09.4. DA-03 does not define a global catalog of domain-specific reason
      values.

09.5. DA-03 does not define object-specific reason naming schemes.

09.6. DA-03 does not require a domain to use a particular reason value for
      an object-specific transition.

09.7. The document that owns a domain, lifecycle, or workflow defines the
      reason values needed by that subject.

09.8. Subject-owned reason values are interpreted through the trace event
      mechanics defined here.

09.9. Reason examples in this document are illustrative when present.

09.10. Illustrative reason examples do not create required domain coverage.


===============================================================================
10. TRACE ATTRIBUTES
===============================================================================

10.1. attributes is the structured key-value data attached to a trace event.

10.2. Attributes carry event-specific evidence.

10.3. Attributes support offline linking between trace evidence and Turnix
       objects.

10.4. Stable identifiers may appear in attributes when the emitting domain
      owns or knows those identifiers.

10.5. Identifier presence varies by domain.

10.6. DA-03 does not define every object-specific attribute set.

10.7. The subject-owning document defines which attributes are required for
      its own events when it defines trace obligations.

10.8. Subject-specific attributes do not redefine the attribute field
      mechanics defined by this document.

10.9. Boundary events and mutation events remain linkable to their owning
      objects when the owning domain requires that linkage.


===============================================================================
11. TRACE STORAGE AND LOSS EVIDENCE
===============================================================================

11.1. Trace data is:
      - best-effort persistence
      - non-blocking
      - bounded or rotating
      - safe to drop under pressure

11.2. Failure to write trace evidence does not block execution.

11.3. Failure to write trace evidence does not affect committed-state
      correctness.

11.4. Trace loss is acceptable.

11.5. Silent trace loss is not acceptable.

11.6. Trace retention is policy-driven.

11.7. Turnix may:
      - retain traces indefinitely
      - rotate traces
      - prune traces
      - drop traces under severe pressure

11.8. Turnix does not silently claim trace completeness that does not exist.

11.9. If trace evidence is pruned, dropped, corrupted, partially exported,
      or otherwise incomplete, retained evidence represents that limitation
      explicitly when the limitation is known.

11.10. Loss evidence allows an auditor to distinguish:
       - not provable due to missing evidence
       from
       - not recorded by Turnix at the time of action

11.11. Trace retention policy does not affect committed state correctness.

11.12. These rules remain [constrained-by: DA-04 03.2],
       [constrained-by: DA-08 17], and [constrained-by: DA-08 18].

11.13. DA-03 owns trace-substrate reason identifiers for trace storage,
       trace pruning, trace continuity, and trace incompleteness.

11.14. The following trace-substrate reasons are defined:

       - TraceStorageWriteSucceeded
       - TraceStorageWriteFailed
       - TraceStorageWriteDropped
       - TracePruneRequested
       - TracePruneApplied
       - TracePruneFailed
       - TraceContinuityAnchorCreated
       - TraceContinuityBroken
       - TraceEvidenceDeclaredIncomplete

11.15. TraceStorageWriteSucceeded is emitted when a trace storage write unit
       is persisted successfully.

11.16. TraceStorageWriteFailed is emitted when a trace storage write attempt
       fails.

11.17. TraceStorageWriteDropped is emitted when trace events are intentionally
       dropped due to pressure, quota, or non-blocking trace policy.

11.18. TracePruneRequested is emitted when trace pruning is initiated.

11.19. TracePruneApplied is emitted when trace pruning is successfully
       applied.

11.20. TracePruneFailed is emitted when trace pruning fails.

11.21. TraceContinuityAnchorCreated is emitted when a trace continuity anchor
       is created.

11.22. TraceContinuityBroken is emitted when trace continuity cannot be
       maintained.

11.23. TraceEvidenceDeclaredIncomplete is emitted when Turnix explicitly
       declares retained trace evidence incomplete for a scope.


===============================================================================
12. TRACE ACCESS
===============================================================================

12.1. Trace access occurs through controlled Turnix surfaces.

12.2. Trace consumers do not rely on direct file scraping.

12.3. Trace consumers rely on structured access.

12.4. Trace access may support:
      - TraceMonitor UI
      - backend APIs
      - debugger tooling
      - future export formats

12.5. A trace access surface preserves causal meaning.

12.6. A trace access surface does not fabricate reasons.

12.7. A trace access surface does not hide known evidence loss.


===============================================================================
13. USER-FACING DIAGNOSTICS
===============================================================================

13.1. Trace events may be surfaced to users as:
      - warnings
      - explanations
      - decision prompts
      - error dialogs
      - debugger inspection views

13.2. User-facing views may simplify presentation.

13.3. User-facing views preserve causality.

13.4. User-facing views do not fabricate reasons.

13.5. Raw trace remains developer-facing unless a surface explicitly exposes
      it.


===============================================================================
14. ERROR ENVELOPE CORRELATION
===============================================================================

14.1. Surfaced Turnix errors include trace correlation evidence when
      available.

14.2. A surfaced error includes or remains linkable to:
      - error code
      - human-readable message
      - reason
      - trace correlation identifiers when available

14.3. Error envelopes do not replace trace evidence.

14.4. Surfaced errors and trace evidence remain linked when trace evidence
      exists for the failing operation.

14.5. A surfaced error maps to one or more trace events when the error
      occurs inside Turnix-traced execution.

14.6. If trace evidence for a surfaced error is unavailable, incomplete,
      pruned, dropped, corrupt, or not yet initialized, that limitation
      remains explicit.

14.7. Concrete error-envelope schemas are defined by the owning protocol or
      implementation specification.

14.8. The owning protocol or implementation specification remains
      constrained by the trace-correlation semantics defined here.

14.9. DD-02 defines the interface doctrine for surfaced failure structure.

14.10. DA-03 defines only the trace-correlation obligation for surfaced
       errors.


===============================================================================
15. DEVELOPMENT-MODE TRACE ENHANCEMENTS
===============================================================================

15.1. Development mode may add trace and diagnostic enrichment.

15.2. Development-mode trace enrichment may include:
      - verbose traces
      - speculative branches
      - why-not-chosen explanations
      - layer-resolution breakdowns
      - debugger-specific annotations

15.3. Development-mode trace enrichment is explicit.

15.4. Development-mode trace enrichment is never enabled silently.

15.5. Development-mode trace enrichment does not change runtime behavior.

15.6. Development-mode trace enrichment does not create authoritative state.

15.7. Development-mode trace enrichment does not create additional
      authority.

15.8. Development-mode trace enrichment does not replace domain-owned trace
      obligations.


===============================================================================
16. NO HEURISTIC TRACE SUPPRESSION
===============================================================================

16.1. Turnix does not infer reasons without recording them.

16.2. Turnix does not fabricate causal links from temporal ordering.

16.3. Turnix does not hide known trace loss.

16.4. Turnix does not present incomplete trace evidence as complete.

16.5. A subject-owning document may decide that a subject does not emit
      trace evidence for a given action.

16.6. If a subject-owning document requires trace evidence for an action,
      the trace substrate does not silently suppress that evidence.


===============================================================================
17. MIGRATED DOMAIN TRACE OBLIGATIONS
===============================================================================

17.1. Earlier revisions of this document contained domain-specific trace
      obligations.

17.2. Those obligations are moving to the documents that own the affected
      domains.

17.3. Discovery-specific trace obligations belong in the discovery-owning
      design area.

17.4. Resolution-specific trace obligations belong in the resolution-owning
      design area.

17.5. Activation-specific trace obligations belong in the activation-owning
      design area.

17.6. Capability-specific trace obligations belong in the capability-owning
      design area.

17.7. State and transaction trace obligations belong in the state-owning
      design area.

17.8. Configuration trace obligations belong in the configuration-owning
      design area.

17.9. Window trace obligations belong in the Window-owning design area.

17.10. Debugger trace obligations belong in the debugger-owning design
       area.

17.11. RPC trace obligations belong in the RPC implementation
       specification or a higher RPC semantic document when one exists.

17.12. This migration changes ownership location.

17.13. This migration does not intentionally weaken valid domain trace
       obligations that are preserved in their owning documents.

      - DA-03 owns the tracing substrate
      - DA-03 does not own object-specific lifecycle truth
      - tracing is structured evidence rather than logging
      - trace-substrate reason identifiers are owned by DA-03
      - trace evidence explains what happened, what caused what, and why
      - retained trace evidence is authoritative for retained causal proof
      - trace evidence is not authoritative committed state
      - trace evidence does not define rollback eligibility by itself
      - the trace DAG is the canonical causal structure
      - temporal order does not define causality
      - parent links express explicit causal parentage
      - trace spans have begin and end events
      - span end does not exist without a reachable span begin
      - exported or snapshotted trace evidence does not include span end
        without the corresponding span begin
      - trace roots and span stratification are representable
      - reason is a machine-readable event field
      - domain documents own their own reason values and required
        attributes
      - trace retention is policy-driven
      - trace loss is acceptable only when represented honestly
      - trace access occurs through controlled Turnix surfaces
      - user-facing diagnostics preserve causality and do not fabricate
        reasons
      - surfaced errors correlate with trace evidence where available
      - development-mode trace enrichment is explicit and non-authoritative
      - development-mode enhancements do not change runtime behavior
      - domain-specific trace obligations live in the documents that own
        those domains

      - concrete formats and tooling refinements only
