===============================================================================
DESIGN AREA DA-23 - EXECUTION BOUNDARIES, ADAPTERS AND CODE RUNTIMES
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the Turnix execution boundary for declared pack code, the abstract
role of execution adapters, code artifact recognition and activation
phases, PackContext authority, and the RPC-shaped boundary through which
adapter-executed code participates in an AppInstance.

This document owns the abstract execution-boundary authority model. Adapter
lifecycle, trust, isolation, concrete adapter classes, adapter-domain trace
reason identifiers, and runtime-host deployment semantics are defined by
DA-27.

Purpose:
--------
This document defines how Turnix treats executable pack code across host
languages and runtime environments without allowing host-language convenience
to bypass Kernel authority.

It defines:
- Kernel authority over adapter-executed code participation
- execution adapter meaning at the abstract boundary level
- code artifact recognition, declaration, binding, permission, and
  activation phases
- executable artifact declaration requirements
- pack-provided code versus backend implementation code
- PackContext authority and limits
- RPC-shaped execution boundary semantics
- relationship between adapters and transports
- relationship to AppPack, SavePack, and AppInstance
- relationship to activation, imports, capabilities, decisions, Jobs,
  Views, Orchestration Units, and runtime state
- direct-call transport optimization boundaries

It does NOT define:
- generic resolution mechanics (see DA-11)
- importing and module-loading selector details (see DA-12)
- activation lifecycle beyond execution-boundary constraints (see DA-14)
- capability permission semantics (see DA-16)
- user decision and remembered policy semantics (see DA-17)
- Job lifecycle, timeout, cancellation, or progress semantics (see DA-19)
- AppInstance lifecycle or runtime identity semantics (see DA-20)
- abstract RPC, channel, and transport semantics (see DA-21)
- UI, View, and frontend runtime presentation semantics (see DA-22)
- Orchestration Unit lifecycle or mutation authority semantics (see DA-24)
- adapter lifecycle, adapter trust and isolation, concrete adapter classes,
  adapter-domain trace reason identifiers, or runtime-host deployment
  semantics (see DA-27)
- concrete RPC message schema (see IS-01)
- concrete WebSocket flow (see IS-02)
- concrete Python adapter implementation
- concrete JavaScript loader implementation
- concrete CLI command syntax
- concrete workspace locking implementation

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines execution-boundary, adapter, and code-runtime
      authority semantics within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. This design area answers:

          "How does Turnix execute declared pack code without allowing host
           language, process location, transport choice, or frontend
           convenience to bypass Kernel authority?"

02.2. Kernel owns authority.

02.3. Execution adapters execute declared pack code.

02.4. Transports carry messages.

02.5. RPC defines the logical communication boundary.

02.6. Capabilities define what adapter-executed code may do.

02.7. SavePack stores durable executed state under Kernel-owned persistence
      rules.

02.8. The host language used by pack code does not grant backend authority.

02.9. Pack-provided Python is not backend implementation code merely because
      Turnix backend implementation is Python.

02.10. Pack-provided JavaScript is not frontend authority merely because it
       runs in a browser.

02.11. Pack code participates in an AppInstance only through Turnix-managed
       context, capability, RPC-shaped, lifecycle, policy, trace, and state
       boundaries.

02.12. Direct calls may be used as a transport optimization.

02.13. Direct backend authority is not a valid authority path for pack code.

02.14. Later sections refine and apply these rules. They do not weaken them.


===============================================================================
03. KERNEL AUTHORITY BOUNDARY
===============================================================================

03.1. Kernel is the trusted Turnix authority owner.

03.2. Kernel owns:
      - platform authority
      - AppInstance lifecycle coordination
      - orchestration entry boundaries
      - authoritative state mutation boundaries
      - SavePack persistence authority
      - capability routing
      - policy decision application
      - permission enforcement
      - trace substrate participation
      - adapter boundary translation

03.3. Kernel authority is not delegated merely because code runs in the same
      process.

03.4. Kernel authority is not delegated merely because code runs in the same
      language.

03.5. Kernel authority is not delegated merely because a transport is
      in-process.

03.6. Kernel authority is not delegated merely because a frontend can send a
      message.

03.7. Kernel authority is not delegated merely because a pack declares code.

03.8. Kernel may expose authority through capabilities, policies, and
      Turnix-controlled contexts.

03.9. Exposed authority remains scoped by the capability, policy, context,
      AppInstance, SavePack, and lifecycle boundaries that grant it.


===============================================================================
04. EXECUTION ADAPTER
===============================================================================

04.1. An execution adapter is a Turnix-managed environment capable of
      loading and executing declared pack code.

04.2. An execution adapter is a boundary participant, not authority by
      itself.

04.3. An execution adapter may run code:
      - in the browser
      - in the backend process
      - in another process
      - over a local socket or pipe
      - over a LAN transport
      - inside a container or virtual machine
      - through another explicitly defined runtime environment

04.4. Execution location does not define authority.

04.5. Adapter mediation does not imply sandboxing.

04.6. Adapter sandboxing, trust classification, lifecycle, concrete adapter
      classes, and adapter trace reason ownership are defined by DA-27.

04.7. The minimum abstract adapter role is to:
      - load declared executable artifacts
      - execute code through declared lifecycle entrypoints
      - provide a Turnix-controlled context to pack code
      - communicate with Kernel through an RPC-shaped boundary
      - report structured result, error, readiness, and failure information
      - preserve trace correlation metadata where available
      - remain subordinate to capability and permission boundaries

04.8. An adapter does not own authoritative persistent state unless a later
      authoritative document defines an explicit delegated state authority.

04.9. An adapter does not become Kernel authority because it is trusted.

04.10. An adapter does not become Kernel authority because it is in-process.

04.11. An adapter does not become Kernel authority because its transport is
       direct.


===============================================================================
05. CODE ARTIFACT PHASES
===============================================================================

05.1. Recognizing a file as code is not permission to execute it.

05.2. Turnix separates executable code handling into these phases:
      - recognition
      - declaration
      - binding
      - permission
      - activation
      - execution

05.3. Recognition means Turnix identifies a file or manifest entry as a code
      artifact.

05.4. Declaration means pack metadata identifies the code artifact as
      executable and states its intended role.

05.5. Binding means Turnix determines which adapter kind or adapter provider
      can execute the declared artifact.

05.6. Permission means Turnix determines whether the declared artifact may
      be executed in the current pack, AppInstance, SavePack, trust, policy,
      and user-decision context.

05.7. Activation means Turnix dispatches the declared lifecycle entrypoint
      through the selected adapter.

05.8. Execution means adapter-executed code runs within the Turnix context
      and boundary provided for that activation or operation.

05.9. No phase silently implies a later phase.

05.10. Recognition does not imply declaration.

05.11. Declaration does not imply binding success.

05.12. Binding does not imply execution permission.

05.13. Permission does not imply activation success.

05.14. Activation dispatch does not imply execution success.

05.15. Execution success does not imply committed-state mutation.


===============================================================================
06. DECLARED EXECUTABLE ARTIFACTS
===============================================================================

06.1. Executable artifacts are declared explicitly.

06.2. A pack does not execute code merely because a file with a recognized
      extension exists.

06.3. A declaration identifies at least:
      - artifact path or identity
      - adapter kind or runtime requirement
      - lifecycle role
      - entrypoint when applicable
      - requested capabilities when applicable
      - trust or policy requirements when applicable

06.4. Declaration format is owned by the relevant manifest, pack, activation,
      or implementation specification documents.

06.5. This document owns the authority meaning of declaration, not the final
      manifest field syntax.

06.6. Undeclared executable code is not activated by Turnix.

06.7. Declared executable code remains subject to adapter binding,
      permission, activation, capability, and policy checks.

06.8. A declaration does not grant capability permission by itself.

06.9. A declaration does not grant persistent I/O authority by itself.

06.10. A declaration does not grant committed-state mutation authority by
       itself.


===============================================================================
07. PACK CODE VERSUS BACKEND IMPLEMENTATION CODE
===============================================================================

07.1. Backend implementation code is trusted platform implementation code.

07.2. Pack-provided code is content supplied by packs.

07.3. Pack-provided code is not backend implementation code merely because
      it is written in the backend implementation language.

07.4. Pack-provided code is not backend implementation code merely because
      it runs in the backend process.

07.5. Pack-provided code is not frontend authority merely because it runs in
      the browser.

07.6. Pack-provided code receives controlled participation through
      PackContext, capability calls, RPC-shaped messages, and declared
      lifecycle entrypoints.

07.7. Pack-provided code does not import backend internals as an authority
      path.

07.8. Pack-provided code does not directly mutate Kernel-owned authoritative
      state.

07.9. Pack-provided code does not directly write SavePack state as an
      authority path.

07.10. Pack-provided code does not bypass tracing, policy, capability,
       permission, lifecycle, or transaction boundaries through native
       language convenience.

07.11. Any allowed same-process or same-language integration remains a
       Turnix-controlled boundary, not a backend authority grant.


===============================================================================
08. PACKCONTEXT
===============================================================================

08.1. PackContext is the Turnix-controlled context object or context
      interface provided to adapter-executed pack code.

08.2. PackContext exposes scoped participation in Turnix.

08.3. PackContext may expose:
      - pack identity
      - AppInstance identity when applicable
      - View identity when applicable
      - capability call surfaces
      - event or request surfaces
      - state access surfaces when allowed
      - logging or trace correlation helpers
      - cancellation or lifecycle helpers when allowed

08.4. PackContext does not expose general backend internals.

08.5. PackContext does not expose unrestricted filesystem authority.

08.6. PackContext does not expose unrestricted committed-state mutation.

08.7. PackContext does not expose unrestricted capability invocation.

08.8. PackContext is scoped to the pack, adapter, AppInstance, lifecycle,
      permission, and policy context that created it.

08.9. PackContext authority does not outlive the lifecycle boundary that
      created it unless an explicit retained handle model is defined.

08.10. PackContext may be represented differently across languages and
       adapters.

08.11. Host-language representation differences do not change PackContext
       authority meaning.


===============================================================================
09. RPC-SHAPED EXECUTION BOUNDARY
===============================================================================

09.1. Adapter-executed code communicates with Kernel authority through an
      RPC-shaped boundary.

09.2. RPC-shaped boundary means the interaction preserves:
      - message or call envelope semantics
      - route or capability target identity
      - permission checks
      - lifecycle ownership
      - trace correlation
      - structured result, error, cancellation, and readiness reporting
      - AppInstance and SavePack state authority boundaries

09.3. RPC-shaped boundary is not equivalent to WebSocket.

09.4. RPC-shaped boundary is not necessarily networked.

09.5. RPC-shaped boundary may be implemented by:
      - in-process call
      - local queue
      - WebSocket
      - stdio
      - local socket
      - named pipe
      - TCP socket
      - postMessage
      - another explicitly defined transport

09.6. The concrete transport does not define authority.

09.7. Direct in-process calls may implement an RPC-shaped boundary.

09.8. Direct in-process calls do not bypass capability routing, permission
      checks, lifecycle ownership, trace correlation, structured error
      handling, Orchestration Unit boundaries, transaction boundaries, or
      AppInstance state authority.

09.9. A call that bypasses those boundaries is not treated as valid pack
      execution authority merely because it works technically.

09.10. DA-21 owns abstract RPC, channel, and transport semantics.

09.11. IS-01 owns concrete RPC message schema.

09.12. IS-02 owns concrete WebSocket flow.


===============================================================================
10. RELATION TO TRANSPORTS
===============================================================================

10.1. Transport is the concrete mechanism carrying messages or call
      envelopes.

10.2. Transport may be local, remote, in-process, process-boundary,
      browser-boundary, or networked.

10.3. Transport does not define authority.

10.4. Transport does not grant capability permission.

10.5. Transport does not grant adapter execution authorization.

10.6. Transport does not define AppInstance lifecycle.

10.7. Transport does not define Job lifecycle.

10.8. Transport failure does not by itself define adapter failure.

10.9. Adapter failure does not by itself define transport failure.

10.10. Transport timeout does not by itself define Job timeout or adapter
       timeout.

10.11. DA-21 defines the abstract separation between RPC, channels, and
       transports.

10.12. DA-27 defines adapter lifecycle and adapter failure semantics.


===============================================================================
11. RELATION TO APPPACK, SAVEPACK, AND APPINSTANCE
===============================================================================

11.1. AppPack defines executable application behavior and entrypoints.

11.2. SavePack stores persisted executed state for an AppPack.

11.3. AppInstance is the live runtime identity created by loading an AppPack
      with a SavePack.

11.4. Kernel load formula is:

          AppPack + SavePack + RuntimeContext -> AppInstance

11.5. Adapter-executed code participates in an AppInstance through the
      Kernel-owned authority model.

11.6. Adapter-executed code does not own the AppInstance merely because it
      runs application logic.

11.7. Adapter-executed code does not own the SavePack merely because its
      work affects persisted state.

11.8. SavePack persistence remains under Kernel-owned persistence authority.

11.9. A writable SavePack is not written through adapter-owned state
      authority.

11.10. AppInstance lifecycle and runtime identity are defined by DA-20.

11.11. SavePack persistence safety is constrained by DA-04 and DA-07.


===============================================================================
12. RELATION TO ACTIVATION
===============================================================================

12.1. Activation lifecycle semantics are defined by DA-14.

12.2. Activation dispatch through an adapter is one activation mechanism.

12.3. Adapter activation dispatch does not imply activation success.

12.4. Adapter artifact loading does not imply activation success.

12.5. Adapter readiness does not imply activation success.

12.6. Activation success does not imply committed-state mutation.

12.7. Activation success does not imply unrestricted capability permission.

12.8. This document constrains activation only where activation crosses the
      execution adapter boundary.

12.9. DA-27 defines adapter lifecycle states and adapter-domain trace
      reason identifiers used around adapter loading and dispatch.


===============================================================================
13. RELATION TO IMPORTS AND MODULE LOADING
===============================================================================

13.1. Importing and module-loading semantics are defined by DA-12.

13.2. Native language imports cannot bypass Turnix semantics.

13.3. Python pack code does not import backend internals as an authority
      path.

13.4. JavaScript pack code does not bypass Turnix resolution, visibility,
      activation, or capability boundaries through native or arbitrary
      loading mechanisms.

13.5. turnixImport-style mechanisms remain Turnix-controlled loading
      surfaces where JavaScript pack code needs cross-pack access.

13.6. Host-language module systems may be used only within boundaries
      allowed by DA-12 and this document.

13.7. A successful native import does not imply Turnix permission,
      activation, state authority, or capability permission.


===============================================================================
14. RELATION TO CAPABILITIES AND PERMISSIONS
===============================================================================

14.1. Capability and permission semantics are defined by DA-16.

14.2. Adapter-executed code receives no general backend access.

14.3. Adapter-executed code receives capability-scoped access through its
      Turnix-controlled context.

14.4. Capability permission is checked by Kernel-owned authority paths.

14.5. Adapter identity may participate in permission decisions when the
      governing permission model defines such participation.

14.6. Adapter identity does not grant permission by itself.

14.7. Pack identity does not grant permission by itself.

14.8. Transport identity does not grant permission by itself.

14.9. Capability invocation that mutates state remains constrained by DA-04
      and DA-24.

14.10. Capability invocation through RPC remains constrained by DA-21.


===============================================================================
15. RELATION TO DECISIONS AND POLICY
===============================================================================

15.1. User decision and policy handling are defined by DA-17.

15.2. Execution authorization may require policy evaluation or user
      decision.

15.3. Trust decisions for unsafe or unsandboxed adapter execution are
      policy-scoped decisions.

15.4. A policy decision authorizes only the scope it states.

15.5. A remembered policy decision does not silently broaden itself to new
      adapter kinds, new packs, new capabilities, new SavePacks, or new
      trust classes.

15.6. Policy decision storage does not become pack authority.

15.7. DA-27 defines adapter trust and isolation classification semantics
      consumed by policy decisions.


===============================================================================
16. RELATION TO JOBS AND ORCHESTRATION UNITS
===============================================================================

16.1. Job lifecycle semantics are defined by DA-19.

16.2. Orchestration Unit lifecycle and mutation authority semantics are
      defined by DA-24.

16.3. Adapter-executed work may be associated with a Job when the owning
      workflow defines Job-backed behavior.

16.4. Adapter-executed work may create or enter an Orchestration Unit when
      it seeks Turnix mutation guarantees or lifecycle authority.

16.5. Adapter execution acceptance is not Job success.

16.6. Adapter execution failure is not Job failure by default.

16.7. Adapter disconnect is not Job cancellation by default.

16.8. Adapter execution failure is not Orchestration Unit failure by default.

16.9. Job and Orchestration Unit relationships to adapter-executed work are
      explicit and trace-visible when they exist.

16.10. Adapter lifecycle and trace semantics are defined by DA-27.


===============================================================================
17. RELATION TO VIEWS AND FRONTEND RUNTIME
===============================================================================

17.1. View and frontend runtime semantics are defined by DA-22.

17.2. Browser JavaScript pack code runs in a frontend adapter environment.

17.3. Browser state is not authoritative AppInstance state.

17.4. Browser reload, tab close, client disconnect, or frontend state loss
      does not redefine committed state.

17.5. Frontend adapters may hold:
      - render state
      - temporary UI state
      - input fields
      - scroll position
      - local cache
      - pending view interaction state

17.6. Frontend adapters do not own authoritative AppInstance state.

17.7. Frontend adapter participation remains constrained by DA-22, DA-23,
      DA-27, DA-21, IS-01, and IS-02.


===============================================================================
18. RELATION TO DA-27
===============================================================================

18.1. DA-23 owns the abstract execution adapter authority boundary.

18.2. DA-23 owns:
      - code artifact phases
      - declared executable artifact authority meaning
      - pack-provided code versus backend implementation code
      - PackContext authority
      - RPC-shaped execution boundary semantics
      - relation between execution boundary and Kernel authority

18.3. DA-27 owns:
      - adapter mediation versus sandboxing
      - adapter trust and isolation
      - initial adapter classes
      - adapter lifecycle
      - adapter failure and disconnect meaning
      - adapter-domain trace reason identifiers
      - adapter trace context
      - runtime-host and local-first deployment interpretation

18.4. DA-27 does not weaken DA-23 authority boundaries.

18.5. DA-23 does not define concrete adapter lifecycle state machines.

18.6. DA-23 does not define concrete adapter trust or isolation taxonomy.

18.7. If DA-23 and DA-27 appear to conflict, the interpretation that
      preserves Kernel authority, capability mediation, and the RPC-shaped
      execution boundary is used until the documents are corrected.


===============================================================================
19. DIRECT-CALL OPTIMIZATION BOUNDARY
===============================================================================

19.1. Direct in-process calls may implement transport or dispatch
      optimization.

19.2. Direct in-process calls do not bypass the RPC-shaped authority
      boundary.

19.3. A direct call remains valid only when it preserves:
      - route or capability target identity
      - permission checks
      - lifecycle ownership
      - trace correlation
      - structured result and error handling
      - Orchestration Unit boundaries when mutation or lifecycle authority
        is involved
      - transaction and commit boundaries when state mutation occurs
      - AppInstance and SavePack state authority

19.4. A direct call that bypasses those requirements is backend
      implementation behavior, not pack-code authority.

19.5. Pack-provided code does not acquire direct-call authority merely
      because a direct-call implementation path exists.

19.6. Direct-call optimization must remain explainable and trace-visible
      where the equivalent message-shaped path would be trace-visible.


===============================================================================
20. WHAT IS EXPLICITLY NOT SUPPORTED
===============================================================================

20.1. The following are not supported:
      - executing code because a file exists
      - treating recognized code artifacts as executable by default
      - treating pack-provided Python as backend implementation code
      - treating same-language execution as authority
      - treating same-process execution as authority
      - treating direct calls as authority bypass
      - treating transport choice as authority
      - treating WebSocket as the RPC model itself
      - treating browser state as authoritative AppInstance state
      - treating adapter identity as capability permission
      - treating adapter execution authorization as capability permission
      - treating pack code as a SavePack writer
      - treating PackContext as general backend access
      - bypassing tracing, lifecycle, permission, policy, capability,
        Orchestration Unit, transaction, or commit boundaries through native
        language convenience

22. LOCKED SUMMARY
===============================================================================

22.1. LOCKED:
      - Kernel owns authority
      - execution adapters execute declared pack code
      - transports carry messages
      - RPC defines the logical communication boundary
      - capabilities define what adapter-executed code may do
      - SavePack stores durable executed state under Kernel authority
      - host language does not grant backend authority
      - pack-provided Python is not backend implementation code
      - recognizing a file as code is not permission to execute it
      - executable artifacts are declared explicitly
      - declaration does not grant capability permission or state mutation
        authority
      - PackContext exposes scoped participation, not general backend access
      - adapter-executed code communicates with Kernel authority through an
        RPC-shaped boundary
      - direct in-process calls may implement a transport optimization but
        do not bypass capability routing, permission checks, lifecycle,
        trace correlation, Orchestration Unit boundaries, transaction
        boundaries, or AppInstance state authority
      - transport choice does not define authority
      - adapter-executed code does not own AppInstance or SavePack state
      - browser state is not authoritative AppInstance state
      - native language imports cannot bypass Turnix semantics
      - DA-27 owns adapter lifecycle, trust, isolation, adapter-domain trace
        reason ownership, adapter classes, and runtime-host deployment
        interpretation

22.2. OPEN:
      - exact executable artifact declaration schema
      - exact adapter binding schema
      - exact PackContext API shape
      - exact language-specific runtime implementation details
      - exact direct-call optimization implementation shape
