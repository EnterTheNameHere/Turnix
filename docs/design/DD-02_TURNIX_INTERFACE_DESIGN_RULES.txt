===============================================================================
DESIGN DOCTRINE DD-02 - TURNIX INTERFACE DESIGN RULES
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines interface design rules for Turnix APIs, boundaries, and helper
usage.

Purpose:
--------
This document defines how Turnix interfaces communicate meaning, reduce
accidental misuse, and preserve platform integrity without unnecessarily
restricting creators.

It defines:
- interface responsibility
- misuse resistance
- execution boundaries
- surfaced failure structure
- semantic naming expectations
- canonical helper reuse

It does NOT define:
- specific pack APIs
- specific capability schemas
- frontend protocol details
- exact error taxonomies
- tracing data structures

Upstream Documents:
------------------
- CMR-01
- CMR-02
- CMR-03
- CMR-04
- CMR-05


===============================================================================
01. INTERFACE RESPONSIBILITY
===============================================================================

01.1. If a Turnix interface is likely to be used incorrectly by a competent
      creator, the interface design requires review.

01.2. Turnix interface design makes the intended path explicit.

01.3. Turnix interface design makes common failure modes visible.

01.4. Turnix interface design reduces accidental misuse where doing so does
      not compromise creator freedom.

01.5. Turnix does not assume that documentation alone compensates for a
      weak interface.

01.6. Documentation may explain constraints, but when misuse is probable
      and detectable, interface design and runtime structure both matter.


===============================================================================
02. ADULT INTERFACE MODEL
===============================================================================

02.1. Incorrect use is not automatically forbidden.

02.2. Turnix interfaces do not attempt to block every incorrect action by
      default.

02.3. Creators remain free to do unusual, risky, or unconventional things.

02.4. Turnix interface design focuses on:
      - reducing accidental misuse where practical
      - detecting important misuse where practical
      - preserving platform integrity always

02.5. This model assumes competent adult creators and does not treat
      default prohibition as the starting point.


===============================================================================
03. DOCUMENTED FAILURE MODES AS DETECTION OPPORTUNITIES
===============================================================================

03.1. When documentation states that a certain failure mode exists and must
      be handled, that statement creates a potential runtime-detection
      opportunity.

03.2. Example:
      - a mod calls a capability
      - capability permission is rejected
      - the mod does not handle the failure
      - the error escapes the mod boundary

03.3. If Turnix can identify that pattern reliably, Turnix may record it,
      report it, or surface it in post-run diagnostics.

03.4. Turnix does not introduce broad detection logic from vague hints.

03.5. Detection logic enters the platform only when the pattern is clear,
      the signal is reliable, and the handling is worth the added
      complexity.


===============================================================================
04. EXECUTION CONTAINMENT BOUNDARIES
===============================================================================

04.1. Turnix defines containment boundaries for execution.

04.2. Typical containment boundaries include:
      - mod execution boundaries
      - capability execution boundaries
      - pipeline step boundaries
      - pack lifecycle boundaries

04.3. A boundary either contains failure correctly or it does not.

04.4. If failure crosses a boundary that Turnix defined as containing that
      class of failure, Turnix has a boundary defect.

04.5. Boundary behavior is part of interface design, not merely an
      implementation detail.


===============================================================================
05. FAILURE TRANSLATION
===============================================================================

05.1. Raw failure at a boundary is translated into a structured runtime
      outcome.

05.2. Structured outcomes support:
      - tracing
      - diagnosis
      - creator feedback
      - safe continuation decisions
      - platform integrity review

05.3. A raw exception escaping without translation at a Turnix-defined
      containment edge indicates a likely containment defect.

05.4. Translation does not erase the original cause. Translation preserves
      the cause in a form suitable for runtime handling.


05.1 SURFACED FAILURE STRUCTURE
-------------------------------

05.1.1. A surfaced Turnix failure is not presented as a bare message alone.

05.1.2. Surfaced failures preserve structured meaning sufficient for:
        - diagnosis
        - creator feedback
        - safe continuation decisions
        - platform integrity review

05.1.3. A surfaced failure includes or remains linkable to:
        - error code
        - human-readable message
        - reason
        - trace correlation evidence when available

05.1.4. Failure translation does not erase traceability.

05.1.5. When trace evidence is unavailable, incomplete, pruned, dropped,
        corrupt, or not yet initialized, that limitation remains explicit.

05.1.6. Interface surfaces do not fabricate explanation evidence merely to
        present a cleaner error.


===============================================================================
06. EXPLICIT SEMANTIC NAMING
===============================================================================

06.1. Turnix code favors explicit semantic naming.

06.2. Names communicate meaning at the point of use.

06.3. Short names that rely on easy local deduction are not preferred
      merely because they are conventional.

06.4. Examples of preferred semantic naming include names such as:
      - traceRecordId
      - sessionId
      - indexElement
      - wasErrorRaised

06.5. Semantic explicitness reduces ambiguity for both humans and language
      model assistants.

06.6. Incidental implementation detail does not belong in a name unless
      that detail matters semantically.

06.7. Type-encoding style inherited from older ecosystems is not a default
      Turnix naming strategy.

06.8. Domain meaning matters more than decorative type noise.


===============================================================================
07. CANONICAL HELPERS AND CONTRACT REUSE
===============================================================================

07.1. When a stable semantic check or contract already has a canonical
      helper, code uses that helper instead of re-expressing the same rule
      inline.

07.2. Inline duplication of stable logic increases drift risk.

07.3. Canonical helpers centralize contract meaning.

07.4. Canonical helpers make later tightening, auditing, and review easier.

07.5. A helper is introduced when it strengthens clarity, stability, or
      contract reuse without imposing more cost than the problem it
      solves.

07.6. Helper introduction is not automatic. It remains a design decision.


===============================================================================
08. INTERFACE CLARITY OVER FALSE TIGHTNESS
===============================================================================

08.1. Turnix does not treat tighter code as an automatic virtue.

08.2. Interface clarity, stable meaning, and reviewability matter more
      than local cleverness.

08.3. A design that is less compact but much harder to misuse may be the
      better Turnix design.

08.4. Post-hoc tightening that changes scope, weakens clarity, or starts a
      second refactor cycle is not treated as an automatic improvement.


===============================================================================
09. SUMMARY
===============================================================================

09.1. Turnix interface design aims for explicit meaning, accidental-misuse
      resistance, creator freedom, and strong containment boundaries.

09.2. Interfaces are judged not only by elegance, but by whether they help
      preserve platform truth and reduce unnecessary ambiguity.

09.3. Surfaced failures remain structured, explainable, and linked to
      trace correlation evidence when such evidence is available.
