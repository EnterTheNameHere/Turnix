===============================================================================
DESIGN AREA DA-17 - USER INTERACTION, DECISION POINTS & POLICY HANDLING
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines how Turnix involves the user in decisions, when execution is allowed
to pause, how decisions are persisted, and how policy reduces repeated
friction without hiding uncertainty.

Purpose:
--------
This document defines the Turnix decision-handling model for situations where
the system cannot proceed safely or deterministically on its own.

It defines:
- user interaction as a kernel concern
- decision-point types
- hard and soft decision handling
- scoped pause-and-resume behavior
- policy handling and policy persistence
- run, save, and global policy scope precedence
- remembered permission-policy key semantics
- remembered fingerprint-mismatch-policy key semantics
- trace-backed decision explanation
- frontend responsibility at decision time
- failure behavior while a decision is pending

It does NOT define:
- generic resolution mechanics (see DA-11)
- config storage file placement (see DA-06)
- importing and module-loading rules (see DA-12)
- activation lifecycle or execution guarantees (see DA-14)
- capability schemas and permission-class definitions (see DA-16)
- tracing schema or storage details (see DA-03)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines user interaction, decision points, and policy
      handling within that shared interpretation model.


===============================================================================
02. CORE INTENT
===============================================================================

02.1. Turnix does not make ambiguous decisions silently.

02.2. Turnix does not delegate authority implicitly.

02.3. Once a decision is made, later behavior remains deterministic within
      the scope governed by that decision.

02.4. User interaction exists only to resolve situations where Turnix cannot
      proceed safely or deterministically on its own.

02.5. These situations are limited to:
      - ambiguity
      - permission
      - conflict
      - recovery
      - suspicious content divergence

02.6. If a decision affects identity, persistence, authority, safety, or
      known content equivalence, the decision is explicit and trace-backed.

02.7. Decision handling reduces uncertainty by recording an explicit user or
      policy outcome. It does not prove authenticity, trust, or correctness.


===============================================================================
03. USER INTERACTION IS A SYSTEM CONCERN (LOCKED)
===============================================================================

03.1. User interaction is not:
      - UI-specific
      - mod-specific
      - frontend-specific

03.2. User interaction is:
      - kernel-mediated
      - policy-driven
      - serializable
      - trace-backed

03.3. All decision points originate in the kernel.

03.4. Mods and packs may trigger decision points.

03.5. Mods and packs do not own, implement, or bypass decision handling.


===============================================================================
04. DECISION POINT TYPES (LOCKED)
===============================================================================

04.1. Turnix recognizes the following decision-point types:
      - Resolution decisions
      - Permission decisions
      - Recovery decisions
      - Configuration-scope decisions
      - Fingerprint-mismatch decisions


04.1 RESOLUTION DECISIONS (LOCKED)
----------------------------------

04.1.1. Resolution decisions occur when Turnix cannot deterministically
        select one safe option.

04.1.2. Examples include:
        - multiple eligible packs
        - semantic-version mismatch override
        - pinned pack missing
        - last-known-good fallback
        - conflicting pack requests

04.1.3. Resolution-decision outcomes may include:
        - chosen pack or version
        - persisted resolution rule when applicable
        - trace explaining why the selected option was chosen


04.2 PERMISSION DECISIONS (LOCKED)
----------------------------------

04.2.1. Permission decisions occur when an action would cross an authority
        or trust boundary.

04.2.2. Examples include:
        - external network access
        - filesystem writes through IoManager
        - mod installation
        - invoking a hard-classified capability

04.2.3. Permission decisions apply only to actions that:
        - mutate persisted state
        - delete persisted state
        - intentionally cross ownership boundaries

04.2.4. Permission decisions are not raised for:
        - computed-only effects
        - runtime-only transforms
        - observation or watching of values

04.2.5. Read access is excluded from permission decisions:
        - reads either succeed or return NotFoundOrPermissionDenied
        - user interaction is never used for reads
        - prompting, confirmation, and policy do not apply to reads

04.2.6. This preserves determinism, prevents probing, and avoids decision
        spam.

04.2.7. Permission-decision outcomes may include:
        - allow
        - deny
        - for this run
        - remember

04.2.8. For capability-based permission decisions, the remembered
        permission identity uses the exact versioned capability identifier.

04.2.9. A capability permission for storage.export@1 is remembered against
        storage.export@1, not against storage.export, storage, or a wildcard
        family.

04.2.10. This document does not introduce wildcard, prefix, or versionless
         fallback lookup for capability permission policies.


04.3 RECOVERY DECISIONS (LOCKED)
--------------------------------

04.3.1. Recovery decisions occur when persisted state cannot be used
        safely.

04.3.2. Examples include:
        - corrupted config file
        - unreadable SavePack
        - failed migration
        - checksum mismatch
        - missing dependency for a SavePack

04.3.3. Recovery-decision outcomes may include:
        - restore from backup
        - reset to defaults
        - quarantine or rename broken data
        - abort startup of the affected unit

04.3.4. Recovery decisions reference:
        - last-known-good state
        - available backups
        - scope of impact such as engine, pack, or save


04.4 CONFIGURATION-SCOPE DECISIONS (LOCKED)
-------------------------------------------

04.4.1. Configuration-scope decisions occur when persistence intent is
        ambiguous rather than unsafe.

04.4.2. These decisions answer:
        "Where should this value live, if it is persisted at all?"

04.4.3. Examples include:
        - modifying a value that exists in both global and save scope
        - promoting a save-local value to pack-global or engine scope
        - deleting a value that shadows defaults
        - attempting to persist a value produced only by computation

04.4.4. Configuration-scope decisions exclude:
        - computed-only values
        - derived runtime effects
        - accessibility or reactive transforms

04.4.5. Configuration-scope-decision outcomes may include:
        - chosen scope such as engine, pack-global, or save
        - persisted rule when applicable
        - trace linking the decision to the key and scope


04.5 FINGERPRINT-MISMATCH DECISIONS (LOCKED)
--------------------------------------------

04.5.1. Fingerprint-mismatch decisions occur when Turnix has an expected
        pack fingerprint and the computed fingerprint differs.

04.5.2. Fingerprint mismatch means content divergence from an expected pack
        representation. It does not prove malicious action or authenticity
        failure.

04.5.3. Fingerprint-mismatch decision triggers are:
        - same canonicalPackId and same packVersion with different
          fingerprints
        - SavePack expected fingerprint mismatch
        - manifest-declared dependency fingerprint mismatch

04.5.4. Manifest-declared dependency fingerprint mismatch applies only when
        the manifest actually declares an expected fingerprint.

04.5.5. Origin-only collision is not a fingerprint-mismatch decision trigger.

04.5.6. A first-party-looking source, repo source, third-party source,
        custom source, save-local source, or externally-managed content
        source does not by itself prove which candidate is authentic.

04.5.7. Candidate origin may be shown as context to the user, but origin
        alone is not treated as proof.

04.5.8. Fingerprint-mismatch-decision outcomes may include:
        - allow
        - deny
        - for this run
        - remember

04.5.9. Allowing for this run does not update the stored expected
        fingerprint.

04.5.10. Remembering a fingerprint-mismatch decision records acceptance of
         the specific mismatch under the selected policy scope.


===============================================================================
05. HARD AND SOFT DECISIONS (LOCKED)
===============================================================================

05.1. Turnix distinguishes between soft decisions and hard decisions.


05.1 SOFT DECISIONS (LOCKED)
----------------------------

05.1.1. Soft-decision characteristics:
        - execution may continue if denied
        - denial does not break identity
        - policy may auto-allow where permitted

05.1.2. Examples include:
        - version mismatch warnings
        - optional capability usage
        - non-critical override refusals


05.2 HARD DECISIONS (LOCKED)
----------------------------

05.2.1. Hard-decision characteristics:
        - execution pauses
        - denial blocks progress
        - user response is mandatory unless an accepted policy already
          supplies the outcome

05.2.2. Examples include:
        - replacing a pinned pack
        - destructive migration
        - restoring from backup
        - persisting a change that alters identity

05.2.3. Hard decisions always:
        - generate trace events
        - require explicit acknowledgement of the effective outcome

05.2.4. Fingerprint-mismatch decisions may be hard or soft depending on the
        surrounding consumer context and impact. SavePack reproducibility or
        exact dependency expectations may require hard handling.


===============================================================================
06. PAUSING MODEL (LOCKED)
===============================================================================

06.1. When a decision is required:
      - the triggering execution unit pauses
      - unrelated execution may continue
      - frontends are notified through a kernel-mediated event
      - execution resumes only after decision handling completes

06.2. Rules:
      - no busy waiting occurs
      - no polling occurs
      - no global freeze occurs

06.3. Pause remains scoped and trace-linked.

06.4. While a decision is pending:
      - no persisted mutation may be committed
      - no configuration write may be finalized
      - no SavePack mutation may complete

06.5. Computed or runtime-only effects may continue if they do not persist
      state and do not cross the paused authority boundary.


===============================================================================
07. POLICY SYSTEM (LOCKED)
===============================================================================

07.1. Policies define how Turnix behaves when a decision arises.

07.2. Policy sources include:
      - global defaults
      - user-defined overrides
      - pack-provided hints that remain non-binding

07.3. Policies may express:
      - ask always
      - allow for this run
      - deny for this run
      - allow and remember
      - deny and remember

07.4. Policies influence default responses.

07.5. Policies do not redefine authority.

07.6. Policies do not:
      - override ownership rules
      - bypass access levels
      - grant mutation rights forbidden by system rules

07.7. Policy exists to reduce friction rather than to weaken safety or
      determinism.

07.8. Policy evaluation order is:
      1) run policy
      2) save policy
      3) global policy

07.9. Run policy overrides save policy.

07.10. Save policy overrides global policy.

07.11. Run policy is memory-only and ends with the current Turnix run.

07.12. Save policy is persisted inside the save-bound policy storage defined
       by DA-06.

07.13. Global policy is persisted inside the global policy storage defined
       by DA-06.


===============================================================================
08. POLICY PERSISTENCE (LOCKED)
===============================================================================

08.1. When a user makes a remembered decision:
      - it is persisted centrally in the selected scope
      - future matching decisions become deterministic within that scope
      - trace links the decision to its origin

08.2. Persisted policies are:
      - explicit
      - inspectable
      - editable through Turnix management surfaces

08.3. Silent or implicit policy creation is forbidden.

08.4. Run-scope policy is not persisted.

08.5. Save-scope and global-scope policy storage locations are defined by
      DA-06.

08.6. Persisted policy files contain distinct remembered-decision classes
      for permission policies and fingerprint policies.

08.7. Exact JSON record layout inside policy files is defined outside this
      document and remains storage-format detail unless another document
      locks it.


===============================================================================
09. PERMISSION POLICY KEYS (LOCKED)
===============================================================================

09.1. Remembered general permission policy is keyed by:
      - packCanonicalId
      - exact versioned capability id

09.2. The exact versioned capability id is the capability id that caused the
      permission decision.

09.3. Versionless capability family lookup is not used for remembered
      permission policy.

09.4. Wildcard capability matching is not used for remembered permission
      policy.

09.5. Broader prefix lookup is not used for remembered permission policy.

09.6. Permission policy records store comparison metadata, including:
      - tierMode
      - acceptedAtPackVersion
      - remembered outcome

09.7. tierMode controls when the user is asked again for pack-version
      changes.

09.8. Pedantic tier asks again when the pack major version changes.

09.9. Second tier asks again when the pack minor version changes.

09.10. Normal tier asks again only when no remembered decision applies.

09.11. The pack version is comparison metadata. It is not part of the
       permission policy identity key.


===============================================================================
10. FINGERPRINT POLICY KEYS (LOCKED)
===============================================================================

10.1. Remembered fingerprint-mismatch policy is keyed by:
      - canonicalPackId
      - packVersion
      - expectedFingerprint
      - actualFingerprint

10.2. expectedFingerprint and actualFingerprint are full fingerprint
      strings.

10.3. The fingerprint algorithm/version is part of the full fingerprint
      string rather than a separate policy-key field.

10.4. Fingerprint-mismatch policy is specific to the exact mismatch pair.

10.5. Remembering one fingerprint mismatch does not bless all future
      mismatches for the same pack or version.

10.6. For-this-run fingerprint acceptance does not update persisted policy
      and does not update an expected fingerprint.

10.7. Remembered fingerprint acceptance may update the applicable policy
      record for the selected scope.

10.8. SavePack fingerprint mismatch defaults to save-scope persistence when
      the user chooses remember, unless a higher explicit policy says
      otherwise.

10.9. General permission decisions default to global-scope persistence when
      the user chooses remember, unless a higher explicit policy says
      otherwise.


===============================================================================
11. TRACE INTEGRATION (LOCKED)
===============================================================================

11.1. Every decision point emits trace events including:
      - decision type
      - triggering cause
      - affected scope such as engine, pack, or save
      - available options
      - selected outcome
      - policy involvement when applicable

11.2. Fingerprint-mismatch decisions additionally trace:
      - canonicalPackId
      - packVersion
      - expectedFingerprint
      - actualFingerprint
      - candidate origin context when available

11.3. Permission decisions additionally trace:
      - packCanonicalId
      - exact versioned capability id when capability-based
      - tierMode when policy participates
      - acceptedAtPackVersion when a remembered policy participates

11.4. User-facing explanations are derived from trace data.

11.5. User-facing surfaces do not invent rationale.

11.6. For denied mutation attempts, trace additionally records:
      - owner domain
      - caller domain
      - access level
      - authority failure reason

11.7. This applies even when no user interaction occurs.


===============================================================================
12. FRONTEND RESPONSIBILITY (LOCKED)
===============================================================================

12.1. Frontends:
      - present clear explanation
      - show consequences
      - indicate scope of effect
      - offer only valid choices

12.2. Frontends do not:
      - invent defaults
      - auto-confirm without policy
      - bypass kernel authority
      - obscure the impact of denial

12.3. For fingerprint-mismatch decisions, frontends may show origin context,
      but must not present origin as authenticity proof.


===============================================================================
13. MULTIPLE FRONTENDS (LOCKED)
===============================================================================

13.1. If multiple frontends are connected:
      - the decision is shown on all connected frontends
      - the first valid response wins
      - all other pending presentations are dismissed

13.2. The kernel remains the single arbiter.


===============================================================================
14. FAILURE MODES (LOCKED)
===============================================================================

14.1. If a frontend disconnects during a decision:
      - the affected execution remains paused
      - timeout policy may apply
      - the user may reconnect through another valid frontend

14.2. If the decision handler fails:
      - execution aborts safely
      - failure is trace-logged
      - persisted state remains unchanged


===============================================================================
15. MOD INVOLVEMENT (LOCKED)
===============================================================================

15.1. Mods may trigger decision points.

15.2. Mods do not handle decision points directly.

15.3. Mods do not block while awaiting frontend interaction.

15.4. Mods observe decision outcomes only through Turnix-provided context
      and results.


===============================================================================
16. RELATION TO DEBUGGING (LOCKED)
===============================================================================

16.1. Decision points:
      - are trace-visible
      - may trigger breakpoints
      - remain inspectable after the fact

16.2. Debuggers may simulate decisions only in development mode.


===============================================================================
17. WHAT IS EXPLICITLY NOT SUPPORTED
===============================================================================

17.1. The following are not supported:
      - silent auto-approval without policy
      - console-based prompting
      - blocking the entire system for frontend interaction
      - mod-defined decision UIs
      - persistence of computed-only values
      - prompting the user to allow read access to restricted state
      - treating origin alone as authenticity proof
      - wildcard capability permission matching
      - versionless capability permission fallback


===============================================================================
18. EXPLICITLY OPEN QUESTIONS
===============================================================================

18.1. OPEN questions include:
      - bulk decision-management user interfaces
      - policy import and export
      - scripted policy rules

18.2. Cross-scope copy and promotion are accepted conceptually, but the
      concrete API and user-interaction model remain OPEN.

18.3. OPEN cross-scope copy and promotion questions include:
      - exact API shape
      - whether copy and promotion are direct API calls, decision-driven
        flows, or both
      - naming of copy, promote, clone, move, or related operations
      - overwrite and conflict behavior when the target scope already
        contains a value
      - whether the operation is exact value copy, structural subtree copy,
        or re-materialization through normal config materialization rules
      - whether subtree copy or promotion is supported
      - whether metadata, revision markers, migration markers, or version
        fields are copied, recomputed, or omitted
      - whether copy and promotion may occur between all scope pairs or only
        some scope pairs
      - whether owner-domain boundaries impose additional permission checks
      - trace shape for copy, promotion, overwrite, refusal, discard, and
        conflict outcomes
      - interaction with existing higher-layer or lower-layer overrides
      - whether promoting a value also removes or preserves the source-layer
        value
      - whether copy and promotion may be batched
      - whether copy and promotion may be policy-remembered
      - how user-facing wording distinguishes:
        - keep local only
        - promote to broader scope
        - copy without removing source
        - move semantics if move is ever allowed
      - how computed-only or derived values are rejected cleanly in these
        flows
      - failure behavior when part of the requested change cannot be applied
      - whether the operation is atomic when multiple keys or subtrees are
        involved


===============================================================================
19. LOCKED SUMMARY
===============================================================================

19.1. LOCKED:
      - decision authority is kernel-mediated
      - decision types remain distinct
      - configuration-scope resolution is explicit
      - fingerprint-mismatch decision handling is explicit and trace-backed
      - origin alone is not authenticity proof
      - execution pauses and resumes in a scoped way
      - policy is persistent and inspectable where persisted
      - run policy overrides save policy and save policy overrides global
        policy
      - permission policy keys use packCanonicalId and exact versioned
        capability id
      - fingerprint policy keys use canonicalPackId, packVersion,
        expectedFingerprint, and actualFingerprint
      - explanations are derived only from trace-backed rationale

19.2. OPEN:
      - policy automation
      - user-experience refinement only
      - detailed cross-scope copy and promotion model
