===============================================================================
DESIGN AREA DA-21 - RPC, CHANNELS, TRANSPORTS & MESSAGE FLOW
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines Turnix RPC as a transport-independent communication boundary, defines
RPC channels and transports, and separates logical message flow from concrete
wire schemas such as JSON envelopes and WebSocket frames.

This document owns abstract RPC semantics. Concrete message schema and
concrete WebSocket flow remain implementation specifications.

Purpose:
--------
This document defines the Turnix model for RPC communication between runtime
participants, execution adapters, frontend surfaces, tools, and
kernel-mediated execution paths.

It defines:
- RPC as a logical communication boundary
- RPC message-flow categories
- RPCChannel as the transport-neutral communication interface
- transport as an interchangeable carrier of RPC messages
- peer identity and connection context
- request, emit, subscribe, cancel, reply, error, and acknowledgement
  meaning
- routing boundaries for object and capability messages
- relationship between RPC and execution adapters
- separation between RPC transport lifecycle and Job lifecycle
- relationship between RPC requests and Job-backed work
- transport timeout and disconnect semantics at the abstract level
- message correlation and trace obligations
- relation between DA-level RPC semantics and implementation specifications

It does NOT define:
- concrete JSON envelope fields (see IS-01)
- concrete WebSocket handshake and frame flow (see IS-02)
- trace event structure, trace DAG semantics, or span mechanics (see DA-03)
- capability permission semantics (see DA-16)
- user decision and remembered policy semantics (see DA-17)
- Job lifecycle, Job terminal outcomes, or Job result semantics (see DA-19)
- AppInstance lifecycle or runtime identity semantics (see DA-20)
- execution adapter boundary semantics (see DA-23)
- frontend UI layout or presentation behavior (see DA-22)
- RuntimeHost identity, workspace ownership, startup modes, entry surfaces, or
  command-surface semantics (see DA-28)

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines RPC, channel, transport, routing, and abstract
      message-flow semantics within that shared interpretation model.


===============================================================================
02. CORE CONCEPT
===============================================================================

02.1. RPC answers:

          "How do Turnix participants communicate across a boundary without
           confusing transport delivery, permission authority, execution
           lifecycle, and final work outcome?"

02.2. RPC is a logical communication boundary.

02.3. RPC is not a transport.

02.4. RPC is not WebSocket.

02.5. RPC is not JSON.

02.6. RPC is not capability authority by itself.

02.7. RPC is not adapter authority by itself.

02.8. RPC is not Job lifecycle by itself.

02.9. A transport carries RPC messages.

02.10. A channel provides the transport-neutral interface through which RPC
       messages are received and sent.

02.11. Message delivery, message acknowledgement, request acceptance,
       handler execution, Job creation, and Job terminal outcome are
       distinct events.

02.12. Turnix does not treat successful message delivery as successful work
       completion.

02.13. Turnix does not treat transport failure as Job failure.

02.14. Adapter-executed code may communicate through RPC-shaped boundaries
       under DA-23.

02.15. RPC carries adapter-originated or adapter-targeted messages without
       making the transport or channel an authority owner.

02.16. RuntimeHost entry surfaces and command surfaces may use RPC to
       communicate with Kernel-mediated runtime paths.

02.17. RPC use by a RuntimeHost entry surface does not make RPC the
       RuntimeHost authority owner.

02.18. RuntimeHost entry-surface, command-surface, startup-mode, workspace
       ownership, and host lifecycle semantics remain owned by DA-28.


===============================================================================
03. RPC, CHANNEL, AND TRANSPORT TERMS
===============================================================================

03.1. RPC means the logical request, reply, event, subscription,
      cancellation, and error flow between Turnix participants.

03.2. RPCChannel is the transport-neutral interface consumed by the RPC host
      or equivalent runtime owner.

03.3. Transport is the concrete carrier that moves bytes, text, frames,
      streams, or messages.

03.4. Example transports may include:
      - WebSocket
      - local in-process channel
      - testing channel
      - future inter-process communication transport
      - future stdio transport

03.5. A transport may disconnect.

03.6. A channel may become unavailable.

03.7. RPC semantics remain defined independently of any one transport.

03.8. Transport-specific framing, handshake, encoding, buffering, and
      reconnection rules belong to implementation specifications.

03.9. A transport adapter translates concrete transport events into
      RPCChannel events.

03.10. A transport adapter does not gain authority merely because it carries
       RPC messages.

03.11. A transport adapter must not smuggle transport-specific authority,
       fallback, or policy into RPC semantics.

03.12. A transport adapter is distinct from an Execution Adapter under
       DA-23, even when one implementation object performs both roles.

03.13. A transport adapter is distinct from a RuntimeHost entry surface under
       DA-28, even when one implementation object participates in both roles.


===============================================================================
04. RPC MESSAGE CATEGORIES
===============================================================================

04.1. RPC message categories include:
      - control messages
      - request messages
      - emit messages
      - reply messages
      - error messages
      - subscription messages
      - cancellation messages
      - state or update messages
      - acknowledgement messages

04.2. Control messages establish or maintain communication state.

04.3. Request messages ask for a correlated result, acceptance, or error.

04.4. Emit messages report or trigger one-way behavior where no final result
      is expected by the caller.

04.5. Reply messages carry successful immediate results or accepted-work
      responses according to operation contract.

04.6. Error messages carry structured failure information.

04.7. Subscription messages establish continuing delivery of updates.

04.8. Cancellation messages request cancellation or withdrawal of interest.

04.9. State or update messages carry state snapshots, state deltas, progress
      updates, or other subscribed update material according to operation
      contract.

04.10. Acknowledgement messages confirm message receipt or transport-level
       handling only.

04.11. Acknowledgement does not imply request success.

04.12. Acknowledgement does not imply capability execution.

04.13. Acknowledgement does not imply adapter execution.

04.14. Acknowledgement does not imply Job creation.

04.15. Acknowledgement does not imply Job success.

04.16. Concrete message type names and concrete field layout belong to
       IS-01 or another implementation specification.


===============================================================================
05. ROUTING MODEL
===============================================================================

05.1. RPC routing identifies the logical target of a message.

05.2. RPC routes may target:
      - objects
      - capabilities
      - adapter-facing endpoints
      - control endpoints
      - subscription endpoints

05.3. RPC routing does not grant permission.

05.4. RPC routing does not activate packs.

05.5. RPC routing does not select or authorize an execution adapter.

05.6. RPC routing does not bypass capability mediation.

05.7. A capability-routed RPC message remains subject to DA-16.

05.8. An adapter-facing RPC message remains subject to DA-23.

05.9. A RuntimeHost control-surface RPC message remains subject to DA-28.

05.10. RuntimeHost control-surface routing does not grant workspace ownership,
       command acceptance, AppInstance creation, or LAN exposure by itself.

05.11. If routing fails before capability execution, adapter execution,
       RuntimeHost command acceptance, or Job creation, the failure remains an
       RPC routing or dispatch failure rather than a Job terminal outcome.

05.12. If routing succeeds and a capability accepts Job-backed work, later
       Job lifecycle semantics are owned by DA-19.


===============================================================================
06. REQUEST, ACCEPTANCE, AND JOB-BACKED WORK
===============================================================================

06.1. An RPC request may end in:
      - immediate reply
      - structured error
      - accepted Job-backed work
      - accepted adapter-mediated work
      - accepted RuntimeHost operation
      - rejected-before-Job outcome

06.2. Request acceptance means the request crossed the RPC dispatch boundary
      and was accepted by the target operation.

06.3. Request acceptance is not final operation success unless the operation
      contract explicitly defines successful acceptance as the result.

06.4. A request accepted as Job-backed work creates or exposes Job lifecycle
      correlation.

06.5. A request accepted as adapter-mediated work may create or expose
      adapter lifecycle correlation.

06.6. A request accepted as RuntimeHost operation may create or expose
      runtime-host operation correlation.

06.7. Adapter lifecycle semantics remain owned by DA-23.

06.8. RuntimeHost operation, command-surface, and entry-surface semantics
      remain owned by DA-28.

06.9. Once Job-backed work exists, DA-19 owns Job lifecycle semantics.

06.10. A request rejected before Job-backed work exists does not produce a
       Job terminal outcome.

06.11. A request rejected before adapter-mediated work exists does not
       produce adapter execution failure unless DA-23 classifies that
       relationship explicitly.

06.12. A request rejected before RuntimeHost command acceptance does not
       produce RuntimeHost operation completion or failure unless DA-28
       classifies that relationship explicitly.

06.13. A request rejected before Job-backed work exists may still produce
       RPC trace evidence and a structured error envelope.

06.14. RPC request identity and jobId remain distinct.

06.15. RPC request identity and adapter instance identity remain distinct.

06.16. RPC request identity and runtimeHostId remain distinct.

06.17. RPC correlation may link request identity to jobId when a Job exists.

06.18. RPC correlation may link request identity to adapter identity when
       adapter-mediated work exists.

06.19. RPC correlation may link request identity to runtimeHostId and
       RuntimeHost operation identity when host-level work exists.

06.20. A request accepted as Job-backed work may return a Job handle,
       immediate accepted-work response, subscription relationship, or other
       operation-defined lifecycle surface.

06.21. The exact concrete envelope shape for Job-backed request acceptance
       remains owned by IS-01, IS-02, or a future implementation
       specification constrained by DA-19.

06.22. The exact concrete envelope shape for RuntimeHost-operation acceptance
       remains owned by IS-01, IS-02, or a future implementation
       specification constrained by DA-28.


===============================================================================
07. EMIT, SUBSCRIBE, AND UPDATE FLOW
===============================================================================

07.1. Emit flow represents one-way communication where the sender does not
      expect a final correlated result.

07.2. Emit flow may still be acknowledged at the transport or RPC handling
      level.

07.3. Emit acknowledgement does not imply that downstream work completed.

07.4. Subscribe flow establishes continuing delivery of updates until the
      subscription ends, fails, disconnects, or is cancelled.

07.5. Subscription delivery may carry state updates, event updates, Job
      progress, adapter lifecycle updates, RuntimeHost operation updates, or
      other operation-defined update material.

07.6. Subscription delivery does not by itself define Job lifecycle state.

07.7. Subscription delivery does not by itself define adapter lifecycle
      state unless it carries adapter-domain evidence owned by DA-23.

07.8. Subscription delivery does not by itself define RuntimeHost lifecycle or
      command outcome unless it carries RuntimeHost-domain evidence owned by
      DA-28.

07.9. If subscription delivery reports Job progress or terminal outcome,
      the reported Job meaning remains owned by DA-19.

07.10. If subscription delivery reports adapter lifecycle state, the reported
       adapter meaning remains owned by DA-23.

07.11. If subscription delivery reports RuntimeHost lifecycle or operation
       state, the reported RuntimeHost meaning remains owned by DA-28.

07.12. Unsubscribe withdraws interest in future subscription delivery.

07.13. Unsubscribe does not by itself cancel the underlying work unless the
       operation contract or policy explicitly connects unsubscribe to
       cancellation.


===============================================================================
08. TRANSPORT TIMEOUT, DISCONNECT, AND CANCELLATION
===============================================================================

08.1. Transport timeout is an RPC, channel, or transport concern.

08.2. Job timeout is a Job lifecycle concern.

08.3. Adapter timeout, if defined, is an adapter lifecycle concern.

08.4. RuntimeHost timeout, if defined, is a RuntimeHost lifecycle or operation
      concern under DA-28.

08.5. Transport timeout does not automatically transition a Job to TimedOut.

08.6. Transport timeout does not automatically fail an adapter instance.

08.7. Transport timeout does not automatically fail a RuntimeHost.

08.8. Job timeout does not mean the transport failed.

08.9. Adapter failure does not mean the transport failed.

08.10. RuntimeHost failure does not mean the transport failed.

08.11. Transport disconnect means the communication carrier became
       unavailable or closed.

08.12. Transport disconnect does not automatically cancel a Job.

08.13. Transport disconnect does not automatically supersede a Job.

08.14. Transport disconnect does not automatically discard a Job result.

08.15. Transport disconnect does not automatically close or fail an execution
       adapter.

08.16. Transport disconnect does not automatically stop, fail, or close a
       RuntimeHost.

08.17. If accepted or in-flight work exists when transport disconnect occurs,
       policy may decide whether work continues, cancels, supersedes,
       detaches, retains result, or discards result.

08.18. DA-17 owns decision and policy handling.

08.19. DA-19 owns Job terminal outcomes when a policy outcome affects a Job.

08.20. DA-23 owns adapter lifecycle outcomes when a policy outcome affects
       adapter execution.

08.21. DA-28 owns RuntimeHost lifecycle or host-operation outcomes when a
       policy outcome affects RuntimeHost behavior.

08.22. RPC cancellation messages are requests.

08.23. RPC cancellation messages do not directly kill arbitrary execution.

08.24. When cancellation targets Job-backed work, the cancellation request is
       translated into Job lifecycle handling under DA-19.

08.25. When cancellation targets adapter-mediated work, the cancellation
       request is translated into adapter and work lifecycle handling under
       DA-23 and owning workflow documents.

08.26. When cancellation targets a RuntimeHost operation, the cancellation
       request is translated into RuntimeHost operation handling under DA-28
       and owning workflow documents.

08.27. When cancellation targets a subscription, cancellation may withdraw
       delivery interest without ending the underlying work.

08.28. When cancellation targets transport-local pending work, cancellation
       may resolve or reject local waiting state without defining a Job,
       adapter, or RuntimeHost outcome.


===============================================================================
09. RPCCHANNEL CONTRACT
===============================================================================

09.1. RPCChannel is the abstract communication surface used by the RPC host
      or equivalent runtime owner.

09.2. RPCChannel exposes peer context.

09.3. RPCChannel exposes connection state.

09.4. RPCChannel receives incoming transport events.

09.5. RPCChannel sends outgoing RPC payloads through the active transport.

09.6. RPCChannel may report:
      - text payload
      - binary payload
      - disconnect
      - transport-specific event classified as other

09.7. RPCChannel does not define message schema.

09.8. RPCChannel does not define WebSocket behavior.

09.9. RPCChannel does not define Job semantics.

09.10. RPCChannel does not define adapter lifecycle semantics.

09.11. RPCChannel does not define RuntimeHost lifecycle or command-surface
       semantics.

09.12. Concrete channel implementations adapt concrete transports into this
       abstract interface.

09.13. A channel implementation preserves peer identity and connection
       context when the transport provides them.

09.14. A channel implementation reports loss of connection explicitly.

09.15. A channel implementation does not silently transform disconnect into
       work cancellation.

09.16. A channel implementation does not silently transform disconnect into
       adapter failure.

09.17. A channel implementation does not silently transform disconnect into
       RuntimeHost stop, failure, or command cancellation.


===============================================================================
10. TRACE AND OBSERVABILITY
===============================================================================

10.1. RPC message flow remains trace-visible where trace evidence exists.

10.2. RPC trace evidence distinguishes:
      - message receipt
      - message acknowledgement
      - routing or dispatch
      - rejection before accepted work exists
      - request accepted as Job-backed work
      - request accepted as adapter-mediated work
      - request accepted as RuntimeHost operation
      - reply sent
      - delivery failure
      - cancellation requested
      - transport timeout
      - generation or session mismatch
      - channel disconnect
      - disconnect policy application
      - incomplete or malformed envelope rejection

10.3. RPC trace evidence records enough context, when available, to explain:
      - message identity
      - correlation identity
      - route
      - lane or logical stream
      - peer identity
      - channel identity
      - transport kind
      - runtimeHostId when applicable
      - RuntimeHost operation identity when applicable
      - appInstanceId
      - adapter kind when applicable
      - adapter instance identity when applicable
      - capability identifier when applicable
      - jobId when applicable
      - timeout category when applicable
      - disconnect category when applicable

10.4. DA-03 defines the tracing substrate.

10.5. DA-16 owns capability-domain trace reasons.

10.6. DA-17 owns decision-domain trace reasons.

10.7. DA-19 owns Job-domain trace reasons.

10.8. DA-23 owns adapter-domain trace reasons.

10.9. DA-28 owns RuntimeHost-domain trace reasons.

10.10. IS-02 owns concrete WebSocket RPC transport trace reasons unless this
       document later promotes those reason identifiers into the abstract RPC
       domain.

10.11. RPC observability does not replace trace evidence owned by
       capability, decision, Job, debugger, AppInstance, adapter,
       RuntimeHost, or Window documents.

10.12. RPC observability preserves causal links between RPC messages and
       higher-domain events when those events exist.


===============================================================================
11. RELATION TO IMPLEMENTATION SPECIFICATIONS
===============================================================================

11.1. IS-01 defines the concrete RPC message envelope schema.

11.2. IS-02 defines the concrete WebSocket RPC flow.

11.3. IS documents may define:
       - concrete field names
       - JSON representation
       - WebSocket frame flow
       - handshake sequence
       - serialization checks
       - concrete transport reason identifiers
       - concrete adapter metadata fields when constrained by DA-23
       - concrete RuntimeHost metadata fields when constrained by DA-28

11.4. IS documents do not redefine:
       - RPC as a logical boundary
       - channel and transport separation
       - capability authority
       - adapter authority
       - RuntimeHost authority boundaries
       - Job lifecycle
       - decision and policy semantics

11.5. If an IS document needs a semantic rule not defined here, the DA is
       updated first or alongside the IS document.

11.6. A future non-WebSocket transport specification remains constrained by
       this document.

11.7. A future RPC schema revision remains constrained by this document.

11.8. A future adapter transport specification remains constrained by this
       document and DA-23.

11.9. A future RuntimeHost control-surface or LAN-client transport
      specification remains constrained by this document and DA-28.


===============================================================================
12. EXPLICITLY OPEN QUESTIONS
===============================================================================

12.1. OPEN questions include:
      - exact abstract RPC trace reason ownership versus IS-02 concrete
        transport reason ownership
      - non-WebSocket transport implementation specifications
      - whether subscription flow becomes its own design area or remains
        inside RPC
      - retained-result retrieval flow for Job-backed RPC
      - exact relationship between RPC request promises and Job handles
      - exact channel identity representation
      - exact peer identity representation beyond concrete transport context
      - exact adapter metadata representation in RPC envelopes
      - exact RuntimeHost metadata representation in RPC envelopes

12.2. RuntimeHost identity, entry-surface semantics, command-surface
      semantics, startup-mode semantics, workspace ownership, and
      RuntimeHost-domain trace reason ownership are defined by DA-28.


===============================================================================
13. LOCKED SUMMARY
===============================================================================

13.1. LOCKED:
      - RPC is a logical communication boundary, not a transport
      - transport carries RPC messages but does not define RPC semantics
      - RPCChannel is the transport-neutral interface
      - WebSocket is one concrete transport, not the RPC model itself
      - JSON envelope shape belongs to IS-01
      - concrete WebSocket flow belongs to IS-02
      - RPC is not adapter authority by itself
      - RPC is not RuntimeHost authority by itself
      - acknowledgement means transport or message handling, not work
        success
      - acknowledgement does not imply adapter execution
      - acknowledgement does not imply RuntimeHost operation completion
      - request acceptance is distinct from final result
      - request accepted as Job-backed work is distinct from Job success
      - request accepted as adapter-mediated work is distinct from adapter
        success
      - request accepted as RuntimeHost operation is distinct from
        RuntimeHost operation success
      - request rejected before Job-backed work exists has no Job terminal
        outcome
      - request rejected before adapter-mediated work exists has no adapter
        lifecycle outcome unless DA-23 defines that relation explicitly
      - request rejected before RuntimeHost command acceptance has no
        RuntimeHost operation outcome unless DA-28 defines that relation
        explicitly
      - emit acknowledgement does not imply downstream work completion
      - unsubscribe does not by itself cancel underlying work
      - transport timeout is not Job timeout
      - transport timeout is not adapter timeout
      - transport timeout is not RuntimeHost timeout
      - transport disconnect is not Job cancellation
      - transport disconnect is not adapter failure
      - transport disconnect is not RuntimeHost stop or failure
      - RPC cancellation is a request, not direct execution kill
      - capability-routed RPC remains capability-mediated under DA-16
      - adapter-facing RPC remains adapter-mediated under DA-23
      - RuntimeHost control-surface RPC remains constrained by DA-28
      - Job lifecycle remains owned by DA-19
      - adapter lifecycle remains owned by DA-23
      - RuntimeHost lifecycle and command-surface semantics remain owned by
        DA-28
      - policy handling remains owned by DA-17
      - RPC trace evidence preserves correlation across message, adapter,
        capability, RuntimeHost, AppInstance, and Job identity where available

13.2. OPEN:
      - trace reason ownership refinement
      - transport implementation expansion
      - subscription, Job-result retrieval, RuntimeHost metadata, and adapter
        metadata details
