===============================================================================
OPTIONAL DESIGN AREA ODA-B - DEVELOPMENT, DIAGNOSTICS & OBSERVABILITY
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines optional developer-facing diagnostics, reporting, and observability
surfaces that consume Turnix trace evidence and related system evidence.

This area improves trust, explainability, and maintainability without
introducing new runtime correctness semantics.

Purpose:
--------
This document defines optional but strongly useful development,
diagnostics, and observability expectations for Turnix tooling.

It defines:
- observability principles for developer experience
- tooling expectations for trace viewing and reporting
- activation and resolution explanation requirements as surfaced in tooling
- development-versus-production diagnostics behavior
- snapshot-tooling expectations for bug reports and regression testing

It does NOT define:
- trace event schemas, trace reasons, or trace retention semantics
  (see DA-03)
- debugger semantics, pause rules, or mutation safety (see DA-16)
- authoritative state, transactions, or window semantics (see DA-04)
- new runtime correctness guarantees or authority boundaries

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines optional development, diagnostics, and
      observability tooling expectations within that shared interpretation
      model.


===============================================================================
02. CORE PROBLEM
===============================================================================

02.1. Turnix is:
      - modular
      - dynamic
      - policy-driven
      - runtime-activated
      - partially user-governed

02.2. Without strong diagnostics, failures may become:
      - non-reproducible
      - misattributed
      - silently masked
      - explained away as "magic"

02.3. This area exists to improve:
      - explainability
      - inspectability
      - debuggability
      - auditability

02.4. This area remains optional in tier, but it is strategically useful for
      development and operational trust.


===============================================================================
03. OBSERVABILITY PRINCIPLE (LOCKED)
===============================================================================

03.1. Meaningful decisions remain explainable.

03.2. Every meaningful decision should be:
      - traceable
      - attributable
      - reproducible where the underlying evidence permits reproduction

03.3. If something happened, the system should be able to answer:
      - what happened
      - why it happened
      - who or what decided it
      - what alternatives existed when such alternatives were part of the
        decision context

03.4. ODA-B does not define the trace model itself.
      DA-03 defines the trace model.
      ODA-B defines how tooling should consume that evidence.


===============================================================================
04. TRACE SYSTEM FOUNDATION (LOCKED)
===============================================================================

04.1. Turnix uses the structured tracing system defined in DA-03.

04.2. This document does not define:
      - trace event schemas
      - trace reason vocabularies
      - trace span rules
      - trace retention, pruning, or loss-evidence rules

04.3. This document defines:
      - developer-facing tooling expectations
      - diagnostics workflows and reports
      - optional user-interface surfaces that consume trace data
      - development-mode enhancements that do not change runtime behavior


===============================================================================
05. TRACE DOMAINS (LOCKED)
===============================================================================

05.1. Important trace domains may include:
      - discovery
      - indexing
      - resolution
      - activation
      - permissions
      - capabilities
      - io
      - persistence
      - runtime errors

05.2. Trace domains are additive rather than exclusive.


===============================================================================
06. TRACE VISIBILITY (LOCKED)
===============================================================================

06.1. Trace events remain recordable even when no user interface is present.

06.2. Trace evidence remains queryable later through structured access.

06.3. Trace persistence remains policy-driven and best-effort under DA-03.

06.4. ODA-B tooling treats trace evidence as potentially incomplete.

06.5. If trace evidence is incomplete because of pruning, dropping,
      corruption, or partial export, tooling surfaces that incompleteness
      explicitly and does not guess.

06.6. Trace event levels and trace domains are defined by DA-03.

06.7. ODA-B tooling uses the DA-03 definitions rather than inventing new
      local ones.


===============================================================================
07. TRACE VIEWER (OPTIONAL UI)
===============================================================================

07.1. A trace-viewer surface may exist to:
      - inspect trace DAGs and spans
      - filter by domain, reason, and scope identifiers
      - collapse or expand causal branches
      - jump to related entities such as packs, links, memory, or Jobs

07.2. The trace-viewer user interface is optional.

07.3. The trace data model and reason vocabulary remain defined by DA-03.

07.4. Viewer rules:
      - the viewer does not fabricate reasons
      - the viewer does not fabricate causal links
      - if evidence incompleteness is declared under DA-03, the viewer
        surfaces that incompleteness explicitly


===============================================================================
08. PACK RESOLUTION EXPLANATIONS (LOCKED)
===============================================================================

08.1. When a pack is:
      - selected
      - rejected
      - overridden
      - downgraded

      the resolver trace remains sufficient, under DA-03 and DA-09, to
      explain:
      - candidate set
      - applied filters
      - scoring or ordering inputs
      - tie-breakers
      - final choice

08.2. This explanatory requirement is defined by DA-03 and DA-09 rather
      than by ODA-B.

08.3. ODA-B adds the following tooling expectations:
      - development tooling may provide "why not chosen" views when
        evidence is available
      - if available trace evidence is not sufficient to explain why a
        candidate was not chosen, tooling must say so explicitly and must
        not guess


===============================================================================
09. CAPABILITY USAGE REPORTING (LOCKED)
===============================================================================

09.1. In development mode, Turnix may produce capability-usage reports.

09.2. Reports may be produced per:
      - KernelRunSpan
      - AppInstanceActiveSpan
      - selected window
      - selected time range

09.3. Any such report is derived from trace evidence and registry state.

09.4. Reports distinguish:
      - not used
      from
      - not provable because of missing trace evidence


===============================================================================
10. ACTIVATION ORDER DIAGNOSTICS (LOCKED)
===============================================================================

10.1. Tooling should be able to explain:
      - activation order
      - why pack X ran before pack Y
      - why reordering was impossible
      - where user intervention occurred

10.2. These explanations remain trace-backed as defined by DA-03 and
      constrained by DA-12.

10.3. Reordering failures remain visible.

10.4. If evidence is incomplete, tooling surfaces that incompleteness
      explicitly.


===============================================================================
11. RUNTIME SAFETY WARNINGS (OPTIONAL)
===============================================================================

11.1. Optional diagnostics may warn about patterns such as:
      - background tasks outside Turnix orchestration
      - timers not owned by the kernel
      - untracked async loops
      - direct filesystem access attempts

11.2. These are warnings rather than correctness errors unless a higher
      document explicitly classifies the situation otherwise.


===============================================================================
12. DEVELOPMENT-MODE BEHAVIOR (LOCKED)
===============================================================================

12.1. In development mode:
      - warnings may be more verbose
      - unused declarations may be reported
      - trace verbosity may be higher
      - stricter diagnostics may be enabled

12.2. Development-mode enhancements do not change runtime correctness
      semantics.

12.2.1. Development-mode versus production-mode diagnostics do not weaken
        tracing requirements defined elsewhere.

12.2.2. If tracing is required by DA-03 or another authoritative document,
        that tracing remains required regardless of mode.

12.3. In production mode:
      - warnings are quieter
      - traces may still be recorded according to DA-03 policy
      - user-facing diagnostic noise is reduced


===============================================================================
13. ERROR, WARNING, AND INFO LEVELS (LOCKED)
===============================================================================

13.1. ODA-B tooling may classify surfaced diagnostics using the levels:
      - ERROR
      - WARNING
      - INFO

13.2. In ODA-B tooling:
      - ERROR means correctness was violated or a blocking failure occurred
      - WARNING means a policy, hygiene, or diagnostics concern exists
        without the same correctness implication
      - INFO means expected behavior or informational outcome

13.3. Correctness errors are not silently downgraded by tooling.


===============================================================================
14. STATE SNAPSHOTS (OPTIONAL)
===============================================================================

14.1. Optional tooling may snapshot:
      - pack registry
      - dependency graph
      - activation state
      - memory layers

14.2. Snapshot tooling may be used for:
      - bug reports
      - reproducibility work
      - regression testing

14.3. Snapshot tooling does not create new authority semantics by itself.


===============================================================================
15. REPLAY AND POST-MORTEM (OPEN)
===============================================================================

15.1. Future tooling may support:
      - replaying activation with the same inputs
      - inspecting divergence
      - comparing traces

15.2. This area remains OPEN and is not locked by this document.


===============================================================================
16. WHAT THIS AREA DOES NOT DO
===============================================================================

16.1. This area does not define:
      - automatic fixing
      - AI-driven debugging as a required feature
      - speculative corrections
      - hiding errors


===============================================================================
17. LOCKED SUMMARY
===============================================================================

17.1. LOCKED:
      - observability remains a first-class development concern
      - tooling consumes DA-03 trace evidence without redefining it
      - resolver explanations remain trace-backed
      - capability-usage reporting distinguishes absence from missing proof
      - activation diagnostics remain trace-backed
      - development and production diagnostics remain distinct

17.2. OPTIONAL OR OPEN:
      - trace-viewer UI
      - snapshots
      - replay tools
