===============================================================================
DESIGN AREA DA-02 - PLATFORM CORE, CONTRACT BOUNDARIES &
IMPLEMENTATION INDEPENDENCE
===============================================================================

Status:
-------
EVOLVING

Description:
------------
Defines the architectural relationship between Turnix platform truth,
platform core, adapters, interoperability envelopes, and replaceable
implementation layers.

Purpose:
--------
This document defines how Turnix remains one platform even when its current
implementation stack, host-language realizations, execution adapters,
transports, or first-party modular components change over time.

It defines:
- platform identity and architectural scope
- platform core and guarantee source
- contract boundaries
- first-party modularity
- adapter boundaries and host-language realization
- execution adapter authority boundaries
- interoperability envelope role
- implementation independence and replaceability

It does NOT define:
- concrete RPC schemas
- concrete transport protocols
- concrete frontend rendering rules
- concrete language adapter internals
- pack loading or activation mechanics in detail
- concrete execution adapter protocols

Upstream Documents:
------------------
- CMR-01
- CMR-02
- CMR-03
- CMR-04
- CMR-05

===============================================================================
01. INTERPRETATION BASIS
===============================================================================

01.1. This document expresses normative meaning through explicit,
      deterministic prose.

01.2. Normative meaning in Turnix derives from sentence structure and
      document context rather than from RFC-style keywords.

01.3. The CORE META-RULES documents define the general rule language and
      interpretation model used by this document.

01.4. This document defines architectural boundaries and implementation-
      independence rules within that shared interpretation model.


===============================================================================
02. PLATFORM IDENTITY AND ARCHITECTURAL SCOPE
===============================================================================

02.1. Turnix platform truth is defined by contracts, guarantees,
      orchestration semantics, and continuity behavior rather than by any
      specific implementation language, framework, or current runtime
      stack.

02.2. The current implementation stack is a realization of Turnix rather
      than the identity of Turnix.

02.3. Backend language, frontend technology, adapter strategy, transport,
      or current implementation vehicle do not by themselves define what
      Turnix is.

02.4. Turnix remains one platform even when different technical layers are
      implemented in different languages.

02.5. Architectural scope in this document includes:
      - platform core
      - contract boundaries
      - first-party modular components
      - host-language adapters
      - execution adapters
      - interoperability envelopes

02.6. Architectural scope in this document excludes concrete schema,
      adapter-protocol, and transport details. Those details belong in
      later DA or IS documents.

02.7. Runtime authority follows the project-level ordering defined by
      DA-01:
      - Kernel owns authority
      - adapters execute declared pack code
      - transports carry messages
      - RPC defines the logical communication boundary
      - capabilities define what code may do
      - SavePack stores durable executed state

02.8. This ordering is architectural and survives implementation
      replacement.


===============================================================================
03. PLATFORM CORE AND GUARANTEE SOURCE
===============================================================================

03.1. The Turnix platform core is the source of platform guarantee meaning.

03.2. The platform core defines:
      - orchestration semantics
      - boundary semantics
      - continuity expectations
      - contract meaning
      - guarantee meaning

03.3. The platform core does not need to be monolithic to remain the source
      of platform truth.

03.4. A component belongs to the platform core when its semantics are
      required to preserve platform truth rather than merely to realize it
      in one implementation environment.

03.5. Adapters, envelopes, transports, and host-language realizations do not
      redefine the meaning established by the platform core.

03.6. If platform guarantee meaning changes, the change is architectural
      rather than a mere implementation refactor.

03.7. The Kernel is the runtime authority owner within the runtime
      architecture defined by later design areas.

03.8. Kernel authority may be realized by replaceable implementation code,
      but the authority meaning is platform-semantic rather than accidental
      implementation privilege.


===============================================================================
04. CONTRACT BOUNDARIES
===============================================================================

04.1. Turnix uses contract boundaries to separate platform truth from
      implementation realization.

04.2. A contract boundary defines:
      - what a component may rely on
      - what a component must provide
      - what meanings remain stable across implementations
      - what outcomes are guaranteed or not guaranteed

04.3. Contract meaning remains primary. Internal realization remains
      secondary.

04.4. If an implementation change preserves contract meaning, platform
      truth remains preserved.

04.5. If an implementation change alters contract meaning, guarantee
      behavior, or boundary truth, the change is a platform-semantic
      change rather than a mere implementation substitution.

04.6. Contract boundaries are part of platform architecture. They are not
      decorative documentation.

04.7. Same-language or same-process implementation does not erase a
      contract boundary.

04.8. In-process calls may realize a boundary only when they preserve the
      same contract meaning that an explicit boundary would have provided.


===============================================================================
05. FIRST-PARTY MODULARITY
===============================================================================

05.1. Modularity in Turnix applies to creator-provided extensions and to
      first-party platform-realized functionality where modular treatment
      preserves platform truth and controllability.

05.2. First-party modularity may include areas such as:
      - UI
      - TTS
      - LLM-facing components
      - adapters
      - other platform-realized subsystems

05.3. Modularity does not remove the existence of a platform core.

05.4. A first-party component may be replaceable or modular while still
      remaining constrained by platform contracts and guarantees.

05.5. First-party modularity is acceptable only when contract truth,
      orchestration semantics, and boundary discipline remain preserved.

05.6. Replaceable first-party realization does not imply replaceable
      platform meaning.

05.7. A first-party module that executes pack code remains constrained by
      execution-boundary semantics rather than by implementation proximity.


===============================================================================
06. ADAPTER BOUNDARIES AND HOST-LANGUAGE REALIZATION
===============================================================================

06.1. A host-language adapter or realization layer expresses Turnix in a
      particular technical environment.

06.2. A host-language adapter preserves:
      - platform meaning
      - contract truth
      - boundary semantics
      - guarantee expectations

06.3. A host-language adapter may adapt syntax, calling style, or surface
      mechanics when that adaptation does not weaken platform truth.

06.4. A host-language adapter does not silently reinterpret platform
      contracts, continuity guarantees, permission meaning, tracing
      expectations, or orchestration boundaries.

06.5. If a host-language realization cannot preserve a required platform
      guarantee, the limitation is made explicit rather than hidden behind
      host-language convenience.

06.6. Host-language convenience does not take precedence over contract
      truth or boundary truth.

06.7. Creator-facing cross-language coherence is constrained further by
      Design Doctrine documents that define language-facing doctrine.

06.8. Same host language does not imply same authority.

06.9. Pack-provided code is not platform core implementation code merely
      because it is written in the same language as the platform core.

06.10. Execution adapter semantics are defined by DA-23.

06.11. Host-language adaptation does not bypass execution adapter,
       activation, capability, persistence, or AppInstance boundaries.


===============================================================================
07. INTEROPERABILITY ENVELOPE ROLE
===============================================================================

07.1. Turnix requires interoperability mechanisms for communication across
      execution domains, host-language boundaries, adapter boundaries, or
      architectural layers.

07.2. An interoperability envelope carries structured interaction across
      such boundaries.

07.3. The interoperability envelope exists to preserve:
      - boundary clarity
      - request and response meaning
      - tracing and observability
      - permission and capability semantics
      - continuity-safe interaction

07.4. The interoperability envelope is an architectural boundary rather
      than incidental transport plumbing.

07.5. RPC is the design-area realization of the logical communication
      boundary in the current runtime architecture.

07.6. Transport is the concrete carrier of messages and does not define the
      authority meaning of the envelope.

07.7. Concrete message schemas, transport protocols, and wire details are
      not defined in this document.

07.8. Later DA and IS documents may define those concrete details, but the
      architectural role of the envelope remains constrained by this
      design area.


===============================================================================
08. IMPLEMENTATION INDEPENDENCE AND REPLACEABILITY
===============================================================================

08.1. Current implementation choices are pragmatic and may change.

08.2. Turnix platform truth remains valid across implementation
      replacement when platform contracts and guarantees remain preserved.

08.3. Backend replacement does not change Turnix identity when platform
      contracts and guarantees remain preserved.

08.4. Frontend replacement does not change Turnix identity when creator-
      facing contract meaning and platform guarantees remain preserved.

08.5. Adapter replacement does not change Turnix identity when boundary
      semantics and contract truth remain preserved.

08.6. Transport replacement does not change Turnix identity when logical
      boundary semantics and contract truth remain preserved.

08.7. A replaceable implementation layer does not by itself possess
      authority to redefine platform meaning.

08.8. Implementation independence is not an aesthetic goal. It is a
      consequence of contract primacy and architectural truth.

08.9. If platform truth does not survive implementation replacement, the
      architecture relied too heavily on implementation accident.


===============================================================================
09. EXPLICIT EXCLUSIONS AND NON-GOALS
===============================================================================

09.1. This design area does not treat the current backend language as the
      identity of Turnix.

09.2. This design area does not treat the current frontend stack as the
      identity of Turnix.

09.3. This design area does not treat native host-language convention as
      superior to Turnix contracts or guarantees.

09.4. This design area does not treat interoperability envelopes as mere
      incidental wiring.

09.5. This design area does not treat modularity as permission to weaken
      platform guarantees or boundary discipline.

09.6. This design area does not define exact loader, adapter, protocol, or
      transport mechanics.

09.7. This design area does not treat FastAPI, WebSocket, browser
      JavaScript, Python, CLI, subprocesses, containers, or network
      topology as platform identity.

09.8. This design area does not treat direct calls as authority-valid merely
      because they are convenient.


===============================================================================
10. WHY THIS AREA EXISTS
===============================================================================

10.1. This area exists to prevent Turnix architecture from collapsing into
      current implementation choices, framework assumptions, or
      host-language accident.

10.2. Without this area, later specifications risk treating current
      technical realization as platform identity.

10.3. Without this area, later specifications also risk weakening:
      - contract truth
      - adapter accountability
      - interoperability boundary meaning
      - first-party modularity discipline
      - implementation independence

10.4. This document anchors the architectural distinction between platform
      truth and implementation realization.
