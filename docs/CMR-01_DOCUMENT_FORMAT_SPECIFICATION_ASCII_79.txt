===============================================================================
CORE META-RULES CMR-01 - DOCUMENT FORMAT SPECIFICATION (ASCII-79)
===============================================================================

Status:
-------
FROZEN

Description:
------------
Physical format for all authoritative documents.
Changes here have cascading impact and must be rare and explicit.

Purpose:
--------
This document defines the canonical plain-text format used by Turnix
specification documents.

It defines:
- maximum line length
- allowed characters
- main section headers
- subsection headers
- lists and indentation
- tables
- doc-meta block
- publication boundaries for chat

It does NOT define:
- authority tiers or authority decision semantics
- document title banner rules (see CMR-02)
- document change discipline (see CMR-03)
- document lifecycle rules

Those are defined by higher-level documentation rules.

Upstream Documents:
-------------------
[NOTE] Listed for retrieval hygiene. See CMR-02 08.1 and 08.5.3.

- CMR-04

===============================================================================
01. DOCUMENT RULE LANGUAGE
===============================================================================

01.1. Turnix specifications express rules using direct normative prose
      rather than RFC-style keyword markers.

01.2. Normative meaning derives from explicit actor-action-constraint
      statements and from document context.

01.3. A sentence is normative when it directly states requirements,
      prohibitions, guarantees, invariants, or allowed variation.

01.4. Statements describing permitted alternatives express implementation
      freedom only within the constraints explicitly defined by the
      specification.

01.5. Normative meaning does not depend on reserved RFC-style keywords.


===============================================================================
02. HARD LINE-LENGTH RULE
===============================================================================

02.1. All authored lines are limited to a maximum of 79 characters.

02.2. Authored lines never reach or exceed 80 characters.

02.3. doc-meta is tooling-managed and is not authored content.

02.3.1. "Not authored content" means the text is not produced by the
        drafting author and is instead owned and maintained by tooling.

02.3.2. doc-meta remains part of the document artifact when present and
        follows the structural rules defined in section 10.

02.3.3. In committed authoritative documents, humans do not manually edit
        doc-meta fields. Tooling maintains them.

02.4. The 79-character rule applies to all authored content, including:
      - prose
      - document banner and end marker
      - section headers and subsection headers
      - lists
      - tables
      - separators
      - examples

02.5. A limited exception exists for lines whose first non-space token is
      exactly one of the following:

      - "file:"
      - "DOC_FILE:"
      - "uri:"
      - "link:"

      Such lines may exceed 79 characters.

02.5.1. "DOC_FILE:" appears only inside doc-meta.

02.5.2. "file:", "uri:", and "link:" appear in authored content and
        examples and are reserved for literal paths, URIs, or Turnix
        link addresses embedded in text.

02.5.3. This exception exists only to keep long literal references on a
        single line and is not used for prose.

02.6. Authored content is all document text produced by the drafting author
      excluding the doc-meta block.

02.6.1. Authored content forms the normative and descriptive body of the
        document. doc-meta exists only for retrieval and version hygiene.

02.7. Examples are authored content and follow all format rules.


===============================================================================
03. CHARACTER SET
===============================================================================

03.1. Documents use plain ASCII characters.

03.2. Documents do not include:
      - Unicode punctuation
      - Unicode whitespace
      - non-ASCII symbols

03.3. Only the following conventions are used:
      - "-" for dashes
      - "->" for arrows
      - straight quotes: " and '

03.4. If an editor, chat client, or copy operation introduces non-ASCII
      characters, the drafting author corrects them before publication
      or commit.


===============================================================================
04. MAIN SECTION HEADERS
===============================================================================

04.1. Each main section follows this structure:

      Line 1: exactly 79 "=" characters
      Line 2: Section title
      Line 3: exactly 79 "=" characters
      Line 4: blank line
      Line 5+: section content

04.2. Section titles begin with a two-digit number ranging from "00."
      through "99.".

04.3. Section titles use uppercase text.

04.4. Section titles never contain state markers defined in CMR-03.

04.5. Two consecutive blank lines separate main sections.

04.6. A main section ends when two consecutive blank lines appear.
      The next line is either the next main section header or the
      end-of-document marker defined by CMR-02.

04.7. Authored content does not contain two consecutive blank lines except
      when terminating a main section.

04.7.1. One exception exists between the document preamble and the first
        numbered main section.

04.7.2. This exception provides a visual separation between the preamble
        and section "01.".

04.7.3. A main section cannot contain multi-paragraph breaks using two
        blank lines. Drafting authors use subsections, lists, or single
        blank lines to structure content within a main section.

04.8. A blank line contains zero characters.
      Lines containing spaces or tabs are not blank lines.


===============================================================================
05. SUBSECTION HEADERS
===============================================================================

05.1. Subsections may appear inside a main section.

05.2. A subsection header follows this structure:

      (REQUIRED blank line)
      <subsection-number><space><subsection title>
      <underline>
      (blank line)

05.3. The underline uses "-" characters only.

05.3.1. The underline length exactly matches the subsection header line
        length excluding the line ending.

05.4. Example:

      04.1 Sub-header one
      -------------------

      04.1.2 Exactly as length of the text
      ------------------------------------

05.5. Subsection content begins immediately after the required blank line.

05.5.1. The blank line appears even if the subsection header is the first
        content in the section.

05.6. Subsections do not change the rule that the enclosing main section
      ends with exactly two blank lines.

05.7. Subsection numbers contain digits and "." characters only.

05.8. Subsection numbers begin with the enclosing main section number.

05.9. Preamble blocks may use a label-block structure:

      <Label>:
      <underline>
      <free text>

05.10. In label blocks:
       - the label is free text and is not a subsection number
       - the underline uses "-" characters
       - underline length is advisory rather than exact

05.11. Label blocks appear only inside the document preamble defined by
       CMR-02. Label blocks are not treated as subsection headers.

[NOTE] The underline rule is intentionally strict and designed to be
       machine-checkable.


===============================================================================
06. LISTS AND INDENTATION
===============================================================================

06.1. Lists may be used.

06.2. The preferred list format is a bullet list beginning with "-"
      followed by one space.

06.3. Numbered lists may also be used. When used, they use "1.", "2.",
      and so on.

06.4. Indentation uses spaces only.

06.5. Tabs do not appear in documents.

06.6. Continuation lines align with the first character of the list item's
      text on the first line.

06.7. This alignment rule applies to:
      - bullet lists
      - numbered lists
      - in-body numbered clauses

06.8. If numbering tokens become excessively long and cause indentation
      problems, the drafting author may restructure the section.


===============================================================================
07. TABLES
===============================================================================

07.1. Tables may be used.

07.2. The preferred format is a boxed ASCII table using "+", "-", and "|".

07.3. A minimal pipe-separated table format is also acceptable.

07.4. Table lines follow the 79-character rule.

07.5. Columns use space padding so rows align consistently.

07.6. In boxed tables:
      - borders use "+" corners and "-" fill
      - rows begin and end with "|"
      - separators use "|"

07.7. In pipe tables:
      - "|" separates columns
      - the final "|" may be omitted
      - if the final "|" is omitted, padding is still applied so that
        visible column text aligns to the same column widths as other rows

07.8. Example (boxed):

      +----------+-------------+
      | Column 1 | Last column |
      +----------+-------------+
      | Value    | Text        |
      +----------+-------------+

07.9. Example (pipe):

      | Column 1 | Last column
      | Value    | Text

07.10. If a table becomes fragile under the line-length rule, the drafting
       author may replace it with prose or key-value blocks.


===============================================================================
08. EMPHASIS AND TAGS
===============================================================================

08.1. Documents do not rely on Markdown semantics for meaning.

08.2. Emphasis may appear as:
      - ALL CAPS
      - leading tags

08.3. Tags appear at the start of the line.

      [NOTE] Informational note.
      [WARNING] Behavioral constraint.


===============================================================================
09. DOCUMENT BANNER
===============================================================================

09.1. Every document begins with a document banner defined by CMR-02.

09.2. The banner and preamble follow the same 79-character rule as all
      authored content.


===============================================================================
10. DOCUMENT FOOTER (doc-meta)
===============================================================================

10.1. Committed authoritative documents end with a doc-meta block placed
      after all authored content.

10.1.1. A committed authoritative document is a repository version whose
        Canonical Index entry lists DOC_AUTHORITATIVE_STATUS as
        AUTHORITATIVE.

10.1.2. Draft copies may omit doc-meta.

10.2. If present, doc-meta uses the following structure:

      --- doc-meta ---
      DOC_ID: <canonical document identifier>
      DOC_FILE: <canonical repository path>
      DOC_REV: <revision number>
      DOC_GIT: <7-hex git commit prefix>
      ----------------

10.2.1. No additional keys appear in doc-meta.

10.2.2. In committed authoritative documents, DOC_GIT contains exactly
        seven lowercase hexadecimal characters.

10.2.2.1. In draft or in-progress copies, DOC_GIT may be temporary,
          missing, or non-final when doc-meta is present.

10.3. doc-meta does not influence behavioral semantics such as
      guarantees, policy, or mechanisms.

10.3.1. doc-meta may be used for retrieval and version hygiene.

10.3.2. In committed authoritative documents it must exist.

10.3.3. In drafts it may exist or be omitted.

10.4. DOC_ID is the canonical identifier used for cross-document
      reference and cross-thread retrieval.

10.5. In committed authoritative documents, DOC_ID and DOC_FILE match
      the Canonical Index entry.

10.6. DOC_FILE identifies the canonical repository location of the
      current authoritative revision of the document.

10.7. DOC_REV is a non-negative integer representing the revision number
      of the committed authoritative document.

10.7.1. On each commit that changes authored content, DOC_REV is greater
        than the prior committed DOC_REV for the same DOC_ID.

10.7.2. Draft or in-progress copies may exist in chat threads without
        doc-meta updates until commit.

10.7.3. If authored content differs from the revision identified by
        DOC_REV when doc-meta is present, the copy is treated as an
        EVOLVING working copy and is non-authoritative until committed
        and indexed.

10.8. The long-line exception from section 02.5 applies here.

10.9. Manual edits do not modify doc-meta in committed authoritative
      documents.

10.9.1. In draft or in-progress copies, doc-meta may be missing or
        non-final, and tooling may rewrite it.


===============================================================================
11. DOCUMENT BOUNDARIES IN CHAT
===============================================================================

11.1. When published in chat, an authoritative document appears entirely
      inside one chat message.

11.1.1. For this section, a publishable document artifact is a single
        authoritative document printed as a complete artifact in chat.

11.1.2. This rule applies to publication of an authoritative document
        artifact. Review bundles may include multiple documents in one
        message but are not treated as publishable artifacts.

11.1.3. Any chat message that contains more than one document banner is
        treated as a review bundle and is not treated as a publishable
        authoritative document artifact.

11.2. A publishable document message contains only the document, with no
      extra text before the document banner or after the doc-meta block.

11.3. Humans may optionally use a code block. For assistant-generated
      documents intended for reuse or cross-thread retrieval, see
      11.3.2.

11.3.1. If a code block is used, the code block contains only the
        document, with no extra text before or after inside the block.

11.3.1.1. If a code block is used for a publishable authoritative
          document artifact, the chat message contains nothing except
          that single code block.

11.3.2. When a language model assistant outputs a full document intended
        for reuse or cross-thread retrieval, the assistant wraps the
        document in a plain-text code block using a "text" fence.

11.3.3. [NOTE] Code blocks reduce copy/paste corruption.

11.4. The transport container does not alter document meaning.


===============================================================================
12. AUTHORING ROLES AND FORMAT OWNERSHIP
===============================================================================

12.1. In this document, specification text may be drafted by humans or by
      language model assistants acting as drafting authors.

12.2. The drafting author produces or edits document text.

12.3. A reviewer evaluates the produced text.

12.4. Acceptance authority remains human.

12.5. Drafting authors may propose changes to any part of the document.

12.6. This section defines local drafting and format-ownership terminology
      for CMR-01 and does not override the broader authority and workflow
      rules defined by CMR-03.

12.7. For format ownership in this document, authored content is produced
      by the drafting author, while doc-meta is maintained by tooling.


===============================================================================
13. WHITESPACE HYGIENE
===============================================================================

13.1. Documents use LF line endings.

13.2. Lines do not contain trailing whitespace.

13.3. Files end with a single trailing newline.

13.4. Whitespace rules exist for readability and tooling hygiene and do
      not represent behavioral semantics.
