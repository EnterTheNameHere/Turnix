===============================================================================
CORE META-RULES CMR-01 - DOCUMENT FORMAT SPECIFICATION (ASCII-79)
===============================================================================

Status:
-------
FROZEN

Description:
------------
Physical format for all authoritative documents.
Changes here have cascading impact and must be rare and explicit.

Purpose:
--------
This document defines the canonical plain-text format used by Turnix
specification documents.

It defines:
- maximum line length
- allowed characters
- main section headers
- subsection headers
- lists and indentation
- tables
- doc-meta block
- publication boundaries for chat

It does NOT define:
- authority tiers or document roles
- document title banner rules (see CMR-02)
- document change discipline (see CMR-03)
- document lifecycle rules

Those are defined by higher-level documentation rules.

Upstream Documents:
------------------
[NOTE] Listed for retrieval hygiene. See CMR-02 08.1 and 08.5.3.
- CMR-02
- CMR-03
- CMR-04
- CMR-05


===============================================================================
01. NORMATIVE KEYWORDS
===============================================================================

01.1. The following keywords are normative and are used across ALL Turnix
      documents:

      - MUST
      - MUST NOT
      - MAY

01.2. No other normative keywords are recognized.

01.3. Keywords not listed above have no normative meaning.


===============================================================================
02. HARD LINE-LENGTH RULE
===============================================================================

02.1. All authored lines MUST be at most 79 characters long.

02.2. No authored line MAY reach 80 characters.

02.3. doc-meta is tooling-managed and is not human-authored content.

02.3.1. "Not authored content" means "not human-authored": it is owned and
        maintained by tooling.

02.3.2. doc-meta is still part of the document artifact when present, and
        is subject to the format rules defined in section 10.

02.3.3. Tooling-managed means humans MUST NOT edit it manually in committed
        authoritative documents; its format is still normatively defined.

02.4. The 79-character rule applies to all authored content, including:
      - prose
      - document banner and end marker
      - section headers and subsection headers
      - lists
      - tables
      - separators
      - examples

02.5. Exception: Any line whose first non-space text is exactly one of:
      - "file:"
      - "DOC_FILE:"
      - "uri:"
      - "link:"
      MAY exceed 79 characters.

02.5.1. The "DOC_FILE:" form applies to doc-meta only.

02.5.2. The "file:" and "uri:" and "link:" forms apply to authored content
        and examples, and are reserved for literal paths, URIs, or Turnix
        link addresses embedded in text.

02.5.3. This exception exists only to keep long literals on a single line.
        It MUST NOT be used for prose or other content.

02.6. Authored content is all document text excluding the doc-meta block.

02.6.1. Authored content is the normative and descriptive body of the
        document. doc-meta is retrieval/version hygiene only.

02.7. Examples are authored content and MUST obey all format rules.


===============================================================================
03. CHARACTER SET
===============================================================================

03.1. Documents MUST use plain ASCII characters only.

03.2. Documents MUST NOT use:
      - Unicode punctuation
      - Unicode whitespace
      - non-ASCII symbols

03.3. Use only:
      - "-" for dashes
      - "->" for arrows
      - straight quotes: " and '

03.4. If an editor, chat client, or copy operation introduces non-ASCII
      characters, authors MUST correct them before publication or commit.


===============================================================================
04. MAIN SECTION HEADERS
===============================================================================

04.1. Each main section MUST use the following structure:

      Line 1: exactly 79 "=" characters
      Line 2: Section title
      Line 3: exactly 79 "=" characters
      Line 4: blank line
      Line 5+: section content

04.2. Section titles MUST start with a two-digit number:

      "00." through "99."

04.3. Section titles MUST be uppercase.

04.4. Section titles MUST NOT contain state markers (see CMR-03 03.7.)

04.5. Main sections MUST be separated by exactly two consecutive blank lines.

04.6. A main section ends when exactly two consecutive blank lines occur.
      The next line MUST be either:
      - the next main section header, or
      - the end marker (end-of-document marker) defined by CMR-02.

04.7. Authored content MUST NOT contain two consecutive blank lines except to
      end a main section.

04.7.1. Exception: The boundary between the document preamble (CMR-02) and
        the first numbered main section MAY contain exactly two consecutive
        blank lines.

04.7.2. This exception exists to allow a consistent visual separation between
        preamble blocks and section "01." while keeping the two-blank-line
        sequence otherwise reserved for main section boundaries.

04.7.3. This means a main section cannot contain multi-paragraph breaks using
        two blank lines. Authors MUST use subsections, lists, or single blank
        lines to structure content within a main section.

04.8. A blank line contains zero characters.
      Lines containing spaces or tabs are NOT blank lines.


===============================================================================
05. SUBSECTION HEADERS
===============================================================================

05.1. Subsections MAY be used inside a main section.

05.2. A subsection header MUST use the following structure:

      (REQUIRED blank line)
      <subsection-number><space><subsection title>
      <underline>
      (blank line)

05.3. The underline MUST:
      - use "-" characters only
      - have EXACTLY the same character count as the subsection header line
      - count ALL characters excluding the line ending

05.4. Example:

      04.1 Sub-header one
      -------------------

      04.1.2 Exactly as length of the text
      ------------------------------------

05.5. Subsection content follows immediately after the blank line.

05.5.1. The required blank line MUST be present even when the subsection
        header is the first content inside the main section.

05.6. Subsections do NOT change the rule that the enclosing main section MUST
      end with exactly two blank lines.

05.7. <subsection-number> MUST use digits and "." characters only.

05.8. <subsection-number> MUST begin with the enclosing main section number.
      Example: inside section "05.", valid subsection numbers include:
      - 05.1
      - 05.2.3

05.9. Preamble blocks MAY use the following "label block" structure:

      <Label>:
      <underline>
      <free text>

05.10. In label blocks:
      - <Label> is free text and is NOT a subsection number
      - the underline MUST use "-" characters only
      - the underline length is advisory for readability and is NOT required
        to match the label line length

05.11. Label blocks are permitted only in the document preamble as defined by
       CMR-02. Label blocks MUST NOT be treated as subsection headers.

[NOTE] The underline exact-length rule is intentionally strict.
       Authors should treat the underline as machine-checkable syntax.


===============================================================================
06. LISTS AND INDENTATION
===============================================================================

06.1. Lists MAY be used.

06.2. The preferred list format is a bullet list using "-" followed by one
      space.

06.3. Numbered lists are also allowed and MUST use "1." "2." etc.

06.4. Indentation MUST use spaces only.

06.5. Tabs MUST NOT be used.

06.6. Continuation lines of a list item MUST be indented to align with the
      first character of the list item's text on the first line.

06.7. This rule applies to:
      - bullet lists
      - numbered lists
      - in-body numbered clauses (for example: 10.7.3.)

06.8. If a numbering token becomes excessively long and causes excessive
      indentation, authors MAY restructure the section to reduce depth.


===============================================================================
07. TABLES
===============================================================================

07.1. Tables MAY be used.

07.2. The preferred table format is a boxed ASCII table using "+", "-",
      and "|".

07.3. A minimal pipe table format using "|" separators is also allowed.

07.4. All table lines MUST obey the 79-character line limit.

07.5. Table columns MUST be padded with spaces so that rows align to the same
      column widths within the table.

07.6. In boxed tables:
      - the top and bottom borders MUST use "+" corners and "-" fill
      - each row MUST begin with "|" and end with "|"
      - inner separators MUST use "|"

07.7. In pipe tables:
      - "|" separates columns
      - the rightmost column MAY omit the trailing "|" character
      - if the trailing "|" is omitted, padding MUST still be applied so that
        the visible column text aligns to the same widths as other rows

07.8. Example (boxed, valid):

      +----------+-------------+
      | Column 1 | Last column |
      +----------+-------------+
      | Value    | Text        |
      +----------+-------------+

07.9. Example (pipe, valid):

      | Column 1 | Last column
      | Value    | Text

07.10. If a table becomes fragile under the 79-character rule, authors MAY
       replace it with key-value blocks or prose.


===============================================================================
08. EMPHASIS AND TAGS
===============================================================================

08.1. Documents MUST NOT rely on Markdown semantics for meaning.

08.2. Emphasis MAY be expressed using:
      - ALL CAPS
      - leading tags

08.3. Tags MUST appear at the start of the line.

      [NOTE] Informational note.
      [WARNING] Behavioral constraint.


===============================================================================
09. DOCUMENT BANNER
===============================================================================

09.1. Each document MUST begin with a document banner as defined by CMR-02.

09.2. The document banner and the preamble (as defined by CMR-02) are
      authored content and MUST obey the 79-character rule.


===============================================================================
10. DOCUMENT FOOTER (doc-meta)
===============================================================================

10.1. Each committed authoritative document MUST end with a doc-meta block
      placed after all authored content.

10.1.1. For the purpose of this section, a 'committed authoritative document'
        means a repository version of a document whose Canonical Index entry
        has DOC_AUTHORITATIVE_STATUS: AUTHORITATIVE.

10.1.2. Draft or in-progress copies MAY omit the doc-meta block.

10.2. If present, the doc-meta block MUST use this exact structure:

      --- doc-meta ---
      DOC_ID: <canonical document identifier>
      DOC_FILE: <canonical repository path>
      DOC_REV: <revision number>
      DOC_GIT: <7-hex git commit prefix>
      ----------------

10.2.1. No additional keys are permitted in doc-meta.

10.2.2. DOC_GIT MUST be exactly 7 lowercase hexadecimal characters.

10.3. If present, the doc-meta block:
      - MUST be ignored for behavioral semantics
        (guarantees, policy, mechanisms)
      - MAY be used for retrieval and version hygiene

10.3.1. In committed authoritative documents, doc-meta MUST be present.

10.3.2. In draft or in-progress copies, doc-meta MAY be present or omitted.

10.4. DOC_ID is the canonical document identifier used for cross-document
      references and cross-thread retrieval.

10.5. In committed authoritative documents, DOC_ID and DOC_FILE MUST match
      the Canonical Index entry for this document.

10.6. DOC_FILE is the canonical repository path of the current authoritative
      revision of the document.

10.7. DOC_REV is a non-negative integer representing the revision number of
      the committed authoritative document.

10.7.1. On each commit that changes authored content, DOC_REV MUST be greater
        than the prior committed DOC_REV for the same DOC_ID.

10.7.2. Draft or in-progress copies MAY exist in chat threads without
        doc-meta updates until commit.

10.7.3. If authored content differs from the revision identified by DOC_REV
        (when doc-meta is present), the copy MUST be treated as an EVOLVING
        working copy (non-authoritative) until committed and indexed.

10.8. Exception: See section 02.5.

10.9. In committed authoritative documents, manual edits MUST NOT modify
      doc-meta content.

10.9.1. In draft or in-progress copies, doc-meta MAY be missing or non-final,
        and tooling MAY rewrite it.


===============================================================================
11. DOCUMENT BOUNDARIES IN CHAT
===============================================================================

11.1. When published in chat, an authoritative document MUST be fully
      contained within a single chat message.

11.1.1. For this section, "publishable document artifact" means a single
        authoritative document printed as a complete artifact in chat.

11.1.2. This rule applies to publication of an authoritative document
        artifact; review bundles may include multiple documents in one
        message but must not be treated as publishable artifacts.

11.1.3. Any chat message that contains more than one document banner MUST be
        treated as a review bundle and MUST NOT be treated as a publishable
        authoritative document artifact.

11.2. When publishing a single authoritative document artifact, the chat
      message MUST contain only the document, with no extra text before
      the document banner or after the doc-meta block.

11.3. For humans, a code block is optional. For assistant-generated documents
      intended for reuse, see 11.3.2.

11.3.1. If a code block is used, the code block MUST contain only the
        document, with no extra text before or after inside the block.

11.3.1.1. If a code block is used for a publishable authoritative document
          artifact, the chat message MUST contain nothing except that single
          code block.

11.3.2. When a language model assistant outputs a full document intended for
        reuse or cross-thread retrieval, it MUST wrap the document in a
        plain-text code block using a "text" fence (```text)..

11.3.3. [NOTE] Code blocks reduce copy/paste corruption caused by chat
        clients.

11.4. The transport container (plain message or code block) MUST NOT be
      treated as part of the document's meaning.


===============================================================================
12. WHITESPACE HYGIENE
===============================================================================

12.1. Documents MUST be committed with line feed (LF) line endings.

12.2. Lines MUST NOT contain trailing whitespace.

12.3. Files MUST end with a single trailing newline.

12.4. Whitespace rules exist for human readability and optional tooling.
      They MUST NOT be interpreted as authority semantics.
